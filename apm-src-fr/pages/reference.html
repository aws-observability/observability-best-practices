Limites et quotas de service

Comprendre les limites et quotas de CloudWatch Application Signals.

Limites et quotas

Application Signals repose sur CloudWatch et AWS X-Ray. Les limites (ingestion, taux de requetes et quotas) sont regies par ces services et peuvent changer au fil du temps. Utilisez la documentation officielle pour les valeurs actuelles.

  • Application Signals : Documentation Application Signals
  • Quotas CloudWatch : Quotas de service CloudWatch
  • Quotas AWS X-Ray : Limitation et limites de l'API X-Ray

Retention des metriques CloudWatch (reference courante)

CloudWatch conserve les donnees de metriques a differentes resolutions. Consultez la documentation CloudWatch pour le modele de retention complet.

  • Resolution de 1 minute : 15 jours
  • Resolution de 5 minutes : 63 jours
  • Resolution de 1 heure : 15 mois

Quotas de service

Application Signals :

  • Services par compte : 1 000
  • Operations par service : 100
  • Metriques personnalisees : 30 par service

RUM :

  • Moniteurs d'application par compte : 20
  • Evenements par seconde : 10 000
  • Taille d'evenement : 64 Ko

Synthetics :

  • Canaries par compte : 100
  • Duree d'execution des canaries : 15 minutes maximum
  • Taille des canaries : 1 Go (decompresse)

Glossaire

Termes et concepts cles de CloudWatch Application Signals.

Concepts fondamentaux

Application Performance Monitoring (APM)

La pratique de la surveillance et de la gestion de la performance et de la disponibilite des applications logicielles.

Tracage distribue

Une methode de suivi des requetes a travers les systemes distribues, vous permettant de comprendre le chemin et la performance des requetes a travers plusieurs services.

Carte de services

Une representation visuelle de l'architecture de votre application montrant comment les services se connectent et interagissent.

Span

Une operation unique au sein d'une trace, representant une unite de travail telle qu'une requete de base de donnees ou un appel API.

Trace

Un parcours complet de bout en bout d'une requete a travers votre application, compose de plusieurs spans.

Termes d'instrumentation

Auto-instrumentation

Ajout automatique de code d'observabilite aux applications sans necessiter de modifications du code.

Instrumentation manuelle

Ajout explicite de code de tracage et de metriques au code source de l'application.

OpenTelemetry

Un framework d'observabilite open source fournissant des API neutres vis-a-vis des fournisseurs pour l'instrumentation.

Echantillonnage

Le processus de selection des requetes a tracer pour equilibrer l'observabilite avec le cout et la performance.

Propagation du contexte

Le mecanisme de transmission du contexte de trace entre les services pour maintenir la continuite de la trace.

Metriques et surveillance

Signaux dores

Les quatre metriques cles pour surveiller la sante d'une application : latence, debit, taux d'erreur et saturation.

Latence

Le temps necessaire pour traiter une requete, generalement mesure en percentiles (P50, P90, P99).

Debit

Le taux de traitement des requetes, generalement mesure en requetes par seconde.

Taux d'erreur

Le pourcentage de requetes qui entrainent des erreurs.

Saturation

Le degre d'utilisation des ressources systeme.

Objectif de niveau de service (SLO)

Un niveau cible de qualite de service convenu entre les fournisseurs et les consommateurs de services.

Indicateur de niveau de service (SLI)

Un aspect mesurable du niveau de service fourni, tel que la latence ou la disponibilite.

RUM et Synthetics

Real User Monitoring (RUM)

Surveillance des interactions reelles des utilisateurs avec les applications web en production.

Surveillance synthetique

Tests automatises des applications a l'aide d'interactions utilisateur simulees.

Canary

Un script de surveillance synthetique qui execute des tests automatises contre votre application.

Core Web Vitals

Les metriques de Google pour mesurer l'experience utilisateur sur le web : LCP, FID et CLS.

Largest Contentful Paint (LCP)

Le temps necessaire pour que le plus grand element de contenu soit affiche a l'ecran.

First Input Delay (FID)

Le temps entre la premiere interaction de l'utilisateur avec une page et la reponse du navigateur.

Cumulative Layout Shift (CLS)

Une mesure de la stabilite visuelle, quantifiant les changements de mise en page inattendus.

Termes specifiques a AWS

Application Signals

La solution d'observabilite unifiee d'AWS combinant APM, RUM et synthetics.

X-Ray

Le service de tracage distribue d'AWS, desormais integre avec Application Signals.

CloudWatch

Le service de surveillance et d'observabilite d'AWS qui collecte les metriques, les logs et les traces.

ADOT (AWS Distro for OpenTelemetry)

La distribution d'OpenTelemetry par AWS optimisee pour les services AWS.

Environnements d'execution et frameworks pris en charge

Langages de programmation et frameworks pris en charge par Application Signals.

Prise en charge de l'auto-instrumentation

Pour la matrice de compatibilite et d'environnement d'execution officielle, voir Systemes pris en charge. Points cles :

  • Java : JVM 8/11/17/21/23 (la prise en charge des frameworks suit l'instrumentation Java ADOT/OpenTelemetry).
  • Python : Python 3.9+ (la prise en charge des frameworks suit l'instrumentation Python ADOT/OpenTelemetry).
  • Node.js : Node 14/16/18/20/22 (la prise en charge des frameworks suit l'instrumentation JS ADOT/OpenTelemetry).
  • .NET : pris en charge sur x86-64 ou ARM64 ; Linux x64/ARM64 et Windows Server 2022 x64 (la prise en charge des bibliotheques suit l'instrumentation .NET ADOT/OpenTelemetry).
  • PHP/Ruby/Go : pris en charge via les chemins d'auto-instrumentation/zero-code OpenTelemetry (pas de SDK ADOT pour ces langages).

Notes et considerations connues

  • Node.js ESM : la prise en charge ESM est limitee ; preferez CommonJS sauf si vous suivez la configuration specifique ESM dans la documentation.
  • Python : les applications conteneurisees peuvent necessiter la definition de PYTHONPATH (probleme connu d'auto-instrumentation OTel).
  • .NET : ADOT .NET a des limitations avec AWS SDK pour .NET V4 ; AWS SDK V3 est recommande pour une prise en charge complete.

Instrumentation manuelle

Tous les principaux langages de programmation sont pris en charge via les SDK OpenTelemetry :

  • Java, Scala, Kotlin
  • Python, Ruby
  • JavaScript/TypeScript, Go
  • .NET (C#, F#, VB.NET)
  • PHP, Rust, C++

Reference de configuration

Options de configuration courantes et variables d'environnement.

Variables d'environnement

Configuration OpenTelemetry :

  • OTEL_SERVICE_NAME - Nom de votre service
  • OTEL_TRACES_EXPORTER - Exportateur de traces (otlp, jaeger, zipkin)
  • OTEL_EXPORTER_OTLP_ENDPOINT - URL du point de terminaison OTLP
  • OTEL_TRACES_SAMPLER - Strategie d'echantillonnage
  • OTEL_TRACES_SAMPLER_ARG - Parametre d'echantillonnage

Configuration specifique AWS :

  • AWS_REGION - Region AWS
  • AWS_XRAY_DAEMON_ADDRESS - Adresse du daemon X-Ray
  • OTEL_RESOURCE_ATTRIBUTES - Attributs du service (egalement utilises pour le regroupement personnalise dans la carte applicative)

Configuration de l'echantillonnage

Strategies d'echantillonnage :

  • always_on - Echantillonner toutes les traces
  • always_off - N'echantillonner aucune trace
  • traceidratio - Echantillonner selon un ratio (par ex., 0.1 pour 10%)
  • parentbased_always_on - Respecter la decision d'echantillonnage du parent
  • parentbased_traceidratio - Echantillonnage par ratio base sur le parent

Optimisation des couts

Strategies pour controler les couts d'observabilite.

Strategies d'echantillonnage

  • Production : taux d'echantillonnage de 5-10%
  • Developpement : taux d'echantillonnage de 50-100%
  • Traces d'erreur : toujours echantillonner les traces avec des erreurs
  • Operations a haute valeur : echantillonnage a 100% pour les chemins critiques

Politiques de retention des donnees

La retention est controlee par les magasins de telemetrie sous-jacents (par exemple, les metriques/logs CloudWatch et les traces AWS X-Ray). Configurez la retention dans ces services et alignez-la avec vos exigences de depannage et de conformite.

Optimisation des ressources

  • Exports par lots : reduire la surcharge reseau
  • Compression : activer la compression des donnees
  • Filtrage : supprimer les spans et attributs inutiles
  • Mise en memoire tampon : utiliser des tailles de tampon appropriees
