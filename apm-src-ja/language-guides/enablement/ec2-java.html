1. 概要

このガイドでは、AWS CloudWatch Application Signals を有効にした状態で、Amazon EC2 上に Quarkus Java アプリケーションをデプロイする手順を説明します。 Application Signals は、アプリケーションの健全性、パフォーマンス、および依存関係を監視するための自動インストルメンテーションを提供します。

構築するもの:
  • 8 つ以上のエンドポイントを持つ Quarkus Java REST API アプリケーション
  • Application Signals 向けに設定された CloudWatch エージェント
  • 自動インストルメンテーション用の AWS Distro for OpenTelemetry (ADOT) Java エージェント
  • テスト用の自動トラフィック生成
  • CloudWatch Application Signals による完全なオブザーバビリティ

アーキテクチャ

このソリューションは以下で構成されています:

  • Quarkus アプリケーション: REST エンドポイントを持つ Java 21 アプリケーション
  • ADOT Java エージェント: トレースとメトリクスのためにアプリケーションを自動インストルメント
  • CloudWatch エージェント: テレメトリデータを収集して AWS に送信
  • Application Signals: サービスの健全性、トレース、およびメトリクスを可視化

2. 前提条件

必要なツール

  • 適切なプロファイルで設定済みの AWS CLI
  • EC2 インスタンス (Amazon Linux 2 または Amazon Linux 2023)
  • Java 21 (Amazon Corretto)
  • Maven 3.8+
  • SSM Session Manager または SSH 経由での EC2 インスタンスへのアクセス

AWS Permissions

EC2 インスタンスには、以下を含む IAM ロールが必要です:

  • CloudWatchAgentServerPolicy
  • AmazonSSMManagedInstanceCore (SSM を使用する場合)
  • S3 読み取りアクセス (S3 経由でデプロイする場合)

ネットワーク要件

  • セキュリティグループがポート 8080 を許可している (アプリケーション)
  • 依存関係のダウンロードのためのアウトバウンドインターネットアクセス

3. Application Signals の有効化

1AWS Console で有効化する

  1. AWS Console を開き、CloudWatch に移動します
  2. 次に移動: Application Signals → 始める
  3. クリック: Enable Application Signals
  4. これはアカウントレベルの一回限りの設定です
注意: Application Signals は、続行する前に AWS アカウントで有効化されている必要があります。これにより、CloudWatch がアプリケーションからテレメトリデータを受信して表示できるようになります。

4. IAM ロールのセットアップ

既存の IAM ロールの確認

# EC2 インスタンスにアタッチされている IAM ロールを取得する aws ec2 describe-instances \ --instance-ids i-YOUR-INSTANCE-ID \ --query 'Reservations[0].Instances[0].IamInstanceProfile.Arn' \ --profile YOUR_PROFILE \ --region us-east-1

CloudWatch ポリシーのアタッチ

# ARN からロール名を抽出する ROLE_NAME=$(aws ec2 describe-instances \ --instance-ids i-YOUR-INSTANCE-ID \ --query 'Reservations[0].Instances[0].IamInstanceProfile.Arn' \ --output text \ --profile YOUR_PROFILE \ --region us-east-1 | cut -d'/' -f2) echo "Role name: ${ROLE_NAME}" # CloudWatchAgentServerPolicy をアタッチする aws iam attach-role-policy \ --role-name ${ROLE_NAME} \ --policy-arn arn:aws:iam::aws:policy/CloudWatchAgentServerPolicy \ --profile YOUR_PROFILE

ポリシーのアタッチを確認する

# アタッチされているポリシーを一覧表示する aws iam list-attached-role-policies \ --role-name ${ROLE_NAME} \ --profile YOUR_PROFILE # CloudWatchAgentServerPolicy を具体的に確認する aws iam list-attached-role-policies \ --role-name ${ROLE_NAME} \ --profile YOUR_PROFILE \ --query 'AttachedPolicies[?PolicyName==`CloudWatchAgentServerPolicy`]'

5. CloudWatch Agent のインストール

重要: これらのコマンドを実行する前に、EC2 インスタンスに SSH 接続するか、SSM Session Manager を使用してください。

ダウンロードとインストール

# CloudWatch エージェントをダウンロードする wget https://amazoncloudwatch-agent.s3.amazonaws.com/amazon_linux/amd64/latest/amazon-cloudwatch-agent.rpm # エージェントのインストール sudo rpm -U ./amazon-cloudwatch-agent.rpm

Application Signals 用の設定

# 必要に応じて設定ディレクトリを作成 sudo mkdir -p /opt/aws/amazon-cloudwatch-agent/etc # Application Signals の設定を作成 sudo tee /opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json > /dev/null <<'EOF' { "traces": { "traces_collected": { "application_signals": {} } }, "logs": { "metrics_collected": { "application_signals": {} } } } EOF # 設定ファイルを確認 cat /opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json

CloudWatch Agent の起動

# Application Signals の設定でエージェントを起動 sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \ -a fetch-config \ -m ec2 -s \ -c file:/opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json # ステータスを確認 sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \ -a status \ -m ec2
期待される出力: JSON 出力に "status": "running" が表示されるはずです。

ポートの確認

# Application Signals のポートがリッスン状態かどうかを確認 sudo netstat -tlnp | grep 431 # 期待値: ポート 4315 および 4316 が LISTEN 状態
ポート情報:
  • 4315: OTLP gRPC レシーバー
  • 4316: OTLP HTTP レシーバー（ADOT Java エージェントが使用）

6. Quarkus アプリケーションのビルド

注意: 以下のコマンドは、ソースコードが存在する ローカルマシン で実行してください。

アプリケーションのビルド

# プロジェクトディレクトリに移動 cd ec2-quarkus-java # デプロイメントディレクトリを作成 mkdir -p deployment # アプリケーションをビルド（ビルドを高速化するためにテストをスキップ） mvn clean package -DskipTests # デプロイ用のtarballを作成 cd target tar -czf quarkus-app.tar.gz quarkus-app/ cd ..

ADOT Javaエージェントのダウンロード

# AWS Distro for OpenTelemetry Javaエージェントをダウンロード wget -O deployment/aws-opentelemetry-agent.jar \ https://github.com/aws-observability/aws-otel-java-instrumentation/releases/latest/download/aws-opentelemetry-agent.jar echo "Build complete! Files ready for deployment."

EC2へのアップロード

以下のいずれかの方法でEC2インスタンスにファイルを転送します：

オプションA：S3経由

# S3にアップロード aws s3 cp target/quarkus-app.tar.gz s3://YOUR-BUCKET/ \ --profile YOUR_PROFILE # 次にEC2上でS3からダウンロード aws s3 cp s3://YOUR-BUCKET/quarkus-app.tar.gz /home/ec2-user/

オプションB：SCP経由（SSHを使用する場合）

scp -i YOUR-KEY.pem target/quarkus-app.tar.gz ec2-user@YOUR-EC2-IP:/home/ec2-user/ scp -i YOUR-KEY.pem deployment/aws-opentelemetry-agent.jar ec2-user@YOUR-EC2-IP:/home/ec2-user/

オプションC：EC2上でADOTエージェントを直接ダウンロード

# EC2インスタンス上で cd /home/ec2-user wget -O aws-opentelemetry-agent.jar \ https://github.com/aws-observability/aws-otel-java-instrumentation/releases/latest/download/aws-opentelemetry-agent.jar

7. EC2へのアプリケーションのデプロイ

重要： 以下のコマンドを EC2インスタンス SSH または SSM Session Manager 経由で実行してください。

Java 21のインストール

# Amazon Corretto 21をインストール sudo yum install -y java-21-amazon-corretto-devel # デフォルトのJavaバージョンとして設定 sudo alternatives --config java # Java 21オプションを選択 # Javaバージョンを確認 java -version # 期待される出力：openjdk version "21.x.x"

アプリケーションの展開

cd /home/ec2-user # 既存のアプリケーションを停止 pkill -f quarkus-run.jar || true # 古いデプロイメントのクリーンアップ rm -rf quarkus-app # アプリケーションの展開 tar -xzf quarkus-app.tar.gz # 展開の確認 ls -la quarkus-app/

起動スクリプトの作成

# OpenTelemetry 設定を含む起動スクリプトの作成 cat > start-app.sh <<'EOF' #!/bin/bash export OTEL_RESOURCE_ATTRIBUTES="service.name=ec2-quarkus-java,deployment.environment=ec2:production" export OTEL_EXPORTER_OTLP_TRACES_ENDPOINT="http://localhost:4316/v1/traces" export OTEL_METRICS_EXPORTER="none" export OTEL_LOGS_EXPORTER="none" export OTEL_AWS_APPLICATION_SIGNALS_ENABLED="true" export OTEL_AWS_APPLICATION_SIGNALS_EXPORTER_ENDPOINT="http://localhost:4316/v1/metrics" java -javaagent:/home/ec2-user/aws-opentelemetry-agent.jar \ -jar /home/ec2-user/quarkus-app/quarkus-run.jar EOF # 実行可能にする chmod +x start-app.sh
環境変数の説明:
  • OTEL_RESOURCE_ATTRIBUTES: Application Signals ダッシュボード用のサービス名と環境を設定する
  • OTEL_EXPORTER_OTLP_TRACES_ENDPOINT: トレース用の CloudWatch エージェントエンドポイント
  • OTEL_AWS_APPLICATION_SIGNALS_ENABLED: Application Signals 固有の機能を有効にする
  • OTEL_AWS_APPLICATION_SIGNALS_EXPORTER_ENDPOINT: メトリクス用の CloudWatch エージェントエンドポイント
  • OTEL_METRICS_EXPORTER/OTEL_LOGS_EXPORTER: Application Signals エクスポーターのみを使用するために無効化

アプリケーションの起動

# バックグラウンドでアプリケーションを起動 nohup ./start-app.sh > app.log 2>&1 & # 起動を待機 sleep 5 # プロセスが実行中かどうかを確認 ps aux | grep quarkus # アプリケーションログを確認 tail -30 app.log

アプリケーションのローカルテスト

# ルートエンドポイントのテスト curl http://localhost:8080 # ヘルスエンドポイントのテスト curl http://localhost:8080/health # hello エンドポイントのテスト curl http://localhost:8080/hello/TestUser
Expected Response: 
{"environment":"ec2","message":"Hello from Quarkus EC2 Application!","timestamp":1760817129049,"status":"running"}

利用可能なエンドポイント

エンドポイント メソッド 説明
/ GET 基本情報を含むルートエンドポイント
/health GET ヘルスチェックエンドポイント
/hello/{name} GET パーソナライズされた挨拶
/slow GET 低速操作のシミュレーション（2〜5秒）
/error GET エラーのシミュレーション（500ステータス）
/external GET 外部API呼び出しのシミュレーション
/s3 GET S3操作のシミュレーション
/database GET データベースクエリのシミュレーション
/complex GET マルチステップの複雑な操作

8. トラフィックの生成

Application Signals にデータを蓄積するために、アプリケーションへの継続的なトラフィックを生成します。

トラフィックジェネレータースクリプトの作成

# On EC2 instance cd /home/ec2-user # Create traffic generator that runs once cat > traffic-generator.sh <<'EOF' #!/bin/bash # Array of endpoints to test endpoints=( "/" "/health" "/hello/User1" "/hello/User2" "/hello/User3" "/hello/TestUser" "/external" "/s3" "/database" "/slow" "/complex" ) # Make 5 random requests for i in {1..5}; do # Select random endpoint endpoint=${endpoints[$RANDOM % ${#endpoints[@]}]} # Make request response=$(curl -s -o /dev/null -w "%{http_code}" http://localhost:8080${endpoint}) # Log request timestamp=$(date '+%Y-%m-%d %H:%M:%S') echo "[$timestamp] ${endpoint} - Status: ${response}" >> /home/ec2-user/traffic-generator.log # Small delay between requests sleep 2 done EOF # 実行可能にする chmod +x traffic-generator.sh

Cron でスケジュール設定

# 毎分実行するように crontab に追加 (crontab -l 2>/dev/null; echo "* * * * * /home/ec2-user/traffic-generator.sh") | crontab - # crontab を確認 crontab -l
トラフィックパターン: これにより、1分あたり5リクエスト（1時間あたり300リクエスト）がランダムなエンドポイントにヒットし、多様なトレースとメトリクスが生成されます。

トラフィックの監視

# トラフィックログをリアルタイムで確認 tail -f /home/ec2-user/traffic-generator.log # 総リクエスト数をカウント wc -l /home/ec2-user/traffic-generator.log # 直近30件のリクエストを確認 tail -30 /home/ec2-user/traffic-generator.log

トラフィックジェネレーターの停止

# crontab から削除 crontab -r # 削除を確認 crontab -l

9. Application Signals の確認

データの待機

重要: Application Signals にデータが表示されるまで、トラフィック生成を開始してから5〜10分お待ちください。

CloudWatch コンソールへのアクセス

  1. AWS コンソールを開く: https://console.aws.amazon.com/cloudwatch/
  2. 次に移動 Application Signals → サービス
  3. サービスを探す: ec2-quarkus-java

サービスの詳細

Application Signals では、以下が表示されます:

  • サービスマップ: サービスと依存関係のビジュアル表示
  • サービスオペレーション: メトリクスを含むすべてのエンドポイントの一覧
  • トレース: X-Ray による個別リクエストトレース
  • メトリクス: レイテンシー、エラー率、リクエスト率
  • サービスヘルス: 全体的なヘルス指標と SLO

X-Ray でトレースを表示する

  1. 次に移動: X-Ray → トレース
  2. サービス名でフィルタリング: ec2-quarkus-java
  3. 個々のトレースをクリックして詳細なタイミング情報を確認する

CloudWatch でメトリクスを表示する

  1. 次に移動: CloudWatch → メトリクス
  2. 次を検索: ApplicationSignals 名前空間
  3. サービス、オペレーション、ディメンションごとにメトリクスを探索する
成功の指標:
  • サービス「ec2-quarkus-java」がサービスマップに表示される
  • X-Ray でトレースが適切なサービス名で表示される
  • リクエスト率、レイテンシー、エラーを示すメトリクス
  • サービスのヘルスステータスが「正常」である

10. トラブルシューティング

アプリケーションが起動しない

Java バージョンを確認する

java -version # Java 21 が表示されるはずです # バージョンが異なる場合は切り替えてください: sudo alternatives --config java

アプリケーションログを確認する

tail -100 /home/ec2-user/app.log grep -i error /home/ec2-user/app.log

プロセスが実行中かどうかを確認する

ps aux | grep quarkus netstat -tlnp | grep 8080

CloudWatch エージェントの問題

エージェントのステータスを確認する

sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \ -a status \ -m ec2

エージェントログの確認

sudo tail -100 /opt/aws/amazon-cloudwatch-agent/logs/amazon-cloudwatch-agent.log

CloudWatch Agent の再起動

sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \ -a stop \ -m ec2 sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \ -a fetch-config \ -m ec2 -s \ -c file:/opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json

ポートの確認

sudo netstat -tlnp | grep 431 # ポート 4315 および 4316 がリッスン状態であることを確認

Application Signals にデータが表示されない

チェックリスト

  • CloudWatch agent が起動しており、ポート 4315/4316 でリッスンしている
  • アプリケーションが起動しており、リクエストを受信している
  • トラフィックジェネレーターが有効である（crontab とログを確認）
  • IAM ロールに CloudWatchAgentServerPolicy がアタッチされている
  • Application Signals が AWS アカウントで有効になっている
  • 初期データが表示されるまで 5〜10 分待つ

OTLP 接続の確認

# OTLP エラーのアプリケーションログを確認 grep -i "otlp\|exporter" /home/ec2-user/app.log | tail -20 # トラフィックがアプリケーションに到達しているか確認 tail -20 /home/ec2-user/traffic-generator.log

ポートがすでに使用中

# ポートを使用しているプロセスを確認 sudo netstat -tlnp | grep 4315 # プロセスを終了する（PID を置き換えてください） sudo kill -9 PID # または CloudWatch agent のすべてのプロセスを終了 sudo pkill -9 amazon-cloudwatch-agent # その後、再起動する sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \ -a fetch-config \ -m ec2 -s \ -c file:/opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json

権限拒否エラー

# IAM ロールポリシーを確認 aws iam list-attached-role-policies \ --role-name YOUR_ROLE_NAME \ --profile YOUR_PROFILE # CloudWatch agent に書き込み権限があるか確認 sudo tail -50 /opt/aws/amazon-cloudwatch-agent/logs/amazon-cloudwatch-agent.log | grep -i "permission\|denied\|403"

アプリケーションログ

リアルタイムログの表示

tail -f /home/ec2-user/app.log

エラーの検索

grep -i "error\|exception\|failed" /home/ec2-user/app.log | tail -30

Restart Everything

# アプリケーションを停止する pkill -f quarkus-run.jar # CloudWatch エージェントを再起動する sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \ -a fetch-config \ -m ec2 -s \ -c file:/opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json # エージェントの起動を待機する sleep 5 # アプリケーションを再起動する cd /home/ec2-user nohup ./start-app.sh > app.log 2>&1 & # 両方が実行中であることを確認する sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl -a status -m ec2 ps aux | grep quarkus

よくあるエラーメッセージ

"Failed to connect to localhost:4316"

解決策: CloudWatch エージェントが実行されていないか、正しく設定されていません。エージェントを再起動してください。

"class file version 65.0"

解決策: アプリケーションが Java 21 でコンパイルされていますが、古い Java で実行されています。Java 21 をインストールしてください。

"Port 8080 already in use"

解決策: 別のプロセスがポート 8080 を使用しています。そのプロセスを終了するか、別のポートを使用してください。

"Permission denied" CloudWatch への書き込み

解決策: IAM ロールに CloudWatchAgentServerPolicy がありません。ポリシーをアタッチしてください。
