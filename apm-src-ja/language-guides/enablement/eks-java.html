概要

このガイドでは、 ステップバイステップの手順 AWS Application Signals の自動計装を使用して、Amazon EKS 上に Java Spring Boot アプリケーションをデプロイする方法を説明します。これは Lambda Java 関数の EKS 版であり、同じビジネスロジックとエンドポイントを再利用しています。

達成できること：
  • EKS 上に Java Spring Boot アプリをデプロイする
  • AWS Application Signals の自動計装を有効にする
  • CloudWatch Application Signals コンソールでサービスを確認する
  • サービスのメトリクス、トレース、およびオペレーションを監視する
完了までの時間： 30〜45 分

必要なツール

  • AWS CLI v2（最新バージョン）
  • Docker Desktop
  • kubectl（最新バージョン）

アーキテクチャ

EKS Observability アドオン（標準的なアプローチ）： このガイドでは、公式のEKSオブザーバビリティアドオンを使用します。これは、EKSクラスターでApplication Signalsを有効にするための推奨される標準的な方法です。このアドオンは、CloudWatch Agentを手動で設定することなく、テレメトリの収集を自動的に処理します。

EKSクラスターで実行されるもの：

  1. Java Spring Boot Application：自動インストルメンテーションアノテーションを備えたWeb API
  2. EKS Observability Add-on：トレースとメトリクス収集を自動化する公式AWSアドオン
  3. CloudWatch Agent（アドオンによる管理）：アドオンによって自動的にデプロイおよび設定される

フロー：

EKS Cluster
  → Java Spring Boot App (auto-instrumented via annotation)
  → EKS Observability Add-on
  → CloudWatch Agent (managed)
  → CloudWatch Application Signals
EKSオブザーバビリティアドオンの主なメリット：
  • CloudWatch Agentの手動設定が不要
  • サポートされている言語への自動インストルメンテーション
  • Application Signalsの組み込みサポート
  • デプロイと管理の簡素化

package com.appsignals.controller;

import org.springframework.web.bind.annotation.*;

import java.time.Instant;
import java.util.HashMap;
import java.util.Map;
import java.util.Random;

@RestController
@RequestMapping("/api")
public class HelloController {

    @GetMapping("/")

        Map<String, Object> response = new HashMap<>();
        response.put("message", "AWS Application Signals EKS Java Demo");
        response.put("timestamp", Instant.now().toEpochMilli());
        response.put("service", "java-eks-service");

        response.put("status", "healthy");
        response.put("timestamp", Instant.now().toEpochMilli());
        response.put("service", "java-eks-service");
        return response;
    }

    @GetMapping("/hello/{name}")

        try {
            // Simulate some processing time
            Thread.sleep(random.nextInt(100) + 10);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
        
        Map<String, Object> response = new HashMap<>();
        response.put("message", "Hello, " + name + "!");
        response.put("timestamp", Instant.now().toEpochMilli());
        response.put("service", "java-eks-service");
        return response;
    }

    @GetMapping("/slow")
    public Map<String, Object> slow() {
        try {
            // Simulate slow operation
            Thread.sleep(random.nextInt(1000) + 500);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
        
        Map<String, Object> response = new HashMap<>();
        response.put("message", "Slow operation completed");
        response.put("timestamp", Instant.now().toEpochMilli());
        response.put("service", "java-eks-service");
        return response;
    }

    public Map<String, Object> error() {
        if (random.nextBoolean()) {
            throw new RuntimeException("Simulated error for testing Application Signals error tracking");
        }

        response.put("message", "No error occurred");
        response.put("timestamp", Instant.now().toEpochMilli());

ステップ3：EKSへのデプロイ

3.1 EKSクラスターの作成

# Create EKS cluster
eksctl create cluster --name appsignals-cluster --region us-east-1 --nodegroup-name workers --node-type t3.medium --nodes 2 --nodes-min 1 --nodes-max 3

# Create namespace
kubectl create namespace appsignals-demo

3.2 EKSオブザーバビリティアドオンのインストール（必須）

重要： EKSオブザーバビリティアドオンは、EKSでApplication Signalsを有効にするための標準かつ推奨される方法です。このアドオンはテレメトリの収集を自動的に処理し、自動インストルメンテーション機能を提供します。 
# Install the EKS observability add-on (REQUIRED for Application Signals)
aws eks create-addon --cluster-name appsignals-cluster --addon-name amazon-cloudwatch-observability --region us-east-1

# Verify the add-on is installed
aws eks describe-addon --cluster-name appsignals-cluster --addon-name amazon-cloudwatch-observability --region us-east-1
EKSオブザーバビリティアドオンが提供するもの：
  • CloudWatch Agentの自動デプロイ
  • Application Signalsの組み込みサポート
  • Java、Python、Node.js、および.NETの自動インストルメンテーション
  • 手動設定が不要

ステップ4：トラフィックの生成とモニタリング

4.1 トラフィックの生成

作成 generate-traffic.sh:

#!/bin/bash

# Port forward the service
kubectl port-forward service/java-eks-service 8080:80 -n appsignals-demo &
PORT_FORWARD_PID=$!

# Wait for port forward to be ready
sleep 10

BASE_URL="http://localhost:8080"

echo "Generating traffic for Java EKS Application Signals demo..."

# Generate traffic for 5 minutes
for i in {1..300}; do
    echo "Request $i"

    # Root endpoint
    curl -s "$BASE_URL/api/" > /dev/null

    # Health check
    curl -s "$BASE_URL/api/health" > /dev/null

    # Hello endpoint
    curl -s "$BASE_URL/api/hello/User$i" > /dev/null

    # Slow endpoint
    curl -s "$BASE_URL/api/slow" > /dev/null

    # Error endpoint
    curl -s "$BASE_URL/api/error" > /dev/null

    # External call
    curl -s "$BASE_URL/api/external" > /dev/null

    sleep 1
done

echo "Traffic generation complete!"

# Clean up port forward
kill $PORT_FORWARD_PID

実行可能にして実行する：

chmod +x generate-traffic.sh
./generate-traffic.sh

4.2 CloudWatchでのモニタリング

  1. CloudWatch コンソールを開く

    • AWS コンソール → CloudWatch に移動する
    • 「Application Signals」に移動する

  2. サービスマップを表示する

    • 「java-eks-service」サービスを探す
    • サービスの依存関係と正常性を確認する

  3. メトリクスを確認する

    • レイテンシー、エラーレート、スループットを表示する
    • Lambda メトリクスと比較する

  4. トレースを分析する

    • サービスをクリックして詳細なトレースを表示する
    • オペレーションレベルのメトリクスを表示する

トラブルシューティング

よくある問題

  1. アプリケーションが起動しない

    • Pod のステータスを確認する： kubectl get pods -n appsignals-demo
    • Pod のログを確認する： kubectl logs -n appsignals-demo -l app=java-eks-service
    • EKS オブザーバビリティアドオンがインストールされていることを確認する

  2. Application Signals にデータが表示されない

    • 自動インストルメンテーションのアノテーションが存在することを確認する
    • 環境変数が正しく設定されていることを確認する
    • EKS オブザーバビリティアドオンが実行中であることを確認する

  3. ポートフォワードの問題

    • サービスが実行中であることを確認する： kubectl get svc -n appsignals-demo
    • Pod が準備完了状態であることを確認する： kubectl get pods -n appsignals-demo
