ベストプラクティス

インストルメンテーションのベストプラクティス

効果的なインストルメンテーションは、オブザーバビリティデータから価値あるインサイトを得るための鍵です。以下のベストプラクティスに従って、オーバーヘッドを最小限に抑えながら最大限の価値を提供するインストルメンテーションを実現してください。

インストルメント対象

アプリケーションの最も重要な部分のインストルメントに集中してください：

優先度：高

  • パブリック API エンドポイントとユーザー向け操作
  • データベースクエリと外部サービス呼び出し
  • ビジネスクリティカルなワークフローとトランザクション
  • エラーパスと例外処理

優先度：中

  • 内部サービス間通信
  • バックグラウンドジョブとスケジュールタスク
  • キャッシュ操作

優先度：低

  • シンプルなユーティリティ関数
  • 静的コンテンツの配信
  • ヘルスチェックエンドポイント

命名規則

一貫した命名により、トレースの検索とフィルタリングが容易になります：

スパン名：

  • ハイフン区切りの小文字を使用する： http-get-user-profile
  • 操作タイプを含める： db-query-users, redis-get-session
  • 説明的かつ簡潔にする：「process」や「handle」のような汎用的な名前は避けてください

サービス名：

  • 環境全体で一貫したサービス命名を使用する
  • マルチ環境構成ではサービス名に環境を含める
  • 組織の命名規則に従う

属性とタグ

属性を使用してスパンとトレースにコンテキストを追加してください：

標準属性：

  • http.method - HTTP メソッド（GET、POST など）
  • http.url - リクエスト URL（PII に注意）
  • http.status_code - HTTP レスポンスステータス
  • db.system - データベースタイプ（mysql、postgres など）
  • db.name - データベース名
  • db.operation - データベース操作（SELECT、INSERT など）

カスタム属性：

  • user.id - ユーザー識別子（PIIの場合はハッシュ化）
  • business.operation - ビジネス操作タイプ
  • version - アプリケーションバージョン
  • environment - デプロイ環境

PIIとセキュリティに関する考慮事項

テレメトリ内の機密データを保護してください：

絶対に含めないもの：

  • パスワード、APIキー、または認証トークン
  • メールアドレスや電話番号などの個人識別情報（PII）
  • 機密データを含むクエリパラメータ付きの完全なURL
  • クレジットカード番号または財務情報

安全な代替手段：

  • ハッシュ化またはトークン化された識別子を使用する
  • URLのパス部分のみを含める
  • 具体的な値の代わりに汎用的なステータス指標を使用する
  • インストルメンテーションレベルでデータマスキングを実装する

サンプリング戦略

サンプリングは、オブザーバビリティのカバレッジを維持しながらコストを管理するのに役立ちます。

サンプリング戦略の選択

レートベースサンプリング：

  • リクエストの一定割合をサンプリングする（例：10%）
  • 実装と理解が簡単
  • 重要だが稀なイベントを見逃す可能性がある

テールサンプリング：

  • トレースデータを収集した後にサンプリングを決定する
  • エラーや高レイテンシのトレースを優先できる
  • より複雑だが、よりインテリジェント

ルールベースサンプリング：

  • 特定の条件（ユーザー、エンドポイントなど）に基づいてサンプリングする
  • 重要なパスが常にトレースされることを保証する
  • 慎重なルール定義が必要

推奨サンプリングレート

  • 開発環境： 完全な可視性のために100%サンプリング
  • ステージング環境： テスト用に50〜100%サンプリング
  • 本番環境： 費用対効果の高いカバレッジのために10〜20%
  • 高トラフィックサービス： ボリュームを制御するために1〜5%
  • 重要なパス： 他のルールに関係なく100%サンプリング

パフォーマンス最適化

インストルメンテーションのパフォーマンスへの影響を最小限に抑える。

オーバーヘッドの削減

  • 可能な限り非同期スパン処理を使用する
  • ネットワーク呼び出しを減らすためにスパンエクスポートをバッチ処理する
  • 適切なバッファサイズを設定する
  • インストルメンテーションのパフォーマンスメトリクスを監視する

リソース管理

  • スパンエクスポートに合理的なタイムアウトを設定する
  • バックエンド障害に対してサーキットブレーカーを実装する
  • コレクター用に接続プーリングを使用する
  • 適切なリトライポリシーを設定する

アラートと監視

アプリケーションの健全性を把握するための効果的なアラートを設定する。

監視すべき主要メトリクス

  • レイテンシー： P95/P99レスポンスタイム
  • エラーレート： 失敗したリクエストの割合
  • スループット： 1秒あたりのリクエスト数
  • 飽和度： リソース使用率
  • トレース量： 収集されたトレース数

SLOベースのアラート

サービスレベル目標を定義し、リスクがある場合にアラートを発する：

  • レイテンシーSLO：リクエストの95%が500ms未満
  • エラーSLO：99.9%の成功率
  • 可用性SLO：99.95%の稼働時間

コスト管理

価値を維持しながらオブザーバビリティのコストを管理する。

コスト最適化戦略

  • 適切なサンプリングレートを実装する
  • データ保持ポリシーを使用する
  • 不要なスパンと属性をフィルタリングする
  • 古いデータをより安価なストレージにアーカイブする
  • データ量を監視して最適化する

データ保持ポリシー

  • ホットストレージ: アクティブなトラブルシューティング用の最近のデータ（7〜30日）
  • ウォームストレージ: トレンド分析用の過去のデータ（30〜90日）
  • コールドストレージ: コンプライアンスおよびまれなクエリ用のアーカイブデータ（90日以上）
