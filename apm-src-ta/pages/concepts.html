CloudWatch APM ஆவணங்கள்

கருத்துக்கள் & அடிப்படைகள்

முக்கிய கருத்துக்கள்

Application Performance Monitoring (APM) இன் அடிப்படைக் கருத்துக்களைப் புரிந்துகொள்வது CloudWatch Application Signals-ஐ திறம்பட பயன்படுத்துவதற்கு அவசியமாகும். இந்தப் பகுதி நவீன observability-யின் அடித்தளத்தை உருவாக்கும் முக்கிய கட்டுமானத் தொகுதிகளை உள்ளடக்கியது.

பரவலான Tracing

பரவலான tracing என்பது பரவலான அமைப்புகள் வழியாக கோரிக்கைகள் செல்லும்போது அவற்றைக் கண்காணிக்கும் நடைமுறையாகும். Microservices கட்டமைப்புகளில், ஒரு பயனர் கோரிக்கை பல சேவைகள், தரவுத்தளங்கள் மற்றும் வெளிப்புற API-களை கடக்கலாம். பரவலான tracing இந்த முழு பயணத்தின் மீது தெரிவுநிலையை வழங்குகிறது.

முக்கிய கூறுகள்:

  • Trace: உங்கள் அமைப்பு வழியாக ஒரு கோரிக்கையின் முழுமையான தொடக்கம்-முதல்-முடிவு வரையிலான பயணம்
  • Span: ஒரு trace-க்குள் ஒரு தனிச் செயல்பாடு (எ.கா., ஒரு தரவுத்தள வினவல் அல்லது API அழைப்பு)
  • Trace Context: சேவை எல்லைகள் முழுவதும் span-களை இணைக்கும் மெட்டாடேட்டா

Traces, Spans, மற்றும் Segments

பரவலான tracing-ல் பயன்படுத்தப்படும் அடிப்படை தரவு கட்டமைப்புகள் இவை:

Traces

ஒரு trace உங்கள் பயன்பாடு வழியாக ஒரு கோரிக்கையின் முழுமையான பாதையைக் குறிக்கிறது. இது நேர தகவல், மெட்டாடேட்டா மற்றும் கோரிக்கையின் பயணத்தை உருவாக்கும் அனைத்து span-களையும் கொண்டுள்ளது.

Spans

ஒரு span ஒரு trace-க்குள் ஒரு தனிப்பட்ட வேலை அலகைக் குறிக்கிறது. ஒவ்வொரு span-ம் கொண்டிருப்பவை:

  • ஒரு பெயர் மற்றும் செயல்பாட்டு வகை
  • தொடக்க மற்றும் முடிவு நேர முத்திரைகள்
  • கூடுதல் சூழலுக்கான tags மற்றும் attributes
  • parent spans-க்கான குறிப்புகள்

Segments

AWS X-Ray சொல்லாட்சியில், segments என்பது trace தரவின் அடிப்படை அலகுகளாகும். ஒவ்வொரு segment-ம் ஒரு span-க்கு ஒத்திருக்கிறது மற்றும் trace-ன் அந்தப் பகுதிக்காக செய்யப்பட்ட வேலையைப் பற்றிய தகவல்களைக் கொண்டுள்ளது.

Service Maps

Service maps உங்கள் பயன்பாட்டின் கட்டமைப்பு மற்றும் சார்புகளின் காட்சிப் பிரதிநிதித்துவத்தை வழங்குகின்றன. அவை காட்டுபவை:

  • சேவைகள் ஒன்றோடொன்று எவ்வாறு இணைகின்றன
  • சேவைகளுக்கு இடையே போக்குவரத்து ஓட்டம்
  • ஒவ்வொரு சேவைக்கும் செயல்திறன் அளவீடுகள்
  • பிழை விகிதங்கள் மற்றும் தாமத வடிவங்கள்

Service maps trace தரவிலிருந்து தானாகவே உருவாக்கப்படுகின்றன, உங்கள் அமைப்பின் இடவியலை ஒரு பார்வையில் புரிந்துகொள்ள உதவுகின்றன.

Instrumentation

Instrumentation என்பது உங்கள் பயன்பாடுகளுக்கு observability code-ஐ சேர்க்கும் செயல்முறையாகும். பல அணுகுமுறைகள் உள்ளன:

Auto-Instrumentation

தானியங்கு instrumentation-க்கு code மாற்றங்கள் தேவையில்லை. நூலகங்கள் மற்றும் agents பொதுவான frameworks மற்றும் libraries-ஐ தானாகவே instrument செய்கின்றன.

கைமுறை Instrumentation

கைமுறை instrumentation உங்கள் பயன்பாட்டிற்கு tracing code-ஐ சேர்ப்பதை உள்ளடக்கியது. இது நுண்ணிய கட்டுப்பாட்டை வழங்குகிறது ஆனால் அதிக மேம்பாட்டு முயற்சி தேவைப்படுகிறது.

OpenTelemetry

OpenTelemetry என்பது instrumentation-க்கான விற்பனையாளர்-நடுநிலை API-களை வழங்கும் ஒரு திறந்த-மூல observability framework ஆகும். இது பல நிரலாக்க மொழிகளை ஆதரிக்கிறது மற்றும் பல்வேறு backends-க்கு தரவை ஏற்றுமதி செய்ய முடியும்.

தரவு மாதிரி

Application Signals telemetry தரவை ஒழுங்கமைக்க ஒரு கட்டமைக்கப்பட்ட தரவு மாதிரியைப் பயன்படுத்துகிறது:

Metrics

  • தாமதம்: பதில் நேர அளவீடுகள் (P50, P90, P99)
  • செயல்திறன்: கோரிக்கை விகிதம் (வினாடிக்கு கோரிக்கைகள்)
  • பிழை விகிதம்: தோல்வியடைந்த கோரிக்கைகளின் சதவீதம்
  • நிறைவு: வள பயன்பாடு (CPU, நினைவகம், போன்றவை)

Traces

spans, tags மற்றும் நேர தகவல்களுடன் கூடிய கட்டமைக்கப்பட்ட trace தரவு.

Logs

முழுமையான சூழலுக்காக traces மற்றும் metrics-உடன் தொடர்புபடுத்தப்பட்ட பயன்பாட்டு logs.

Sampling உத்திகள்

Sampling observability-ஐ செலவு மற்றும் செயல்திறனுடன் சமநிலைப்படுத்த எந்த கோரிக்கைகள் trace செய்யப்படுகின்றன என்பதைக் கட்டுப்படுத்துகிறது:

Head Sampling

ஒரு கோரிக்கையின் வாழ்க்கைச் சுழற்சியின் தொடக்கத்தில் trace செய்யலாமா என்பதை முடிவு செய்கிறது.

Tail Sampling

trace தரவை சேகரித்த பிறகு sampling முடிவுகளை எடுக்கிறது, கோரிக்கை பண்புகளின் அடிப்படையில் மிகவும் புத்திசாலித்தனமான sampling-ஐ அனுமதிக்கிறது.

விகிதம் அடிப்படையிலான Sampling

கோரிக்கைகளின் நிலையான சதவீதத்தை trace செய்கிறது (எ.கா., அனைத்து கோரிக்கைகளில் 10%).
