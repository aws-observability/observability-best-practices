1. Overview

This guide walks you through deploying a Quarkus Java application on Amazon EC2 with AWS CloudWatch Application Signals enabled. Application Signals provides automatic instrumentation for monitoring application health, performance, and dependencies.

What You'll Build:
  • Quarkus Java REST API application with 8+ endpoints
  • CloudWatch agent configured for Application Signals
  • AWS Distro for OpenTelemetry (ADOT) Java agent for auto-instrumentation
  • Automated traffic generation for testing
  • Complete observability in CloudWatch Application Signals

Architecture

The solution consists of:

  • Quarkus Application: Java 21 application with REST endpoints
  • ADOT Java Agent: Auto-instruments the application for traces and metrics
  • CloudWatch Agent: Collects telemetry data and sends to AWS
  • Application Signals: Visualizes service health, traces, and metrics

2. Prerequisites

Required Tools

  • AWS CLI configured with appropriate profile
  • EC2 instance (Amazon Linux 2 or Amazon Linux 2023)
  • Java 21 (Amazon Corretto)
  • Maven 3.8+
  • Access to EC2 instance via SSM Session Manager or SSH

AWS Permissions

Your EC2 instance needs an IAM role with:

  • CloudWatchAgentServerPolicy
  • AmazonSSMManagedInstanceCore (if using SSM)
  • S3 read access (if deploying via S3)

Network Requirements

  • Security group allows port 8080 (application)
  • Outbound internet access for downloading dependencies

3. Enable Application Signals

1Enable in AWS Console

  1. Open AWS Console and navigate to CloudWatch
  2. Go to Application Signals → Get started
  3. Click Enable Application Signals
  4. This is a one-time account-level configuration
Note: Application Signals must be enabled in your AWS account before proceeding. This allows CloudWatch to receive and display telemetry data from your applications.

4. IAM Role Setup

Check Existing IAM Role

# Get the IAM role attached to your EC2 instance aws ec2 describe-instances \ --instance-ids i-YOUR-INSTANCE-ID \ --query 'Reservations[0].Instances[0].IamInstanceProfile.Arn' \ --profile YOUR_PROFILE \ --region us-east-1

Attach CloudWatch Policy

# Extract role name from the ARN ROLE_NAME=$(aws ec2 describe-instances \ --instance-ids i-YOUR-INSTANCE-ID \ --query 'Reservations[0].Instances[0].IamInstanceProfile.Arn' \ --output text \ --profile YOUR_PROFILE \ --region us-east-1 | cut -d'/' -f2) echo "Role name: ${ROLE_NAME}" # Attach CloudWatchAgentServerPolicy aws iam attach-role-policy \ --role-name ${ROLE_NAME} \ --policy-arn arn:aws:iam::aws:policy/CloudWatchAgentServerPolicy \ --profile YOUR_PROFILE

Verify Policy Attachment

# List attached policies aws iam list-attached-role-policies \ --role-name ${ROLE_NAME} \ --profile YOUR_PROFILE # Check specifically for CloudWatchAgentServerPolicy aws iam list-attached-role-policies \ --role-name ${ROLE_NAME} \ --profile YOUR_PROFILE \ --query 'AttachedPolicies[?PolicyName==`CloudWatchAgentServerPolicy`]'

5. Install CloudWatch Agent

Important: SSH to your EC2 instance or use SSM Session Manager before running these commands.

Download and Install

# Download CloudWatch agent wget https://amazoncloudwatch-agent.s3.amazonaws.com/amazon_linux/amd64/latest/amazon-cloudwatch-agent.rpm # Install the agent sudo rpm -U ./amazon-cloudwatch-agent.rpm

Configure for Application Signals

# Create configuration directory if needed sudo mkdir -p /opt/aws/amazon-cloudwatch-agent/etc # Create Application Signals configuration sudo tee /opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json > /dev/null <<'EOF' { "traces": { "traces_collected": { "application_signals": {} } }, "logs": { "metrics_collected": { "application_signals": {} } } } EOF # Verify the configuration file cat /opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json

Start CloudWatch Agent

# Start the agent with Application Signals config sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \ -a fetch-config \ -m ec2 -s \ -c file:/opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json # Check status sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \ -a status \ -m ec2
Expected Output: You should see "status": "running" in the JSON output.

Verify Ports

# Check if Application Signals ports are listening sudo netstat -tlnp | grep 431 # Expected: ports 4315 and 4316 in LISTEN state
Port Information:
  • 4315: OTLP gRPC receiver
  • 4316: OTLP HTTP receiver (used by ADOT Java agent)

6. Build Quarkus Application

Note: Run these commands on your local machine where you have the source code.

Build Application

# Navigate to project directory cd ec2-quarkus-java # Create deployment directory mkdir -p deployment # Build the application (skip tests for faster build) mvn clean package -DskipTests # Create tarball for deployment cd target tar -czf quarkus-app.tar.gz quarkus-app/ cd ..

Download ADOT Java Agent

# Download AWS Distro for OpenTelemetry Java agent wget -O deployment/aws-opentelemetry-agent.jar \ https://github.com/aws-observability/aws-otel-java-instrumentation/releases/latest/download/aws-opentelemetry-agent.jar echo "Build complete! Files ready for deployment."

Upload to EC2

Transfer the files to your EC2 instance using one of these methods:

Option A: Via S3

# Upload to S3 aws s3 cp target/quarkus-app.tar.gz s3://YOUR-BUCKET/ \ --profile YOUR_PROFILE # Then on EC2, download from S3 aws s3 cp s3://YOUR-BUCKET/quarkus-app.tar.gz /home/ec2-user/

Option B: Via SCP (if using SSH)

scp -i YOUR-KEY.pem target/quarkus-app.tar.gz ec2-user@YOUR-EC2-IP:/home/ec2-user/ scp -i YOUR-KEY.pem deployment/aws-opentelemetry-agent.jar ec2-user@YOUR-EC2-IP:/home/ec2-user/

Option C: Download ADOT agent directly on EC2

# On EC2 instance cd /home/ec2-user wget -O aws-opentelemetry-agent.jar \ https://github.com/aws-observability/aws-otel-java-instrumentation/releases/latest/download/aws-opentelemetry-agent.jar

7. Deploy Application on EC2

Important: Run these commands on your EC2 instance via SSH or SSM Session Manager.

Install Java 21

# Install Amazon Corretto 21 sudo yum install -y java-21-amazon-corretto-devel # Set as default Java version sudo alternatives --config java # Select Java 21 option # Verify Java version java -version # Expected: openjdk version "21.x.x"

Extract Application

cd /home/ec2-user # Stop any existing application pkill -f quarkus-run.jar || true # Clean up old deployment rm -rf quarkus-app # Extract application tar -xzf quarkus-app.tar.gz # Verify extraction ls -la quarkus-app/

Create Startup Script

# Create startup script with OpenTelemetry configuration cat > start-app.sh <<'EOF' #!/bin/bash export OTEL_RESOURCE_ATTRIBUTES="service.name=ec2-quarkus-java,deployment.environment=ec2:production" export OTEL_EXPORTER_OTLP_TRACES_ENDPOINT="http://localhost:4316/v1/traces" export OTEL_METRICS_EXPORTER="none" export OTEL_LOGS_EXPORTER="none" export OTEL_AWS_APPLICATION_SIGNALS_ENABLED="true" export OTEL_AWS_APPLICATION_SIGNALS_EXPORTER_ENDPOINT="http://localhost:4316/v1/metrics" java -javaagent:/home/ec2-user/aws-opentelemetry-agent.jar \ -jar /home/ec2-user/quarkus-app/quarkus-run.jar EOF # Make executable chmod +x start-app.sh
Environment Variables Explained:
  • OTEL_RESOURCE_ATTRIBUTES: Sets service name and environment for Application Signals dashboard
  • OTEL_EXPORTER_OTLP_TRACES_ENDPOINT: CloudWatch agent endpoint for traces
  • OTEL_AWS_APPLICATION_SIGNALS_ENABLED: Enables Application Signals specific features
  • OTEL_AWS_APPLICATION_SIGNALS_EXPORTER_ENDPOINT: CloudWatch agent endpoint for metrics
  • OTEL_METRICS_EXPORTER/OTEL_LOGS_EXPORTER: Disabled to use only Application Signals exporter

Start Application

# Start application in background nohup ./start-app.sh > app.log 2>&1 & # Wait for startup sleep 5 # Check if process is running ps aux | grep quarkus # Check application logs tail -30 app.log

Test Application Locally

# Test root endpoint curl http://localhost:8080 # Test health endpoint curl http://localhost:8080/health # Test hello endpoint curl http://localhost:8080/hello/TestUser
Expected Response: 
{"environment":"ec2","message":"Hello from Quarkus EC2 Application!","timestamp":1760817129049,"status":"running"}

Available Endpoints

Endpoint Method Description
/ GET Root endpoint with basic info
/health GET Health check endpoint
/hello/{name} GET Personalized greeting
/slow GET Simulated slow operation (2-5s)
/error GET Simulated error (500 status)
/external GET Simulated external API call
/s3 GET Simulated S3 operation
/database GET Simulated database query
/complex GET Multi-step complex operation

8. Traffic Generation

To populate Application Signals with data, generate continuous traffic to your application.

Create Traffic Generator Script

# On EC2 instance cd /home/ec2-user # Create traffic generator that runs once cat > traffic-generator.sh <<'EOF' #!/bin/bash # Array of endpoints to test endpoints=( "/" "/health" "/hello/User1" "/hello/User2" "/hello/User3" "/hello/TestUser" "/external" "/s3" "/database" "/slow" "/complex" ) # Make 5 random requests for i in {1..5}; do # Select random endpoint endpoint=${endpoints[$RANDOM % ${#endpoints[@]}]} # Make request response=$(curl -s -o /dev/null -w "%{http_code}" http://localhost:8080${endpoint}) # Log request timestamp=$(date '+%Y-%m-%d %H:%M:%S') echo "[$timestamp] ${endpoint} - Status: ${response}" >> /home/ec2-user/traffic-generator.log # Small delay between requests sleep 2 done EOF # Make executable chmod +x traffic-generator.sh

Schedule with Cron

# Add to crontab to run every minute (crontab -l 2>/dev/null; echo "* * * * * /home/ec2-user/traffic-generator.sh") | crontab - # Verify crontab crontab -l
Traffic Pattern: This generates 5 requests per minute (300 requests/hour) hitting random endpoints to create diverse traces and metrics.

Monitor Traffic

# Watch traffic log in real-time tail -f /home/ec2-user/traffic-generator.log # Count total requests wc -l /home/ec2-user/traffic-generator.log # See last 30 requests tail -30 /home/ec2-user/traffic-generator.log

Stop Traffic Generator

# Remove from crontab crontab -r # Verify removal crontab -l

9. Verify Application Signals

Wait for Data

Important: Wait 5-10 minutes after starting traffic generation for data to appear in Application Signals.

Access CloudWatch Console

  1. Open AWS Console: https://console.aws.amazon.com/cloudwatch/
  2. Navigate to Application Signals → Services
  3. Look for service: ec2-quarkus-java

Service Details

In Application Signals, you should see:

  • Service Map: Visual representation of your service and dependencies
  • Service Operations: List of all endpoints with metrics
  • Traces: Individual request traces via X-Ray
  • Metrics: Latency, error rate, request rate
  • Service Health: Overall health indicators and SLOs

View Traces in X-Ray

  1. Navigate to X-Ray → Traces
  2. Filter by service name: ec2-quarkus-java
  3. Click on individual traces to see detailed timing information

View Metrics in CloudWatch

  1. Navigate to CloudWatch → Metrics
  2. Look for ApplicationSignals namespace
  3. Explore metrics by service, operation, and dimension
Success Indicators:
  • Service "ec2-quarkus-java" appears in Service Map
  • Traces visible in X-Ray with proper service name
  • Metrics showing request rate, latency, and errors
  • Service health status is "Healthy"

10. Troubleshooting

Application Won't Start

Check Java Version

java -version # Should show Java 21 # If wrong version, switch: sudo alternatives --config java

Check Application Logs

tail -100 /home/ec2-user/app.log grep -i error /home/ec2-user/app.log

Check if Process is Running

ps aux | grep quarkus netstat -tlnp | grep 8080

CloudWatch Agent Issues

Check Agent Status

sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \ -a status \ -m ec2

Check Agent Logs

sudo tail -100 /opt/aws/amazon-cloudwatch-agent/logs/amazon-cloudwatch-agent.log

Restart CloudWatch Agent

sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \ -a stop \ -m ec2 sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \ -a fetch-config \ -m ec2 -s \ -c file:/opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json

Verify Ports

sudo netstat -tlnp | grep 431 # Should show ports 4315 and 4316 listening

No Data in Application Signals

Checklist

  • CloudWatch agent is running and listening on ports 4315/4316
  • Application is running and receiving requests
  • Traffic generator is active (check crontab and logs)
  • IAM role has CloudWatchAgentServerPolicy attached
  • Application Signals is enabled in the AWS account
  • Wait 5-10 minutes for initial data to appear

Verify OTLP Connection

# Check application logs for OTLP errors grep -i "otlp\|exporter" /home/ec2-user/app.log | tail -20 # Check if traffic is reaching the application tail -20 /home/ec2-user/traffic-generator.log

Port Already in Use

# Find what's using the port sudo netstat -tlnp | grep 4315 # Kill the process (replace PID) sudo kill -9 PID # Or kill all CloudWatch agent processes sudo pkill -9 amazon-cloudwatch-agent # Then restart sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \ -a fetch-config \ -m ec2 -s \ -c file:/opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json

Permission Denied Errors

# Check IAM role policies aws iam list-attached-role-policies \ --role-name YOUR_ROLE_NAME \ --profile YOUR_PROFILE # Check CloudWatch agent has write permissions sudo tail -50 /opt/aws/amazon-cloudwatch-agent/logs/amazon-cloudwatch-agent.log | grep -i "permission\|denied\|403"

Application Logs

View Real-time Logs

tail -f /home/ec2-user/app.log

Search for Errors

grep -i "error\|exception\|failed" /home/ec2-user/app.log | tail -30

Restart Everything

# Stop application pkill -f quarkus-run.jar # Restart CloudWatch agent sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \ -a fetch-config \ -m ec2 -s \ -c file:/opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json # Wait for agent to start sleep 5 # Restart application cd /home/ec2-user nohup ./start-app.sh > app.log 2>&1 & # Verify both are running sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl -a status -m ec2 ps aux | grep quarkus

Common Error Messages

"Failed to connect to localhost:4316"

Solution: CloudWatch agent is not running or not configured properly. Restart the agent.

"class file version 65.0"

Solution: Application compiled with Java 21 but running on older Java. Install Java 21.

"Port 8080 already in use"

Solution: Another process is using port 8080. Kill the process or use a different port.

"Permission denied" writing to CloudWatch

Solution: IAM role missing CloudWatchAgentServerPolicy. Attach the policy.
