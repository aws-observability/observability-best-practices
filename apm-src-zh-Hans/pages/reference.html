服务限制和配额

了解 CloudWatch Application Signals 的限制和配额。

限制和配额

Application Signals 依赖于 CloudWatch 和 AWS X-Ray。限制（摄取、查询速率和配额）由这些服务管理，并可能随时间变化。请使用官方文档获取当前值。

  • Application Signals： Application Signals 文档
  • CloudWatch 配额： CloudWatch 服务配额
  • AWS X-Ray 配额： X-Ray API 限流和限制

CloudWatch 指标保留（常用参考）

CloudWatch 以不同分辨率保留指标数据。有关完整的保留模型，请参阅 CloudWatch 文档。

  • 1 分钟分辨率：15 天
  • 5 分钟分辨率：63 天
  • 1 小时分辨率：15 个月

服务配额

Application Signals：

  • 每个账户的服务数：1,000
  • 每个服务的操作数：100
  • 自定义指标：每个服务 30 个

RUM：

  • 每个账户的应用监控器：20
  • 每秒事件数：10,000
  • 事件大小：64 KB

Synthetics：

  • 每个账户的金丝雀：100
  • 金丝雀执行时间：最长 15 分钟
  • 金丝雀大小：1 GB（解压后）

术语表

CloudWatch Application Signals 中的关键术语和概念。

核心概念

应用程序性能监控（APM）

监控和管理软件应用程序性能和可用性的实践。

分布式追踪

一种跟踪请求在分布式系统中流转的方法，使您能够了解请求在多个服务中的路径和性能。

服务地图

应用程序架构的可视化表示，展示服务如何连接和交互。

Span

追踪中的单个操作，表示一个工作单元，如数据库查询或 API 调用。

追踪

请求在您的应用程序中完整的端到端旅程，由多个 span 组成。

检测术语

自动检测

无需代码更改即可自动向应用程序添加 Observability 代码。

手动检测

向应用程序源代码显式添加追踪和指标代码。

OpenTelemetry

一个开源 Observability 框架，提供与供应商无关的 API 用于检测。

采样

选择要追踪哪些请求的过程，以平衡 Observability 与成本和性能。

上下文传播

在服务之间传递追踪上下文以维护追踪连续性的机制。

指标和监控

黄金信号

监控应用程序健康状况的四个关键指标：延迟、吞吐量、错误率和饱和度。

延迟

处理请求所需的时间，通常以百分位数（P50、P90、P99）衡量。

吞吐量

处理请求的速率，通常以每秒请求数衡量。

错误率

导致错误的请求百分比。

饱和度

系统资源被利用的程度。

服务级别目标（SLO）

服务提供者和消费者之间商定的服务质量目标水平。

服务级别指标（SLI）

所提供服务级别的可衡量方面，如延迟或可用性。

RUM 和 Synthetics

真实用户监控（RUM）

监控生产环境中用户与 Web 应用程序的实际交互。

合成监控

使用模拟用户交互对应用程序进行自动化测试。

金丝雀

对您的应用程序运行自动化测试的合成监控脚本。

Core Web Vitals

Google 用于衡量 Web 用户体验的指标：LCP、FID 和 CLS。

最大内容绘制（LCP）

最大内容元素渲染到屏幕上所需的时间。

首次输入延迟（FID）

从用户首次与页面交互到浏览器响应的时间。

累积布局偏移（CLS）

视觉稳定性的度量，量化意外的布局偏移。

AWS 特定术语

Application Signals

AWS 的统一 Observability 解决方案，结合了 APM、RUM 和 Synthetics。

X-Ray

AWS 的分布式追踪服务，现已与 Application Signals 集成。

CloudWatch

AWS 的监控和 Observability 服务，收集指标、日志和追踪。

ADOT（AWS Distro for OpenTelemetry）

AWS 的 OpenTelemetry 发行版，针对 AWS 服务进行了优化。

支持的运行时和框架

Application Signals 支持的编程语言和框架。

自动检测支持

有关权威的兼容性和运行时/版本矩阵，请参阅 支持的系统。要点：

  • Java：JVM 8/11/17/21/23（框架支持遵循 ADOT/OpenTelemetry Java 检测）。
  • Python：Python 3.9+（框架支持遵循 ADOT/OpenTelemetry Python 检测）。
  • Node.js：Node 14/16/18/20/22（框架支持遵循 ADOT/OpenTelemetry JS 检测）。
  • .NET：支持 x86-64 或 ARM64；Linux x64/ARM64 和 Windows Server 2022 x64（库支持遵循 ADOT/OpenTelemetry .NET 检测）。
  • PHP/Ruby/Go：通过 OpenTelemetry 零代码/自动检测路径支持（这些语言没有 ADOT SDK）。

注意事项和已知考量

  • Node.js ESM：ESM 支持有限；除非您按照文档中 ESM 特定设置操作，否则建议使用 CommonJS。
  • Python：容器化应用可能需要设置 PYTHONPATH（已知的 OTel 自动检测问题）。
  • .NET：ADOT .NET 在 AWS SDK for .NET V4 方面存在限制；建议使用 AWS SDK V3 以获得完整支持。

手动检测

通过 OpenTelemetry SDK 支持所有主要编程语言：

  • Java、Scala、Kotlin
  • Python、Ruby
  • JavaScript/TypeScript、Go
  • .NET（C#、F#、VB.NET）
  • PHP、Rust、C++

配置参考

常用配置选项和环境变量。

环境变量

OpenTelemetry 配置：

  • OTEL_SERVICE_NAME - 您的服务名称
  • OTEL_TRACES_EXPORTER - 追踪导出器（otlp、jaeger、zipkin）
  • OTEL_EXPORTER_OTLP_ENDPOINT - OTLP 端点 URL
  • OTEL_TRACES_SAMPLER - 采样策略
  • OTEL_TRACES_SAMPLER_ARG - 采样参数

AWS 特定配置：

  • AWS_REGION - AWS 区域
  • AWS_XRAY_DAEMON_ADDRESS - X-Ray 守护进程地址
  • OTEL_RESOURCE_ATTRIBUTES - 服务属性（也用于应用程序地图中的自定义分组）

采样配置

采样策略：

  • always_on - 采样所有追踪
  • always_off - 不采样追踪
  • traceidratio - 基于比率采样（例如，0.1 表示 10%）
  • parentbased_always_on - 遵循父级采样决策
  • parentbased_traceidratio - 基于父级的比率采样

成本优化

控制 Observability 成本的策略。

采样策略

  • 生产环境：5-10% 采样率
  • 开发环境：50-100% 采样率
  • 错误追踪：始终采样包含错误的追踪
  • 高价值操作：关键路径 100% 采样

数据保留策略

保留由底层遥测存储（例如，CloudWatch 指标/日志和 AWS X-Ray 追踪）控制。在这些服务中配置保留期，并使其与您的故障排除和合规需求保持一致。

资源优化

  • 批量导出：减少网络开销
  • 压缩：启用数据压缩
  • 过滤：移除不必要的 span 和属性
  • 缓冲：使用适当的缓冲区大小
