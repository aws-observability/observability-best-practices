概要

このガイドでは ステップバイステップの手順 AWS Application Signals を有効にした Amazon ECS 上への .NET Core アプリケーションのデプロイ方法を説明します。CDK は不要で、AWS CLI と Docker のみを使用します。

達成できること：
  • ECS Fargate に .NET Core アプリをデプロイする
  • AWS Application Signals の自動インストルメンテーションを有効にする
  • CloudWatch Application Signals コンソールでサービスを確認する
  • サービスのメトリクス、トレース、オペレーションを監視する
所要時間： 30〜45 分

前提条件

必要なツール

  • AWS CLI v2（最新バージョン）
  • Docker Desktop
  • .NET 8.0 SDK（ローカル開発用）
  • 適切な権限を持つ AWS アカウント
  • テキストエディタ

必要な AWS 権限

  • ECS: フルアクセス（クラスター、サービス、タスク定義の作成）
  • ECR: イメージのプッシュ
  • IAM: ロールおよびポリシーの作成
  • CloudWatch: ログおよび Application Signals へのアクセス
  • EC2: VPC、サブネット、セキュリティグループへのアクセス
  • SSM: Parameter Store への書き込みアクセス

前提とする知識

  • AWS CLI の基本的な使い方
  • Docker の基本的なコマンド
  • ECS の概念に関する基本的な理解
  • .NET Core の基本的な知識

アーキテクチャ

ECS タスクには 3 つのコンテナ:

  1. Init コンテナ (init): ADOT .NET 自動インストルメンテーションファイルをコピーします
  2. CloudWatch Agent (ecs-cwagent): トレースとメトリクスを収集します
  3. Application Container (app): .NET Core アプリケーション
互換性に関する重要な注意事項: AWS Distro for OpenTelemetry (ADOT) SDK for .NET は AWS SDK for .NET V4 をサポートしていません。Application Signals の完全なサポートには AWS SDK .NET V3 を使用してください。このガイドでは、AWS SDK .NET V3 を使用した基本的な OpenTelemetry 設定による ECS 実装の動作例を提供します。

フロー:

Init コンテナ（一度だけ実行され、ファイルをコピーします）

↓

アプリケーションが起動します（.NET スタートアップフックによる自動インストルメンテーション）

↓

トレース/メトリクスを CloudWatch Agent に送信します（localhost:4316）

↓

CloudWatch Agent が Application Signals に送信します

1アカウントで Application Signals を有効にする

重要: Application Signals はリージョンごとに AWS アカウントあたり一度だけ有効にする必要があります。

オプション 1: AWS コンソールから有効にする（推奨）

  1. AWS コンソール → CloudWatch → Application Signals に移動します
  2. リージョンを選択します（例： us-east-1)
  3. 「Get Started」または「Enable Application Signals」をクリックします
  4. 有効化されるまで 2〜3 分待ちます

オプション 2: すでに有効になっているか確認する

# Check if service-linked role exists
aws iam get-role \
  --role-name AWSServiceRoleForCloudWatchApplicationSignals \
  --region us-east-1

# If role exists, Application Signals is enabled

出力結果: 有効な場合はロール ARN、無効な場合はエラーが表示されます。

2.2.1 プロジェクトの依存関係

このプロジェクトは Application Signals との互換性のために AWS SDK .NET V3 と OpenTelemetry パッケージを使用しています：

主な依存関係：

  • AWS SDK .NET V3: AWSSDK.Core, AWSSDK.CloudWatch, AWSSDK.XRay （V3.7.400.1）
  • OpenTelemetry: トレースとメトリクスのコアパッケージ (V1.7.0)
  • AWS.DistributedOpenTelemetry: Application Signals 統合用 (V1.0.0)
AWS SDK V3 を使用する理由 AWS Distro for OpenTelemetry (ADOT) SDK for .NET は AWS SDK for .NET V4 をサポートしていません。V3 を使用することで、Application Signals の完全なサポートが保証されます。

3Docker イメージのビルドとプッシュ

3.1 変数の設定

# Replace with your values
export AWS_REGION="us-east-1"
export AWS_ACCOUNT_ID="YOUR_ACCOUNT_ID"
export AWS_PROFILE="your-profile-name"  # Optional if using default
export APP_NAME="my-appsignals-dotnet-app"

アカウント ID の取得:

export AWS_ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
echo "Account ID: $AWS_ACCOUNT_ID"

3.2 ECR リポジトリの作成

aws ecr create-repository \
  --repository-name ${APP_NAME} \
  --region ${AWS_REGION}

出力: リポジトリ URI (保存してください)

3.3 Docker イメージのビルド

重要: Apple Silicon (M1/M2) Mac でビルドする場合は、正しいアーキテクチャ向けにビルドする必要があります。 
cd ~/my-appsignals-dotnet-app

# For Apple Silicon Macs, build for AMD64 architecture
docker build --platform linux/amd64 -t ${APP_NAME} .

# For Intel Macs or Linux, use standard build
# docker build -t ${APP_NAME} .

3.4 ECR へのプッシュ

# Login to ECR
aws ecr get-login-password --region ${AWS_REGION} | \
  docker login --username AWS --password-stdin \
  ${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com

# Tag image
docker tag ${APP_NAME}:latest \
  ${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com/${APP_NAME}:latest

# Push image
docker push ${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com/${APP_NAME}:latest

4CloudWatch Agent 設定の作成

4.1 設定ファイルの作成

cat > /tmp/ecs-cwagent-dotnet.json << 'EOF'
{
  "traces": {
    "traces_collected": {
      "application_signals": {}
    }
  },
  "logs": {
    "metrics_collected": {
      "application_signals": {}
    }
  }

EOF

4.2 SSM Parameter Store へのアップロード

aws ssm put-parameter \
  --name "ecs-cwagent-${APP_NAME}" \
  --type "String" \
  --value "$(cat /tmp/ecs-cwagent-dotnet.json)" \
  --region "${AWS_REGION}" \
  --overwrite

出力: バージョン番号 (新規の場合は 1、更新の場合はインクリメント)

5IAM ロールの作成

5.1 信頼ポリシーの作成

cat > /tmp/task-trust-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "ecs-tasks.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
EOF

5.2 タスクロールの作成

aws iam create-role \
  --role-name ${APP_NAME}-ecs-task-role \
  --assume-role-policy-document file:///tmp/task-trust-policy.json

# Attach required policies
aws iam attach-role-policy \
  --role-name ${APP_NAME}-ecs-task-role \
  --policy-arn arn:aws:iam::aws:policy/CloudWatchAgentServerPolicy

aws iam attach-role-policy \
  --role-name ${APP_NAME}-ecs-task-role \
  --policy-arn arn:aws:iam::aws:policy/AWSXRayDaemonWriteAccess

5.3 タスク実行ロールの作成

aws iam create-role \
  --role-name ${APP_NAME}-ecs-execution-role \
  --assume-role-policy-document file:///tmp/task-trust-policy.json

# Attach policies
aws iam attach-role-policy \
  --role-name ${APP_NAME}-ecs-execution-role \
  --policy-arn arn:aws:iam::aws:policy/service-role/AmazonECSTaskExecutionRolePolicy

aws iam attach-role-policy \
  --role-name ${APP_NAME}-ecs-execution-role \
  --policy-arn arn:aws:iam::aws:policy/AmazonSSMReadOnlyAccess

5.4 CloudWatch Logs 権限の追加

cat > /tmp/logs-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogGroup",
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "arn:aws:logs:*:*:log-group:/ecs/*"
    }
  ]
}
EOF

aws iam put-role-policy \
  --role-name ${APP_NAME}-ecs-execution-role \
  --policy-name CloudWatchLogsPolicy \
  --policy-document file:///tmp/logs-policy.json

7ネットワークのセットアップ

7.1 VPC とサブネットの取得

既に VPC がある場合:

# List VPCs
aws ec2 describe-vpcs --region ${AWS_REGION} \
  --query 'Vpcs[*].{VpcId:VpcId,CIDR:CidrBlock,Name:Tags[?Key==`Name`].Value|[0]}'

# Choose your VPC
export VPC_ID="vpc-xxxxx"

# Get private subnets
aws ec2 describe-subnets --region ${AWS_REGION} \
  --filters "Name=vpc-id,Values=${VPC_ID}" \
  --query 'Subnets[?MapPublicIpOnLaunch==`false`].[SubnetId,AvailabilityZone,CidrBlock]'

# Save subnet IDs
export SUBNET_1="subnet-xxxxx"
export SUBNET_2="subnet-yyyyy"

7.2 セキュリティグループの作成

export SG_NAME="${APP_NAME}-ecs-sg"

# Create security group
SG_ID=$(aws ec2 create-security-group \
  --group-name ${SG_NAME} \
  --description "Security group for ${APP_NAME} ECS tasks" \
  --vpc-id ${VPC_ID} \
  --region ${AWS_REGION} \
  --query 'GroupId' \
  --output text)

echo "Security Group ID: $SG_ID"
export SG_ID

# Add ingress rule (allow HTTP from anywhere for testing)
aws ec2 authorize-security-group-ingress \
  --group-id ${SG_ID} \
  --protocol tcp \
  --port 8080 \
  --cidr 0.0.0.0/0 \
  --region ${AWS_REGION}
本番環境の場合: 置換 0.0.0.0/0 特定のIP範囲またはロードバランサーのセキュリティグループに置き換えてください。

9ECSサービスのデプロイ

aws ecs create-service \
  --cluster ${CLUSTER_NAME} \
  --service-name ${APP_NAME}-service \
  --task-definition ${APP_NAME}-task:1 \
  --desired-count 1 \
  --launch-type FARGATE \
  --network-configuration "awsvpcConfiguration={subnets=[${SUBNET_1},${SUBNET_2}],securityGroups=[${SG_ID}],assignPublicIp=ENABLED}" \
  --region ${AWS_REGION}
注意： assignPublicIp=ENABLED はテスト用です。本番環境ではロードバランサーを使用してください。

トラブルシューティング

問題：Application Signals にサービスが表示されない

確認事項：

# 1. Verify metrics are being published
aws cloudwatch list-metrics \
  --namespace ApplicationSignals \
  --region ${AWS_REGION} \
  --dimensions Name=Service,Value=${APP_NAME}

# 2. Check if Application Signals is enabled
aws iam get-role \
  --role-name AWSServiceRoleForCloudWatchApplicationSignals

# 3. Check container logs for errors
aws logs tail /ecs/${APP_NAME} --region ${AWS_REGION} --since 10m

問題：Init コンテナが失敗する

確認事項：

# Describe task to see init container exit code
aws ecs describe-tasks \
  --cluster ${CLUSTER_NAME} \
  --tasks ${TASK_ARN} \
  --region ${AWS_REGION} \
  --query 'tasks[0].containers[?name==`init`].{name:name,exitCode:exitCode,reason:reason}'

期待される結果： exitCode: 0

ゼロ以外の場合： イメージが利用できない可能性があります。タグを確認してください v1.34.0 存在するか確認：

docker pull public.ecr.aws/aws-observability/adot-autoinstrumentation-dotnet:v1.34.0

問題：アプリケーションが CloudWatch Agent に接続できない

症状：

Connection refused to localhost:4316

確認事項：

  1. CloudWatch agent コンテナが RUNNING 状態であることを確認
  2. コンテナの依存関係を確認 - アプリは以下に依存する必要があります init 条件 SUCCESS を指定して
  3. 環境変数を確認（特にエンドポイント： http://localhost:4316 および http://localhost:2000)

問題：X-Ray にトレースが表示されない

よくある原因：

  1. 不足している DOTNET_STARTUP_HOOKS - .NET 自動インストルメンテーションに必須
  2. 不足している DOTNET_ADDITIONAL_DEPS - .NETインストルメンテーションに必要
  3. 見つかりません DOTNET_SHARED_STORE - .NETインストルメンテーションに必要
  4. ボリュームマウントが見つかりません - .NETインストルメンテーションファイルにアクセス可能である必要があります

問題: IAM権限エラー

症状:

AccessDeniedException: ... logs:CreateLogGroup

修正方法: 実行ロールにCloudWatch Logsの権限を追加する（ステップ5.4を参照）

問題: コンテナ内の「exec format error」

症状:

exec /usr/bin/dotnet: exec format error

原因: アーキテクチャの不一致 - DockerイメージはARM64（Apple Silicon）でビルドされましたが、ECS FargateはX86_64で動作します。

修正方法: 正しいアーキテクチャ向けにDockerイメージを再ビルドする:

# Rebuild for AMD64 architecture
docker build --platform linux/amd64 -t ${APP_NAME} .

# Tag and push the corrected image
docker tag ${APP_NAME}:latest \
  ${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com/${APP_NAME}:latest

docker push ${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com/${APP_NAME}:latest

# Update the ECS service to use the new image
aws ecs update-service \
  --cluster ${CLUSTER_NAME} \
  --service ${APP_NAME}-service \
  --force-new-deployment \
  --region ${AWS_REGION}

イメージ: public.ecr.aws/aws-observability/adot-autoinstrumentation-dotnet:v1.34.0

  • タグが存在しません。特定のバージョンタグを使用してください（確認先: https://github.com/aws-observability/aws-otel-dotnet-instrumentation/releases)

動作内容:

  1. アプリコンテナが起動する前に一度だけ実行されます
  2. ADOTの.NETインストルメンテーションファイルを /autoinstrumentation/ から共有ボリュームにコピーします
  3. コード0で終了します
  4. アプリコンテナはボリュームをマウントし、.NETスタートアップフックを使用してインストルメンテーションを読み込みます

アプリケーションコードの要件

すること：

  • 通常の .NET Core コードを記述する
  • 標準ライブラリを使用する（ASP.NET Core、HttpClient など）
  • コードをクリーンに保つ - OpenTelemetry のインポートは不要
  • 標準の ASP.NET Core 属性を使用する（[ApiController]、[HttpGet] など）

しないこと：

  • インポートしない OpenTelemetry.* パッケージ
  • OpenTelemetry を手動で設定する
  • 手動トレースコードを追加する（カスタムスパンが必要な場合を除く）
  • .csproj に OpenTelemetry の依存関係をインストールする

# Delete ECS service
aws ecs update-service \
  --cluster ${CLUSTER_NAME} \
  --service ${APP_NAME}-service \
  --desired-count 0 \
  --region ${AWS_REGION}

aws ecs delete-service \
  --cluster ${CLUSTER_NAME} \
  --service ${APP_NAME}-service \
  --region ${AWS_REGION}

# Delete ECS cluster
aws ecs delete-cluster \
  --cluster ${CLUSTER_NAME} \
  --region ${AWS_REGION}

# Delete ECR repository
aws ecr delete-repository \
  --repository-name ${APP_NAME} \
  --force \
  --region ${AWS_REGION}

# Delete security group
aws ec2 delete-security-group \
  --group-id ${SG_ID} \
  --region ${AWS_REGION}

# Delete IAM roles
aws iam delete-role \
  --role-name ${APP_NAME}-ecs-task-role

aws iam delete-role \
  --role-name ${APP_NAME}-ecs-execution-role

# Delete SSM parameter
aws ssm delete-parameter \
  --name ecs-cwagent-${APP_NAME} \
  --region ${AWS_REGION}
