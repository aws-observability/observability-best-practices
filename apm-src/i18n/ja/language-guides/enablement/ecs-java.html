概要

このガイドでは ステップバイステップの手順 を提供します。AWS Application Signals を有効にした Java Spring Boot アプリケーションを Amazon ECS にデプロイします。CDK 不要 - 純粋な AWS CLI と Docker を使用します。

達成できること：
  • ECS Fargate に Java Spring Boot アプリをデプロイする
  • AWS Application Signals の自動インストルメンテーションを有効にする
  • CloudWatch Application Signals コンソールでサービスを確認する
  • サービスのメトリクス、トレース、オペレーションを監視する
完了までの時間： 30〜45 分

前提条件

必要なツール

  • AWS CLI v2（最新バージョン）
  • Docker Desktop
  • Java 17+（ローカル開発用）
  • Maven 3.8+（ローカル開発用）
  • 適切な権限を持つ AWS アカウント
  • テキストエディタ

必要な AWS 権限

  • ECS: フルアクセス（クラスター、サービス、タスク定義の作成）
  • ECR: イメージのプッシュ
  • IAM: ロールとポリシーの作成
  • CloudWatch: ログおよび Application Signals へのアクセス
  • EC2: VPC、サブネット、セキュリティグループへのアクセス
  • SSM: Parameter Store への書き込みアクセス

前提とする知識

  • AWS CLI の基本的な使い方
  • Docker の基本的なコマンド
  • ECS の概念に関する基本的な理解
  • Java/Spring Boot の基本的な知識

アーキテクチャ

ECS タスクには以下が含まれます 3 つのコンテナ:

  1. Init Container (init): ADOT Java自動インストルメンテーションファイルをコピーします
  2. CloudWatch Agent (ecs-cwagent): トレースとメトリクスを収集します
  3. Application Container (app): Spring Bootアプリケーション

フロー:

Init Container（1回実行、ファイルをコピー）

↓

アプリケーションが起動（javaagent経由で自動インストルメンテーション）

↓

トレース/メトリクスをCloudWatch Agent（localhost:4316）に送信

↓

CloudWatch AgentがApplication Signalsに送信

1アカウントでApplication Signalsを有効にする

重要: Application Signalsはリージョンごとに、AWSアカウントごとに1回有効にする必要があります。

オプション1: AWSコンソールから有効にする（推奨）

  1. AWSコンソール → CloudWatch → Application Signals に移動
  2. リージョンを選択（例： us-east-1)
  3. 「Get Started」または「Enable Application Signals」をクリック
  4. 有効化まで2〜3分待機

オプション2: すでに有効かどうかを確認する

# Check if service-linked role exists
aws iam get-role \
  --role-name AWSServiceRoleForCloudWatchApplicationSignals \
  --region us-east-1

# If role exists, Application Signals is enabled

出力結果: 有効な場合はロールARN、無効な場合はエラーが表示されます。

3Dockerイメージのビルドとプッシュ

3.1 変数の設定

# Replace with your values
export AWS_REGION="us-east-1"
export AWS_ACCOUNT_ID="YOUR_ACCOUNT_ID"
export AWS_PROFILE="your-profile-name"  # Optional if using default
export APP_NAME="my-appsignals-java-app"

アカウントIDを取得する：

export AWS_ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
echo "Account ID: $AWS_ACCOUNT_ID"

3.2 ECRリポジトリの作成

aws ecr create-repository \
  --repository-name ${APP_NAME} \
  --region ${AWS_REGION}

出力： リポジトリURI（保存しておくこと）

3.3 Dockerイメージのビルド

cd ~/my-appsignals-java-app
docker build -t ${APP_NAME} .

3.4 ECRへのプッシュ

# Login to ECR
aws ecr get-login-password --region ${AWS_REGION} | \
  docker login --username AWS --password-stdin \
  ${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com

# Tag image
docker tag ${APP_NAME}:latest \
  ${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com/${APP_NAME}:latest

# Push image
docker push ${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com/${APP_NAME}:latest

4CloudWatch エージェント設定の作成

4.1 設定ファイルの作成

cat > /tmp/ecs-cwagent-java.json << 'EOF'
{
  "traces": {
    "traces_collected": {
      "application_signals": {}
    }
  },
  "logs": {
    "metrics_collected": {
      "application_signals": {}
    }
  }
}
EOF

4.2 SSM パラメータストアへのアップロード

aws ssm put-parameter \
  --name "ecs-cwagent-${APP_NAME}" \
  --type "String" \
  --value "$(cat /tmp/ecs-cwagent-java.json)" \
  --region "${AWS_REGION}" \
  --overwrite

出力： バージョン番号（新規の場合は1、更新の場合はインクリメント）

5IAMロールの作成

5.1 信頼ポリシーの作成

cat > /tmp/task-trust-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "ecs-tasks.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
EOF

5.2 タスクロールの作成

aws iam create-role \
  --role-name ${APP_NAME}-ecs-task-role \
  --assume-role-policy-document file:///tmp/task-trust-policy.json

# Attach required policies
aws iam attach-role-policy \
  --role-name ${APP_NAME}-ecs-task-role \
  --policy-arn arn:aws:iam::aws:policy/CloudWatchAgentServerPolicy

aws iam attach-role-policy \
  --role-name ${APP_NAME}-ecs-task-role \
  --policy-arn arn:aws:iam::aws:policy/AWSXRayDaemonWriteAccess

5.3 タスク実行ロールの作成

aws iam create-role \
  --role-name ${APP_NAME}-ecs-execution-role \
  --assume-role-policy-document file:///tmp/task-trust-policy.json

# Attach policies
aws iam attach-role-policy \
  --role-name ${APP_NAME}-ecs-execution-role \
  --policy-arn arn:aws:iam::aws:policy/service-role/AmazonECSTaskExecutionRolePolicy

aws iam attach-role-policy \
  --role-name ${APP_NAME}-ecs-execution-role \
  --policy-arn arn:aws:iam::aws:policy/AmazonSSMReadOnlyAccess

5.4 CloudWatch Logs 権限の追加

cat > /tmp/logs-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogGroup",
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "arn:aws:logs:*:*:log-group:/ecs/*"
    }
  ]
}
EOF

aws iam put-role-policy \
  --role-name ${APP_NAME}-ecs-execution-role \
  --policy-name CloudWatchLogsPolicy \
  --policy-document file:///tmp/logs-policy.json

7ネットワークのセットアップ

7.1 VPCとサブネットの取得

すでにVPCがある場合：

# List VPCs
aws ec2 describe-vpcs --region ${AWS_REGION} \
  --query 'Vpcs[*].{VpcId:VpcId,CIDR:CidrBlock,Name:Tags[?Key==`Name`].Value|[0]}'

# Choose your VPC
export VPC_ID="vpc-xxxxx"

# Get private subnets
aws ec2 describe-subnets --region ${AWS_REGION} \
  --filters "Name=vpc-id,Values=${VPC_ID}" \
  --query 'Subnets[?MapPublicIpOnLaunch==`false`].[SubnetId,AvailabilityZone,CidrBlock]'

# Save subnet IDs
export SUBNET_1="subnet-xxxxx"
export SUBNET_2="subnet-yyyyy"

7.2 セキュリティグループの作成

export SG_NAME="${APP_NAME}-ecs-sg"

# Create security group
SG_ID=$(aws ec2 create-security-group \
  --group-name ${SG_NAME} \
  --description "Security group for ${APP_NAME} ECS tasks" \
  --vpc-id ${VPC_ID} \
  --region ${AWS_REGION} \
  --query 'GroupId' \
  --output text)

echo "Security Group ID: $SG_ID"
export SG_ID

# Add ingress rule (allow HTTP from anywhere for testing)
aws ec2 authorize-security-group-ingress \
  --group-id ${SG_ID} \
  --protocol tcp \
  --port 8080 \
  --cidr 0.0.0.0/0 \
  --region ${AWS_REGION}
本番環境の場合： 置き換える 0.0.0.0/0 特定のIP範囲またはロードバランサーのセキュリティグループを使用してください。

11Application Signals を確認する

11.1 CloudWatch ログを確認する

# Application logs
aws logs tail /ecs/${APP_NAME} --region ${AWS_REGION} --since 5m --format short

# CloudWatch agent logs
aws logs tail /ecs/ecs-cwagent-${APP_NAME} --region ${AWS_REGION} --since 5m --format short

確認する内容：

  • アプリケーションログ："Started ApplicationSignalsApplication"
  • エージェントログ："application_signals"、"X-Ray proxy server started"

11.2 メトリクスを確認する

# Wait 2-3 minutes after first requests, then check
aws cloudwatch list-metrics \
  --namespace ApplicationSignals \
  --region ${AWS_REGION} \
  --dimensions Name=Service,Value=${APP_NAME} \
  --query 'Metrics[*].MetricName' \
  --output text | tr '\t' '\n' | sort -u

期待されるメトリクス：

  • JvmHeapMemoryUsed
  • JvmNonHeapMemoryUsed
  • JvmGcDuration
  • JvmThreadCount
  • Latency
  • Error
  • Fault

11.3 トラフィックを生成する

# Generate traffic to create traces
for i in {1..100}; do
  curl -s http://${TASK_IP}:8080/api/hello/User${i} > /dev/null
  curl -s http://${TASK_IP}:8080/api/health > /dev/null
  curl -s http://${TASK_IP}:8080/api/database > /dev/null
done

echo "100 requests sent! Wait 5-10 minutes for Application Signals..."

11.4 Application Signals のサービス検出を確認する

# Wait 5-10 minutes after traffic generation
aws application-signals list-services \
  --region ${AWS_REGION} \
  --start-time $(date -u -v-2H +%s) \
  --end-time $(date -u +%s) \
  --query 'ServiceSummaries[*].{Name:KeyAttributes.Name,Environment:KeyAttributes.Environment}'

期待される結果： サービスは以下の情報とともに表示されるはずです：

  • 名前： ${APP_NAME}

  2. サービスを探す：

トラブルシューティング

問題: アプリケーションシグナルにサービスが表示されない

確認事項:

# 1. Verify metrics are being published
aws cloudwatch list-metrics \
  --namespace ApplicationSignals \
  --region ${AWS_REGION} \
  --dimensions Name=Service,Value=${APP_NAME}

# 2. Check if Application Signals is enabled
aws iam get-role \
  --role-name AWSServiceRoleForCloudWatchApplicationSignals

# 3. Check container logs for errors
aws logs tail /ecs/${APP_NAME} --region ${AWS_REGION} --since 10m

問題: Initコンテナが失敗する

確認事項:

# Describe task to see init container exit code
aws ecs describe-tasks \
  --cluster ${CLUSTER_NAME} \
  --tasks ${TASK_ARN} \
  --region ${AWS_REGION} \
  --query 'tasks[0].containers[?name==`init`].{name:name,exitCode:exitCode,reason:reason}'

期待される結果: exitCode: 0

ゼロ以外の場合: イメージが利用できない可能性があります。タグを確認してください v1.34.0 存在します:

docker pull public.ecr.aws/aws-observability/adot-autoinstrumentation-java:v1.34.0

問題: アプリケーションがCloudWatchエージェントに接続できない

症状:

Connection refused to localhost:4316

確認事項:

  1. CloudWatchエージェントコンテナがRUNNINGであることを確認する
  2. コンテナの依存関係を確認する - アプリは以下に依存する必要があります init 条件がSUCCESSであること
  3. 環境変数を確認する（特にエンドポイント: http://localhost:4316 および http://localhost:2000)

問題: X-Rayにトレースが表示されない

よくある原因:

  1. 不足している JAVA_TOOL_OPTIONS - JavaオートインストルメンテーションにとってCritical
  2. 不足している OTEL_TRACES_SAMPLER_ARG - X-Rayサンプラーエンドポイントが必要
  3. javaagentのパスが間違っている - 以下を指定する必要があります /otel-auto-instrumentation-java/opentelemetry-javaagent.jar
  4. ボリュームマウントが不足している - Javaエージェントファイルにアクセス可能である必要があります

問題: IAMパーミッションエラー

症状:

AccessDeniedException: ... logs:CreateLogGroup

修正方法: 実行ロールにCloudWatch Logsのパーミッションを追加する（ステップ5.4を参照）

Initコンテナの詳細

イメージ: public.ecr.aws/aws-observability/adot-autoinstrumentation-java:v1.34.0

バージョン付きタグを使用する理由（ :latest?

  • The :latest タグはJava ADOTイメージには存在しません
  • 特定のバージョンタグを使用してください（確認先: https://github.com/aws-observability/aws-otel-java-instrumentation/releases)
  • 現在の安定版: v1.34.0

機能の説明:

  1. アプリコンテナが起動する前に一度実行されます
  2. から共有ボリュームにADOT Javaインストルメンテーションファイルをコピーします /autoinstrumentation/ 共有ボリュームへ
  3. コード0で終了します
  4. アプリコンテナはボリュームをマウントし、 JAVA_TOOL_OPTIONS を使用してjavaagentを読み込みます

CloudWatch エージェントの設定

両方のセクションを含める必要があります:

{
  "traces": {
    "traces_collected": {
      "application_signals": {}   // Not just "otlp"!
    }
  },
  "logs": {
    "metrics_collected": {
      "application_signals": {}
    }
  }
}

アプリケーションコードの要件

すべきこと:

  • 通常のSpring Bootコードを記述する
  • 標準ライブラリを使用する（Spring Web、WebFluxなど）
  • コードをクリーンに保つ - OpenTelemetryのインポートは不要
  • 標準のSpring Bootアノテーションを使用する（@RestController、@GetMappingなど）

すべきでないこと:

  • インポート opentelemetry-* パッケージ
  • OpenTelemetryを手動で設定する
  • 手動トレースコードを追加する（カスタムスパンが必要な場合を除く）
  • pom.xmlにOpenTelemetryの依存関係をインストールする

クリーンアップ

すべてのリソースを削除するには:

# Delete ECS service
aws ecs update-service \
  --cluster ${CLUSTER_NAME} \
  --service ${APP_NAME}-service \
  --desired-count 0 \
  --region ${AWS_REGION}

aws ecs delete-service \
  --cluster ${CLUSTER_NAME} \
  --service ${APP_NAME}-service \
  --region ${AWS_REGION}

# Delete ECS cluster
aws ecs delete-cluster \
  --cluster ${CLUSTER_NAME} \
  --region ${AWS_REGION}

# Delete ECR repository
aws ecr delete-repository \
  --repository-name ${APP_NAME} \
  --force \
  --region ${AWS_REGION}

# Delete security group
aws ec2 delete-security-group \
  --group-id ${SG_ID} \
  --region ${AWS_REGION}

# Delete IAM roles
aws iam delete-role \
  --role-name ${APP_NAME}-ecs-task-role

aws iam delete-role \
  --role-name ${APP_NAME}-ecs-execution-role

# Delete SSM parameter
aws ssm delete-parameter \
  --name ecs-cwagent-${APP_NAME} \
  --region ${AWS_REGION}
