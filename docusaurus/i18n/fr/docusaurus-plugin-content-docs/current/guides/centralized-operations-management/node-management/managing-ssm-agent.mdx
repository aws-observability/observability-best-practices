# Gérer l'agent SSM au sein d'une AWS Organization

Cette section décrit comment gérer l'installation de l'agent SSM au sein d'une AWS Organization et fournit des exemples de code CloudFormation, Terraform, Shell, PowerShell, Ansible, Chef et Puppet.

Veuillez garder à l'esprit qu'il s'agit d'exemples de code qui doivent être soigneusement testés et validés dans un environnement de développement avant toute utilisation dans un environnement de production.

## Comprendre l'agent SSM

Comme discuté dans les sections précédentes, pour commencer à gérer vos noeuds dans votre environnement en utilisant Systems Manager, les noeuds doivent être [gérés](https://docs.aws.amazon.com/systems-manager/latest/userguide/operating-systems-and-machine-types.html#supported-machine-types), ce qui signifie que l'agent SSM est installé sur la machine et que l'agent peut communiquer avec les points de terminaison du service Systems Manager avec les permissions requises.

AWS Systems Manager Agent (SSM Agent) est un logiciel Amazon qui s'exécute sur les instances Amazon Elastic Compute Cloud (Amazon EC2), les appareils edge, les serveurs sur site et les machines virtuelles (VM). Le code source de l'agent SSM est disponible sur [GitHub](https://github.com/aws/amazon-ssm-agent). L'agent SSM permet à Systems Manager de mettre à jour, gérer et configurer ces ressources. L'agent traite les requêtes du service Systems Manager dans le AWS Cloud, puis les exécute comme spécifié dans la requête.

Par conséquent, s'assurer que l'agent SSM est installé sur les noeuds est nécessaire et constitue la toute première étape à aborder lors du début du parcours avec les opérations de gestion des noeuds utilisant le service AWS Systems Manager.

## Support des systèmes d'exploitation

L'agent SSM est pris en charge sur une large gamme de systèmes d'exploitation et de types de machines. Consultez [Systèmes d'exploitation et types de machines supportés](https://docs.aws.amazon.com/systems-manager/latest/userguide/operating-systems-and-machine-types.html) dans le guide utilisateur pour vérifier que votre système d'exploitation (OS), votre version d'OS et votre type de machine sont supportés.

## AMI avec l'agent SSM préinstallé

Pour commencer avec Systems Manager, AWS Systems Manager Agent (SSM Agent) est préinstallé sur certaines Amazon Machine Images (AMI) fournies par AWS et des tiers de confiance. Consultez [Trouver les AMI avec l'agent SSM préinstallé](https://docs.aws.amazon.com/systems-manager/latest/userguide/ami-preinstalled-agent.html), pour obtenir la liste des systèmes d'exploitation où il est préinstallé.

## Méthodes d'installation de l'agent SSM

Maintenant que nous comprenons quels systèmes d'exploitation supportent l'agent SSM, explorons les différentes méthodes pour installer l'agent sur les instances qui n'ont pas l'agent SSM préinstallé. L'approche d'installation peut être catégorisée en deux scénarios principaux :

* **Déploiement de nouvelles instances** : Installer l'agent SSM lors du lancement de nouvelles instances.
* **Gestion d'instances existantes** : Installer l'agent SSM sur des instances déjà en cours d'exécution.

### Installer l'agent SSM sur de nouvelles instances

Lors du lancement de nouvelles instances, l'incorporation de l'installation de l'agent SSM dans votre processus de configuration initiale assure des capacités de gestion cohérentes dès le départ. Cette approche élimine le besoin d'installation post-déploiement. Vous pouvez y parvenir par diverses méthodes telles que les Golden AMI, les scripts de données utilisateur ou les outils d'infrastructure as code, etc. Le choix de la méthode dépend souvent des pratiques de déploiement et des exigences de mise à l'échelle de votre organisation.

#### Approche Golden AMI

Une Golden AMI est une Amazon Machine Image personnalisée qui sert de modèle standardisé pour le lancement d'instances EC2. Pour l'intégration avec Systems Manager, les organisations peuvent créer des Golden AMI avec l'agent SSM préinstallé, assurant que toutes les instances lancées à partir de ces AMI sont immédiatement gérables via Systems Manager. Cette approche élimine le besoin de scripts d'installation post-lancement et réduit le risque d'échecs de déploiement.

Créer une Golden AMI

1. Manuellement : La création manuelle d'une Golden AMI peut se faire en lançant une instance, en installant et configurant l'agent SSM, en testant la configuration, puis en créant une AMI à partir de cette instance.
1. EC2 Image Builder : Si vous utilisez EC2 Image Builder pour créer des images golden, configurez la [recette d'image](https://docs.aws.amazon.com/imagebuilder/latest/userguide/start-build-image-pipeline.html#start-build-image-step2) et sélectionnez l'option pour garder l'agent SSM dans l'image de sortie.
1. Hashicorp Packer : Si vous utilisez Hashicorp Packer, utilisez le provisioner shell de Packer pour exécuter des commandes shell afin d'installer l'agent SSM.

```json
{
  "provisioners": [
    {
      "type": "shell",
      "inline": [
        "if [ -f /etc/system-release ]; then",
        "  # Amazon Linux",
        "  sudo yum install -y amazon-ssm-agent",
        "  sudo systemctl enable amazon-ssm-agent",
        "  sudo systemctl start amazon-ssm-agent",
        "elif [ -f /etc/lsb-release ]; then",
        "  # Ubuntu",
        "  sudo snap install amazon-ssm-agent —classic",
        "  sudo systemctl enable snap.amazon-ssm-agent.amazon-ssm-agent.service",
        "  sudo systemctl start snap.amazon-ssm-agent.amazon-ssm-agent.service",
        "fi"
      ]
    }
  ]
}
```

Alternativement, fournissez le chemin vers le script qui contient les commandes shell d'installation de l'agent.

```json
{
  "provisioners": [{
    "type": "shell",
    "script": "scripts/install_ssm.sh"
  }]
}
```

Lors de l'utilisation de Packer avec le provisioner script, le fichier `install_ssm.sh` doit être stocké relativement à votre fichier de configuration Packer (`.json` ou `.pkr` `.hcl`).

#### Group Policy (pour Windows)

Bien que Group Policy supporte traditionnellement les packages .msi, l'agent AWS Systems Manager (SSM) est livré sous forme de fichier .exe. Pour surmonter cette limitation, nous utiliserons une combinaison de Group Policy et de tâches planifiées pour déployer l'agent SSM dans votre environnement Windows.

1. Créez (ou modifiez un existant) GPO dans la hiérarchie/OU appropriée dans Active Directory. Modifiez la politique pour utiliser la configuration de script de tâche planifiée de l'ordinateur.

2. L'option se trouve sous Computer Configuration | Preferences | Control Panel Settings | Scheduled Tasks | Clic droit et sélectionnez New - Immediate Task

3. Sélectionnez **Create** pour l'action, donnez un nom à la tâche et spécifiez `NT AUTHORITY\System` comme compte utilisateur pour exécuter la tâche. Cliquez sur le bouton radio pour **Run whether user is logged on or not** et cliquez sur l'onglet **Triggers** pour passer à la section suivante.

![Capture d'écran du gestionnaire de tâches pour les propriétés de SSMAgent](/img/cloudops/guides/centralized-operations-management/node-management/task-manager-ssm-agent.png "Capture d'écran du gestionnaire de tâches pour les propriétés de SSMAgent")

4. Pour les déclencheurs, choisissez une fois.

5. Créez une nouvelle action pour **Start a program** et choisissez `powershell.exe` et pour l'**Argument** utilisez ce qui suit :

```powershell
-ExecutionPolicy Bypass -Command "$dir = $env:TEMP + '\ssm'; New-Item -ItemType directory -Path $dir -Force; Set-Location $dir; (New-Object System.Net.WebClient).DownloadFile('https://s3.amazonaws.com/ec2-downloads-windows/SSMAgent/latest/windows_amd64/AmazonSSMAgentSetup.exe', $dir + '\AmazonSSMAgentSetup.exe'); Start-Process .\AmazonSSMAgentSetup.exe -ArgumentList @('/q', '/log', 'install.log') -Wait"
```

![Capture d'écran du gestionnaire de tâches pour ajouter une nouvelle action](/img/cloudops/guides/centralized-operations-management/node-management/task-manager-new-action.png "Capture d'écran du gestionnaire de tâches pour ajouter une nouvelle action")

6. Cliquez sur **OK** dans la fenêtre New Action et cliquez sur **OK** dans la fenêtre Properties pour fermer ces dialogues.

7. Une fois cela terminé, tout ordinateur lié devrait recevoir l'installateur lorsque la politique GP se met à jour ou lors de l'utilisation de `gpudate /force` depuis l'invite de commande pour forcer cette action.

Pour plus d'informations, consultez [Automatiser l'enregistrement des noeuds gérés Windows dans AWS Systems Manager](https://github.com/aws-samples/sample-ssm-automate-windows-hybrid-install).

#### SSH (pour Linux)

Pour les instances Linux accessibles via SSH, vous pouvez automatiser l'installation de l'agent SSM en utilisant le script exemple suivant. Ce script détecte le type d'OS et installe le package approprié.

```bash
#!/bin/bash
USERNAME=ec2-user
HOSTS=("192.168.1.121" "192.168.1.122" "192.168.1.123")
KEY_PATH="/path/to/your/key.pem"

for HOSTNAME in ${HOSTS}; do
    # Detect OS type
    OS_TYPE=$(ssh -i "${KEY_PATH}" -l ${USERNAME} ${HOSTNAME} "cat /etc/os-release | grep '^ID=' | cut -d= -f2 | tr -d '\"'")

    case ${OS_TYPE} in
        "amzn"|"rhel"|"centos")
            SCRIPT="sudo yum install -y s3.amazonaws.com/ec2-downloads-windows/SSMAgent/latest/linux_amd64/amazon-ssm-agent.rpm && sudo systemctl start amazon-ssm-agent"
            ;;
        "ubuntu"|"debian")
            SCRIPT="wget s3.amazonaws.com/ec2-downloads-windows/SSMAgent/latest/debian_amd64/amazon-ssm-agent.deb && sudo dpkg -i amazon-ssm-agent.deb && sudo systemctl start amazon-ssm-agent"
            ;;
        *)
            echo "Unsupported OS on ${HOSTNAME}"
            continue
            ;;
    esac

    ssh -i "${KEY_PATH}" -l ${USERNAME} ${HOSTNAME} "${SCRIPT}"
done
```

## Gérer les permissions IAM pour l'agent SSM

Des permissions IAM appropriées sont cruciales pour le fonctionnement de l'agent SSM. AWS fournit plusieurs approches pour gérer ces permissions, de l'attachement manuel de profils d'instance aux solutions automatisées à l'échelle de l'organisation.

### Default Host Management Configuration (DHMC)

DHMC est l'approche recommandée par AWS pour gérer les permissions de l'agent SSM à travers votre organisation. Elle permet de gérer les instances EC2 sans avoir à créer manuellement un profil d'instance AWS Identity and Access Management (IAM). Au lieu de cela, Default Host Management Configuration crée et applique un rôle IAM par défaut pour s'assurer que Systems Manager a les permissions de gérer toutes les instances dans le compte AWS et la région AWS où elle est activée.

#### Méthodes pour activer DHMC :
* **Via l'expérience de console unifiée** : Lors de l'[intégration à l'expérience de console unifiée](https://docs.aws.amazon.com/systems-manager/latest/userguide/systems-manager-setting-up-organizations.html), DHMC est automatiquement activé. Ceci est activé en utilisant une association State Manager créée pendant l'intégration et activera DHMC dans chaque compte et région cible.
* **Via Fleet Manager** : Vous pouvez activer la Default Host Management Configuration depuis la console Fleet Manager. Vous devez activer la Default Host Management Configuration une par une dans chaque région où vous souhaitez que vos instances Amazon EC2 soient gérées. Reportez-vous aux étapes [ici](https://docs.aws.amazon.com/systems-manager/latest/userguide/setup-instance-permissions.html#default-host-management).
* **Via CLI/SDK** : Cela se fait en créant le rôle requis et en mettant à jour le paramètre de service en utilisant l'AWS CLI/SDK. Vous devez activer la Default Host Management Configuration une par une dans chaque région où vous souhaitez que vos instances Amazon EC2 soient gérées. Reportez-vous aux étapes [ici](https://docs.aws.amazon.com/systems-manager/latest/userguide/fleet-manager-default-host-management-configuration.html#dhmc-activate).
* **Via Quick Setup** : Avec Quick Setup, vous pouvez activer la Default Host Management Configuration pour tous les comptes et régions qui ont été ajoutés à votre organisation dans AWS Organizations. Reportez-vous aux étapes [ici](https://docs.aws.amazon.com/systems-manager/latest/userguide/fleet-manager-default-host-management-configuration.html#dhmc-activate).
* **Via CloudFormation** : Vous pouvez également utiliser CloudFormation pour activer DHMC dans plusieurs comptes et régions en utilisant StackSets. Reportez-vous à ce [blog](https://aws.amazon.com/blogs/mt/enable-management-of-your-amazon-ec2-instances-in-aws-systems-manager-using-default-host-management-configuration/).

### Permissions d'instance individuelles

Si vous n'utilisez pas DHMC, les permissions pour l'agent SSM doivent être accordées au niveau de chaque instance en utilisant les profils d'instance IAM. Nous recommandons de créer le profil d'instance avec la politique gérée Amazon recommandée pour l'agent SSM, AmazonSSMManagedInstanceCore. Des permissions supplémentaires à ce profil d'instance seront requises selon le cas d'utilisation. Consultez [Considérations de politiques supplémentaires pour les instances gérées](https://docs.aws.amazon.com/systems-manager/latest/userguide/setup-instance-permissions.html#instance-profile-policies-overview) dans le guide utilisateur qui traite de la création d'un profil d'instance IAM pour l'utilisation de l'agent SSM et des considérations de politiques supplémentaires.

Lors du déploiement de groupes Auto Scaling, la propriété de ressource Launch Template peut référencer le profil d'instance.

```yaml
# ASG with SSM Instance Profile
Resources:
  AutoScalingGroup:
    Type: AWS::AutoScaling::AutoScalingGroup
    Properties:
      LaunchTemplate:
        LaunchTemplateId: !Ref LaunchTemplate
        Version: !GetAtt LaunchTemplate.LatestVersionNumber

  LaunchTemplate:
    Type: AWS::EC2::LaunchTemplate
    Properties:
      LaunchTemplateData:
        IamInstanceProfile:
          Name: !Ref SSMInstanceProfile
```

Si vous utilisez Terraform pour lancer des instances, vous pouvez créer une ressource de profil d'instance IAM et la référencer dans votre ressource d'instance.

```json
# IAM Role and Instance Profile
resource "aws_iam_role" "ssm_role" {
  name = "SSMManagedInstanceRole"

  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Action = "sts:AssumeRole"
        Effect = "Allow"
        Principal = {
          Service = "ec2.amazonaws.com"
        }
      }
    ]
  })
}

resource "aws_iam_role_policy_attachment" "ssm_policy" {
  role       = aws_iam_role.ssm_role.name
  policy_arn = "arn:aws:iam::aws:policy/AmazonSSMManagedInstanceCore"
}

resource "aws_iam_instance_profile" "ssm_profile" {
  name = "SSMManagedInstanceProfile"
  role = aws_iam_role.ssm_role.name
}

# EC2 Instance with Profile
resource "aws_instance" "example" {
  # ... other configuration ...
  iam_instance_profile = aws_iam_instance_profile.ssm_profile.name
}
```

#### Lancer des instances en utilisant les runbooks Systems Manager Automation

[AWS-CreateManagedLinuxInstance](https://docs.aws.amazon.com/systems-manager-automation-runbooks/latest/userguide/automation-aws-createmanagedlinuxinstance.html) et [AWS-CreateManagedWindowsInstance](https://docs.aws.amazon.com/systems-manager-automation-runbooks/latest/userguide/automation-aws-createmanagedwindowsinstance.html) sont des runbooks Systems Manager Automation préconfigurés qui automatisent le processus de création et de configuration d'instances gérées. Ces runbooks assurent que le profil d'instance a les permissions SSM requises attachées aux instances lancées.

### Attacher des profils d'instance à des instances EC2 déjà en cours d'exécution

* Via Quick Setup (recommandé) : En utilisant [la gestion d'hôtes Amazon EC2 via Quick Setup](https://docs.aws.amazon.com/systems-manager/latest/userguide/quick-setup-host-management.html), vous pouvez configurer l'ajout des politiques IAM requises aux profils d'instance existants attachés à vos instances, ou permettre à Quick Setup de créer les politiques IAM et les profils d'instance avec les permissions nécessaires pour la configuration que vous choisissez. Cette option peut cibler des instances dans plusieurs comptes dans l'organisation ou des OUs spécifiques.

* Via Systems Manager Automation : Vous pouvez utiliser le runbook d'automatisation [AWS-SetupManagedRoleOnEc2Instance](https://docs.aws.amazon.com/systems-manager-automation-runbooks/latest/userguide/automation-aws-setupmanagedroleonec2instance.html) pour configurer une instance avec le rôle IAM géré SSMRoleForManagedInstance pour l'accès Systems Manager. Si le rôle spécifié n'existe pas, il sera créé par l'automatisation. Pour cibler des instances dans plusieurs comptes et régions, exécutez l'automatisation en mode [multi-comptes/multi-régions](https://docs.aws.amazon.com/systems-manager/latest/userguide/running-automations-multiple-accounts-regions.html).
