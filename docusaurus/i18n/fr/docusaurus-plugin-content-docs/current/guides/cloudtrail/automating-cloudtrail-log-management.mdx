# Automatisation de la gestion des journaux CloudTrail avec Amazon Athena

---

AWS CloudTrail fournit une surveillance et une journalisation complètes de l'activité des comptes à travers votre infrastructure AWS, capturant les appels API et les événements pour l'audit, la conformité et les investigations de sécurité. Cependant, à mesure que les environnements AWS s'étendent, en particulier dans les organisations multi-comptes, le volume des journaux CloudTrail peut croître considérablement, entraînant des temps de requête et des coûts accrus lors de l'analyse des journaux dans Amazon Athena. Pour y remédier, nous recommandons d'automatiser la création et la gestion de tables Athena partitionnées pour optimiser les performances des requêtes et réduire les coûts. Ce guide de bonnes pratiques décrit comment implémenter une solution automatisée utilisant AWS Lambda et Amazon Athena pour partitionner les journaux CloudTrail par compte, région et date, permettant une analyse efficace des journaux pour les cas d'utilisation de sécurité, de conformité et opérationnels.

## Configurer des tables Athena partitionnées pour les journaux CloudTrail

Pour optimiser les performances des requêtes et réduire les coûts, configurez des tables Athena partitionnées pour les journaux CloudTrail en utilisant une solution automatisée. Cette approche implique le déploiement de deux fonctions AWS Lambda pour créer et maintenir des tables Athena spécifiques aux comptes et à l'ensemble de l'organisation, partitionnées par identifiant de compte, région et date. Le partitionnement garantit qu'Athena ne parcourt que les données pertinentes lors des requêtes, améliorant significativement la vitesse des requêtes et réduisant les coûts par rapport au parcours d'ensembles de données non partitionnés. Par exemple, lors de l'investigation d'une activité suspecte pour un compte AWS spécifique dans un intervalle de temps défini, Athena ne parcourra que les partitions pertinentes, plutôt que l'ensemble des données de journaux.

- **[Déployer ce template CloudFormation pour l'automatisation](https://github.com/aws-samples/sample-automation-cloudtrail-athena)** : Utilisez ce template CloudFormation pour déployer la solution, qui inclut deux fonctions Lambda, des rôles IAM, une base de données AWS Glue et des requêtes Athena prédéfinies. Le template automatise la configuration des tables partitionnées, éliminant la configuration manuelle et assurant la cohérence entre les environnements. La solution GitHub est disponible [ici](https://github.com/aws-samples/sample-automation-cloudtrail-athena).
- **[Journaux CloudTrail dans Athena utilisant la projection de partition](https://docs.aws.amazon.com/athena/latest/ug/create-cloudtrail-table-partition-projection.html)** : Cette solution utilise la projection de partitionnement d'Athena qui, étant donné que les journaux CloudTrail ont une structure connue avec des schémas de partition prédéfinis, vous permet de réduire le temps d'exécution des requêtes et d'automatiser la gestion des partitions en utilisant la fonctionnalité de projection de partition d'Athena. La projection de partition ajoute automatiquement de nouvelles partitions à mesure que de nouvelles données sont ajoutées. Cela élimine le besoin d'ajouter manuellement des partitions en utilisant `ALTER TABLE ADD PARTITION`.
- **[Spécifier le bucket S3 et le préfixe CloudTrail](https://docs.aws.amazon.com/firehose/latest/dev/dynamic-partitioning-s3bucketprefix.html)** : Fournissez le nom du bucket S3 CloudTrail et le préfixe approprié comme entrées du template CloudFormation. Pour les trails organisationnels, le préfixe suit le format `ORG_ID/AWSLogs/ORG_ID`, où `ORG_ID` est l'identifiant de l'organisation AWS. Pour les trails mono-compte, utilisez le préfixe par défaut `AWSLogs/`. Vérifiez le préfixe en utilisant la commande AWS CLI : `aws cloudtrail describe-trails --trail-name-list TRAIL_NAME`. Si un préfixe personnalisé est utilisé, ajoutez-le au préfixe par défaut (par ex., `PREFIX/ORG_ID/AWSLogs/ORG_ID`).
- **[Configurer l'emplacement des résultats de requête Athena](https://docs.aws.amazon.com/athena/latest/ug/query-results-specify-location.html)** : Assurez-vous qu'un bucket S3 est configuré dans Athena pour les résultats de requête. Spécifiez ce bucket (sans le préfixe `s3://`) dans le template CloudFormation pour stocker les sorties de requête.

## Utiliser des tables spécifiques aux comptes et consolidées pour une analyse flexible

La solution crée deux types de tables Athena pour prendre en charge différents cas d'utilisation :

- **Tables spécifiques aux comptes** : La fonction Lambda `CloudTrailLogsPartitionedByAccount` crée des tables Athena dédiées (par ex., `trail_123456789012`) pour chaque compte AWS découvert dans le bucket S3 CloudTrail. Ces tables sont partitionnées par région et date, permettant aux équipes d'analyser les journaux spécifiques à leur compte sans parcourir de données non pertinentes. Cela est particulièrement utile pour les équipes de sécurité enquêtant sur des incidents spécifiques à un compte ou les équipes opérationnelles dépannant des activités régionales.
- **Table consolidée à l'échelle de l'organisation** : La fonction Lambda `CloudTrailLogsPartitionedAllAccounts` maintient une table unifiée (par ex., `all_accounts_trail`) contenant les journaux de tous les comptes de l'organisation AWS, partitionnée par identifiant de compte, région et date. Cette table prend en charge les investigations inter-comptes et les audits à l'échelle de l'organisation, permettant aux administrateurs de sécurité d'interroger l'activité dans l'ensemble de l'environnement de manière efficace.

Les deux fonctions utilisent la projection de partition pour inclure automatiquement les nouveaux comptes et partitions à mesure que l'organisation évolue, éliminant le besoin de mises à jour manuelles. Configurez ces tables pour livrer les journaux dans un bucket S3 centralisé dans une frontière de sécurité séparée avec un accès restreint pour appliquer des contrôles de sécurité stricts et la séparation des responsabilités.

## Activer les mises à jour quotidiennes des tables pour la synchronisation

Pour garantir que les tables Athena restent synchronisées avec la structure de l'organisation AWS, configurez les fonctions Lambda pour exécuter des mises à jour quotidiennes. Ces mises à jour détectent les nouveaux comptes et créent les partitions correspondantes, assurant que les journaux des comptes ou régions nouvellement ajoutés sont inclus dans les tables Athena. Cette automatisation réduit la charge administrative et assure une couverture complète des journaux, répondant aux exigences de conformité telles que FedRAMP ou PCI-DSS qui imposent une surveillance continue de l'activité des comptes.

## Optimiser les coûts avec le partitionnement et la surveillance

Le partitionnement des journaux CloudTrail réduit la quantité de données parcourues par Athena, diminuant directement les coûts de requête. De plus, implémentez les pratiques de gestion des coûts suivantes :

- **Surveiller les coûts des fonctions Lambda** : La solution utilise deux fonctions Lambda, chacune invoquée une fois par jour. Surveillez leur temps d'exécution pour estimer les coûts en utilisant la page de tarification AWS Lambda. Ajustez la fréquence d'invocation si nécessaire pour équilibrer coût et besoins de mise à jour.
- **Suivre les coûts de requête Athena** : Athena facture en fonction de la quantité de données parcourues par requête. Le partitionnement minimise les données parcourues, mais surveillez les coûts de requête en utilisant la page de tarification Amazon Athena. Utilisez AWS Budgets pour définir des seuils de coûts et des alertes pour les dépenses CloudTrail et Athena.
- **Utiliser AWS Cost Anomaly Detection** : Configurez AWS Cost Anomaly Detection pour surveiller les dépenses CloudTrail et Athena. Ce service utilise l'apprentissage automatique pour détecter les pics de coûts inattendus, permettant une gestion proactive des coûts.

## Nettoyer les ressources pour éviter les coûts inutiles

Pour éviter les frais continus, supprimez la pile CloudFormation lorsque la solution n'est plus nécessaire. Cela supprime les fonctions Lambda, les rôles IAM, la base de données Glue et les tables Athena, garantissant qu'aucun coût résiduel n'est engendré. Examinez régulièrement les rapports AWS Budgets et Cost Anomaly Detection pour identifier et traiter toute dépense inattendue liée à la solution.

## Conclusion

L'automatisation des tables Athena partitionnées pour les journaux CloudTrail optimise les performances des requêtes, réduit les coûts et améliore les investigations de sécurité et de conformité. En déployant des tables spécifiques aux comptes et à l'échelle de l'organisation, en activant les mises à jour quotidiennes et en sécurisant le bucket S3, cette solution fournit une approche évolutive et efficace pour l'analyse des journaux CloudTrail. L'intégration avec CloudWatch Logs et l'utilisation de requêtes prédéfinies rationalisent davantage les investigations, tandis que les pratiques de gestion des coûts assurent l'accessibilité financière. Implémentez cette solution pour obtenir une analyse plus rapide, économique et sécurisée des journaux CloudTrail, soutenant la gouvernance, la conformité et l'excellence opérationnelle dans votre environnement AWS.
