# Activer JITNA avec l'Infrastructure-as-Code (IaC)

La section suivante decrit comment deployer l'acces juste-a-temps aux noeuds en utilisant CloudFormation dans une organisation AWS. Pour deployer l'acces juste-a-temps aux noeuds, vous devez d'abord activer l'acces juste-a-temps aux noeuds pour votre environnement, puis vous pouvez deployer les ressources d'acces juste-a-temps aux noeuds qui incluent les politiques d'approbation, les preferences de session et les configurations de notification.

:::info
La console unifiee Systems Manager et l'acces juste-a-temps aux noeuds peuvent egalement etre actives par compte/region en effectuant des deploiements locaux plutot qu'en activant les fonctionnalites dans le compte administrateur delegue.
:::

## Activer l'acces juste-a-temps aux noeuds

1. Dans le compte de gestion AWS de votre organisation, specifiez un compte administrateur delegue pour Systems Manager.
1. Deployez la [console unifiee Systems Manager](https://docs.aws.amazon.com/systems-manager/latest/userguide/systems-manager-setting-up-organizations.html).
1. Deployez l'[acces juste-a-temps aux noeuds](https://docs.aws.amazon.com/systems-manager/latest/userguide/systems-manager-just-in-time-node-access.html).

Voici un exemple de diagramme d'architecture pour le deploiement des gestionnaires de configuration Quick Setup pour la console unifiee et l'acces juste-a-temps aux noeuds.

![Exemple de diagramme d'architecture pour le deploiement des gestionnaires de configuration Quick Setup pour la console unifiee et l'acces juste-a-temps aux noeuds](/img/cloudops/recipes/centralized-operations-management/just-in-time-node-access/jitna-organization.png "Diagramme d'architecture de l'acces juste-a-temps aux noeuds")

1. Creez une pile CloudFormation dans le compte administrateur delegue pour Systems Manager.
1. La pile CloudFormation cree des gestionnaires de configuration Quick Setup.
1. Le gestionnaire de configuration Quick Setup pour la console unifiee est deploye dans toutes les OU de l'organisation, A, B et C.
1. Le gestionnaire de configuration Quick Setup pour JITNA est deploye dans les OU A et B.

:::info
L'acces juste-a-temps aux noeuds peut etre active soit a l'echelle de votre organisation AWS, soit pour des unites organisationnelles (OU) specifiques au sein de l'organisation. Les regions deployables pour l'acces juste-a-temps aux noeuds incluent toutes les regions que vous avez activees pour la console unifiee Systems Manager. Vous ne pouvez pas activer JITNA pour les regions ou la console unifiee n'est pas activee.
:::

### Deployer les gestionnaires de configuration Quick Setup

Pour activer l'acces juste-a-temps aux noeuds en utilisant l'infrastructure-as-code (IaC), vous devez :

1. Deployer les roles de service IAM Quick Setup.
    :::info
    Pour plus d'informations sur les roles de service IAM pour Quick Setup, consultez [Manual onboarding for working with Quick Setup API programatically](https://docs.aws.amazon.com/systems-manager/latest/userguide/quick-setup-getting-started.html#quick-setup-api-manual-onboarding).
    :::
1. Deployer deux [gestionnaires de configuration Quick Setup](https://docs.aws.amazon.com/AWSCloudFormation/latest/UserGuide/aws-resource-ssmquicksetup-configurationmanager.html) :
    1. Console unifiee Systems Manager.
    1. Acces juste-a-temps aux noeuds Systems Manager.

Le modele CloudFormation suivant inclut la creation des deux roles de service IAM Quick Setup requis et deploie les deux gestionnaires de configuration Quick Setup pour la console unifiee Systems Manager et JITNA.

## Configurer l'acces juste-a-temps aux noeuds

Apres avoir active l'acces juste-a-temps aux noeuds pour votre environnement, vous pouvez alors commencer a deployer les politiques d'approbation, les preferences de session et les configurations de notification.

* [Politiques d'approbation pour vos noeuds](https://docs.aws.amazon.com/systems-manager/latest/userguide/systems-manager-just-in-time-node-access-approval-policies.html).
* [Preferences de session](https://docs.aws.amazon.com/systems-manager/latest/userguide/systems-manager-just-in-time-node-access-session-preferences.html) pour les sessions juste-a-temps.
* [Configurations de notification](https://docs.aws.amazon.com/systems-manager/latest/userguide/systems-manager-just-in-time-node-access-notifications.html) pour les demandes d'acces juste-a-temps.

### Deployer les politiques d'approbation pour vos noeuds

Voici un exemple de diagramme d'architecture pour le deploiement des politiques d'approbation et des preferences de session de l'acces juste-a-temps aux noeuds.

![Exemple de diagramme d'architecture pour le deploiement des politiques d'approbation et des preferences de session de l'acces juste-a-temps aux noeuds](/img/cloudops/recipes/centralized-operations-management/just-in-time-node-access/jitna-resources.png "Diagramme d'architecture des politiques d'approbation et preferences de l'acces juste-a-temps aux noeuds")

1. Creez un StackSet CloudFormation pour deployer une politique d'approbation automatique, une ou plusieurs politiques d'approbation manuelle et des preferences de session sous forme de documents Systems Manager.
1. Dans chaque compte et region cible, les piles CloudFormation deploient les ressources requises.
1. Creez une pile CloudFormation dans le compte administrateur delegue pour Systems Manager pour deployer la politique d'acces en refus uniquement.

:::info
Vous ne pouvez avoir qu'une seule politique d'acces en refus par organisation. Vous ne pouvez avoir qu'une seule politique d'approbation automatique par compte et region.

Vous pouvez creer jusqu'a 50 politiques d'approbation manuelle par compte AWS et region AWS. Cependant, lorsque plusieurs politiques d'approbation manuelle s'appliquent a un noeud, cela cree un conflit et les utilisateurs ne peuvent pas demander l'acces au noeud. Gardez cela a l'esprit lors de la creation de vos politiques d'approbation manuelle car certaines instances peuvent avoir plusieurs tags selon votre cas.
:::

Le modele CloudFormation suivant cree :

1. Une politique d'approbation automatique pour permettre l'acces automatique aux noeuds geres de developpement identifies par la paire cle-valeur de tag de ressource `ENV:DEV`.
1. Deux politiques d'approbation manuelle :
    1. Une politique d'approbation manuelle pour l'application E-commerce qui s'applique aux noeuds geres avec la paire cle-valeur de tag de ressource `Workload:Ecommerce`.
    1. Une politique d'approbation manuelle pour l'application Finance qui s'applique aux noeuds geres avec la paire cle-valeur de tag de ressource `Workload:Finance`.
1. Un document de preferences de session a utiliser lors des sessions d'acces juste-a-temps aux noeuds.
