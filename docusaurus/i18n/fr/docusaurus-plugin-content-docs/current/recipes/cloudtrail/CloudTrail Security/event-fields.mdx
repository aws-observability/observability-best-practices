# Comprendre l'importance securitaire des champs d'evenements CloudTrail

Les champs d'evenements AWS CloudTrail fournissent des details granulaires sur les activites API, les identites des utilisateurs et les interactions avec les ressources, permettant un suivi et une analyse precis lors des investigations de securite. Cette section presente les champs d'evenements CloudTrail critiques, leur importance pour la securite et les cas d'utilisation recommandes pour l'analyse forensique et la reponse aux incidents.

## Champs d'evenements CloudTrail cles pour les investigations de securite

Voici une ventilation detaillee des champs d'evenements CloudTrail les plus critiques, leur importance dans les investigations de securite et comment les exploiter efficacement lors de la reponse aux incidents.

:::tip Decouvrez le contenu des enregistrements CloudTrail
Pour explorer tous les details du **contenu des enregistrements AWS CloudTrail**, y compris les champs cles comme `eventName`, `eventSource` et `requestParameters`, **[consultez notre documentation complete](https://docs.aws.amazon.com/awscloudtrail/latest/userguide/cloudtrail-event-reference-record-contents.html)**. Ameliorez votre analyse forensique de securite et votre reponse aux incidents avec des informations completes sur les journaux.
:::

### Champs d'evenements critiques

- **1. userIdentity**
  - **Description** : Contient les details sur l'identite IAM effectuant la requete, y compris `type` (par ex., IAMUser, AssumedRole, Root), `principalId`, `arn`, `accountId`, `userName` et `sessionContext`.
  - **Importance securitaire** : Identifie l'acteur derriere un incident, permettant de determiner si un utilisateur, un role ou un compte root compromis etait implique. Prend egalement en charge le tracage des acces inter-comptes.
  - **Cas d'utilisation** : Utilisez `userIdentity` pour determiner si un acteur malveillant a utilise des identifiants IAM voles ou assume un role pour escalader les privileges. Si `type` est `Root`, escaladez immediatement en raison du risque eleve de compromission du compte root.

- **2. userIdentity.accessKeyId**
  - **Description** : Identifie la cle d'acces utilisee dans la requete.
  - **Importance securitaire** : Suit les identifiants compromis a travers plusieurs evenements, car les acteurs malveillants reutilisent souvent des cles d'acces volees.
  - **Cas d'utilisation** : Correlez les activites suspectes en utilisant `accessKeyId` pour determiner l'etendue d'une violation. Si vide (par ex., pour les actions de console), croisez avec `userIdentity.sessionContext` pour confirmer le type de session.

- **3. userIdentity.userName**
  - **Description** : Specifie le nom de l'utilisateur IAM.
  - **Importance securitaire** : Suit l'activite specifique a un utilisateur, en particulier avec plusieurs cles d'acces.
  - **Cas d'utilisation** : Croisez `userName` avec `userIdentity.principalId` pour un suivi d'identite coherent.

- **4. userIdentity.principalId**
  - **Description** : Fournit un identifiant unique pour l'entite effectuant la requete.
  - **Importance securitaire** : Correle les actions entre les sessions, en particulier pour les identifiants temporaires.
  - **Cas d'utilisation** : Suivez le mouvement de l'acteur malveillant a travers les roles assumes ou les sessions en utilisant `principalId`.

- **5. userIdentity.sessionContext**
  - **Description** : Detaille les roles assumes ou les sessions federees, y compris `sessionIssuer` et [`sourceIdentity`](https://docs.aws.amazon.com/awscloudtrail/latest/userguide/cloudtrail-event-reference-user-identity.html#STS-API-source-identity).
  - **Importance securitaire** : Critique pour comprendre les acces inter-comptes ou les attaques par chaine de roles. Si la requete a ete effectuee avec des identifiants de securite temporaires, sessionContext fournit des informations sur la session creee pour ces identifiants. Vous creez une session lorsque vous appelez toute API qui retourne des identifiants temporaires. Les utilisateurs creent egalement des sessions lorsqu'ils travaillent dans la console et effectuent des requetes avec des API qui incluent l'authentification multi-facteurs.
  - **Cas d'utilisation** : Analysez `sessionContext.sourceIdentity` pour tracer l'origine de l'attaque dans les scenarios d'assumption de role.

- **6. userIdentity.type**
  - **Description** : Specifie le type d'identite (par ex., `IAMUser`, `Root`, `AssumedRole`).
  - **Importance securitaire** : Identifie les actions a haut risque, telles que l'utilisation de `Root`, necessitant une escalade immediate.
  - **Cas d'utilisation** : Filtrez par `userIdentity.type` pour prioriser les investigations impliquant des identites privilegiees.

- **7. userIdentity.arn**
  - **Description** : Fournit le nom de ressource Amazon (ARN) du principal.
  - **Importance securitaire** : Assure une identification precise des acteurs dans des environnements complexes.
  - **Cas d'utilisation** : Tracez les actions vers des entites IAM specifiques en utilisant `arn`.

- **8. userIdentity.accountId**
  - **Description** : Identifie le compte AWS proprietaire de l'entite.
  - **Importance securitaire** : Isole le compte source dans les scenarios inter-comptes.
  - **Cas d'utilisation** : Croisez avec `recipientAccountId` pour detecter les acces non autorises.

- **9. userIdentity.sessionIssuer**
  - **Description** : Identifie l'entite emettant les identifiants de session (par ex., un role).
  - **Importance securitaire** : Suit la source des identifiants temporaires dans les attaques basees sur les roles. Si un utilisateur effectue une requete avec des identifiants de securite temporaires, sessionIssuer fournit des informations sur la facon dont l'utilisateur a obtenu les identifiants. Par exemple, s'il a obtenu des identifiants de securite temporaires en assumant un role, cet element fournit des informations sur le role assume. S'il a obtenu des identifiants avec les identifiants root ou d'un utilisateur IAM pour appeler AWS STS GetFederationToken, l'element fournit des informations sur le compte root ou l'utilisateur IAM.
  - **Cas d'utilisation** : Analysez `sessionIssuer` dans les evenements `AssumedRole` pour l'analyse d'escalade de privileges.

- **10. userIdentity.invokedBy**
  - **Description** : Indique si un service AWS (par ex., Lambda) a effectue la requete.
  - **Importance securitaire** : Distingue les actions initiees par un service de l'activite utilisateur, identifiant les mauvaises configurations ou les roles exploites.
  - **Cas d'utilisation** : Investiguez `invokedBy` pour les anomalies dans le comportement des services.

- **11. eventTime**
  - **Description** : Enregistre l'horodatage UTC de l'activite.
  - **Importance securitaire** : Etablit la chronologie d'une attaque, identifiant le point initial de compromission et correlant les evenements entre les systemes.
  - **Cas d'utilisation** : Sequencez les actions de l'acteur malveillant et determinez la duree de l'acces non autorise. Un pic d'appels API a des heures inhabituelles peut indiquer une activite malveillante.

- **12. eventName**
  - **Description** : Specifie l'action API AWS (par ex., `GetCallerIdentity`, `CreateAccessKey`, `DeleteBucket`).
  - **Importance securitaire** : Revele les techniques de l'acteur malveillant, telles que la reconnaissance (`ListUsers`) ou l'escalade de privileges (`CreateAccessKey`).
  - **Cas d'utilisation** : Analysez `eventName` pour identifier les actions a haut risque comme la modification des politiques IAM ou la desactivation de la journalisation, indiquant une intention de persistance ou d'evasion de la detection.

- **13. eventSource**
  - **Description** : Identifie le service AWS cible (par ex., `iam.amazonaws.com`, `s3.amazonaws.com`).
  - **Importance securitaire** : Determine les services affectes, aidant a prioriser les efforts de remediation.
  - **Cas d'utilisation** : Investiguez `eventSource` pour `sts.amazonaws.com` avec `eventName` `AssumeRole` pour detecter les attaques par chaine de roles. Filtrez les journaux pour vous concentrer sur les services compromis.

- **14. sourceIPAddress**
  - **Description** : Indique l'adresse IP d'ou la requete a ete emise.
  - **Importance securitaire** : Identifie l'origine geographique d'une attaque et les acteurs de menace potentiels.
  - **Cas d'utilisation** : Croisez `sourceIPAddress` avec des flux de renseignements sur les menaces pour detecter des origines suspectes. Si elle indique "AWS Internal/#", investiguez les roles de service ou les mauvaises configurations.

- **15. userAgent**
  - **Description** : Detaille l'application ou l'outil utilise (par ex., `aws-cli`, `aws-sdk-java`).
  - **Importance securitaire** : Identifie les outils suspects ou les applications non standard utilisees par les acteurs malveillants.
  - **Cas d'utilisation** : Surveillez `userAgent` pour les anomalies, telles que les SDK non-AWS, qui peuvent indiquer un outillage malveillant deviant des modeles normaux de l'organisation.

- **16. errorCode et errorMessage**
  - **Description** : Indique si un appel API a echoue et pourquoi (par ex., `AccessDenied`).
  - **Importance securitaire** : Revele les tentatives d'attaque echouees et les limites de permissions, fournissant un apercu de la portee de l'acteur malveillant.
  - **Cas d'utilisation** : Plusieurs erreurs `AccessDenied` pour `CreateAccessKey` peuvent indiquer un test de permissions. Utilisez ces champs pour affiner les controles de securite.

- **17. requestParameters**
  - **Description** : Contient les parametres envoyes avec l'appel API, tels que les noms de ressources ou les configurations. Les elements de reponse, le cas echeant, pour les actions qui effectuent des modifications (actions de creation, mise a jour ou suppression). Pour les API en lecture seule, ce champ est null. Si l'action ne retourne pas d'elements de reponse, ce champ est null. Les elements de reponse pour les actions sont documentes dans la documentation de reference API du service AWS appropriate.
  - **Importance securitaire** : Revele les entrees specifiques d'un appel API et les details de configuration envoyes par les actions d'un acteur malveillant.
  - **Cas d'utilisation** : Analysez `requestParameters` dans un appel `CreateBucket` pour identifier le nom du compartiment pour la remediation.

- **18. responseElements**
  - **Description** : Inclut la sortie de l'appel API, telle que les cles d'acces nouvellement creees ou les ARN de ressources. Les elements de reponse, le cas echeant, pour les actions qui effectuent des modifications (actions de creation, mise a jour ou suppression). Pour les API en lecture seule, ce champ est `null`. Si l'action ne retourne pas d'elements de reponse, ce champ est `null`. Les elements de reponse pour les actions sont documentes dans la documentation de reference API du service AWS appropriate.
  - **Importance securitaire** : Contient des informations critiques comme les ressources nouvellement creees, les cles d'acces et d'autres artefacts.
  - **Cas d'utilisation** : Utilisez `responseElements` dans un evenement `CreateAccessKey` pour desactiver les nouvelles cles et inverser les modifications non autorisees.

- **19. resources**
  - **Description** : Liste les ressources AWS accedees ou modifiees, y compris les ARN et les types de ressources.
  - **Importance securitaire** : Identifie les actifs affectes, tels que les compartiments S3 ou les roles IAM, pour la remediation.
  - **Cas d'utilisation** : Utilisez `resources` dans un evenement `DeleteObject` pour identifier l'objet S3 affecte et prioriser la recuperation.

- **20. eventType**
  - **Description** : Categorise l'evenement (par ex., `AwsApiCall`, `AwsConsoleSignIn`).
  - **Importance securitaire** : Distingue les attaques basees sur les API, les connexions a la console ou les actions initiees par les services.
  - **Cas d'utilisation** : Filtrez les evenements `AwsConsoleSignIn` pour detecter les acces non autorises a la console et verifier le MFA.

- **21. recipientAccountId**
  - **Description** : Identifie le compte AWS recevant la requete.
  - **Importance securitaire** : Suit l'activite inter-comptes dans les organisations multi-comptes.
  - **Cas d'utilisation** : Investiguez si `recipientAccountId` differe de `userIdentity.accountId` pour un acces inter-comptes non autorise.

- **22. awsRegion**
  - **Description** : Specifie la region AWS de la requete.
  - **Importance securitaire** : Etablit les modeles geographiques et la portee de l'incident.
  - **Cas d'utilisation** : Filtrez les journaux par `awsRegion` pour identifier les anomalies dans les regions inutilisees.

- **23. readOnly**
  - **Description** : Indique si l'appel API etait en lecture seule (`true`) ou en ecriture (`false`).
  - **Importance securitaire** : Distingue la reconnaissance des actions de modification.
  - **Cas d'utilisation** : Concentrez-vous sur les operations d'ecriture pour la falsification potentielle de donnees ou la persistance.

- **24. eventID**
  - **Description** : Identifie de maniere unique chaque evenement CloudTrail.
  - **Importance securitaire** : Permet de referencer des evenements specifiques et de les correler avec des systemes externes comme les outils SIEM.
  - **Cas d'utilisation** : Utilisez `eventID` comme cle primaire pour l'agregation de journaux et le suivi.

- **25. serviceEventDetails**
  - **Description** : Fournit un contexte specifique au service, tel que les evenements de federation IAM Identity Center.
  - **Importance securitaire** : Enrichit les investigations avec des details de service specialises.
  - **Cas d'utilisation** : Analysez `serviceEventDetails` pour les evenements IAM Identity Center afin d'auditer les actions des utilisateurs federes.

Obtenez des informations pour une analyse forensique de securite et une reponse aux incidents efficaces.

:::note Bonnes pratiques pour CloudTrail
Pour maximiser l'efficacite de CloudTrail pour les investigations de securite, activez CloudTrail pour toutes les regions et tous les comptes de votre organisation AWS. Assurez-vous que les journaux sont stockes en securite dans un compartiment S3 avec un acces restreint et activez la validation de l'integrite des fichiers journaux pour detecter les alterations. Analysez regulierement ces champs en utilisant des outils comme Amazon Athena ou CloudTrail Lake pour identifier les anomalies et repondre rapidement aux incidents.
:::

## Exemple de workflow d'analyse pour une cle d'acces AWS compromise

1. **Identifier l'acteur** : Utilisez les champs `userIdentity` (`type`, `userName`, `arn`, `accountId`, `principalId`) pour determiner l'identite impliquee. Escaladez si `Root` est detecte.
2. **Etablir la chronologie** : Utilisez `eventTime` pour sequencer les evenements et identifier la duree de l'attaque.
3. **Analyser les actions** : Examinez `eventName`, `eventSource` et `readOnly` pour comprendre les techniques de l'acteur malveillant et les services affectes.
4. **Suivre les ressources** : Utilisez `resources`, `requestParameters` et `responseElements` pour identifier les actifs cibles ou modifies pour la remediation.
5. **Investiguer les acces** : Croisez `sourceIPAddress`, `userAgent`, `recipientAccountId` et `sessionContext` pour detecter les acces non autorises ou les chaines de roles.
6. **Examiner les erreurs** : Analysez `errorCode` et `errorMessage` pour identifier les actions tentees bloquees par les politiques IAM.
7. **Correler les evenements** : Utilisez `eventID` et `serviceEventDetails` pour integrer avec les outils SIEM et effectuer une analyse forensique detaillee.
