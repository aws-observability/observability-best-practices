# 従来のモニタリングにおける問題と課題

## 最新アプリケーションにおけるオブザーバビリティのギャップ

従来の監視アプローチは、よりシンプルなモノリシックアプリケーションを対象として設計されていました。組織がマイクロサービス、サーバーレス、クラウドネイティブアーキテクチャを採用するにつれて、レガシー監視ソリューションの限界がますます明らかになってきています。

### 断片化されたモニタリングの状況

ほとんどの組織は、統一された可視性を提供しない寄せ集めの監視ツールに悩まされています。

| モニタリングレイヤー | 一般的な課題 |
|---|---|
| **Infrastructure** | アプリケーションコンテキストが限定的 |
| **Application Performance** | サイロ化されたメトリクス、相関なし |
| **Distributed Tracing** | サンプリングのギャップ、コスト制約 |
| **Logs** | トレースとの相関が困難 |
| **Business Metrics** | 技術データから分断されている |

### 従来の監視の主な制限事項

**可視性のギャップ**
- **不完全なデータカバレッジ**: サンプリングと集計により、重要なエッジケースや異常が隠れてしまう
- **サービス境界の盲点**: マイクロサービスの境界をまたいでリクエストをトレースすることが困難
- **顧客固有の問題**: 集計メトリクスにより、個々の顧客体験の問題が見えにくくなる
- **断続的な問題**: 一時的な問題が平均化されたメトリクスの中に埋もれてしまう

**コストと複雑性**
- **ツールの乱立**: 複数の監視ソリューションにより、ライセンスおよび運用コストが増加します
- **データサイロ**: メトリクス、トレース、ログのためのストレージシステムが分断されています
- **手動による相関分析**: エンジニアがツール間でデータを結びつけるために多大な時間を費やします
- **スケーリングの課題**: 従来のツールはクラウドネイティブアプリケーションのデータ量に対応しきれません

**運用上の非効率性**
- **平均検出時間 (MTTD) の長期化**: プロアクティブなモニタリングではなく、顧客からのクレームによって問題が発見される
- **平均解決時間 (MTTR) の長期化**: 複数のツールやデータソースにまたがる複雑なトラブルシューティング
- **アラート疲労**: 連携していないモニタリングシステムによる高い誤検知率
- **コンテキストスイッチング**: モニタリングインターフェース間の切り替えによるエンジニアの生産性低下


## 最新のオブザーバビリティ要件

今日のクラウドネイティブアプリケーションは、オブザーバビリティに対して根本的に異なるアプローチを求めています。モノリシックから分散アーキテクチャへの移行は、顧客の期待や規制要件の高まりと相まって、統合された包括的な可視性を必要としています。

**統合されたアプリケーション中心のビュー**
- **サービスディスカバリー**: アプリケーションコンポーネントの自動識別とマッピング
- **ゴールデンシグナルメトリクス**: すべてのサービスにわたるレート、エラー、レイテンシー、および飽和度
- **ビジネスコンテキストの統合**: 技術的なパフォーマンスをビジネス成果に結びつける
- **カスタマージャーニートラッキング**: 分散トランザクション全体にわたるエンドツーエンドの可視性

**リアルタイムインテリジェンス**
- **プロアクティブな異常検知**: 顧客に影響が出る前に問題を特定
- **インテリジェントなアラート**: 誤検知を減らしたコンテキスト対応の通知
- **根本原因分析**: メトリクス、トレース、ログ全体にわたる自動相関
- **パフォーマンス最適化**: 継続的な改善のためのデータドリブンなインサイト

**高度な分析とインサイト**
- **完全なトランザクション可視性**: すべてのリクエストが重要です。特に高価値顧客に対しては
- **高度なクエリ機能**: ビジネスコンテキストを含むテレメトリデータの柔軟な分析
- **Machine Learning 統合**: 予測分析とパターン認識
- **カスタムビジネスメトリクス**: 技術的なテレメトリからビジネス KPI を導出
