# ட்ரேசிங் ஏஜெண்டை தேர்ந்தெடுத்தல்

## சரியான ஏஜெண்டை தேர்ந்தெடுங்கள்

AWS நேரடியாக [ட்ரேஸ்](../signals/traces.md) சேகரிப்புக்கு இரண்டு கருவி தொகுப்புகளை ஆதரிக்கிறது (மேலும் எங்கள் பரந்த [observability பங்காளிகள்](https://aws.amazon.com/products/management-and-governance/partners/)):

* [AWS Distro for OpenTelemetry](https://aws-otel.github.io/), பொதுவாக ADOT என்று அழைக்கப்படுகிறது
* X-Ray [SDKs](https://docs.aws.amazon.com/xray/latest/devguide/xray-instrumenting-your-app.html) மற்றும் [daemon](https://docs.aws.amazon.com/xray/latest/devguide/xray-daemon.html)

உங்கள் observability தீர்வை மேம்படுத்தும்போது எந்த கருவி அல்லது கருவிகளைப் பயன்படுத்த வேண்டும் என்பது நீங்கள் எடுக்க வேண்டிய முக்கிய முடிவாகும். இந்த கருவிகள் பரஸ்பரம் பிரத்தியேகமானவை அல்ல, தேவைக்கேற்ப அவற்றை ஒன்றாக கலக்கலாம். இந்த தேர்வுக்கான சிறந்த நடைமுறை உள்ளது. இருப்பினும், முதலில் [OpenTelemetry (OTEL)](https://opentelemetry.io/) இன் தற்போதைய நிலையைப் புரிந்துகொள்ள வேண்டும்.

OTEL என்பது observability சிக்னலிங்கிற்கான தற்போதைய தொழில் தரநிலை விவரக்குறிப்பு ஆகும், இது மூன்று முக்கிய சிக்னல் வகைகளுக்கான வரையறைகளைக் கொண்டுள்ளது: [மெட்ரிக்குகள்](../signals/metrics.md), [ட்ரேஸ்கள்](../signals/traces.md), மற்றும் [லாக்குகள்](../signals/logs.md). இருப்பினும், OTEL எப்போதும் இருந்ததில்லை, இது [OpenMetrics](https://openmetrics.io) மற்றும் [OpenTracing](https://opentracing.io) போன்ற முந்தைய விவரக்குறிப்புகளிலிருந்து உருவாகியுள்ளது. Observability விற்பனையாளர்கள் சமீப ஆண்டுகளில் OpenTelemetry Line Protocol (OTLP)-ஐ வெளிப்படையாக ஆதரிக்கத் தொடங்கினர்.

AWS X-Ray மற்றும் CloudWatch ஆகியவை OTEL விவரக்குறிப்புக்கு முன்பே இருந்தன, மற்ற முன்னணி observability தீர்வுகளும் அவ்வாறே. இருப்பினும், AWS X-Ray சேவை ADOT-ஐ பயன்படுத்தி OTEL ட்ரேஸ்களை உடனடியாக ஏற்றுக்கொள்கிறது. ADOT-ல் X-Ray-க்கு நேரடியாகவும், மற்ற ISV தீர்வுகளுக்கும் டெலிமெட்ரியை வெளியிடுவதற்கான ஒருங்கிணைப்புகள் ஏற்கனவே உள்ளன.

எந்த பரிவர்த்தனை ட்ரேசிங் தீர்வுக்கும் சிக்னல்களை சேகரிக்க ஒரு ஏஜெண்ட் மற்றும் அடிப்படை பயன்பாட்டில் ஒரு ஒருங்கிணைப்பு தேவை. இது, சோதிக்கப்பட வேண்டிய, பராமரிக்கப்பட வேண்டிய மற்றும் மேம்படுத்தப்பட வேண்டிய நூலகங்கள் வடிவத்தில் தொழில்நுட்ப கடனை உருவாக்குகிறது, மேலும் எதிர்காலத்தில் உங்கள் தீர்வை மாற்ற நீங்கள் தேர்வு செய்தால் மறுகருவியாக்கம் தேவைப்படலாம்.

X-Ray உடன் சேர்க்கப்பட்ட SDKகள் AWS வழங்கும் இறுக்கமாக ஒருங்கிணைக்கப்பட்ட instrumentation தீர்வின் ஒரு பகுதியாகும். ADOT என்பது X-Ray பல ட்ரேசிங் தீர்வுகளில் ஒன்று மட்டுமே என்ற பரந்த தொழில் தீர்வின் ஒரு பகுதியாகும். இரண்டு அணுகுமுறையையும் பயன்படுத்தி X-Ray-ல் முழுமையான ட்ரேசிங்கை செயல்படுத்தலாம், ஆனால் உங்களுக்கு மிகவும் பயனுள்ள அணுகுமுறையைத் தீர்மானிக்க வேறுபாடுகளைப் புரிந்துகொள்வது முக்கியம்.

:::info
	பின்வருவன உங்களுக்குத் தேவையெனில் AWS Distro for OpenTelemetry மூலம் உங்கள் பயன்பாட்டை instrument செய்ய பரிந்துரைக்கிறோம்:

    * உங்கள் குறியீட்டை மறு-instrument செய்யாமல் பல்வேறு ட்ரேசிங் பின்தளங்களுக்கு ட்ரேஸ்களை அனுப்பும் திறன். உதாரணமாக, X-Ray கன்சோலில் இருந்து [Zipkin](https://zipkin.io)-க்கு மாற விரும்பினால், collector-ன் கட்டமைப்பு மட்டுமே மாறும், உங்கள் பயன்பாட்டு குறியீடு மாறாமல் இருக்கும்.

    * OpenTelemetry சமூகத்தால் பராமரிக்கப்படும் ஒவ்வொரு மொழிக்கும் அதிக எண்ணிக்கையிலான நூலக instrumentations ஆதரவு.
:::

:::info
	பின்வருவன உங்களுக்குத் தேவையெனில் உங்கள் பயன்பாட்டை instrument செய்ய X-Ray SDK-ஐ தேர்ந்தெடுக்க பரிந்துரைக்கிறோம்:

    * இறுக்கமாக ஒருங்கிணைக்கப்பட்ட ஒற்றை-விற்பனையாளர் தீர்வு.

    * Node.js, Python, Ruby அல்லது .NET பயன்படுத்தும்போது X-Ray மையப்படுத்தப்பட்ட sampling விதிகளுடன் ஒருங்கிணைப்பு, X-Ray கன்சோலில் இருந்து sampling விதிகளை கட்டமைக்கும் திறன் மற்றும் பல hosts-ல் தானாகவே அவற்றைப் பயன்படுத்தும் திறன் உட்பட.
:::
