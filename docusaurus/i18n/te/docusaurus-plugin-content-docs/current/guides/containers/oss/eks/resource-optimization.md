# Kubernetes వర్క్‌లోడ్‌ల కోసం రిసోర్స్ ఆప్టిమైజేషన్ బెస్ట్ ప్రాక్టీసెస్
Kubernetes అడాప్షన్ వేగవంతంగా కొనసాగుతోంది, చాలా మంది మైక్రోసర్వీస్ ఆధారిత ఆర్కిటెక్చర్‌లకు మారుతున్నారు. ప్రారంభ దృష్టి అప్లికేషన్‌లకు మద్దతు ఇవ్వడానికి కొత్త క్లౌడ్ నేటివ్ ఆర్కిటెక్చర్‌లను డిజైన్ చేయడం మరియు నిర్మించడంపై ఉంది. ఎన్విరాన్‌మెంట్‌లు పెరుగుతున్నప్పుడు, కస్టమర్‌ల నుండి రిసోర్స్ కేటాయింపును ఆప్టిమైజ్ చేయడంపై దృష్టి పెట్టడం ప్రారంభమవుతోంది. సెక్యూరిటీ తర్వాత ఆపరేషన్స్ టీమ్ అడిగే రెండవ అత్యంత ముఖ్యమైన ప్రశ్న రిసోర్స్ ఆప్టిమైజేషన్.
Kubernetes ఎన్విరాన్‌మెంట్‌లలో రిసోర్స్ కేటాయింపును ఆప్టిమైజ్ చేయడం మరియు అప్లికేషన్‌లను సరైన పరిమాణానికి మార్చడం గురించి మార్గదర్శకత్వం చూద్దాం. ఇందులో managed node groups, self-managed node groups మరియు AWS Fargate తో డిప్లాయ్ చేయబడిన Amazon EKS పై రన్ అవుతున్న అప్లికేషన్‌లు ఉన్నాయి.

## Kubernetes లో అప్లికేషన్‌లను సరైన పరిమాణానికి మార్చడానికి కారణాలు
Kubernetes లో, రిసోర్స్ right-sizing అప్లికేషన్‌లపై రిసోర్స్ స్పెసిఫికేషన్‌లు సెట్ చేయడం ద్వారా జరుగుతుంది. ఈ సెట్టింగ్‌లు నేరుగా ప్రభావితం చేస్తాయి:

* పెర్ఫార్మెన్స్ — సరైన రిసోర్స్ స్పెసిఫికేషన్‌లు లేకుండా Kubernetes అప్లికేషన్‌లు రిసోర్సుల కోసం ఏకపక్షంగా పోటీ పడతాయి, ఇది అప్లికేషన్ పెర్ఫార్మెన్స్‌ను ప్రతికూలంగా ప్రభావితం చేయవచ్చు.
* కాస్ట్ ఆప్టిమైజేషన్ — అధిక పరిమాణంలో రిసోర్స్ స్పెసిఫికేషన్‌లతో డిప్లాయ్ చేయబడిన అప్లికేషన్‌లు పెరిగిన ఖర్చులు మరియు తక్కువగా వినియోగించబడిన ఇన్‌ఫ్రాస్ట్రక్చర్‌కు దారితీస్తాయి.
* ఆటోస్కేలింగ్ — Kubernetes Cluster Autoscaler మరియు Horizontal Pod Autoscaling పనిచేయడానికి రిసోర్స్ స్పెసిఫికేషన్‌లు అవసరం.

Kubernetes లో అత్యంత సాధారణ రిసోర్స్ స్పెసిఫికేషన్‌లు [CPU మరియు memory requests మరియు limits](https://kubernetes.io/docs/concepts/configuration/manage-resources-containers/#requests-and-limits) కోసం ఉంటాయి.

## Requests మరియు Limits

కంటైనరైజ్డ్ అప్లికేషన్‌లు Kubernetes పై Pods గా డిప్లాయ్ చేయబడతాయి. CPU మరియు memory requests మరియు limits Pod డెఫినిషన్‌లో ఐచ్ఛిక భాగం. CPU [Kubernetes CPUs](https://kubernetes.io/docs/concepts/configuration/manage-resources-containers/#meaning-of-cpu) యూనిట్లలో స్పెసిఫై చేయబడుతుంది, memory bytes లో, సాధారణంగా [mebibytes (Mi)](https://simple.wikipedia.org/wiki/Mebibyte) గా స్పెసిఫై చేయబడుతుంది.

Request మరియు limits ప్రతి ఒక్కటి Kubernetes లో వేర్వేరు ఫంక్షన్‌లను అందిస్తాయి మరియు షెడ్యూలింగ్ మరియు రిసోర్స్ ఎన్‌ఫోర్స్‌మెంట్‌ను భిన్నంగా ప్రభావితం చేస్తాయి.

## సిఫారసులు
అప్లికేషన్ ఓనర్ తమ CPU మరియు memory రిసోర్స్ requests కోసం "సరైన" విలువలను ఎంచుకోవాలి. ఆదర్శ మార్గం డెవలప్‌మెంట్ ఎన్విరాన్‌మెంట్‌లో అప్లికేషన్‌ను లోడ్ టెస్ట్ చేసి observability టూలింగ్ ఉపయోగించి రిసోర్స్ వాడకాన్ని కొలవడం. అది మీ సంస్థ యొక్క అత్యంత క్లిష్టమైన అప్లికేషన్‌లకు అర్థవంతంగా ఉన్నప్పటికీ, మీ క్లస్టర్‌లో డిప్లాయ్ చేయబడిన ప్రతి కంటైనరైజ్డ్ అప్లికేషన్‌కు ఇది సాధ్యం కాదు. వర్క్‌లోడ్‌లను ఆప్టిమైజ్ చేయడానికి మరియు సరైన పరిమాణానికి మార్చడానికి సహాయపడే టూల్స్ గురించి చూద్దాం:

### Vertical Pod Autoscaler (VPA)
[VPA](https://github.com/kubernetes/autoscaler/tree/master/vertical-pod-autoscaler) Autoscaling special interest group (SIG) యాజమాన్యంలోని Kubernetes సబ్-ప్రాజెక్ట్. ఇది గమనించిన అప్లికేషన్ పెర్ఫార్మెన్స్ ఆధారంగా స్వయంచాలకంగా Pod requests సెట్ చేయడానికి రూపొందించబడింది. VPA డీఫాల్ట్‌గా [Kubernetes Metric Server](https://github.com/kubernetes-sigs/metrics-server) ఉపయోగించి రిసోర్స్ వాడకాన్ని సేకరిస్తుంది, కానీ Prometheus ను డేటా సోర్స్‌గా కాన్ఫిగర్ చేయవచ్చు.
VPA లో రికమెండేషన్ ఇంజిన్ ఉంది, ఇది అప్లికేషన్ పెర్ఫార్మెన్స్‌ను కొలిచి సైజింగ్ రికమెండేషన్‌లు చేస్తుంది. VPA రికమెండేషన్ ఇంజిన్‌ను స్టాండ్-అలోన్‌గా డిప్లాయ్ చేయవచ్చు, తద్వారా VPA ఏ ఆటోస్కేలింగ్ చర్యలు చేయదు. ఇది ప్రతి అప్లికేషన్ కోసం VerticalPodAutoscaler కస్టమ్ రిసోర్స్ సృష్టించడం ద్వారా కాన్ఫిగర్ చేయబడుతుంది మరియు VPA ఆబ్జెక్ట్ యొక్క status ఫీల్డ్‌ను రిసోర్స్ సైజింగ్ రికమెండేషన్‌లతో అప్‌డేట్ చేస్తుంది.
మీ క్లస్టర్‌లోని ప్రతి అప్లికేషన్ కోసం VerticalPodAutoscaler ఆబ్జెక్ట్‌లు సృష్టించడం మరియు JSON ఫలితాలను చదవడం మరియు అర్థం చేసుకోవడానికి ప్రయత్నించడం స్కేల్‌లో సవాలుగా ఉంటుంది. [Goldilocks](https://github.com/FairwindsOps/goldilocks) అనేది దీనిని సులభతరం చేసే ఓపెన్ సోర్స్ ప్రాజెక్ట్.

### Goldilocks
Goldilocks అనేది Fairwinds నుండి వచ్చిన ఓపెన్ సోర్స్ ప్రాజెక్ట్, ఇది సంస్థలకు వారి Kubernetes అప్లికేషన్ రిసోర్స్ requests "సరిగ్గా సరైన" స్థాయిలో పొందడానికి సహాయపడటానికి రూపొందించబడింది. Goldilocks యొక్క డీఫాల్ట్ కాన్ఫిగరేషన్ opt-in మోడల్. నేమ్‌స్పేస్‌కు goldilocks.fairwinds.com/enabled: true లేబుల్ జోడించడం ద్వారా ఏ వర్క్‌లోడ్‌లను మానిటర్ చేయాలో మీరు ఎంచుకుంటారు.


![Goldilocks-Architecture](../../../../images/goldilocks-architecture.png)

Metrics Server వర్కర్ నోడ్‌లలో రన్ అవుతున్న Kubelet నుండి రిసోర్స్ మెట్రిక్స్ సేకరించి Vertical Pod Autoscaler ఉపయోగించడానికి Metrics API ద్వారా వాటిని ఎక్స్‌పోజ్ చేస్తుంది. Goldilocks controller goldilocks.fairwinds.com/enabled: true లేబుల్ ఉన్న నేమ్‌స్పేస్‌లను గమనించి ఆ నేమ్‌స్పేస్‌లలోని ప్రతి వర్క్‌లోడ్ కోసం VerticalPodAutoscaler ఆబ్జెక్ట్‌లను సృష్టిస్తుంది.

రిసోర్స్ రికమెండేషన్ కోసం నేమ్‌స్పేస్‌లను ఎనేబుల్ చేయడానికి, క్రింది కమాండ్ రన్ చేయండి:

```
kubectl create ns javajmx-sample
kubectl label ns javajmx-sample goldilocks.fairwinds.com/enabled=true
```

Amazon EKS క్లస్టర్‌లో goldilocks డిప్లాయ్ చేయడానికి, క్రింది కమాండ్ రన్ చేయండి:

```
helm repo add fairwinds-stable https://charts.fairwinds.com/stable
helm upgrade --install goldilocks fairwinds-stable/goldilocks --namespace goldilocks --create-namespace --set vpa.enabled=true
```

Goldilocks-dashboard డాష్‌బోర్డ్‌ను పోర్ట్ 8080 లో ఎక్స్‌పోజ్ చేస్తుంది మరియు రిసోర్స్ రికమెండేషన్ పొందడానికి దానిని యాక్సెస్ చేయవచ్చు. డాష్‌బోర్డ్ యాక్సెస్ చేయడానికి క్రింది కమాండ్ రన్ చేయండి:

```
kubectl -n goldilocks port-forward svc/goldilocks-dashboard 8080:80
```
ఆపై మీ బ్రౌజర్‌లో http://localhost:8080 తెరవండి

![Goldilocks-Dashboard](../../../../images/goldilocks-dashboard.png)


Goldilocks అందించిన రికమెండేషన్‌లను చూడటానికి శాంపిల్ నేమ్‌స్పేస్‌ను విశ్లేషిద్దాం. డిప్లాయ్‌మెంట్ కోసం రికమెండేషన్‌లను చూడగలగాలి.
![Goldilocks-Recommendation](../../../../images/goldilocks-recommendation.png)

javajmx-sample వర్క్‌లోడ్ కోసం request & limit రికమెండేషన్‌లను చూడవచ్చు. ప్రతి Quality of Service (QoS) కింద Current కాలమ్ ప్రస్తుతం కాన్ఫిగర్ చేయబడిన CPU మరియు Memory request మరియు limits ను సూచిస్తుంది. Guaranteed మరియు Burstable కాలమ్ సంబంధిత QoS కోసం సిఫారసు చేయబడిన CPU మరియు Memory request limits ను సూచిస్తుంది.

మనం రిసోర్సులను అధికంగా ప్రొవిజన్ చేశామని స్పష్టంగా గమనించవచ్చు మరియు goldilocks CPU మరియు Memory request ను ఆప్టిమైజ్ చేయడానికి రికమెండేషన్‌లు చేసింది. Guaranteed QoS కోసం CPU request మరియు limits 100m మరియు 300m తో పోలిస్తే 15m మరియు 15m గా సిఫారసు చేయబడింది మరియు Memory request మరియు limits 180Mi మరియు 300Mi తో పోలిస్తే 105M మరియు 105M గా సిఫారసు చేయబడింది.
మీరు ఆసక్తి ఉన్న QoS క్లాస్ కోసం సంబంధిత manifest ఫైల్‌ను కాపీ చేసి, సరైన పరిమాణానికి మార్చబడిన మరియు ఆప్టిమైజ్ చేయబడిన వర్క్‌లోడ్‌లను డిప్లాయ్ చేయవచ్చు.

### cAdvisor మెట్రిక్ ఉపయోగించి throttling అర్థం చేసుకోవడం మరియు రిసోర్స్ ను తగిన విధంగా కాన్ఫిగర్ చేయడం
limits కాన్ఫిగర్ చేసినప్పుడు, నిర్దిష్ట సమయంలో నిర్దిష్ట కంటైనరైజ్డ్ అప్లికేషన్ ఎంత సేపు రన్ అవ్వవచ్చో Linux నోడ్‌కు చెప్తున్నాము. నోడ్‌లోని మిగిలిన వర్క్‌లోడ్‌లను అసమంజసమైన CPU సైకిల్‌లు తీసుకునే ప్రాసెసుల నుండి రక్షించడానికి ఇలా చేస్తాము. మదర్‌బోర్డ్‌పై ఉన్న అనేక భౌతిక "cores" ను నిర్వచించడం లేదు; అయితే, ఇతర అప్లికేషన్‌లను overwhelm చేయకుండా ఉండటానికి ఒకే కంటైనర్‌లో ఉన్న ప్రాసెసులు లేదా థ్రెడ్‌ల సమూహం తాత్కాలికంగా పాజ్ చేయడానికి ముందు ఎంత సేపు రన్ అవ్వవచ్చో కాన్ఫిగర్ చేస్తున్నాము.

`container_cpu_cfs_throttled_seconds_total` అనే cAdvisor మెట్రిక్ అన్ని throttled 5 ms slices ను జోడించి quota కంటే ప్రాసెస్ ఎంత ముందున్నదో మనకు ఒక ఐడియా ఇస్తుంది. ఈ మెట్రిక్ సెకన్లలో ఉంది, కాబట్టి 100 ms పొందడానికి విలువను 10 తో భాగిస్తాము, ఇది కంటైనర్‌తో అనుసంధానించబడిన నిజమైన సమయ వ్యవధి.

100 ms సమయంలో టాప్ మూడు పాడ్‌ల CPU వాడకాన్ని అర్థం చేసుకోవడానికి PromQL క్వెరీ.
```
topk(3, max by (pod, container)(rate(container_cpu_usage_seconds_total{image!="", instance="$instance"}[$__rate_interval]))) / 10
```
 400 ms vCPU వాడకం గమనించబడింది.

![Throttled-Period](../../../../images/throttled-period.png)

PromQL మనకు ప్రతి సెకను throttling ఇస్తుంది, సెకనుకు 10 periods తో. ప్రతి period throttling పొందడానికి 10 తో భాగిస్తాము. Limits సెట్టింగ్‌ను ఎంత పెంచాలో తెలుసుకోవాలంటే, 10 తో గుణించవచ్చు (ఉదా., 400 ms * 10 = 4000 m).

పై టూల్స్ రిసోర్స్ ఆప్టిమైజేషన్ కోసం అవకాశాలను గుర్తించే మార్గాలను అందిస్తుండగా, అప్లికేషన్ టీమ్‌లు ఇచ్చిన అప్లికేషన్ CPU / Memory ఇంటెన్సివ్ అనేది గుర్తించడానికి సమయం కేటాయించాలి మరియు throttling / over-provisioning నిరోధించడానికి రిసోర్సులను కేటాయించాలి.
