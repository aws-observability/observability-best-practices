# మెట్రిక్స్

మెట్రిక్స్ అనేవి మీ సిస్టమ్ పనితీరు గురించిన డేటా. సిస్టమ్ లేదా వనరులకు సంబంధించిన అన్ని మెట్రిక్స్‌ను ఒక కేంద్రీకృత ప్రదేశంలో కలిగి ఉండటం వలన మెట్రిక్స్‌ను పోల్చడం, పనితీరును విశ్లేషించడం మరియు వనరులను స్కేల్-అప్ లేదా స్కేల్-ఇన్ చేయడం వంటి మెరుగైన వ్యూహాత్మక నిర్ణయాలు తీసుకునే సామర్థ్యాన్ని మీకు అందిస్తుంది. వనరుల ఆరోగ్యాన్ని తెలుసుకోవడానికి మరియు ముందస్తు చర్యలు తీసుకోవడానికి కూడా మెట్రిక్స్ ముఖ్యమైనవి.

మెట్రిక్ డేటా పునాదిగా ఉండి [అలారంలు](../signals/alarms.md), అనామలీ డిటెక్షన్, [ఈవెంట్‌లు](../signals/events.md), [డాష్‌బోర్డ్‌లు](./dashboards.md) మరియు మరిన్నింటిని నడిపించడానికి ఉపయోగించబడుతుంది.

## వెండెడ్ మెట్రిక్స్

[CloudWatch మెట్రిక్స్](https://docs.aws.amazon.com/AmazonCloudWatch/latest/monitoring/working_with_metrics.html) మీ సిస్టమ్‌ల పనితీరు గురించిన డేటాను సేకరిస్తుంది. డిఫాల్ట్‌గా, చాలా AWS సేవలు తమ వనరులకు ఉచిత మెట్రిక్స్ అందిస్తాయి. ఇందులో [Amazon EC2](https://aws.amazon.com/ec2/) ఇన్‌స్టాన్స్‌లు, [Amazon RDS](https://aws.amazon.com/rds/), [Amazon S3](https://aws.amazon.com/s3/?p=pm&c=s3&z=4) బకెట్‌లు మరియు మరిన్ని ఉన్నాయి.

మేము ఈ మెట్రిక్స్‌ను *వెండెడ్ మెట్రిక్స్* అని సూచిస్తాము. మీ AWS ఖాతాలో వెండెడ్ మెట్రిక్స్ సేకరణకు ఎటువంటి ఛార్జ్ లేదు.

:::info
	CloudWatch కు మెట్రిక్స్ ఎమిట్ చేసే AWS సేవల పూర్తి జాబితా కోసం [ఈ పేజీ](https://docs.aws.amazon.com/AmazonCloudWatch/latest/monitoring/aws-services-cloudwatch-metrics.html) చూడండి.
:::
## మెట్రిక్స్ క్వెరీ చేయడం

CloudWatch లో [metric math](https://docs.aws.amazon.com/AmazonCloudWatch/latest/monitoring/using-metric-math.html) ఫీచర్‌ను ఉపయోగించి బహుళ మెట్రిక్స్‌ను క్వెరీ చేయవచ్చు మరియు మరింత గ్రాన్యులారిటీ కోసం మెట్రిక్స్‌ను విశ్లేషించడానికి మాథ్ ఎక్స్‌ప్రెషన్‌లను ఉపయోగించవచ్చు. ఉదాహరణకు, Lambda ఎర్రర్ రేట్‌ను కనుగొనడానికి ఒక metric math ఎక్స్‌ప్రెషన్ ఇలా రాయవచ్చు:

	Errors/Requests

CloudWatch కన్సోల్‌లో ఇది ఎలా కనిపిస్తుందో క్రింద ఒక ఉదాహరణ చూడవచ్చు:

![Metric math ఉదాహరణ](../images/metrics1.png)

:::info
	మీ డేటా నుండి అత్యధిక విలువను పొందడానికి మరియు ప్రత్యేక డేటా సోర్స్‌ల పనితీరు నుండి విలువలను డెరైవ్ చేయడానికి metric math ఉపయోగించండి.
:::
CloudWatch కండిషనల్ స్టేట్‌మెంట్‌లను కూడా సపోర్ట్ చేస్తుంది. ఉదాహరణకు, లేటెన్సీ ఒక నిర్దిష్ట థ్రెషోల్డ్ కంటే ఎక్కువగా ఉన్న ప్రతి టైమ్‌సిరీస్‌కు `1` విలువను రిటర్న్ చేయడానికి, మరియు అన్ని ఇతర డేటా పాయింట్‌లకు `0` రిటర్న్ చేయడానికి, క్వెరీ ఇలా ఉంటుంది:

	IF(latency>threshold, 1, 0)

CloudWatch కన్సోల్‌లో ఈ లాజిక్‌ను ఉపయోగించి బూలియన్ విలువలు సృష్టించవచ్చు, ఇవి [CloudWatch అలారంలు](./alarms.md) లేదా ఇతర చర్యలను ట్రిగ్గర్ చేయగలవు. డెరైవ్ చేసిన డేటాపాయింట్‌ల నుండి ఆటోమేటిక్ చర్యలను ఎనేబుల్ చేయగలదు. CloudWatch కన్సోల్ నుండి ఒక ఉదాహరణ క్రింద ఉంది:

![డెరైవ్ చేసిన విలువ నుండి అలారం సృష్టి](../images/metrics2.png)

:::info
	డెరైవ్ చేసిన విలువలకు పనితీరు థ్రెషోల్డ్‌లను మించినప్పుడు అలారంలు మరియు నోటిఫికేషన్‌లను ట్రిగ్గర్ చేయడానికి కండిషనల్ స్టేట్‌మెంట్‌లను ఉపయోగించండి.
:::
ఏదైనా మెట్రిక్‌కు టాప్ `n` చూపించడానికి `SEARCH` ఫంక్షన్ కూడా ఉపయోగించవచ్చు. పెద్ద సంఖ్యలో టైమ్‌సిరీస్‌లలో (ఉదా. వేలాది సర్వర్లు) ఉత్తమ లేదా చెత్త పనితీరు కలిగిన మెట్రిక్స్‌ను విజ్యువలైజ్ చేసేటప్పుడు ఈ విధానం మీకు అత్యంత ముఖ్యమైన డేటాను మాత్రమే చూడడానికి అనుమతిస్తుంది. గత ఐదు నిమిషాల్లో సగటున టాప్ రెండు CPU-వినియోగించే EC2 ఇన్‌స్టాన్స్‌లను రిటర్న్ చేసే సెర్చ్ ఉదాహరణ ఇక్కడ ఉంది:
```
	SLICE(SORT(SEARCH('{AWS/EC2,InstanceId} MetricName="CPUUtilization"', 'Average', 300), MAX, DESC),0, 2)
```
CloudWatch కన్సోల్‌లో అదే వ్యూ:

![CloudWatch మెట్రిక్స్‌లో సెర్చ్ క్వెరీ](../images/metrics3.png)

:::info
	మీ పర్యావరణంలో విలువైన లేదా చెత్త పనితీరు కలిగిన వనరులను త్వరగా ప్రదర్శించడానికి `SEARCH` విధానాన్ని ఉపయోగించండి, ఆపై వాటిని [డాష్‌బోర్డ్‌లు](./dashboards.md)లో ప్రదర్శించండి.
:::
## మెట్రిక్స్ సేకరణ

మీ EC2 ఇన్‌స్టాన్స్‌లకు మెమరీ లేదా డిస్క్ స్పేస్ వినియోగం వంటి అదనపు మెట్రిక్స్ కావాలంటే, మీ తరఫున ఈ డేటాను CloudWatch కు పుష్ చేయడానికి [CloudWatch ఏజెంట్](./cloudwatch_agent.md) ఉపయోగించవచ్చు. లేదా గ్రాఫికల్‌గా విజ్యువలైజ్ చేయాల్సిన కస్టమ్ ప్రాసెసింగ్ డేటా ఉంటే, మరియు ఈ డేటాను CloudWatch మెట్రిక్‌గా ఉండాలంటే, CloudWatch కు కస్టమ్ మెట్రిక్స్ పబ్లిష్ చేయడానికి [`PutMetricData` API](https://docs.aws.amazon.com/AmazonCloudWatch/latest/APIReference/API_PutMetricData.html) ఉపయోగించవచ్చు.

:::info
	బేర్ API కాకుండా CloudWatch కు మెట్రిక్ డేటాను పుష్ చేయడానికి [AWS SDKs](https://aws.amazon.com/developer/tools/) లో ఒకదానిని ఉపయోగించండి.
:::
`PutMetricData` API కాల్‌లు క్వెరీల సంఖ్య ఆధారంగా ఛార్జ్ చేయబడతాయి. `PutMetricData` API ని సరైన విధంగా ఉపయోగించడం ఉత్తమ పద్ధతి. ఈ API లో Values మరియు Counts మెథడ్ ఉపయోగించి, ఒక `PutMetricData` రిక్వెస్ట్‌తో ప్రతి మెట్రిక్‌కు 150 విలువల వరకు పబ్లిష్ చేయవచ్చు, మరియు ఈ డేటాపై పర్సంటైల్ స్టాటిస్టిక్స్ పొందడానికి సపోర్ట్ చేస్తుంది. అందువల్ల, ప్రతి డేటాపాయింట్‌కు ప్రత్యేక API కాల్ చేయడానికి బదులుగా, మీ అన్ని డేటాపాయింట్‌లను కలిపి ఒకే `PutMetricData` API కాల్‌లో CloudWatch కు పుష్ చేయాలి. ఈ విధానం వినియోగదారునికి రెండు విధాలుగా ప్రయోజనం చేకూరుస్తుంది:

1. CloudWatch ప్రైసింగ్
1. `PutMetricData` API థ్రాట్లింగ్ నివారించవచ్చు

:::info
	`PutMetricData` ఉపయోగించేటప్పుడు, సాధ్యమైనప్పుడల్లా మీ డేటాను ఒకే `PUT` ఆపరేషన్‌లలో బ్యాచ్ చేయడం ఉత్తమ పద్ధతి.
:::
:::info
	CloudWatch లోకి పెద్ద సంఖ్యలో మెట్రిక్స్ ఎమిట్ చేయబడితే ప్రత్యామ్నాయ విధానంగా [Embedded Metric Format](https://docs.aws.amazon.com/AmazonCloudWatch/latest/monitoring/CloudWatch_Embedded_Metric_Format_Manual.html) పరిగణించండి. Embedded Metric Format `PutMetricData` ను ఉపయోగించదు లేదా దానికి ఛార్జ్ చేయదు, అయినప్పటికీ [CloudWatch Logs](./logs/index.md) వినియోగం నుండి బిల్లింగ్ ఏర్పడుతుంది.
:::
## అనామలీ డిటెక్షన్

CloudWatch లో [anomaly detection](https://docs.aws.amazon.com/AmazonCloudWatch/latest/monitoring/CloudWatch_Anomaly_Detection.html) ఫీచర్ ఉంది, ఇది రికార్డ్ చేసిన మెట్రిక్స్ ఆధారంగా *సాధారణం* ఏమిటో నేర్చుకోవడం ద్వారా మీ observability వ్యూహాన్ని మెరుగుపరుస్తుంది. ఏదైనా మెట్రిక్ సిగ్నల్ సేకరణ వ్యవస్థకు అనామలీ డిటెక్షన్ వాడకం ఒక [ఉత్తమ పద్ధతి](../signals/metrics.md#use-anomaly-detection-algorithms).

అనామలీ డిటెక్షన్ రెండు వారాల కాలంలో ఒక మోడల్‌ను నిర్మిస్తుంది.

:::warning
	అనామలీ డిటెక్షన్ సృష్టించిన సమయం నుండి ముందుకు మాత్రమే తన మోడల్‌ను నిర్మిస్తుంది. ఇది గతంలో ఔట్‌లయర్‌లను కనుగొనడానికి వెనుకకు ప్రొజెక్ట్ చేయదు.
:::

:::warning
	అనామలీ డిటెక్షన్‌కు ఒక మెట్రిక్‌కు *మంచి* ఏమిటో తెలియదు, స్టాండర్డ్ డీవియేషన్ ఆధారంగా *సాధారణం* ఏమిటో మాత్రమే తెలుసు.
:::

:::info
	సాధారణ ప్రవర్తన అపేక్షించే రోజులో సమయాలను మాత్రమే విశ్లేషించడానికి మీ అనామలీ డిటెక్షన్ మోడల్‌లను శిక్షణ ఇవ్వడం ఉత్తమ పద్ధతి. శిక్షణ నుండి మినహాయించడానికి కాల వ్యవధులను నిర్వచించవచ్చు (రాత్రులు, వారాంతాలు, లేదా సెలవు దినాలు వంటివి).
:::
అనామలీ డిటెక్షన్ బ్యాండ్ యొక్క ఒక ఉదాహరణ ఇక్కడ చూడవచ్చు, బ్యాండ్ గ్రే రంగులో ఉంది.

![అనామలీ డిటెక్షన్ బ్యాండ్](../images/metrics4.png)

అనామలీ డిటెక్షన్ కోసం ఎక్స్‌క్లూజన్ విండోలను CloudWatch కన్సోల్, [CloudFormation](https://docs.aws.amazon.com/AWSCloudFormation/latest/UserGuide/aws-properties-cloudwatch-anomalydetector-configuration.html), లేదా AWS SDKs లో ఒకదానిని ఉపయోగించి సెట్ చేయవచ్చు.
