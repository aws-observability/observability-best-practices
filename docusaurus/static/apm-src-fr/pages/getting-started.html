Premiers pas avec CloudWatch APM

Vue d'ensemble

La surveillance des performances des applications (APM) est essentielle pour comprendre et optimiser la santé et les performances des applications distribuées modernes. Sans APM, les équipes peinent avec des temps de réponse lents, des pannes fréquentes et des difficultés à localiser les problèmes dans des architectures de microservices complexes. Amazon CloudWatch Application Signals résout ces défis en fournissant une observabilité complète qui instrumente automatiquement vos applications, collecte les données de télémétrie et fournit des informations exploitables via des tableaux de bord préconfigurés et des alertes intelligentes.

CloudWatch Application Signals prend en charge les capacités APM clés, notamment les métriques dorées (latence, débit, taux d'erreurs et saturation) pour une évaluation rapide de la santé, l'analyse des traces et spans pour une analyse détaillée du flux des requêtes, et une compatibilité complète avec OpenTelemetry (OTel). OpenTelemetry est crucial car il fournit des normes indépendantes des fournisseurs pour la collecte et l'exportation des données de télémétrie, garantissant que vous pouvez instrumenter vos applications une seule fois et envoyer les données vers plusieurs plateformes d'observabilité sans verrouillage fournisseur. Cela accélère votre parcours du développement à la production en permettant une surveillance cohérente entre les environnements, en réduisant le temps moyen de résolution (MTTR) et en assurant des applications stables et fiables grâce à la détection proactive des problèmes et aux informations automatisées.

Qu'est-ce que CloudWatch APM ?

CloudWatch Application Performance Monitoring (APM) est la solution d'observabilité complète d'AWS construite sur Application Signals - notre produit phare qui fournit une visibilité de bout en bout sur vos applications, des services backend aux expériences utilisateur frontend.

Application Signals découvre automatiquement la topologie de votre application, instrumente votre code et fournit trois capacités de surveillance intégrées qui travaillent ensemble pour vous offrir une observabilité complète.

Les trois piliers de la surveillance des performances des applications (APM)

CloudWatch Application Signals

CloudWatch Application Signals est la plateforme phare d'observabilité d'AWS qui découvre automatiquement la topologie de votre application, instrumente votre code et fournit trois capacités de surveillance intégrées qui travaillent ensemble pour offrir une observabilité complète.

Ce que vous obtenez :

  • Découverte et instrumentation automatiques des services
  • APM, RUM et Synthetics intégrés dans une seule plateforme
  • Tableaux de bord préconfigurés et informations automatisées
  • Visibilité inter-comptes et inter-régions

Surveillance des utilisateurs réels (RUM)

Capturez les expériences réelles des utilisateurs en production. Surveillez comment les vrais utilisateurs interagissent avec vos applications web à travers différents navigateurs, appareils et zones géographiques. Le RUM suit les performances des pages, les erreurs JavaScript et les Core Web Vitals pour vous aider à comprendre et optimiser l'expérience utilisateur.

Ce que vous obtenez :

  • Temps de chargement des pages et performances de navigation
  • Erreurs JavaScript avec traces de pile
  • Suivi des Core Web Vitals (LCP, FID, CLS)
  • Replays de sessions utilisateur et analyse des parcours
  • Informations sur les performances par zone géographique et par appareil

Surveillance synthétique

Testez proactivement les flux critiques avant les utilisateurs. Exécutez des tests automatisés 24h/24 et 7j/7 depuis plusieurs régions AWS pour détecter les problèmes avant qu'ils n'impactent les utilisateurs. Les canaries synthétiques valident la fonctionnalité des API, surveillent les parcours utilisateur critiques et garantissent que votre application respecte les SLA.

Ce que vous obtenez :

  • Vérifications de santé des API planifiées (toutes les 1 à 60 minutes)
  • Tests de navigateur scriptés pour les flux utilisateur critiques
  • Surveillance de la disponibilité multi-région
  • Suivi des performances de référence
  • Alertes proactives avant l'impact utilisateur

Comment ils fonctionnent ensemble

La puissance d'Application Signals provient de la façon dont ces trois capacités se complètent pour fournir une visibilité complète.

Scénario : Investigation complète d'un incident

Synthetics détecte l'échec du flux de paiement → Alerte déclenchée
Les traces APM montrent que le service de paiement lance des exceptions
Le RUM confirme que les vrais utilisateurs rencontrent les mêmes échecs
Résolution : la trace APM révèle qu'un déploiement récent a introduit un bogue

Scénario : Optimisation des performances

Le RUM identifie des chargements de page lents sur mobile (4,5s LCP)
Les traces APM montrent que les API backend sont rapides (200ms)
Le RUM révèle que le goulot d'étranglement est une image principale de 3 Mo
Synthetics valide le correctif après optimisation

Démarrer avec Application Signals

Avec Application Signals, vous n'avez besoin d'aucune modification de code pour commencer. Activez-le simplement dans votre environnement AWS, et il instrumentera automatiquement vos applications et découvrira les services. Une fois activé, Application Signals continue de découvrir automatiquement les changements dans la topologie de votre application sans configuration supplémentaire.

Vous n'avez pas besoin de tout activer en même temps. Voici le parcours recommandé :

Phase 1 : Commencer avec l'APM (Semaine 1)

  • Activer Application Signals pour les services backend
  • Obtenir une visibilité sur les dépendances de services
  • Établir les lignes de base de performance

Phase 2 : Ajouter Synthetics (Semaines 2-3)

  • Créer des canaries pour les parcours utilisateur critiques
  • Mettre en place une surveillance proactive
  • Établir les lignes de base de disponibilité

Phase 3 : Activer le RUM (Semaine 4+)

  • Ajouter le RUM à votre application web
  • Suivre les expériences des utilisateurs réels
  • Corréler avec les performances backend

Développeurs : Instrumenter votre première application

Ce que vous allez accomplir
En 15 minutes, Application Signals collectera des traces APM et des métriques à partir de vos services backend, sans aucune modification de code pour les environnements d'exécution pris en charge.

Prérequis

  • Application s'exécutant sur AWS (ECS, EKS, EC2 ou Lambda)
  • Environnement d'exécution pris en charge : Java, Python, .NET, Node.js ou Go
  • Permissions IAM pour activer Application Signals

Étape 1 : Choisir votre approche d'instrumentation

Option A : Auto-instrumentation (Recommandée)

  • Aucune modification de code requise
  • Fonctionne avec Java, Python, .NET, Node.js
  • Collecte automatique des métriques et traces APM

Option B : Instrumentation manuelle OpenTelemetry

  • Contrôle total sur ce que vous instrumentez
  • Portable entre les plateformes d'observabilité
  • Spans et attributs personnalisés pour APM, RUM et Synthetics

Option C : AWS Distro for OpenTelemetry (ADOT)

  • OpenTelemetry optimisé pour AWS
  • Préconfiguré pour Application Signals

Étape 2 : Activer Application Signals

Suivez les instructions d'activation spécifiques à la plateforme (EKS, ECS, Lambda) dans le guide développeur. Après activation, attendez 5 à 10 minutes pour que les données apparaissent dans la console.

Étape 3 : Vérifier la collecte des données APM

  1. Attendre 5 à 10 minutes pour que les données apparaissent
  2. Ouvrir la console CloudWatch → Application Signals
  3. Vérifier la carte des applications — vous devriez voir vos services avec les métriques APM
  4. Cliquer sur un service pour voir les traces distribuées et les métriques de performance

Étape 4 : Ajouter une instrumentation personnalisée (Optionnel)

Enrichissez vos traces APM avec des spans personnalisés et un contexte métier (exemples de code disponibles dans la documentation développeur).

Étape 5 : Activer le RUM pour la surveillance frontend (Optionnel)

Ajoutez le client web RUM pour suivre les expériences utilisateur.

Étape 6 : Créer des canaries synthétiques (Optionnel)

Mettez en place une surveillance proactive avec des tests synthétiques.

Dépannage pour les développeurs

  • Vous ne voyez pas les traces APM ? Vérifiez les permissions IAM (cloudwatch:PutMetricData, xray:PutTraceSegments) et confirmez que l'agent est en cours d'exécution (kubectl logs -n amazon-cloudwatch ...).
  • Les données RUM n'apparaissent pas ? Vérifiez que le snippet RUM se charge avant les autres scripts et consultez la console du navigateur pour les erreurs.
  • Les canaries synthétiques échouent ? Examinez les journaux des canaries et testez les scripts localement ; vérifiez la connectivité réseau depuis le VPC du canary.
  • Surcharge élevée ? Ajustez le taux d'échantillonnage APM et envisagez l'échantillonnage en tête ou l'échantillonnage des sessions RUM.

Opérateurs : Surveiller les systèmes en production

Ce que vous allez accomplir
Mettez en place une surveillance, des alertes et des tableaux de bord complets couvrant APM, RUM et Synthetics pour vos applications en production en 30 minutes.

Votre checklist de surveillance

Surveillance de la santé des services (APM)

  • La carte des applications montre tous les services backend critiques
  • Tableaux de bord des métriques APM clés créés (latence, erreurs, débit)
  • SLO définis pour les services critiques
  • Alertes configurées pour les violations de SLO

Surveillance de l'expérience utilisateur (RUM)

  • RUM activé sur les applications web en production
  • Suivi des Core Web Vitals configuré
  • Alertes d'erreurs frontend mises en place
  • Lignes de base de performance géographiques établies

Surveillance proactive (Synthetics)

  • Canaries créés pour les parcours utilisateur critiques
  • Vérifications de santé des API exécutées toutes les 5 minutes
  • Surveillance de la disponibilité multi-région activée
  • Alertes d'échec des tests synthétiques configurées

Étape 1 : Créer votre tableau de bord unifié

Utilisez le modèle de tableau de bord Application Signals et ajoutez des widgets pour les signaux dorés APM, l'expérience utilisateur RUM, la santé Synthetics et les dépendances de services.

Étape 2 : Configurer des alertes intelligentes

Créez des alarmes composites couvrant APM, RUM et Synthetics (exemples dans la documentation).

Étape 3 : Implémenter l'automatisation des runbooks

Liez les runbooks aux alertes et implémentez l'auto-remédiation lorsque c'est sûr (exemples : redémarrer les conteneurs, monter en charge en cas de latence élevée, invalider le cache CDN).

Étape 4 : Établir des revues opérationnelles

Revue hebdomadaire des métriques et analyses approfondies mensuelles (optimisation des coûts, analyse de la fatigue d'alerte, rétrospectives d'incidents).

Architectes : Concevoir votre stratégie d'observabilité

Ce que vous allez accomplir
Concevez une architecture Application Signals évolutive et rentable qui répond aux besoins de votre organisation pour les 2 à 3 prochaines années.

Cadre de décision architecturale

Stratégie d'adoption

Recommandation : Commencez par l'APM pour la visibilité backend, ajoutez Synthetics pour la surveillance proactive, puis activez le RUM pour les informations sur l'expérience utilisateur.

Stratégie d'instrumentation

Recommandation : Utilisez OpenTelemetry pour les nouvelles applications afin de maintenir la flexibilité ; utilisez AWS natif pour une adoption rapide d'Application Signals.

Stratégie multi-cloud

Les options incluent Application Signals pour AWS uniquement, OpenTelemetry pour une approche multi-cloud unifiée, ou un pont de collecteur OTel pour une flexibilité hybride.

Stratégie de rétention des données

  • Données de traces APM : Chaud (7 jours), Tiède (30 jours), Froid (90+ jours archivés)
  • Données de sessions RUM : Chaud (30 jours), Froid (90+ jours)
  • Résultats Synthetics : Chaud (90 jours), Froid (1+ ans)
  • Rétention des métriques par résolution : 1 min (15 jours), 5 min (63 jours), 1 heure (15 mois)

Modèles d'architecture de référence

Exemples : Microservices sur EKS avec ADOT, Serverless avec couches Lambda, Cloud hybride avec collecteurs OTel.

Planification de la capacité

Estimez les coûts d'observabilité et utilisez l'échantillonnage/filtrage et les politiques d'archivage pour optimiser les dépenses.

Sécurité et conformité

  • Chiffrer les traces APM, les données RUM et les résultats Synthetics au repos et en transit
  • Implémenter le masquage des données pour les informations personnelles dans les traces et les sessions RUM
  • Utiliser IAM pour un contrôle d'accès granulaire
  • Activer CloudTrail pour la journalisation d'audit

Considérations de conformité : HIPAA, PCI-DSS, RGPD, SOC 2 — suivez les conseils de rétention et de masquage des données dans chaque cas.

Planification de la migration

De X-Ray à Application Signals

  1. Exécution en parallèle (2-4 semaines) : activez Application Signals en parallèle de X-Ray, ajoutez le RUM, créez des canaries, comparez la couverture
  2. Migration progressive (1-2 mois) : migrez les services non critiques, étendez le RUM, mettez à jour les tableaux de bord
  3. Basculement complet (1 mois) : migrez les services restants et décommissionnez X-Ray

Depuis un APM tiers

  • Évaluer l'instrumentation actuelle
  • Planifier la migration OTel si nécessaire
  • Envisager l'écriture double pendant la transition
  • Prévoir 3 à 6 mois pour la migration

Accompagnement des équipes

  • Semaine 1 : Atelier fondamentaux (aperçu APM, RUM, Synthetics)
  • Semaines 2-3 : Laboratoires pratiques (instrumenter les services, activer le RUM, créer des canaries)
  • Semaine 4 : Pilote en production avec les trois signaux
  • Mois 2+ : Permanences et support

Établir un centre d'excellence : champions, documentation interne, tableaux de bord réutilisables et bonnes pratiques.

Scénarios courants et solutions

Scénario : Déboguer un point de terminaison API lent

Approche développeur : vérifier le RUM, trouver la trace APM lente, identifier le span goulot d'étranglement, ajouter une instrumentation personnalisée, déployer et vérifier avec le RUM et Synthetics.

Approche opérateur : vérifier Synthetics, examiner les traces APM, vérifier le RUM pour l'impact utilisateur, chercher des problèmes d'infrastructure corrélés, escalader avec des preuves.

Scénario : Se préparer pour le Black Friday

Revoir la planification de la capacité, établir des SLO à travers les signaux, définir des politiques de mise à l'échelle et préparer des runbooks.
