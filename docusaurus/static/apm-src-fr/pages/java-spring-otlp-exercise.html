Java Spring + OTLP + CloudWatch (Exercice pratique)

Ce guide documente l'exercice complet pour instrumenter un service Java Spring Boot avec OpenTelemetry, exporter la télémétrie vers AWS, déployer sur ECS/Fargate et vérifier le comportement des traces/journaux/Application Signals.

Il est rédigé dans le style d'un exemple d'implémentation et peut être utilisé comme runbook ou support d'atelier.

1) Ce que cet exercice démontre

  • Télémétrie de bout en bout depuis une application Java :
    • Traces (X-Ray / Transaction Search)
    • Journaux (point de terminaison OTLP CloudWatch Logs ou awslogs sur ECS)
    • Métriques/Application Signals (chemin via le sidecar CloudWatch Agent)
  • La différence entre :
    • Le pont de traçage Micrometer (micrometer-tracing-bridge-otel) pour le pontage de l'API de traçage
    • Les registres Micrometer (micrometer-registry-cloudwatch2, micrometer-registry-otlp) qui sont des destinations uniquement pour les métriques
  • Le câblage des points de terminaison OTLP via des variables d'environnement (OTEL_*), et non des constantes Java codées en dur.

4) Correspondance des points de terminaison OTLP

Signal Modèle de point de terminaison Variables d'environnement typiques
Traces https://xray.<region>.amazonaws.com/v1/traces OTEL_EXPORTER_OTLP_TRACES_ENDPOINT, OTEL_EXPORTER_OTLP_TRACES_PROTOCOL=http/protobuf
Métriques https://monitoring.<region>.amazonaws.com/v1/metrics OTEL_EXPORTER_OTLP_METRICS_ENDPOINT (ou chemin App Signals via sidecar)
Journaux https://logs.<region>.amazonaws.com/v1/logs OTEL_EXPORTER_OTLP_LOGS_ENDPOINT, OTEL_EXPORTER_OTLP_LOGS_HEADERS

En mode sidecar ECS, le conteneur d'application pointe les traces/métriques vers http://localhost:4316/... et le CloudWatch Agent gère le transfert.

5) Modèle d'instrumentation utilisé dans ce dépôt

5.1 Dépendances

  • micrometer-tracing-bridge-otel est présent dans pom.xml.
  • Aucune dépendance sur micrometer-registry-cloudwatch2 ou micrometer-registry-otlp pour le traçage.

5.2 Traçage automatique + manuel

  • L'instrumentation automatique de l'agent Java ADOT capture les spans du framework/client.
  • L'enrichissement manuel est ajouté dans DemoService.greet() :
    • attributs : demo.operation, demo.request.name, demo.downstream.status, demo.downstream.elapsed_ms, demo.custom.instrumented
    • événements : demo.greet.start, demo.httpbin.success, demo.httpbin.failure
    • enregistrement d'erreur avec recordException(...) et setStatus(ERROR, ...)

7) Où voir la télémétrie dans la console AWS

Traces (principal)

  1. X-Ray → Traces
  2. Région : us-east-1
  3. Plage horaire : 15 à 60 dernières minutes
  4. Service : java-spring-otel-poc

Pour les champs personnalisés :

  • Ouvrir une trace → segment java-spring-otel-poc → Métadonnées
  • Chercher les clés :
    • demo.operation
    • demo.request.name
    • demo.downstream.status
    • demo.downstream.elapsed_ms
    • demo.custom.instrumented

Journaux

  • Journaux de l'application ECS : /ecs/java-spring-otel
  • Journaux du sidecar CW agent : /ecs/ecs-cwagent-java-spring-otel
  • Groupe de journaux OTLP local (si exécution locale) : /aws/otel/java-spring-otel-poc

Application Signals

  • CloudWatch → Application Signals → Services
  • Chercher java-spring-otel-poc après un trafic soutenu et un transfert réussi des métriques via le chemin du sidecar.

8) Point clé à retenir

L'instrumentation automatique OpenTelemetry peut créer des spans, mais des résultats d'observabilité AWS réussis dépendent encore de :

  1. Un câblage correct des points de terminaison OTLP
  2. Un chemin d'authentification/transfert correct (souvent via sidecar/collecteur pour les métriques/App Signals)
  3. Une distinction correcte entre le pont de traçage et les registres de métriques
  4. Un trafic soutenu et la bonne vue dans la console (X-Ray pour les traces)
