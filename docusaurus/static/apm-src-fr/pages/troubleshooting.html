Documentation CloudWatch APM

Dépannage

Problèmes courants

Cette section couvre les problèmes les plus fréquemment rencontrés lors de la mise en œuvre de CloudWatch Application Signals et comment les résoudre.

Aucune trace n'apparaît dans CloudWatch

Causes possibles :

  • Application Signals non activé pour le service
  • Permissions IAM incorrectes
  • Agent non correctement configuré ou non en cours d'exécution
  • Taux d'échantillonnage trop bas
  • Problèmes de connectivité réseau vers CloudWatch

Solutions :

  1. Vérifier qu'Application Signals est activé dans la console AWS
  2. Vérifier que les permissions IAM incluent cloudwatch:PutMetricData et xray:PutTraceSegments
  3. Confirmer que l'agent est en cours d'exécution et correctement configuré
  4. Augmenter le taux d'échantillonnage pour les tests (définir à 100 % temporairement)
  5. Vérifier que les groupes de sécurité VPC et les ACL réseau autorisent le trafic sortant vers CloudWatch

Traces manquantes ou incomplètes

Problèmes courants :

  • Contexte non propagé entre les services
  • Certains services non instrumentés
  • L'échantillonnage supprime des spans
  • Problèmes de configuration de l'agent

Étapes de débogage :

  1. Vérifier la carte des services pour identifier les services non instrumentés
  2. Vérifier que les en-têtes de contexte de trace sont transmis (par ex., x-amzn-trace-id)
  3. Examiner les journaux de l'agent pour les erreurs de configuration
  4. Tester avec un échantillonnage à 100 % pour exclure les problèmes d'échantillonnage
  5. Utiliser la recherche de traces pour trouver les traces partielles

Surcharge d'instrumentation élevée

Symptômes :

  • Utilisation accrue du CPU
  • Consommation de mémoire plus élevée
  • Latence accrue
  • Dégradation des performances de l'application

Stratégies d'optimisation :

  • Réduire le taux d'échantillonnage (commencer avec 10 % pour la production)
  • Utiliser l'échantillonnage en tête au lieu de l'échantillonnage en queue
  • Configurer des limites de spans appropriées
  • Regrouper les exportations de spans
  • Surveiller les métriques d'instrumentation

Les données RUM n'apparaissent pas

Liste de vérification :

  • Client web RUM correctement installé et chargé
  • ID d'application et configuration corrects
  • En-têtes CORS configurés pour les domaines RUM
  • La politique de sécurité du contenu autorise les scripts RUM
  • La console du navigateur ne montre aucune erreur JavaScript

Vérification :

// Check if RUM is loaded
console.log(window.cwr);

// Verify configuration
console.log(cwr('getConfig'));

Échec des canaries synthétiques

Causes courantes :

  • Problèmes de connectivité réseau depuis le VPC du canary
  • Échecs d'authentification ou d'autorisation
  • Modifications de l'application cassant les scripts de test
  • Environnement d'exécution ou dépendances du canary obsolètes

Débogage :

  1. Examiner les journaux du canary dans CloudWatch Logs
  2. Tester les scripts du canary localement
  3. Vérifier le routage VPC et les groupes de sécurité
  4. Mettre à jour l'environnement d'exécution du canary si nécessaire
  5. Vérifier que les points de terminaison de l'application sont accessibles

Guides de débogage

Guides pas à pas pour diagnostiquer les problèmes complexes.

Débogage de la propagation du contexte de trace

Étape 1 : Vérifier l'instrumentation

Vérifiez que tous les services de votre chaîne d'appels sont correctement instrumentés :

# Check for trace headers in requests
curl -H "x-amzn-trace-id: Root=1-5759e988-bd862e3fe1be46a994272793;Parent=53995c3f42cd8ad8;Sampled=1" \
     http://your-service/endpoint

Étape 2 : Vérifier la configuration de l'agent

Vérifiez les fichiers de configuration de l'agent et les variables d'environnement.

Étape 3 : Tester de bout en bout

Effectuez une requête de test et tracez-la à travers tous les services.

Diagnostic des problèmes de performance

Identifier le goulot d'étranglement :

  1. Vérifier la carte des services pour les services à haute latence
  2. Examiner les chronologies de traces pour les spans lents
  3. Revoir les métriques de ressources (CPU, mémoire, E/S disque)
  4. Chercher des modèles d'erreurs dans les journaux
  5. Vérifier les performances des requêtes de base de données

Goulots d'étranglement courants :

  • Optimisation des requêtes de base de données nécessaire
  • Appels d'API externes expirant
  • Contraintes de ressources (CPU/mémoire)
  • Épuisement du pool de threads
  • Latence réseau entre les services

Dépannage de l'agent

Vérifier l'état de l'agent :

# For Java applications
jps -l | grep opentelemetry

# For Python applications
ps aux | grep opentelemetry

# Check agent logs
tail -f /var/log/aws-opentelemetry-agent.log

Problèmes courants de l'agent :

  • Mauvaise version de l'agent Java pour la JVM
  • Dépendances ou bibliothèques manquantes
  • Erreurs de syntaxe dans le fichier de configuration
  • Problèmes de permissions d'écriture sur les fichiers journaux

Outils et commandes

Outils et commandes utiles pour le dépannage.

Commandes CLI CloudWatch

# List Application Signals services
aws application-signals list-services

# Get service metrics
aws cloudwatch get-metric-data --metric-data-queries '[
  {
    "Id": "m1",
    "MetricStat": {
      "Metric": {
        "Namespace": "AWS/ApplicationSignals",
        "MetricName": "Latency"
      },
      "Period": 300,
      "Stat": "Average"
    }
  }
]'

# Query traces
aws xray get-trace-summaries --start-time 2024-01-01T00:00:00Z --end-time 2024-01-02T00:00:00Z

Analyse des journaux

Utilisez CloudWatch Logs Insights pour analyser les données de télémétrie :

# Find errors in application logs
fields @timestamp, @message
| filter @message like /ERROR/
| sort @timestamp desc
| limit 100

# Correlate traces with logs
fields @timestamp, traceId, @message
| filter ispresent(traceId)
| sort @timestamp desc
