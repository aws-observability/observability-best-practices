CloudWatch APM दस्तावेज़ीकरण

Java Spring + OTLP + CloudWatch (हैंड्स-ऑन अभ्यास)

यह गाइड OpenTelemetry के साथ Java Spring Boot सर्विस को इंस्ट्रूमेंट करने, AWS को टेलीमेट्री एक्सपोर्ट करने, ECS/Fargate पर डिप्लॉय करने, और ट्रेसेज़/लॉग्स/Application Signals व्यवहार को सत्यापित करने के पूर्ण अभ्यास को दस्तावेज़ित करता है।

यह एक इम्प्लीमेंटेशन उदाहरण की शैली में लिखा गया है और इसे रनबुक या वर्कशॉप हैंडआउट के रूप में उपयोग किया जा सकता है।

1) यह अभ्यास क्या प्रदर्शित करता है

  • एक Java ऐप से एंड-टू-एंड टेलीमेट्री:
    • ट्रेसेज़ (X-Ray / Transaction Search)
    • लॉग्स (CloudWatch Logs OTLP एंडपॉइंट या ECS पर awslogs)
    • मेट्रिक्स/Application Signals (CloudWatch Agent साइडकार पाथ)
  • इनके बीच अंतर:
    • Micrometer ट्रेसिंग ब्रिज (micrometer-tracing-bridge-otel) ट्रेसिंग API ब्रिजिंग के लिए
    • Micrometer रजिस्ट्रीज़ (micrometer-registry-cloudwatch2, micrometer-registry-otlp) जो केवल मेट्रिक्स सिंक हैं
  • env vars (OTEL_*) के माध्यम से OTLP एंडपॉइंट वायरिंग, हार्डकोडेड Java कॉन्स्टेंट नहीं।

2) आर्किटेक्चर

+----------------------------------------------------------------------------------+
|                               Developer / Runtime Host                           |
|                                                                                  |
|  loadgen.sh ---> Spring Boot app (:8080)                                         |
|                    |                                                             |
|                    +--> @Observed method (DemoService.greet)                     |
|                    |      +--> custom span attrs/events                          |
|                    |      +--> downstream call to httpbin.org                    |
|                    |                                                             |
|                    +--> ADOT Java agent (-javaagent)                             |
|                           |                                                      |
|                           +--> OTLP traces ----> xray.<region>.amazonaws.com     |
|                           +--> OTLP logs ------> logs.<region>.amazonaws.com     |
|                           +--> AppSignals -----> monitoring.<region>.amazonaws.. |
|                                                                                  |
+----------------------------------------------------------------------------------+

4) OTLP एंडपॉइंट मैपिंग

सिग्नल एंडपॉइंट पैटर्न सामान्य env vars
ट्रेसेज़ https://xray.<region>.amazonaws.com/v1/traces OTEL_EXPORTER_OTLP_TRACES_ENDPOINT, OTEL_EXPORTER_OTLP_TRACES_PROTOCOL=http/protobuf
मेट्रिक्स https://monitoring.<region>.amazonaws.com/v1/metrics OTEL_EXPORTER_OTLP_METRICS_ENDPOINT (या साइडकार के माध्यम से App Signals पाथ)
लॉग्स https://logs.<region>.amazonaws.com/v1/logs OTEL_EXPORTER_OTLP_LOGS_ENDPOINT, OTEL_EXPORTER_OTLP_LOGS_HEADERS

ECS साइडकार मोड में, ऐप कंटेनर ट्रेसेज़/मेट्रिक्स को http://localhost:4316/... की ओर इंगित करता है और CloudWatch Agent फ़ॉरवर्डिंग संभालता है।

5) इस रेपो में उपयोग किया गया इंस्ट्रूमेंटेशन मॉडल

5.1 डिपेंडेंसीज़

  • micrometer-tracing-bridge-otel pom.xml में मौजूद है।
  • ट्रेसिंग के लिए micrometer-registry-cloudwatch2 या micrometer-registry-otlp पर कोई डिपेंडेंसी नहीं।

5.2 ऑटो + मैनुअल ट्रेसिंग

  • ADOT Java एजेंट से ऑटो इंस्ट्रूमेंटेशन फ्रेमवर्क/क्लाइंट स्पैन कैप्चर करता है।
  • मैनुअल एनरिचमेंट DemoService.greet() के अंदर जोड़ा गया है:
    • एट्रिब्यूट: demo.operation, demo.request.name, demo.downstream.status, demo.downstream.elapsed_ms, demo.custom.instrumented
    • इवेंट: demo.greet.start, demo.httpbin.success, demo.httpbin.failure
    • recordException(...) और setStatus(ERROR, ...) के साथ एरर रिकॉर्डिंग

6) स्क्रीनशॉट (ट्रेस विवरण)

वर्कशॉप/डेमो डॉक्स के लिए नीचे दिए गए स्क्रीनशॉट का उपयोग करें:

7) AWS कंसोल में टेलीमेट्री कहां देखें

ट्रेसेज़ (प्राथमिक)

  1. X-Ray → Traces
  2. रीजन: us-east-1
  3. समय सीमा: पिछले 15-60 मिनट
  4. सर्विस: java-spring-otel-poc

कस्टम फ़ील्ड के लिए:

  • एक ट्रेस खोलें → सेगमेंट java-spring-otel-poc → मेटाडेटा
  • इन कीज़ की तलाश करें:
    • demo.operation
    • demo.request.name
    • demo.downstream.status
    • demo.downstream.elapsed_ms
    • demo.custom.instrumented

लॉग्स

  • ECS ऐप लॉग: /ecs/java-spring-otel
  • CW एजेंट साइडकार लॉग: /ecs/ecs-cwagent-java-spring-otel
  • लोकल OTLP लॉग ग्रुप (यदि लोकल चला रहे हैं): /aws/otel/java-spring-otel-poc

Application Signals

  • CloudWatch → Application Signals → Services
  • निरंतर ट्रैफ़िक और साइडकार पाथ के माध्यम से सफल मेट्रिक्स फ़ॉरवर्डिंग के बाद java-spring-otel-poc की तलाश करें।

8) मुख्य निष्कर्ष

OpenTelemetry ऑटो इंस्ट्रूमेंटेशन स्पैन बना सकता है, लेकिन सफल AWS observability परिणाम अभी भी इन पर निर्भर करते हैं:

  1. सही OTLP एंडपॉइंट वायरिंग
  2. सही auth/फ़ॉरवर्डिंग पाथ (अक्सर मेट्रिक्स/App Signals के लिए साइडकार/कलेक्टर के माध्यम से)
  3. ट्रेसिंग ब्रिज बनाम मेट्रिक्स रजिस्ट्रीज़ के बीच सही अंतर
  4. निरंतर ट्रैफ़िक और सही कंसोल व्यू (ट्रेसेज़ के लिए X-Ray)
