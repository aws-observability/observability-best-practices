CloudWatch Application Signals की सीमाओं और कोटा को समझना।

सीमाएं और कोटा

Application Signals CloudWatch और AWS X-Ray पर निर्भर करता है। सीमाएं (इनजेशन, क्वेरी रेट, और कोटा) उन सर्विसेज़ द्वारा नियंत्रित होती हैं और समय के साथ बदल सकती हैं। वर्तमान मानों के लिए आधिकारिक दस्तावेज़ीकरण का उपयोग करें।

  • Application Signals: Application Signals डॉक्स
  • CloudWatch कोटा: CloudWatch सर्विस कोटा
  • AWS X-Ray कोटा: X-Ray API थ्रॉटलिंग और सीमाएं

CloudWatch मेट्रिक रिटेंशन (सामान्य संदर्भ)

CloudWatch विभिन्न रेज़ोल्यूशन पर मेट्रिक डेटा बनाए रखता है। पूर्ण रिटेंशन मॉडल के लिए CloudWatch डॉक्स देखें।

  • 1-मिनट रेज़ोल्यूशन: 15 दिन
  • 5-मिनट रेज़ोल्यूशन: 63 दिन
  • 1-घंटा रेज़ोल्यूशन: 15 महीने

सर्विस कोटा

Application Signals:

  • प्रति अकाउंट सर्विसेज़: 1,000
  • प्रति सर्विस ऑपरेशन: 100
  • कस्टम मेट्रिक्स: प्रति सर्विस 30

RUM:

  • प्रति अकाउंट ऐप मॉनिटर: 20
  • प्रति सेकंड इवेंट: 10,000
  • इवेंट आकार: 64 KB

Synthetics:

  • प्रति अकाउंट कैनरी: 100
  • कैनरी निष्पादन समय: अधिकतम 15 मिनट
  • कैनरी आकार: 1 GB (अनज़िप्ड)

शब्दावली

CloudWatch Application Signals में प्रमुख शब्द और अवधारणाएं।

मूल अवधारणाएं

एप्लिकेशन परफ़ॉर्मेंस मॉनिटरिंग (APM)

सॉफ़्टवेयर एप्लिकेशन के प्रदर्शन और उपलब्धता की निगरानी और प्रबंधन की प्रथा।

डिस्ट्रिब्यूटेड ट्रेसिंग

डिस्ट्रिब्यूटेड सिस्टम में अनुरोधों को ट्रैक करने की एक विधि, जो आपको कई सर्विसेज़ में अनुरोधों के पथ और प्रदर्शन को समझने की अनुमति देती है।

सर्विस मैप

आपके एप्लिकेशन के आर्किटेक्चर का विज़ुअल प्रतिनिधित्व जो दिखाता है कि सर्विसेज़ कैसे कनेक्ट और इंटरैक्ट करती हैं।

स्पैन

एक ट्रेस के भीतर एक एकल ऑपरेशन, जो डेटाबेस क्वेरी या API कॉल जैसे कार्य की एक इकाई का प्रतिनिधित्व करता है।

ट्रेस

आपके एप्लिकेशन के माध्यम से एक अनुरोध की पूर्ण एंड-टू-एंड यात्रा, जिसमें कई स्पैन शामिल होते हैं।

इंस्ट्रूमेंटेशन शब्द

ऑटो-इंस्ट्रूमेंटेशन

कोड परिवर्तन की आवश्यकता के बिना एप्लिकेशन में observability कोड का स्वचालित जोड़ना।

मैनुअल इंस्ट्रूमेंटेशन

एप्लिकेशन सोर्स कोड में ट्रेसिंग और मेट्रिक्स कोड का स्पष्ट जोड़ना।

OpenTelemetry

एक ओपन-सोर्स observability फ्रेमवर्क जो इंस्ट्रूमेंटेशन के लिए वेंडर-न्यूट्रल API प्रदान करता है।

सैंपलिंग

observability और लागत और प्रदर्शन के बीच संतुलन बनाने के लिए कौन से अनुरोधों को ट्रेस करना है यह चुनने की प्रक्रिया।

कॉन्टेक्स्ट प्रोपेगेशन

ट्रेस निरंतरता बनाए रखने के लिए सर्विसेज़ के बीच ट्रेस कॉन्टेक्स्ट पास करने का तंत्र।

मेट्रिक्स और मॉनिटरिंग

गोल्डन सिग्नल

एप्लिकेशन स्वास्थ्य की निगरानी के लिए चार प्रमुख मेट्रिक्स: लेटेंसी, थ्रूपुट, एरर रेट, और सैचुरेशन।

लेटेंसी

एक अनुरोध को प्रोसेस करने में लगने वाला समय, आमतौर पर पर्सेंटाइल (P50, P90, P99) में मापा जाता है।

थ्रूपुट

अनुरोधों को प्रोसेस करने की दर, आमतौर पर प्रति सेकंड अनुरोधों के रूप में मापा जाता है।

एरर रेट

अनुरोधों का वह प्रतिशत जो एरर में परिणत होता है।

सैचुरेशन

सिस्टम रिसोर्स के उपयोग की डिग्री।

सर्विस लेवल ऑब्जेक्टिव (SLO)

सर्विस प्रदाताओं और उपभोक्ताओं के बीच सहमत सर्विस गुणवत्ता का एक लक्ष्य स्तर।

सर्विस लेवल इंडिकेटर (SLI)

प्रदान किए गए सर्विस स्तर का एक मापने योग्य पहलू, जैसे लेटेंसी या उपलब्धता।

RUM और Synthetics

रियल यूज़र मॉनिटरिंग (RUM)

प्रोडक्शन में वेब एप्लिकेशन के साथ वास्तविक उपयोगकर्ता इंटरैक्शन की निगरानी।

सिंथेटिक मॉनिटरिंग

सिम्युलेटेड उपयोगकर्ता इंटरैक्शन का उपयोग करके एप्लिकेशन का स्वचालित परीक्षण।

कैनरी

एक सिंथेटिक मॉनिटरिंग स्क्रिप्ट जो आपके एप्लिकेशन के विरुद्ध स्वचालित परीक्षण चलाती है।

Core Web Vitals

वेब पर उपयोगकर्ता अनुभव मापने के लिए Google के मेट्रिक्स: LCP, FID, और CLS।

Largest Contentful Paint (LCP)

स्क्रीन पर सबसे बड़े कंटेंट एलिमेंट को रेंडर होने में लगने वाला समय।

First Input Delay (FID)

उपयोगकर्ता पहली बार पेज के साथ इंटरैक्ट करने से लेकर ब्राउज़र के रिस्पॉन्ड करने तक का समय।

Cumulative Layout Shift (CLS)

विज़ुअल स्थिरता का एक माप, अप्रत्याशित लेआउट शिफ्ट की मात्रा निर्धारित करता है।

AWS-विशिष्ट शब्द

Application Signals

AWS का यूनिफ़ाइड observability समाधान जो APM, RUM, और Synthetics को जोड़ता है।

X-Ray

AWS की डिस्ट्रिब्यूटेड ट्रेसिंग सर्विस, अब Application Signals के साथ एकीकृत।

CloudWatch

AWS की मॉनिटरिंग और observability सर्विस जो मेट्रिक्स, लॉग्स, और ट्रेसेज़ एकत्र करती है।

ADOT (AWS Distro for OpenTelemetry)

AWS सर्विसेज़ के लिए अनुकूलित OpenTelemetry का AWS वितरण।

समर्थित रनटाइम और फ्रेमवर्क

Application Signals द्वारा समर्थित प्रोग्रामिंग भाषाएं और फ्रेमवर्क।

ऑटो-इंस्ट्रूमेंटेशन सपोर्ट

आधिकारिक संगतता और रनटाइम/वर्ज़न मैट्रिक्स के लिए, देखें समर्थित सिस्टम। मुख्य बातें:

  • Java: JVM 8/11/17/21/23 (फ्रेमवर्क सपोर्ट ADOT/OpenTelemetry Java इंस्ट्रूमेंटेशन का अनुसरण करता है)।
  • Python: Python 3.9+ (फ्रेमवर्क सपोर्ट ADOT/OpenTelemetry Python इंस्ट्रूमेंटेशन का अनुसरण करता है)।
  • Node.js: Node 14/16/18/20/22 (फ्रेमवर्क सपोर्ट ADOT/OpenTelemetry JS इंस्ट्रूमेंटेशन का अनुसरण करता है)।
  • .NET: x86-64 या ARM64 पर समर्थित; Linux x64/ARM64 और Windows Server 2022 x64 (लाइब्रेरी सपोर्ट ADOT/OpenTelemetry .NET इंस्ट्रूमेंटेशन का अनुसरण करता है)।
  • PHP/Ruby/Go: OpenTelemetry ज़ीरो-कोड/ऑटो-इंस्ट्रूमेंटेशन पाथ के माध्यम से समर्थित (इन भाषाओं के लिए कोई ADOT SDK नहीं)।

नोट्स और ज्ञात विचार

  • Node.js ESM: ESM सपोर्ट सीमित है; डॉक्स में ESM-विशिष्ट सेटअप का पालन न करें तो CommonJS को प्राथमिकता दें।
  • Python: कंटेनराइज़्ड ऐप्स को PYTHONPATH सेट करने की आवश्यकता हो सकती है (ज्ञात OTel ऑटोइंस्ट्रूमेंटेशन समस्या)।
  • .NET: ADOT .NET में AWS SDK for .NET V4 के साथ सीमाएं हैं; पूर्ण सपोर्ट के लिए AWS SDK V3 अनुशंसित है।

मैनुअल इंस्ट्रूमेंटेशन

OpenTelemetry SDK के माध्यम से सभी प्रमुख प्रोग्रामिंग भाषाएं समर्थित हैं:

  • Java, Scala, Kotlin
  • Python, Ruby
  • JavaScript/TypeScript, Go
  • .NET (C#, F#, VB.NET)
  • PHP, Rust, C++

कॉन्फ़िगरेशन संदर्भ

सामान्य कॉन्फ़िगरेशन विकल्प और एनवायरनमेंट वेरिएबल।

एनवायरनमेंट वेरिएबल

OpenTelemetry कॉन्फ़िगरेशन:

  • OTEL_SERVICE_NAME - आपकी सर्विस का नाम
  • OTEL_TRACES_EXPORTER - ट्रेस एक्सपोर्टर (otlp, jaeger, zipkin)
  • OTEL_EXPORTER_OTLP_ENDPOINT - OTLP एंडपॉइंट URL
  • OTEL_TRACES_SAMPLER - सैंपलिंग रणनीति
  • OTEL_TRACES_SAMPLER_ARG - सैंपलिंग पैरामीटर

AWS-विशिष्ट कॉन्फ़िगरेशन:

  • AWS_REGION - AWS रीजन
  • AWS_XRAY_DAEMON_ADDRESS - X-Ray डेमन एड्रेस
  • OTEL_RESOURCE_ATTRIBUTES - सर्विस एट्रिब्यूट (Application Map में कस्टम ग्रुपिंग के लिए भी उपयोग किया जाता है)

सैंपलिंग कॉन्फ़िगरेशन

सैंपलिंग रणनीतियां:

  • always_on - सभी ट्रेसेज़ सैंपल करें
  • always_off - कोई ट्रेस सैंपल न करें
  • traceidratio - अनुपात के आधार पर सैंपल करें (जैसे, 10% के लिए 0.1)
  • parentbased_always_on - पैरेंट सैंपलिंग निर्णय का सम्मान करें
  • parentbased_traceidratio - पैरेंट-आधारित अनुपात सैंपलिंग

लागत अनुकूलन

Observability लागत को नियंत्रित करने की रणनीतियां।

सैंपलिंग रणनीतियां

  • प्रोडक्शन: 5-10% सैंपलिंग रेट
  • डेवलपमेंट: 50-100% सैंपलिंग रेट
  • एरर ट्रेसेज़: एरर वाले ट्रेसेज़ को हमेशा सैंपल करें
  • उच्च-मूल्य ऑपरेशन: महत्वपूर्ण पाथ के लिए 100% सैंपलिंग

डेटा रिटेंशन नीतियां

रिटेंशन अंतर्निहित टेलीमेट्री स्टोर (उदाहरण के लिए, CloudWatch मेट्रिक्स/लॉग्स और AWS X-Ray ट्रेसेज़) द्वारा नियंत्रित होता है। उन सर्विसेज़ में रिटेंशन कॉन्फ़िगर करें और इसे अपनी समस्या निवारण और अनुपालन आवश्यकताओं के साथ संरेखित करें।

रिसोर्स अनुकूलन

  • बैच एक्सपोर्ट: नेटवर्क ओवरहेड कम करें
  • कंप्रेशन: डेटा कंप्रेशन सक्षम करें
  • फ़िल्टरिंग: अनावश्यक स्पैन और एट्रिब्यूट हटाएं
  • बफ़रिंग: उचित बफ़र आकार का उपयोग करें
