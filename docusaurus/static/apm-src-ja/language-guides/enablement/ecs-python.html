概要

このガイドでは、 ステップバイステップの手順 を提供します。Python FastAPI アプリケーションを Amazon ECS に AWS Application Signals を有効にしてデプロイします。CDK 不要 - 純粋な AWS CLI と Docker を使用します。

達成できること：
  • Python FastAPI アプリを ECS Fargate にデプロイする
  • AWS Application Signals の自動インストルメンテーションを有効にする
  • CloudWatch Application Signals コンソールでサービスを確認する
  • サービスのメトリクス、トレース、オペレーションを監視する
完了までの時間： 30〜45 分

前提条件

必要なツール

  • AWS CLI v2（最新バージョン）
  • Docker Desktop
  • Python 3.11+
  • 適切な権限を持つ AWS アカウント
  • テキストエディタ

必要な AWS 権限

  • ECS: フルアクセス（クラスター、サービス、タスク定義の作成）
  • ECR: イメージのプッシュ
  • IAM: ロールおよびポリシーの作成
  • CloudWatch: ログおよび Application Signals へのアクセス
  • EC2: VPC、サブネット、セキュリティグループへのアクセス
  • SSM: Parameter Store への書き込みアクセス

前提とする知識

  • AWS CLI の基本的な使い方
  • Docker の基本的なコマンド
  • ECS の概念に関する基本的な理解

アーキテクチャ

ECS タスクには 3 つのコンテナ:

  1. Init コンテナ (init）: ADOT Python 自動インストルメンテーションファイルをコピーします
  2. CloudWatch Agent (ecs-cwagent): トレースとメトリクスを収集
  3. Application Container (your-app): FastAPI アプリケーション

フロー:

Init Container (runs once, copies files) 
  → Application starts (auto-instrumented via PYTHONPATH)
  → Sends traces/metrics to CloudWatch Agent (localhost:4316)
  → CloudWatch Agent sends to Application Signals

ステップ 1: アカウントで Application Signals を有効化する

重要: Application Signals は AWS アカウントごと、リージョンごとに一度有効化する必要があります。

オプション 1: AWS コンソールから有効化する（推奨）

  1. AWS コンソール → CloudWatch → Application Signals に移動する
  2. リージョンを選択する（例： us-east-1)
  3. 「Get Started」または「Enable Application Signals」をクリックする
  4. 有効化されるまで 2〜3 分待つ

オプション 2: すでに有効化されているか確認する

# Check if service-linked role exists
aws iam get-role \
  --role-name AWSServiceRoleForCloudWatchApplicationSignals \
  --region us-east-1

# If role exists, Application Signals is enabled

出力結果: 有効化されている場合はロール ARN、されていない場合はエラーが表示されます。

ステップ 2: Python FastAPI アプリケーションを準備する

2.1 アプリケーションディレクトリを作成する

mkdir -p ~/my-appsignals-app
cd ~/my-appsignals-app

2.2 作成する app.py

"""
Simple FastAPI application for AWS Application Signals
"""
import time
import random
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
import requests

app = FastAPI(
    title="My Application Signals App",
    description="FastAPI app with Application Signals integration",
    version="1.0.0"
)


class HealthResponse(BaseModel):
    status: str
    timestamp: float


@app.get("/", )
async def root():
    return {
        "message": "Application Signals Demo",
        "timestamp": time.time()
    }


@app.get("/health", response_model=HealthResponse)
async def health_check():
    """Health check endpoint"""
    return HealthResponse(
        status="healthy",
        timestamp=time.time()
    )


@app.get("/api/hello/{name}")
async def hello(name: str):
    """Personalized greeting with simulated processing"""
    time.sleep(random.uniform(0.01, 0.1))

    return {
        "message": f"Hello, {name}!",
        "timestamp": time.time()
    }


@app.get("/api/slow")
async def slow_endpoint():
    """Intentionally slow endpoint for latency testing"""
    time.sleep(random.uniform(0.5, 1.5))
    return {"message": "Slow operation completed"}


@app.get("/api/error")
async def error_endpoint():
    """Endpoint that randomly raises errors"""
    if random.random() > 0.5:
        raise HTTPException(status_code=500, detail="Simulated error")
    return {"message": "No error"}


@app.get("/api/external")
async def external_call():
    """Makes external HTTP call for distributed tracing"""
    try:
        response = requests.get("https://httpbin.org/delay/1", timeout=5)
        return {
            "message": "External call completed",
            "status_code": response.status_code
        }
    except Exception as e:
        raise HTTPException(status_code=500, detail=str(e))


if __name__ == "__main__":
    import uvicorn
    uvicorn.run(app, host="0.0.0.0", port=80)

2.3 作成する requirements.txt

# Core dependencies only - OpenTelemetry provided by ADOT init container
fastapi==0.115.0
uvicorn==0.32.1
pydantic==2.10.3
requests==2.32.3

重要: インストールしないでください opentelemetry-* パッケージは ADOT init コンテナによって提供されます。

2.4 作成する start.sh

#!/bin/bash
# Startup script for Application Signals

echo "Starting Application with Application Signals..."
echo "PYTHONPATH: ${PYTHONPATH:-not set}"
echo "Application Signals: ${OTEL_AWS_APPLICATION_SIGNALS_ENABLED:-not set}"

# Run uvicorn directly - NO opentelemetry-instrument wrapper needed
# NO --workers flag - auto-instrumentation requires single worker
exec uvicorn app:app --host 0.0.0.0 --port 80

実行可能にする:

chmod +x start.sh

2.5 作成する Dockerfile

FROM public.ecr.aws/docker/library/python:3.11-slim as builder

WORKDIR /app

# Install system dependencies
RUN apt-get update && apt-get install -y gcc && rm -rf /var/lib/apt/lists/*

# Install Python dependencies
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Copy application code
COPY . .

# Production stage
FROM public.ecr.aws/docker/library/python:3.11-slim

WORKDIR /app

# Install runtime dependencies
RUN apt-get update && apt-get install -y curl && rm -rf /var/lib/apt/lists/*

# Copy from builder
COPY --from=builder /usr/local/lib/python3.11/site-packages /usr/local/lib/python3.11/site-packages
COPY --from=builder /usr/local/bin /usr/local/bin
COPY --from=builder /app .

# Make start script executable
RUN chmod +x start.sh

# Health check
HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \
    CMD curl -f http://localhost:80/health || exit 1

EXPOSE 80

CMD ["./start.sh"]

ステップ 3: Docker イメージをビルドして ECR にプッシュする

3.1 変数を設定する

# Replace with your values
export AWS_REGION="us-east-1"
export AWS_ACCOUNT_ID="YOUR_ACCOUNT_ID"
export AWS_PROFILE="your-profile-name"  # Optional if using default
export APP_NAME="my-appsignals-app"

アカウント ID を取得する:

export AWS_ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
echo "Account ID: $AWS_ACCOUNT_ID"

3.2 ECR リポジトリを作成する

aws ecr create-repository \
  --repository-name ${APP_NAME} \
  --region ${AWS_REGION}

出力: リポジトリ URI（保存しておくこと）

3.3 Docker イメージをビルドする

cd ~/my-appsignals-app
docker build -t ${APP_NAME} .

3.4 ECR にプッシュする

# Login to ECR
aws ecr get-login-password --region ${AWS_REGION} | \
  docker login --username AWS --password-stdin \
  ${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com

# Tag image
docker tag ${APP_NAME}:latest \
  ${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com/${APP_NAME}:latest

# Push image
docker push ${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com/${APP_NAME}:latest

ステップ 4: CloudWatch Agent の設定を作成する

4.1 設定ファイルを作成する

cat > /tmp/ecs-cwagent.json << 'EOF'
{
  "traces": {
    "traces_collected": {
      "application_signals": {}
    }
  },
  "logs": {
    "metrics_collected": {
      "application_signals": {}
    }
  }
}
EOF

4.2 SSM パラメータストアにアップロードする

aws ssm put-parameter \
  --name "ecs-cwagent-${APP_NAME}" \
  --type "String" \
  --value "$(cat /tmp/ecs-cwagent.json)" \
  --region "${AWS_REGION}" \
  --overwrite

出力: バージョン番号（新規の場合は 1、更新の場合はインクリメントされた値）

ステップ 5: IAM ロールを作成する

5.1 信頼ポリシーの作成

cat > /tmp/task-trust-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "ecs-tasks.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
EOF

5.2 タスクロールの作成

aws iam create-role \
  --role-name ${APP_NAME}-ecs-task-role \
  --assume-role-policy-document file:///tmp/task-trust-policy.json

# Attach required policies
aws iam attach-role-policy \
  --role-name ${APP_NAME}-ecs-task-role \
  --policy-arn arn:aws:iam::aws:policy/CloudWatchAgentServerPolicy

aws iam attach-role-policy \
  --role-name ${APP_NAME}-ecs-task-role \
  --policy-arn arn:aws:iam::aws:policy/AWSXRayDaemonWriteAccess

5.3 タスク実行ロールの作成

aws iam create-role \
  --role-name ${APP_NAME}-ecs-execution-role \
  --assume-role-policy-document file:///tmp/task-trust-policy.json

# Attach policies
aws iam attach-role-policy \
  --role-name ${APP_NAME}-ecs-execution-role \
  --policy-arn arn:aws:iam::aws:policy/service-role/AmazonECSTaskExecutionRolePolicy

aws iam attach-role-policy \
  --role-name ${APP_NAME}-ecs-execution-role \
  --policy-arn arn:aws:iam::aws:policy/AmazonSSMReadOnlyAccess

5.4 CloudWatch Logs 権限の追加

cat > /tmp/logs-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogGroup",
        "logs:CreateLogStream",
        "logs:PutLogEvents"

EOF

aws iam put-role-policy \
  --role-name ${APP_NAME}-ecs-execution-role \
  --policy-name CloudWatchLogsPolicy \

ステップ 7: ネットワークのセットアップ

7.1 VPC とサブネットの取得

既に VPC をお持ちの場合:

# List VPCs
aws ec2 describe-vpcs --region ${AWS_REGION} \
  --query 'Vpcs[*].{VpcId:VpcId,CIDR:CidrBlock,Name:Tags[?Key==`Name`].Value|[0]}'

# Choose your VPC
export VPC_ID="vpc-xxxxx"

# Get private subnets
aws ec2 describe-subnets --region ${AWS_REGION} \
  --filters "Name=vpc-id,Values=${VPC_ID}" \
  --query 'Subnets[?MapPublicIpOnLaunch==`false`].[SubnetId,AvailabilityZone,CidrBlock]'

# Save subnet IDs
export SUBNET_1="subnet-xxxxx"
export SUBNET_2="subnet-yyyyy"

7.2 セキュリティグループの作成

export SG_NAME="${APP_NAME}-ecs-sg"

# Create security group
SG_ID=$(aws ec2 create-security-group \
  --group-name ${SG_NAME} \
  --description "Security group for ${APP_NAME} ECS tasks" \
  --vpc-id ${VPC_ID} \
  --region ${AWS_REGION} \
  --query 'GroupId' \
  --output text)

echo "Security Group ID: $SG_ID"
export SG_ID

# Add ingress rule (allow HTTP from anywhere for testing)
aws ec2 authorize-security-group-ingress \
  --group-id ${SG_ID} \
  --protocol tcp \
  --port 80 \
  --cidr 0.0.0.0/0 \
  --region ${AWS_REGION}

本番環境の場合: 置き換えてください 0.0.0.0/0 特定の IP 範囲またはロードバランサーのセキュリティグループに置き換えてください。

ステップ 12: AWS コンソールで確認する

コンソールアクセス

  1. 移動先:

    AWS Console → CloudWatch → Application Signals → Services

    または直接リンク:

    https://${AWS_REGION}.console.aws.amazon.com/cloudwatch/home?region=${AWS_REGION}#application-signals:services

  2. サービスを探す:

    • サービス名: の値 ${APP_NAME}
    • 環境: ecs:${AWS_REGION}

  3. 探索:

    • サービスマップ: ビジュアルトポロジー
    • サービスダッシュボード: レイテンシー、エラー、フォルト
    • オペレーション: エンドポイントごとのメトリクス
    • 依存関係: ダウンストリームサービスの呼び出し

トラブルシューティング

問題: Application Signals にサービスが表示されない

確認事項:

# 1. Verify metrics are being published
aws cloudwatch list-metrics \
  --namespace ApplicationSignals \
  --region ${AWS_REGION} \
  --dimensions Name=Service,Value=${APP_NAME}

# 2. Check if Application Signals is enabled
aws iam get-role \
  --role-name AWSServiceRoleForCloudWatchApplicationSignals

# 3. Check container logs for errors
aws logs tail /ecs/${APP_NAME} --region ${AWS_REGION} --since 10m

問題: Init コンテナが失敗する

# Describe task to see init container exit code
aws ecs describe-tasks \
  --cluster ${CLUSTER_NAME} \
  --tasks ${TASK_ARN} \
  --region ${AWS_REGION} \
  --query 'tasks[0].containers[?name==`init`].{name:name,exitCode:exitCode,reason:reason}'

期待される結果: exitCode: 0

ゼロ以外の場合: イメージが利用できない可能性があります。タグを確認してください v0.12.1 存在するか確認:

docker pull public.ecr.aws/aws-observability/adot-autoinstrumentation-python:v0.12.1

問題: アプリケーションが CloudWatch Agent に接続できない

症状:

ConnectionRefusedError: [Errno 111] Connection refused

確認事項:

  1. CloudWatch agentコンテナがRUNNINGであることを確認する
  2. コンテナの依存関係を確認する - appは以下に依存する必要があります init 条件SUCCESS付き
  3. 環境変数を確認する（特にエンドポイント: http://localhost:4316 および http://localhost:2000)

問題: X-Rayにトレースが表示されない

よくある原因:

  1. 不足している OTEL_PYTHON_DISTRO=aws_distro - Application Signalsに必須
  2. 不足している OTEL_TRACES_SAMPLER_ARG - X-Rayサンプラーエンドポイントが必要
  3. 複数のワーカーを使用している - 単一ワーカーで実行する必要があります（uvicorn app:app なしで --workers)
  4. 使用している opentelemetry-instrument ラッパー - 使用しないでください。インストルメンテーションはPYTHONPATHに任せてください

問題: IAMパーミッションエラー

症状:

AccessDeniedException: ... logs:CreateLogGroup

修正方法: CloudWatch Logsのパーミッションを実行ロールに追加する（手順5.4を参照）

Initコンテナの詳細

イメージ: public.ecr.aws/aws-observability/adot-autoinstrumentation-python:v0.12.1

バージョン付きタグを使用する理由（ :latest?

  • タグはPython ADOTイメージには存在しません :latest ではなく
  • 特定のバージョンタグを使用します（確認してください https://github.com/aws-observability/aws-otel-python-instrumentation/releases)
  • 現在の安定版： v0.12.1

機能の説明：

  1. アプリコンテナが起動する前に一度実行されます
  2. ADOT Python インストルメンテーションファイルをコピーします（コピー元： /autoinstrumentation/ 共有ボリュームへ
  3. コード 0 で終了します
  4. アプリコンテナはボリュームをマウントし、PYTHONPATH を使用してインストルメンテーションを読み込みます。

アプリケーションコードの要件

実行してください：

  • 通常の FastAPI コードを記述します。
  • 標準ライブラリ（requests、psycopg2 など）を使用します。
  • コードをクリーンに保つ - OpenTelemetry のインポートは不要です

やってはいけないこと：

  • インポート opentelemetry-* パッケージ
  • 呼び出し FastAPIInstrumentor.instrument_app()
  • 使用 opentelemetry-instrument CMD のラッパー
  • 以下で実行します。 --workers フラグ（ワーカーは 1 つである必要があります）
  • `requirements.txt` に OpenTelemetry パッケージをインストールします。

クリーンアップ

すべてのリソースを削除するには、次の手順を実行します。

# Delete ECS service
aws ecs update-service \
  --cluster ${CLUSTER_NAME} \
  --service ${APP_NAME}-service \
  --desired-count 0 \
  --region ${AWS_REGION}

aws ecs delete-service \
  --cluster ${CLUSTER_NAME} \
  --service ${APP_NAME}-service \
  --region ${AWS_REGION}

# Delete ECS cluster
aws ecs delete-cluster \
  --cluster ${CLUSTER_NAME} \
  --region ${AWS_REGION}

# Delete ECR repository
aws ecr delete-repository \
  --repository-name ${APP_NAME} \
  --force \
  --region ${AWS_REGION}

# Delete security group
aws ec2 delete-security-group \
  --group-id ${SG_ID} \
  --region ${AWS_REGION}

# Delete IAM roles
aws iam delete-role \
  --role-name ${APP_NAME}-ecs-task-role

aws iam delete-role \
  --role-name ${APP_NAME}-ecs-execution-role

# Delete SSM parameter
aws ssm delete-parameter \
  --name ecs-cwagent-${APP_NAME} \
  --region ${AWS_REGION}
