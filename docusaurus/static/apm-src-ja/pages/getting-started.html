CloudWatch APM ドキュメント

CloudWatch APM を使ってみる

概要

Application Performance Monitoring (APM) は、現代の分散アプリケーションの健全性とパフォーマンスを把握・最適化するために不可欠です。APM がなければ、チームはレスポンスタイムの遅延、頻繁な障害、複雑なマイクロサービスアーキテクチャにおける問題の特定に苦労します。Amazon CloudWatch Application Signals は、アプリケーションを自動的にインストルメント化し、テレメトリデータを収集し、事前構築済みダッシュボードとインテリジェントなアラートを通じて実用的なインサイトを提供する包括的なオブザーバビリティによって、これらの課題を解決します。

CloudWatch Application Signals は、迅速な健全性評価のためのゴールデンメトリクス（レイテンシー、スループット、エラーレート、サチュレーション）、詳細なリクエストフロー分析のためのトレースおよびスパン分析、完全な OpenTelemetry (OTel) 互換性など、主要な APM 機能をサポートしています。OpenTelemetry は、テレメトリデータの収集とエクスポートにベンダー中立な標準を提供し、アプリケーションを一度インストルメント化するだけで、ロックインなしに複数のオブザーバビリティプラットフォームにデータを送信できるため、非常に重要です。これにより、環境全体で一貫したモニタリングが可能になり、平均解決時間 (MTTR) が短縮され、問題の事前検出と自動化されたインサイトによって安定した信頼性の高いアプリケーションが実現し、開発から本番環境への移行が加速されます。

主な機能

混乱から明確さへ

ゼロ設定でシステムアーキテクチャ全体を可視化してトラブルシューティングします。インストルメント済みおよび未インストルメントのサービスの自動検出、AWS アカウントとリージョンをまたいだ分散アプリケーションの自動マッピング、クロスアカウントの可視性、一般的な問題に対する自動運用監査、アプリケーションへの最近の変更により、依存関係を把握し、問題をより迅速に解決できます。

すべてのトレースからビジネスインサイトを

直感的なビジュアルクエリビルダーを使用してトランザクション分析を簡素化します。トランザクショントレーシングをビジネス目標に結び付けることでビジネスインサイトを浮き彫りにします。最小限のセットアップとワンクリック分析で、ビジネスクリティカルなトランザクションのエンドツーエンドのビューを即座に取得できます。

ソースコードと連携した本番テレメトリ

VS Code や GitHub などの IDE でツールを切り替えたりクエリ言語を習得したりすることなく、自然言語を使用して本番環境の問題を調査します。「なぜ請求サービスが支払い処理に失敗しているのか？」のような平易な英語の質問をするだけで、発火中のアラームを調べ、ログの異常を調査し、メトリクスとデプロイメントを相関させ、問題の原因となっているコードを数秒で特定する AI を活用した分析が得られます。GitHub Issues で @awsapm を使用することで、ソースコードと並行してライブテレメトリの自動分析をトリガーし、詳細な修正案を受け取り、AI に正確な解決策を含むプルリクエストを提出させることができます。これにより、本番テレメトリをコードを書く場所に直接持ち込むことができます。

顧客を満足させ、サービスを維持する

SLO と SLI を使用してサービスの信頼性を追跡することで、顧客を満足させ続けます。CloudWatch の統合コンソールを使用してリアルタイムの SLO ステータスとコンプライアンスを追跡し、一貫したサービス品質を提供します。信頼性の問題が発生する前に CloudWatch でビジネス重視の SLO アラートを設定することで、顧客の不満を防ぎます。

ユーザー行動を把握する

Web およびモバイルアプリケーションのリアルユーザーモニタリング (RUM) を追跡することで、顧客がサイトをどのように体験しているかを正確に把握します。CloudWatch で重要な顧客ジャーニーをシミュレートするシンセティックモニタリング（カナリア）を実行することで、顧客より先にサイトの問題を検出します。CloudWatch の組み込みダッシュボードを使用してパフォーマンスの問題とトレンドを即座に把握し、顧客体験を改善するための意思決定を迅速に行います。

CloudWatch Application Performance Monitoring (APM) は、Application Signals（当社のフラッグシップオブザーバビリティプラットフォーム）を通じて、分散システム全体でアプリケーションを把握、トラブルシューティング、最適化するのに役立ちます。コードをデバッグする開発者、本番システムを維持するオペレーター、オブザーバビリティ戦略を設計するアーキテクトのいずれであっても、このガイドを使用してすぐに始めることができます。

どこから始めればよいですか？

開発者 → ジャンプ先 最初のアプリケーションをインストルメント化する
オペレーター → ジャンプ先 本番システムを監視する
アーキテクト → ジャンプ先 オブザーバビリティ戦略を設計する

CloudWatch APM とは何ですか？

CloudWatch Application Performance Monitoring (APM) は、Application Signals（バックエンドサービスからフロントエンドのユーザー体験まで、アプリケーションへのエンドツーエンドの可視性を提供する当社のフラッグシップ製品）上に構築された AWS の包括的なオブザーバビリティソリューションです。

Application Signals は、アプリケーショントポロジーを自動的に検出し、コードをインストルメント化し、完全なオブザーバビリティを提供するために連携して機能する 3 つの統合モニタリング機能を提供します。

Application Performance Monitoring (APM) の 3 つの柱

CloudWatch Application Signals

CloudWatch Application Signals は、アプリケーショントポロジーを自動的に検出し、コードをインストルメント化し、完全なオブザーバビリティを提供するために連携して機能する 3 つの統合モニタリング機能を提供する AWS のフラッグシップオブザーバビリティプラットフォームです。

提供される機能:

  • サービスの自動検出とインストルメント化
  • 1 つのプラットフォームに統合された APM、RUM、Synthetics
  • 事前構築済みダッシュボードと自動化されたインサイト
  • クロスアカウントおよびクロスリージョンの可視性

Real User Monitoring (RUM)

本番環境での実際のユーザー体験をキャプチャします。さまざまなブラウザ、デバイス、地域にわたって実際のユーザーが Web アプリケーションとどのようにインタラクションするかを監視します。RUM はページパフォーマンス、JavaScript エラー、Core Web Vitals を追跡し、ユーザー体験の把握と最適化を支援します。

提供される機能:

  • ページ読み込み時間とナビゲーションパフォーマンス
  • JavaScriptエラーとスタックトレース
  • Core Web Vitals（LCP、FID、CLS）のトラッキング
  • ユーザーセッションのリプレイとジャーニー分析
  • 地域およびデバイスに基づくパフォーマンスインサイト

Synthetic Monitoring

ユーザーより先に重要なフローをプロアクティブにテストします。複数のAWSリージョンから24時間365日自動テストを実行し、ユーザーへの影響が生じる前に問題を検出します。Syntheticカナリアは、APIの機能を検証し、重要なユーザージャーニーを監視し、アプリケーションがSLAを満たしていることを確認します。

取得できる情報：

  • スケジュールされたAPIヘルスチェック（1〜60分ごと）
  • 重要なユーザーフローに対するスクリプト化されたブラウザテスト
  • マルチリージョンの可用性監視
  • ベースラインパフォーマンスのトラッキング
  • ユーザーへの影響が生じる前のプロアクティブなアラート

連携の仕組み

Application Signalsの強みは、これら3つの機能が互いを補完し合い、完全な可視性を提供する点にあります。

シナリオ：インシデントの完全な調査

Syntheticsがチェックアウトフローの失敗を検出 → アラートが発火
APMトレースがpaymentサービスで例外がスローされていることを示す
RUMが実際のユーザーも同じ障害を経験していることを確認
解決策：APMトレースにより、最近のデプロイでバグが混入したことが判明

シナリオ：パフォーマンスの最適化

RUMがモバイルでのページ読み込みの遅さを特定（LCP 4.5秒）
APMトレースがバックエンドAPIは高速であることを示す（200ms）
RUMがボトルネックは3MBのヒーロー画像であることを明らかにする
Syntheticsが最適化後の修正を検証

Application Signalsを始める

Application Signalsでは、開始にあたってコードを変更する必要はありません。AWS環境で有効化するだけで、アプリケーションが自動的にインストルメント化され、サービスが検出されます。有効化後、Application Signalsは追加の設定なしにアプリケーショントポロジの変更を自動検出し続けます。

すべてを一度に有効化する必要はありません。推奨される手順は以下のとおりです：

フェーズ1：APMから開始する（第1週）

  • バックエンドサービスのApplication Signalsを有効化する
  • サービスの依存関係を可視化する
  • パフォーマンスのベースラインを確立する

フェーズ2：Syntheticsを追加する（第2〜3週）

  • 重要なユーザージャーニーに対してカナリアを作成する
  • プロアクティブな監視を設定する
  • 可用性のベースラインを確立する

フェーズ3：RUMを有効化する（第4週以降）

  • WebアプリケーションにRUMを追加する
  • 実際のユーザーエクスペリエンスをトラッキングする
  • バックエンドのパフォーマンスと相関させる

Application Signals全体の主要機能

  • 自動サービス検出
  • 分散トレーシング
  • リアルタイムメトリクス
  • アプリケーションマップ
  • SLO トラッキング
  • インテリジェントアラート

開発者向け：最初のアプリケーションをインストルメント化する

達成できること
15 分以内に、Application Signals がバックエンドサービスから APM トレースとメトリクスを収集できるようになります。サポートされているランタイムであればコードの変更は不要です。

前提条件

  • AWS 上で動作するアプリケーション（ECS、EKS、EC2、または Lambda）
  • サポートされているランタイム：Java、Python、.NET、Node.js、または Go
  • Application Signals を有効にするための IAM 権限

ステップ 1：インストルメンテーション方法を選択する

オプション A：自動インストルメンテーション（推奨）

  • コードの変更は不要
  • Java、Python、.NET、Node.js に対応
  • APM メトリクスとトレースの自動収集

オプション B：OpenTelemetry 手動インストルメンテーション

  • インストルメント化する内容を完全に制御可能
  • 複数のオブザーバビリティプラットフォームに移植可能
  • APM、RUM、および Synthetics 向けのカスタムスパンと属性

オプション C：AWS Distro for OpenTelemetry (ADOT)

  • AWS に最適化された OpenTelemetry
  • Application Signals 向けに事前設定済み

ステップ 2：Application Signals を有効にする

デベロッパーガイドのプラットフォーム別の有効化手順（EKS、ECS、Lambda）に従ってください。有効化後、コンソールにデータが表示されるまで 5〜10 分お待ちください。

ステップ 3：APM データ収集を確認する

  1. データが表示されるまで 5〜10 分待つ
  2. CloudWatch コンソール → Application Signals を開く
  3. アプリケーションマップを確認する — APM メトリクスとともにサービスが表示されているはずです
  4. サービスをクリックして、分散トレースとパフォーマンスメトリクスを表示する

ステップ 4：カスタムインストルメンテーションを追加する（オプション）

カスタムスパンとビジネスコンテキストを使用して APM トレースを強化します（言語別のサンプルはデベロッパードキュメントをご参照ください）。

ステップ 5：フロントエンドモニタリング用に RUM を有効にする（オプション）

RUM ウェブクライアントを追加してユーザーエクスペリエンスを追跡します。

ステップ 6：Synthetic カナリアを作成する（オプション）

Synthetic テストを使用してプロアクティブなモニタリングを設定します。

開発者向けトラブルシューティング

  • APM トレースが表示されない場合は、IAM 権限（cloudwatch:PutMetricData、xray:PutTraceSegments）を確認し、エージェントが実行中であることを確認してください（kubectl logs -n amazon-cloudwatch ...）。
  • RUMデータが表示されない場合は、RUMスニペットが他のスクリプトより前に読み込まれていることを確認し、ブラウザコンソールでエラーをチェックしてください。
  • Syntheticカナリアが失敗する場合は、カナリアログを確認してテストスクリプトをローカルで実行し、カナリアVPCからのネットワーク接続を確認してください。
  • オーバーヘッドが高い場合は、APMサンプリングレートを調整し、ヘッドベースサンプリングまたはRUMセッションのサンプリングを検討してください。

オペレーター：本番システムの監視

達成できること
30分で、本番アプリケーションのAPM、RUM、Syntheticsにわたる包括的な監視、アラート、ダッシュボードを設定します。

監視チェックリスト

サービスヘルス監視（APM）

  • アプリケーションマップにすべての重要なバックエンドサービスが表示されている
  • 主要なAPMメトリクスダッシュボードが作成されている（レイテンシー、エラー、スループット）
  • 重要なサービスのSLOが定義されている
  • SLO違反に対するアラートが設定されている

ユーザーエクスペリエンス監視（RUM）

  • 本番WebアプリケーションでRUMが有効になっている
  • Core Web Vitalsトラッキングが設定されている
  • フロントエンドエラーアラートが設定されている
  • 地理的パフォーマンスのベースラインが確立されている

プロアクティブ監視（Synthetics）

  • 重要なユーザージャーニーに対してカナリアが作成されている
  • APIヘルスチェックが5分ごとに実行されている
  • マルチリージョンの可用性監視が有効になっている
  • Syntheticテスト失敗アラートが設定されている

ステップ1：統合ダッシュボードの作成

Application Signalsダッシュボードテンプレートを使用し、APMゴールデンシグナル、RUMユーザーエクスペリエンス、Syntheticsヘルス、サービス依存関係のウィジェットを追加します。

ステップ2：インテリジェントアラートの設定

APM、RUM、Syntheticsにまたがるコンポジットアラームを作成します（例はドキュメントを参照）。

ステップ3：ランブック自動化の実装

アラートにランブックをリンクし、安全な範囲で自動修復を実装します（例：コンテナの再起動、高レイテンシー時のスケーリング、CDNキャッシュの無効化）。

ステップ4：運用レビューの確立

週次メトリクスレビューと月次詳細分析（コスト最適化、アラート疲労分析、インシデント振り返り）を実施します。

アーキテクト：オブザーバビリティ戦略の設計

達成できること
組織の今後2〜3年のニーズをサポートする、スケーラブルでコスト効率の高いApplication Signalsアーキテクチャを設計します。

アーキテクチャ決定フレームワーク

採用戦略

推奨事項：バックエンドの可視性のためにまずAPMを導入し、プロアクティブ監視のためにSyntheticsを追加し、その後ユーザーエクスペリエンスのインサイトのためにRUMを有効にします。

インストルメンテーション戦略

推奨事項：柔軟性を維持するために新しいアプリケーションにはOpenTelemetryを使用し、Application Signalsの迅速な導入にはAWSネイティブを使用します。

マルチクラウド戦略

オプションには、AWSのみの場合はApplication Signals、統合マルチクラウドアプローチの場合はOpenTelemetry、ハイブリッドな柔軟性のためのOTelコレクターブリッジが含まれます。

データ保持戦略

  • APM トレースデータ: ホット（7日間）、ウォーム（30日間）、コールド（90日以上のアーカイブ）
  • RUM セッションデータ: ホット（30日間）、コールド（90日以上）
  • Synthetics 結果: ホット（90日間）、コールド（1年以上）
  • 解像度別メトリクス保持期間: 1分（15日間）、5分（63日間）、1時間（15ヶ月）

リファレンスアーキテクチャパターン

例: ADOT を使用した EKS 上のマイクロサービス、Lambda レイヤーを使用したサーバーレス、OTel コレクターを使用したハイブリッドクラウド。

キャパシティプランニング

オブザーバビリティのコストを見積もり、サンプリング/フィルタリングおよびアーカイブポリシーを使用して支出を最適化する。

セキュリティとコンプライアンス

  • APM トレース、RUM データ、および Synthetics 結果を保存時および転送時に暗号化する
  • トレースおよび RUM セッション内の PII に対してデータマスキングを実装する
  • きめ細かいアクセス制御に IAM を使用する
  • 監査ログのために CloudTrail を有効にする

コンプライアンスに関する考慮事項: HIPAA、PCI-DSS、GDPR、SOC 2 — それぞれのケースでデータ保持およびマスキングのガイダンスに従う。

移行計画

X-Ray から Application Signals へ

  1. 並行稼働（2〜4週間）: X-Ray と並行して Application Signals を有効化し、RUM を追加し、カナリアを作成して、カバレッジを比較する
  2. 段階的移行（1〜2ヶ月）: 非クリティカルなサービスを移行し、RUM を拡張し、ダッシュボードを更新する
  3. 完全切り替え（1ヶ月）: 残りのサービスを移行し、X-Ray を廃止する

サードパーティ APM から

  • 現在のインストルメンテーションを評価する
  • 必要に応じて OTel 移行を計画する
  • 移行期間中はデュアルライトを検討する
  • 移行には3〜6ヶ月の予算を確保する

チームの有効化

  • 第1週: 基礎ワークショップ（APM、RUM、Synthetics の概要）
  • 第2〜3週: ハンズオンラボ（サービスのインストルメント、RUM の有効化、カナリアの作成）
  • 第4週: 3つのシグナルすべてを使用した本番パイロット
  • 第2ヶ月以降: オフィスアワーとサポート

卓越性センターを設立する: チャンピオン、社内ドキュメント、再利用可能なダッシュボード、およびベストプラクティス。

一般的なシナリオとソリューション

シナリオ: 遅い API エンドポイントのデバッグ

開発者のアプローチ: RUM を確認し、遅い APM トレースを見つけ、ボトルネックスパンを特定し、カスタムインストルメンテーションを追加し、RUM と Synthetics でデプロイして検証する。

オペレーターのアプローチ: Synthetics を確認し、APM トレースをレビューし、ユーザーへの影響を RUM で確認し、相関するインフラの問題を探し、証拠とともにエスカレーションする。

シナリオ: ブラックフライデーの準備

キャパシティプランニングをレビューし、シグナル全体で SLO を確立し、スケーリングポリシーを設定し、ランブックを準備する。

アプリケーションパフォーマンスモニタリングは初めてですか？

15分間の Node.js クイックスタート
実行中のサンプルを取得し、15分以内に最初のトレースを確認する。

  • サンプルをクローンしてインストールします。 cd samples/nodejs-quickstart && npm install
  • ローカルの OpenTelemetry Collector (logging exporter) を実行します。参照してください otel-config.yaml サンプルに含まれています。
  • アプリを起動します。 npm start トラフィックを生成します。 ./generate_requests.sh

サンプル README を開きます。 詳細な手順については、

アプリケーションの健全性について質問する

Application Signals MCP サーバーを使用して、自然言語でアプリケーションの健全性をクエリします。AI アシスタントまたは IDE に接続することで、APM、RUM、および Synthetics データから即座にインサイトを取得できます。

  • MCP サーバーをインストールします。 リファレンスドキュメント
  • Application Signals エンドポイントを使用して設定します。
  • 「なぜサービスが遅いのか？」や「エラーのトレンドを表示して」といった質問をすることができます。

複雑なクエリを使用せずに、トレース、メトリクス、およびアプリケーションマップへの会話形式のアクセスを取得できます。
