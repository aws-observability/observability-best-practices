CloudWatch APM ドキュメント

Java Spring + OTLP + CloudWatch (ハンズオン演習)

このガイドでは、Java Spring Boot サービスを OpenTelemetry でインストルメント化し、テレメトリを AWS にエクスポートし、ECS/Fargate にデプロイして、トレース/ログ/Application Signals の動作を確認するための完全な手順を説明します。

実装例のスタイルで書かれており、ランブックやワークショップの配布資料として使用できます。

1) この演習で示す内容

  • Java アプリのエンドツーエンドテレメトリ：
    • トレース (X-Ray / Transaction Search)
    • ログ (CloudWatch Logs OTLP エンドポイントまたは awslogs ECS 上で)
    • メトリクス/Application Signals (CloudWatch Agent サイドカーパス)
  • 以下の違い：
    • Micrometer トレーシングブリッジ (micrometer-tracing-bridge-otel) トレーシング API ブリッジング用
    • Micrometer レジストリ (micrometer-registry-cloudwatch2, micrometer-registry-otlp) はメトリクス専用のシンクです
  • 環境変数による OTLP エンドポイントの設定 (OTEL_*)、ハードコードされた Java 定数ではありません。

4) OTLP エンドポイントマッピング

シグナル エンドポイントパターン 一般的な環境変数
トレース https://xray.<region>.amazonaws.com/v1/traces OTEL_EXPORTER_OTLP_TRACES_ENDPOINT, OTEL_EXPORTER_OTLP_TRACES_PROTOCOL=http/protobuf
メトリクス https://monitoring.<region>.amazonaws.com/v1/metrics OTEL_EXPORTER_OTLP_METRICS_ENDPOINT （またはサイドカー経由の App Signals パス）
ログ https://logs.<region>.amazonaws.com/v1/logs OTEL_EXPORTER_OTLP_LOGS_ENDPOINT, OTEL_EXPORTER_OTLP_LOGS_HEADERS

ECS サイドカーモードでは、アプリコンテナはトレース/メトリクスを次の場所に送信します。 http://localhost:4316/... CloudWatch Agent が転送を処理します。

5) このリポジトリで使用されているインストルメンテーションモデル

5.1 依存関係

  • micrometer-tracing-bridge-otel に存在します pom.xml.
  • 依存関係なし micrometer-registry-cloudwatch2 または micrometer-registry-otlp トレースのために。

5.2 自動 + 手動トレーシング

  • ADOTJavaエージェントによる自動計装は、フレームワーク/クライアントスパンをキャプチャします。
  • 手動エンリッチメントは内部に追加されます DemoService.greet():
    • attributes: demo.operation, demo.request.name, demo.downstream.status, demo.downstream.elapsed_ms, demo.custom.instrumented
    • events: demo.greet.start, demo.httpbin.success, demo.httpbin.failure
    • によるエラー記録 recordException(...) および setStatus(ERROR, ...)

6) スクリーンショット（トレース詳細）

以下に提供されたスクリーンショットをワークショップ/デモドキュメントに使用してください：

7) AWSコンソールでテレメトリを確認する場所

トレース（主要）

  1. X-Ray → トレース
  2. リージョン： us-east-1
  3. 時間範囲：直近15〜60分
  4. サービス： java-spring-otel-poc

カスタムフィールドの場合：

  • トレースを開く → セグメント java-spring-otel-poc → メタデータ
  • キーを探す：
    • demo.operation
    • demo.request.name
    • demo.downstream.status
    • demo.downstream.elapsed_ms
    • demo.custom.instrumented

ログ

  • ECSアプリログ： /ecs/java-spring-otel
  • CWエージェントサイドカーログ： /ecs/ecs-cwagent-java-spring-otel
  • ローカルOTLPロググループ（ローカル実行時）： /aws/otel/java-spring-otel-poc

Application Signals

  • CloudWatch → Application Signals → サービス
  • 次を探す java-spring-otel-poc サイドカーパスを通じた継続的なトラフィックとメトリクスの正常な転送の後。

8) 重要なポイント

OpenTelemetryの自動計装はスパンを作成できますが、AWSオブザーバビリティの成果を成功させるには以下が必要です：

  1. OTLPエンドポイントの正しい配線
  2. 正しい認証/転送パス（メトリクス/App Signalsにはサイドカー/コレクター経由が多い）
  3. トレーシングブリッジとメトリクスレジストリの正しい区別
  4. 継続的なトラフィックと正しいコンソールビュー（トレースにはX-Ray）
