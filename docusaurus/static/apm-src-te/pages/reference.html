CloudWatch Application Signals కోసం పరిమితులు మరియు కోటాలను అర్థం చేసుకోవడం.

పరిమితులు మరియు కోటాలు

Application Signals CloudWatch మరియు AWS X-Ray పై ఆధారపడుతుంది. పరిమితులు (ఇన్‌జెషన్, క్వెరీ రేట్లు మరియు కోటాలు) ఆ సేవలచే నిర్వహించబడతాయి మరియు కాలక్రమేణా మారవచ్చు. ప్రస్తుత విలువల కోసం అధికారిక డాక్యుమెంటేషన్‌ను ఉపయోగించండి.

  • Application Signals: Application Signals డాక్స్
  • CloudWatch కోటాలు: CloudWatch సేవ కోటాలు
  • AWS X-Ray కోటాలు: X-Ray API థ్రాట్లింగ్ మరియు పరిమితులు

CloudWatch మెట్రిక్ నిల్వ (సాధారణ సూచన)

CloudWatch వివిధ రిజల్యూషన్‌లలో మెట్రిక్ డేటాను నిల్వ చేస్తుంది. పూర్తి నిల్వ మోడల్ కోసం CloudWatch డాక్స్ చూడండి.

  • 1-నిమిషం రిజల్యూషన్: 15 రోజులు
  • 5-నిమిషాల రిజల్యూషన్: 63 రోజులు
  • 1-గంట రిజల్యూషన్: 15 నెలలు

సేవ కోటాలు

Application Signals:

  • ఖాతాకు సేవలు: 1,000
  • సేవకు ఆపరేషన్‌లు: 100
  • కస్టమ్ మెట్రిక్స్: సేవకు 30

RUM:

  • ఖాతాకు యాప్ మానిటర్‌లు: 20
  • సెకనుకు ఈవెంట్‌లు: 10,000
  • ఈవెంట్ పరిమాణం: 64 KB

Synthetics:

  • ఖాతాకు కానరీలు: 100
  • కానరీ అమలు సమయం: గరిష్టంగా 15 నిమిషాలు
  • కానరీ పరిమాణం: 1 GB (అన్‌జిప్డ్)

పదకోశం

CloudWatch Application Signals లో కీలక పదాలు మరియు భావనలు.

ప్రధాన భావనలు

Application Performance Monitoring (APM)

సాఫ్ట్‌వేర్ అప్లికేషన్‌ల పనితీరు మరియు అందుబాటును పర్యవేక్షించడం మరియు నిర్వహించడం యొక్క అభ్యాసం.

డిస్ట్రిబ్యూటెడ్ ట్రేసింగ్

డిస్ట్రిబ్యూటెడ్ సిస్టమ్‌ల ద్వారా అభ్యర్థనలు ప్రవహించేటప్పుడు వాటిని ట్రాక్ చేసే పద్ధతి, బహుళ సేవల అంతటా అభ్యర్థనల మార్గం మరియు పనితీరును అర్థం చేసుకోవడానికి మిమ్మల్ని అనుమతిస్తుంది.

సర్వీస్ మ్యాప్

సేవలు ఎలా కనెక్ట్ అవుతాయి మరియు ఇంటరాక్ట్ చేస్తాయో చూపించే మీ అప్లికేషన్ ఆర్కిటెక్చర్ యొక్క దృశ్య ప్రాతినిధ్యం.

Span

ట్రేస్‌లో ఒకే ఆపరేషన్, డేటాబేస్ క్వెరీ లేదా API కాల్ వంటి పని యూనిట్‌ను సూచిస్తుంది.

ట్రేస్

మీ అప్లికేషన్ ద్వారా అభ్యర్థన యొక్క పూర్తి ఎండ్-టు-ఎండ్ ప్రయాణం, బహుళ span లను కలిగి ఉంటుంది.

ఇన్‌స్ట్రుమెంటేషన్ పదాలు

ఆటో-ఇన్‌స్ట్రుమెంటేషన్

కోడ్ మార్పులు అవసరం లేకుండా అప్లికేషన్‌లకు observability కోడ్‌ను స్వయంచాలకంగా జోడించడం.

మాన్యువల్ ఇన్‌స్ట్రుమెంటేషన్

అప్లికేషన్ సోర్స్ కోడ్‌కు ట్రేసింగ్ మరియు మెట్రిక్స్ కోడ్‌ను స్పష్టంగా జోడించడం.

OpenTelemetry

ఇన్‌స్ట్రుమెంటేషన్ కోసం వెండర్-న్యూట్రల్ APIలను అందించే ఓపెన్-సోర్స్ observability ఫ్రేమ్‌వర్క్.

శాంప్లింగ్

observability ని ఖర్చు మరియు పనితీరుతో సమతులం చేయడానికి ఏ అభ్యర్థనలను ట్రేస్ చేయాలో ఎంచుకునే ప్రక్రియ.

కాంటెక్స్ట్ ప్రాపగేషన్

ట్రేస్ కొనసాగింపును నిర్వహించడానికి సేవల మధ్య ట్రేస్ కాంటెక్స్ట్‌ను పంపే విధానం.

మెట్రిక్స్ మరియు మానిటరింగ్

గోల్డెన్ సిగ్నల్స్

అప్లికేషన్ ఆరోగ్యాన్ని పర్యవేక్షించడానికి నాలుగు కీలక మెట్రిక్స్: లేటెన్సీ, థ్రూపుట్, ఎర్రర్ రేట్ మరియు సాచురేషన్.

లేటెన్సీ

అభ్యర్థన ప్రాసెస్ అవడానికి పట్టే సమయం, సాధారణంగా పర్సంటైల్‌లలో (P50, P90, P99) కొలవబడుతుంది.

థ్రూపుట్

అభ్యర్థనలు ప్రాసెస్ అయ్యే రేటు, సాధారణంగా సెకనుకు అభ్యర్థనలుగా కొలవబడుతుంది.

ఎర్రర్ రేట్

ఎర్రర్‌లకు దారితీసే అభ్యర్థనల శాతం.

సాచురేషన్

సిస్టమ్ వనరులు ఎంత మేరకు ఉపయోగించబడుతున్నాయో దాని స్థాయి.

Service Level Objective (SLO)

సేవ ప్రదాతలు మరియు వినియోగదారుల మధ్య అంగీకరించబడిన సేవ నాణ్యత లక్ష్య స్థాయి.

Service Level Indicator (SLI)

లేటెన్సీ లేదా అందుబాటు వంటి అందించబడిన సేవ స్థాయి యొక్క కొలవగల అంశం.

RUM మరియు Synthetics

Real User Monitoring (RUM)

ప్రొడక్షన్‌లో వెబ్ అప్లికేషన్‌లతో వాస్తవ వినియోగదారు ఇంటరాక్షన్‌ల పర్యవేక్షణ.

సింథటిక్ మానిటరింగ్

అనుకరించిన వినియోగదారు ఇంటరాక్షన్‌లను ఉపయోగించి అప్లికేషన్‌ల యొక్క ఆటోమేటెడ్ టెస్టింగ్.

కానరీ

మీ అప్లికేషన్‌పై ఆటోమేటెడ్ టెస్ట్‌లను నిర్వహించే సింథటిక్ మానిటరింగ్ స్క్రిప్ట్.

Core Web Vitals

వెబ్‌లో వినియోగదారు అనుభవాన్ని కొలవడానికి Google యొక్క మెట్రిక్స్: LCP, FID మరియు CLS.

Largest Contentful Paint (LCP)

స్క్రీన్‌పై అతిపెద్ద కంటెంట్ ఎలిమెంట్ రెండర్ అవడానికి పట్టే సమయం.

First Input Delay (FID)

వినియోగదారు మొదటిసారి పేజీతో ఇంటరాక్ట్ చేసినప్పటి నుండి బ్రౌజర్ ప్రతిస్పందించే వరకు సమయం.

Cumulative Layout Shift (CLS)

దృశ్య స్థిరత్వం యొక్క కొలత, ఊహించని లేఅవుట్ మార్పులను పరిమాణీకరిస్తుంది.

AWS-నిర్దిష్ట పదాలు

Application Signals

APM, RUM మరియు synthetics ను కలిపే AWS యొక్క ఏకీకృత observability పరిష్కారం.

X-Ray

AWS యొక్క డిస్ట్రిబ్యూటెడ్ ట్రేసింగ్ సేవ, ఇప్పుడు Application Signals తో ఇంటిగ్రేట్ చేయబడింది.

CloudWatch

మెట్రిక్స్, లాగ్‌లు మరియు ట్రేస్‌లను సేకరించే AWS యొక్క మానిటరింగ్ మరియు observability సేవ.

ADOT (AWS Distro for OpenTelemetry)

AWS సేవల కోసం ఆప్టిమైజ్ చేయబడిన OpenTelemetry యొక్క AWS డిస్ట్రిబ్యూషన్.

మద్దతు ఉన్న రన్‌టైమ్‌లు మరియు ఫ్రేమ్‌వర్క్‌లు

Application Signals ద్వారా మద్దతు ఉన్న ప్రోగ్రామింగ్ భాషలు మరియు ఫ్రేమ్‌వర్క్‌లు.

ఆటో-ఇన్‌స్ట్రుమెంటేషన్ మద్దతు

అధికారిక అనుకూలత మరియు రన్‌టైమ్/వెర్షన్ మ్యాట్రిక్స్ కోసం, మద్దతు ఉన్న సిస్టమ్‌లు చూడండి. ముఖ్యాంశాలు:

  • Java: JVM 8/11/17/21/23 (ఫ్రేమ్‌వర్క్ మద్దతు ADOT/OpenTelemetry Java ఇన్‌స్ట్రుమెంటేషన్‌ను అనుసరిస్తుంది).
  • Python: Python 3.9+ (ఫ్రేమ్‌వర్క్ మద్దతు ADOT/OpenTelemetry Python ఇన్‌స్ట్రుమెంటేషన్‌ను అనుసరిస్తుంది).
  • Node.js: Node 14/16/18/20/22 (ఫ్రేమ్‌వర్క్ మద్దతు ADOT/OpenTelemetry JS ఇన్‌స్ట్రుమెంటేషన్‌ను అనుసరిస్తుంది).
  • .NET: x86-64 లేదా ARM64 లో మద్దతు; Linux x64/ARM64 మరియు Windows Server 2022 x64 (లైబ్రరీ మద్దతు ADOT/OpenTelemetry .NET ఇన్‌స్ట్రుమెంటేషన్‌ను అనుసరిస్తుంది).
  • PHP/Ruby/Go: OpenTelemetry zero-code/auto-instrumentation మార్గాల ద్వారా మద్దతు (ఈ భాషల కోసం ADOT SDK లేదు).

గమనికలు మరియు తెలిసిన పరిగణనలు

  • Node.js ESM: ESM మద్దతు పరిమితం; డాక్స్‌లో ESM-నిర్దిష్ట సెటప్‌ను అనుసరించకపోతే CommonJS ని ప్రాధాన్యత ఇవ్వండి.
  • Python: కంటెయినరైజ్డ్ యాప్‌లకు PYTHONPATH సెట్ చేయడం అవసరం కావచ్చు (తెలిసిన OTel autoinstrumentation సమస్య).
  • .NET: ADOT .NET కు AWS SDK for .NET V4 తో పరిమితులు ఉన్నాయి; పూర్తి మద్దతు కోసం AWS SDK V3 సిఫార్సు చేయబడింది.

మాన్యువల్ ఇన్‌స్ట్రుమెంటేషన్

అన్ని ప్రధాన ప్రోగ్రామింగ్ భాషలు OpenTelemetry SDKల ద్వారా మద్దతు ఉన్నాయి:

  • Java, Scala, Kotlin
  • Python, Ruby
  • JavaScript/TypeScript, Go
  • .NET (C#, F#, VB.NET)
  • PHP, Rust, C++

కాన్ఫిగరేషన్ సూచన

సాధారణ కాన్ఫిగరేషన్ ఎంపికలు మరియు ఎన్విరాన్మెంట్ వేరియబుల్స్.

ఎన్విరాన్మెంట్ వేరియబుల్స్

OpenTelemetry కాన్ఫిగరేషన్:

  • OTEL_SERVICE_NAME - మీ సేవ పేరు
  • OTEL_TRACES_EXPORTER - ట్రేస్ ఎక్స్‌పోర్టర్ (otlp, jaeger, zipkin)
  • OTEL_EXPORTER_OTLP_ENDPOINT - OTLP ఎండ్‌పాయింట్ URL
  • OTEL_TRACES_SAMPLER - శాంప్లింగ్ వ్యూహం
  • OTEL_TRACES_SAMPLER_ARG - శాంప్లింగ్ పారామీటర్

AWS-నిర్దిష్ట కాన్ఫిగరేషన్:

  • AWS_REGION - AWS రీజియన్
  • AWS_XRAY_DAEMON_ADDRESS - X-Ray daemon చిరునామా
  • OTEL_RESOURCE_ATTRIBUTES - సేవ అట్రిబ్యూట్‌లు (అప్లికేషన్ మ్యాప్‌లో కస్టమ్ గ్రూపింగ్ కోసం కూడా ఉపయోగించబడుతుంది)

శాంప్లింగ్ కాన్ఫిగరేషన్

శాంప్లింగ్ వ్యూహాలు:

  • always_on - అన్ని ట్రేస్‌లను శాంపిల్ చేయండి
  • always_off - ట్రేస్‌లను శాంపిల్ చేయవద్దు
  • traceidratio - నిష్పత్తి ఆధారంగా శాంపిల్ చేయండి (ఉదా., 10% కోసం 0.1)
  • parentbased_always_on - పేరెంట్ శాంప్లింగ్ నిర్ణయాన్ని గౌరవించండి
  • parentbased_traceidratio - పేరెంట్-ఆధారిత నిష్పత్తి శాంప్లింగ్

శాంప్లింగ్ వ్యూహాలు

  • ప్రొడక్షన్: 5-10% శాంప్లింగ్ రేటు
  • డెవలప్‌మెంట్: 50-100% శాంప్లింగ్ రేటు
  • ఎర్రర్ ట్రేస్‌లు: ఎర్రర్‌లతో ఉన్న ట్రేస్‌లను ఎల్లప్పుడూ శాంపిల్ చేయండి
  • అధిక-విలువ ఆపరేషన్‌లు: క్లిష్టమైన మార్గాల కోసం 100% శాంప్లింగ్

డేటా నిల్వ విధానాలు

నిల్వ అంతర్లీన టెలిమెట్రీ స్టోర్‌ల ద్వారా నియంత్రించబడుతుంది (ఉదాహరణకు, CloudWatch మెట్రిక్స్/లాగ్‌లు మరియు AWS X-Ray ట్రేస్‌లు). ఆ సేవలలో నిల్వను కాన్ఫిగర్ చేయండి మరియు మీ ట్రబుల్‌షూటింగ్ మరియు కంప్లయన్స్ అవసరాలతో సమలేఖనం చేయండి.

వనరుల ఆప్టిమైజేషన్

  • బ్యాచ్ ఎక్స్‌పోర్ట్‌లు: నెట్‌వర్క్ ఓవర్‌హెడ్‌ను తగ్గించండి
  • కంప్రెషన్: డేటా కంప్రెషన్‌ను ప్రారంభించండి
  • ఫిల్టరింగ్: అనవసరమైన span లు మరియు అట్రిబ్యూట్‌లను తొలగించండి
  • బఫరింగ్: తగిన బఫర్ పరిమాణాలను ఉపయోగించండి
