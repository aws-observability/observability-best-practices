Overview

This guide provides step-by-step instructions to deploy a Python FastAPI application on Amazon EKS with AWS Application Signals auto-instrumentation. This is the EKS equivalent of the Lambda Python function, reusing the same business logic and endpoints.

What you'll accomplish:
  • Deploy a Python FastAPI app on EKS
  • Enable AWS Application Signals auto-instrumentation
  • View your service in CloudWatch Application Signals console
  • Monitor service metrics, traces, and operations
Time to complete: 30-45 minutes

Prerequisites

Required Tools

  • AWS CLI v2 (latest version)
  • Docker Desktop
  • kubectl (latest version)
  • Python 3.11+ (for local development)
  • pip (for local development)
  • AWS Account with appropriate permissions
  • Text editor

AWS Permissions Required

  • EKS: Full access (create clusters, node groups, add-ons)
  • IAM: Create roles and policies
  • CloudWatch: Logs and Application Signals access
  • ECR: Push/pull container images
  • VPC: Create and manage VPC resources
  • S3: Read access for testing

Knowledge Assumed

  • Basic AWS CLI usage
  • Basic Docker commands
  • Basic kubectl commands
  • Basic understanding of EKS concepts
  • Basic Python/FastAPI knowledge

Architecture

EKS Observability Add-on (Standard Approach): This guide uses the official EKS observability add-on, which is the recommended and standard way to enable Application Signals on EKS clusters. The add-on automatically handles telemetry collection without requiring manual CloudWatch Agent configuration.

Your EKS cluster will run:

  1. Python FastAPI Application: Your web API with auto-instrumentation annotations
  2. EKS Observability Add-on: Official AWS add-on for automatic tracing and metrics collection
  3. CloudWatch Agent (managed by add-on): Automatically deployed and configured by the add-on

Flow:

EKS Cluster
  → Python FastAPI App (auto-instrumented via annotation)
  → EKS Observability Add-on
  → CloudWatch Agent (managed)
  → CloudWatch Application Signals
Key Benefits of EKS Observability Add-on:
  • No manual CloudWatch Agent configuration required
  • Automatic auto-instrumentation for supported languages
  • Built-in support for Application Signals
  • Simplified deployment and management

Step 1: Create the Application

1.1 Create Project Structure

mkdir eks-python
cd eks-python

1.2 Create the Python Application

Create app.py with the same endpoints as the Lambda function:

"""
AWS Application Signals EKS Python Service
FastAPI application with Application Signals auto-instrumentation
"""
import time
import random
import boto3
import requests
from fastapi import FastAPI, HTTPException
from fastapi.responses import JSONResponse
from typing import Dict, Any
import uvicorn

# Initialize FastAPI app
app = FastAPI(
    title="AWS Application Signals EKS Python Demo",
    description="Python service running on EKS with Application Signals",
    version="1.0.0"
)

# Initialize AWS clients
s3_client = boto3.client('s3')


@app.get("/")
async def root():
    """Root endpoint"""
    return {
        "message": "AWS Application Signals EKS Python Demo",
        "timestamp": time.time(),
        "service": "python-eks-service"
    }


@app.get("/health")
async def health():
    """Health check endpoint"""
    return {
        "status": "healthy",
        "timestamp": time.time(),
        "service": "python-eks-service"
    }


@app.get("/api/hello/{name}")
async def hello(name: str):
    """Personalized greeting with simulated processing"""
    # Simulate some processing time
    time.sleep(random.uniform(0.01, 0.1))
    
    return {
        "message": f"Hello, {name}!",
        "timestamp": time.time(),
        "service": "python-eks-service"
    }


@app.get("/api/slow")
async def slow():
    """Intentionally slow endpoint for latency testing"""
    time.sleep(random.uniform(0.5, 1.5))
    
    return {
        "message": "Slow operation completed",
        "timestamp": time.time(),
        "duration": "1-1.5 seconds",
        "service": "python-eks-service"
    }


@app.get("/api/error")
async def error():
    """Endpoint that randomly raises errors for testing error rates"""
    if random.random() > 0.5:
        raise HTTPException(
            status_code=500, 
            detail="Simulated error for testing Application Signals error tracking"
        )
    
    return {
        "message": "No error occurred",
        "timestamp": time.time(),
        "service": "python-eks-service"
    }


@app.get("/api/external")
async def external():
    """Makes external HTTP call for distributed tracing"""
    try:
        # This will be automatically traced by OpenTelemetry
        response = requests.get("https://httpbin.org/delay/1", timeout=5)
        
        return {
            "message": "External call completed",
            "external_status_code": response.status_code,
            "timestamp": time.time(),
            "service": "python-eks-service"
        }
    except Exception as e:
        raise HTTPException(
            status_code=500,
            detail=f"External call failed: {str(e)}"
        )


@app.get("/api/s3")
async def s3():
    """Makes AWS S3 call for testing AWS SDK instrumentation"""
    try:
        # This will be automatically traced by OpenTelemetry
        response = s3_client.list_buckets()
        bucket_names = [bucket['Name'] for bucket in response['Buckets']]
        
        return {
            "message": "S3 call completed successfully",
            "bucket_count": len(bucket_names),
            "buckets": bucket_names[:5],  # Show first 5 buckets
            "timestamp": time.time(),
            "service": "python-eks-service"
        }
    except Exception as e:
        raise HTTPException(
            status_code=500,
            detail=f"S3 call failed: {str(e)}"
        )


@app.get("/api/metrics")
async def metrics():
    """Custom metrics endpoint"""
    # Simulate some business logic
    processing_time = random.uniform(0.1, 0.5)
    time.sleep(processing_time)
    
    # Simulate different response times based on some logic
    if processing_time > 0.3:
        status = "slow"
    else:
        status = "fast"
    
    return {
        "message": "Metrics endpoint",
        "processing_time": processing_time,
        "status": status,
        "timestamp": time.time(),
        "service": "python-eks-service"
    }


if __name__ == "__main__":
    uvicorn.run(
        "app:app",
        host="0.0.0.0",
        port=8000,
        reload=True,
        log_level="info"
    )

1.4 Create Dockerfile

Create Dockerfile:

# Use Python 3.11 slim image
FROM python:3.11-slim

# Set working directory
WORKDIR /app

# Install system dependencies
RUN apt-get update && apt-get install -y \
    gcc \
    && rm -rf /var/lib/apt/lists/*

# Copy requirements first for better caching
COPY requirements.txt .

# Install Python dependencies
RUN pip install --no-cache-dir -r requirements.txt

# Copy application code
COPY app.py .

# Create non-root user
RUN useradd --create-home --shell /bin/bash app && chown -R app:app /app
USER app

# Expose port
EXPOSE 8000

# Health check
HEALTHCHECK --interval=30s --timeout=30s --start-period=5s --retries=3 \
    CMD curl -f http://localhost:8000/health || exit 1

# Run the application
CMD ["uvicorn", "app:app", "--host", "0.0.0.0", "--port", "8000"]

Step 3: Deploy to EKS

3.1 Create EKS Cluster

# Create EKS cluster
eksctl create cluster --name appsignals-cluster --region us-east-1 --nodegroup-name workers --node-type t3.medium --nodes 2 --nodes-min 1 --nodes-max 3

# Create namespace
kubectl create namespace appsignals-demo

3.2 Install EKS Observability Add-on (Required)

Important: The EKS observability add-on is the standard and recommended way to enable Application Signals on EKS. This add-on automatically handles telemetry collection and provides auto-instrumentation capabilities. 
# Install the EKS observability add-on (REQUIRED for Application Signals)
aws eks create-addon --cluster-name appsignals-cluster --addon-name amazon-cloudwatch-observability --region us-east-1

# Verify the add-on is installed
aws eks describe-addon --cluster-name appsignals-cluster --addon-name amazon-cloudwatch-observability --region us-east-1
What the EKS Observability Add-on provides:
  • Automatic CloudWatch Agent deployment
  • Built-in support for Application Signals
  • Auto-instrumentation for Java, Python, Node.js, and .NET
  • No manual configuration required

3.4 Deploy Application

Create deployment.yaml:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: python-eks-service
  namespace: appsignals-demo
  labels:
    app: python-eks-service
    service: python
spec:
  replicas: 2
  selector:
    matchLabels:
      app: python-eks-service
  template:
    metadata:
      labels:
        app: python-eks-service
        service: python
      annotations:
        # Enable Application Signals auto-instrumentation
        instrumentation.opentelemetry.io/inject-python: "true"
        prometheus.io/scrape: "true"
        prometheus.io/port: "8000"
        prometheus.io/path: "/metrics"
    spec:
      containers:
      - name: python-app
        image: 188524202814.dkr.ecr.us-east-1.amazonaws.com/my-appsignals-python-app:latest
        ports:
        - containerPort: 8000
        env:
        - name: OTEL_SERVICE_NAME
          value: "python-eks-service"
        - name: OTEL_RESOURCE_ATTRIBUTES
          value: "service.name=python-eks-service,service.namespace=appsignals-demo,service.version=1.0.0"
        - name: OTEL_EXPORTER_OTLP_ENDPOINT
          value: "http://cloudwatch-agent.amazon-cloudwatch.svc.cluster.local:4315"
        - name: OTEL_EXPORTER_OTLP_PROTOCOL
          value: "grpc"
        - name: OTEL_EXPORTER_OTLP_TRACES_ENDPOINT
          value: "http://cloudwatch-agent.amazon-cloudwatch.svc.cluster.local:4315"
        - name: OTEL_EXPORTER_OTLP_METRICS_ENDPOINT
          value: "http://cloudwatch-agent.amazon-cloudwatch.svc.cluster.local:4315"
        - name: OTEL_TRACES_EXPORTER
          value: "otlp"
        - name: OTEL_METRICS_EXPORTER
          value: "otlp"
        - name: OTEL_LOGS_EXPORTER
          value: "otlp"
        - name: AWS_REGION
          value: "us-east-1"
        resources:
          requests:
            memory: "256Mi"
            cpu: "250m"
          limits:
            memory: "512Mi"
            cpu: "500m"
        livenessProbe:
          httpGet:
            path: /health
            port: 8000
          initialDelaySeconds: 30
          periodSeconds: 10
        readinessProbe:
          httpGet:
            path: /health
            port: 8000
          initialDelaySeconds: 5
          periodSeconds: 5
---
apiVersion: v1
kind: Service
metadata:
  name: python-eks-service
  namespace: appsignals-demo
  labels:
    app: python-eks-service
    service: python
spec:
  selector:
    app: python-eks-service
  ports:
  - port: 80
    targetPort: 8000
    protocol: TCP
  type: ClusterIP

Deploy the application:

kubectl apply -f deployment.yaml

Step 4: Generate Traffic and Monitor

4.1 Generate Traffic

Create generate-traffic.sh:

#!/bin/bash

# Port forward the service
kubectl port-forward service/python-eks-service 8000:80 -n appsignals-demo &
PORT_FORWARD_PID=$!

# Wait for port forward to be ready
sleep 10

BASE_URL="http://localhost:8000"

echo "Generating traffic for Python EKS Application Signals demo..."

# Generate traffic for 5 minutes
for i in {1..300}; do
    echo "Request $i"

    # Root endpoint
    curl -s "$BASE_URL/" > /dev/null

    # Health check
    curl -s "$BASE_URL/health" > /dev/null

    # Hello endpoint
    curl -s "$BASE_URL/api/hello/User$i" > /dev/null

    # Slow endpoint
    curl -s "$BASE_URL/api/slow" > /dev/null

    # Error endpoint
    curl -s "$BASE_URL/api/error" > /dev/null

    # External call
    curl -s "$BASE_URL/api/external" > /dev/null

    # S3 call
    curl -s "$BASE_URL/api/s3" > /dev/null

    # Metrics endpoint
    curl -s "$BASE_URL/api/metrics" > /dev/null

    sleep 1
done

echo "Traffic generation complete!"

# Clean up port forward
kill $PORT_FORWARD_PID

Make it executable and run:

chmod +x generate-traffic.sh
./generate-traffic.sh

Troubleshooting

Common Issues

  1. Application won't start

    • Check pod status: kubectl get pods -n appsignals-demo
    • Check pod logs: kubectl logs -n appsignals-demo -l app=python-eks-service
    • Verify EKS observability add-on is installed

  2. No data in Application Signals

    • Verify auto-instrumentation annotation is present
    • Check environment variables are set correctly
    • Ensure EKS observability add-on is running

  3. Port forward issues

    • Check service is running: kubectl get svc -n appsignals-demo
    • Verify pods are ready: kubectl get pods -n appsignals-demo
