CloudWatch APM 文档

Java Spring + OTLP + CloudWatch（动手练习）

本指南记录了完整的练习，内容包括使用 OpenTelemetry 为 Java Spring Boot 服务进行插桩、将遥测数据导出到 AWS、部署到 ECS/Fargate，以及验证链路追踪/日志/Application Signals 行为。

本文以实施示例的风格编写，可用作运维手册或研讨会讲义。

1) 本练习演示的内容

  • Java 应用程序的端到端遥测：
    • 链路追踪（X-Ray / Transaction Search）
    • 日志（CloudWatch Logs OTLP 端点或 ECS 上的 awslogs）
    • 指标/Application Signals（CloudWatch Agent sidecar 路径）
  • 以下两者的区别：
    • Micrometer tracing bridge（micrometer-tracing-bridge-otel）用于链路追踪 API 桥接
    • Micrometer registries（micrometer-registry-cloudwatch2、micrometer-registry-otlp）仅用于指标输出
  • 通过环境变量（OTEL_*）进行 OTLP 端点配置，而非硬编码 Java 常量。

4) OTLP 端点映射

信号 端点模式 典型环境变量
链路追踪 https://xray.<region>.amazonaws.com/v1/traces OTEL_EXPORTER_OTLP_TRACES_ENDPOINT、OTEL_EXPORTER_OTLP_TRACES_PROTOCOL=http/protobuf
指标 https://monitoring.<region>.amazonaws.com/v1/metrics OTEL_EXPORTER_OTLP_METRICS_ENDPOINT（或通过 sidecar 的 App Signals 路径）
日志 https://logs.<region>.amazonaws.com/v1/logs OTEL_EXPORTER_OTLP_LOGS_ENDPOINT、OTEL_EXPORTER_OTLP_LOGS_HEADERS

在 ECS sidecar 模式下，应用容器将链路追踪/指标指向 http://localhost:4316/...，由 CloudWatch Agent 负责转发。

5) 本仓库使用的插桩模型

5.1 依赖项

  • micrometer-tracing-bridge-otel 存在于 pom.xml 中。
  • 链路追踪不依赖 micrometer-registry-cloudwatch2 或 micrometer-registry-otlp。

5.2 自动 + 手动追踪

  • ADOT Java agent 的自动插桩捕获框架/客户端 span。
  • 在 DemoService.greet() 中添加了手动增强：
    • 属性：demo.operation、demo.request.name、demo.downstream.status、demo.downstream.elapsed_ms、demo.custom.instrumented
    • 事件：demo.greet.start、demo.httpbin.success、demo.httpbin.failure
    • 使用 recordException(...) 和 setStatus(ERROR, ...) 记录错误

7) 在 AWS 控制台中查看遥测数据

链路追踪（主要）

  1. X-Ray → Traces
  2. 区域：us-east-1
  3. 时间范围：最近 15-60 分钟
  4. 服务：java-spring-otel-poc

对于自定义字段：

  • 打开链路追踪 → 段 java-spring-otel-poc → Metadata
  • 查找以下键：
    • demo.operation
    • demo.request.name
    • demo.downstream.status
    • demo.downstream.elapsed_ms
    • demo.custom.instrumented

日志

  • ECS 应用日志：/ecs/java-spring-otel
  • CW agent sidecar 日志：/ecs/ecs-cwagent-java-spring-otel
  • 本地 OTLP 日志组（如果本地运行）：/aws/otel/java-spring-otel-poc

Application Signals

  • CloudWatch → Application Signals → Services
  • 在持续流量和通过 sidecar 路径成功转发指标后，查找 java-spring-otel-poc。

8) 关键要点

OpenTelemetry 自动插桩可以创建 span，但成功的 AWS Observability 结果仍然取决于：

  1. 正确的 OTLP 端点配置
  2. 正确的认证/转发路径（通常通过 sidecar/collector 处理指标/App Signals）
  3. 正确区分 tracing bridge 与 metrics registries
  4. 持续的流量和正确的控制台视图（X-Ray 用于链路追踪）
