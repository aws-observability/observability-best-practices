デバッグガイド

高速トリアージワークフロー（推奨）

  1. SLO から開始します（定義されている場合）。 異常なサービス/オペレーション/依存関係を特定します。
  2. サービスを使用します。 高レイテンシーまたはフォルトレートでソートして、上位の問題箇所を見つけます。
  3. サービスの詳細を開きます。 レイテンシ/障害のデータポイントを選択するために、オペレーションと依存関係のエッジを検査します。
  4. 相関トレース/スパンにジャンプします。 トレースタイムラインを調べて、遅延または失敗しているダウンストリームコールを特定します。
  5. ログ/ランタイムメトリクスで確認します。 関連する場合は、エラーログと JVM/CPU/GC を相関させます。

参照： Application Signals でオペレーションの健全性をモニタリングする.

「レアスパイク」に対するトランザクション検索の使用

断続的なスパイクの場合、Transaction Search を使用すると、属性ごとにスパンを分析し、影響を受けたユーザー/リクエストを詳しく調べることができます。一般的なパターン：

  • グループ分析 ビジネス属性（例：顧客 ID）によって、影響を受けているユーザーを特定します。
  • フィルター サービス/オペレーションおよびレイテンシー (p99) 別に外れ値を特定します。
  • トレースを開く 外れ値スパンからの依存関係呼び出しパスをたどります。

参照： アプリケーション問題のトラブルシューティング（トランザクション検索）.

「データが表示されないのはなぜ？」という一般的な確認事項

  • アカウントの有効化： Application Signals はアカウント/リージョンごとに 1 回有効化する必要があります（サービスにリンクされたロールが作成されます）。
  • 時間範囲： サービス/オペレーションは、選択した時間枠内にアクティビティがある場合にのみ表示されます。
  • サービスの命名： 確認する service.name / OTEL_RESOURCE_ATTRIBUTES は適切に設定されています（非 EKS では、多くの場合、明示的なサービス/環境名が必要です）。
  • パイプラインの健全性： CloudWatch Agent/コレクターが実行中で OTLP を受信していること。ローカルエンドポイントと IAM 権限を確認してください。

参照： Application Signals のインストールに関するトラブルシューティング.
