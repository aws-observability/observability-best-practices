CloudWatch APM ドキュメント

機能と特長

Application Signals: 提供される機能

CloudWatch Application Signals は、CloudWatch における AWS のアプリケーションパフォーマンスモニタリング (APM) エクスペリエンスです。有効化後、アプリケーションのテレメトリを自動的に収集し、 サービス, オペレーション、および 依存関係 ごとに整理することで、運用上の健全性を迅速にトリアージし、長期的なパフォーマンス目標を追跡できます。

  • 標準アプリケーションメトリクス: 呼び出し量、可用性、レイテンシー、障害、およびエラー。
  • サービスと依存関係のトポロジー: 検出されたサービスと関係から生成されたアプリケーションマップ。
  • 詳細なトラブルシューティング: サービスの健全性から相関トレースおよび関連テレメトリへのドリルダウン。

アプリケーションマップ (トポロジー + 健全性)

アプリケーションマップは、アプリケーションのトポロジーを可視化し、運用上の健全性を強調表示します。クライアント、カナリア、サービス、および依存関係間の接続を表示することで、オペレーターは実行フローを把握し、問題が伝播している箇所を特定できます。

できること

  • SLO/SLI の健全性を確認: 目標を達成していないサービス (および上流/下流への影響) を素早く把握できます。
  • ドリルダウン: ノード/エッジを選択して詳細メトリクスを開き、サービスの詳細に移動します。
  • グループ化とフィルタリング: デフォルトのグループ (関連サービス、環境) を使用し、AWS タグまたは OpenTelemetry リソース属性を使用してカスタムグループを作成します。
  • クロスアカウントの可視性: 複数の AWS アカウントにまたがるサービスを表示します (クロスアカウントオブザーバビリティ経由で設定した場合)。

サービス、オペレーション、および依存関係

Application Signals は、チームが本番システムをデバッグする方法に合わせたモデルにテレメトリを構造化します:

  • サービス: デプロイ可能なユニット (例: ECS サービス、EKS ワークロード、Lambda 関数)。
  • オペレーション: サービス内のリクエストハンドラー (例: HTTP ルートまたは RPC メソッド)。
  • 依存関係: サービスが行うダウンストリーム呼び出し (例: 別のサービスまたは AWS の依存関係)。

この構造により、「どのオペレーションが低下したか」、「これは上流または下流の問題か」、「どの依存関係がレイテンシーや障害の原因となっているか」といった質問に答えやすくなります。

サービスレベル目標 (SLO)

Application Signals が収集するテレメトリから直接 SLO を作成および監視します。SLO は、目標に対するパフォーマンスと信頼性を追跡し、「リスクのある」サービスを早期に表面化するのに役立ちます。

  • 主要な SLI に対して SLO を定義: 可用性、レイテンシー、またはビジネス目標に関連するその他の CloudWatch メトリクス。
  • 健全性の追跡: サービスリストおよび Application Map で SLI の健全性ステータスを確認する。
  • アラート: 停止前にチームが対応できるよう、SLO のステータス/バーンレートに連動したアラームを作成する。

インストルメンテーション（OpenTelemetry 互換）

Application Signals は OpenTelemetry 互換であり、プラットフォームおよび言語に応じてアプリケーションをインストルメント化する複数の方法をサポートしています。

  • サポート対象言語: Java、Python、Node.js、および .NET（さらに OpenTelemetry のゼロコードインストルメンテーションパスを通じた追加ランタイムも対応）。
  • サポート対象システム: Amazon EKS、Kubernetes、Amazon ECS、Amazon EC2（および CloudWatch エージェント + AWS Distro for OpenTelemetry を利用した有効化パターン）。
  • 一般的なセットアップパターン: 環境に応じて、AWS Distro for OpenTelemetry（ADOT）SDK/コレクター、または OpenTelemetry SDK + コレクターを使用する。

RUM および Synthetics との連携

Application Signals は CloudWatch RUM および CloudWatch Synthetics と統合されています。これらのクライアント/カナリアのトレースを有効にすると、Application Map に表示され、ユーザーエクスペリエンスおよびプロアクティブテストを、それらが呼び出すバックエンドサービスに関連付けることができます。

  • RUM クライアント: フロントエンドのパフォーマンスおよび JS エラーをバックエンドのサービス呼び出しに関連付ける。
  • Synthetics カナリア: 重要なフローを検証し、ユーザーより先に問題を検出するのに役立てる。
