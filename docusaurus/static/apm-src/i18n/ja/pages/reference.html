サービスの制限とクォータ

CloudWatch Application Signals の制限とクォータについて。

制限とクォータ

Application Signals は CloudWatch および AWS X-Ray に依存しています。制限（取り込み、クエリレート、クォータ）はそれらのサービスによって管理され、時間の経過とともに変更される場合があります。現在の値については公式ドキュメントをご参照ください。

  • Application Signals: Application Signals ドキュメント
  • CloudWatch クォータ: CloudWatch サービスクォータ
  • AWS X-Ray クォータ: X-Ray API スロットリングと制限

CloudWatch メトリクスの保持期間（共通リファレンス）

CloudWatch はメトリクスデータをさまざまな解像度で保持します。完全な保持モデルについては CloudWatch ドキュメントをご参照ください。

  • 1 分解像度: 15 日
  • 5 分解像度: 63 日
  • 1 時間解像度: 15 か月

サービスクォータ

Application Signals:

  • アカウントあたりのサービス数: 1,000
  • サービスあたりのオペレーション数: 100
  • カスタムメトリクス: サービスあたり 30

RUM:

  • アカウントあたりのアプリモニター数: 20
  • 1 秒あたりのイベント数: 10,000
  • イベントサイズ: 64 KB

Synthetics:

  • アカウントあたりの Canary 数: 100
  • Canary の実行時間: 最大 15 分
  • Canary のサイズ: 1 GB（解凍後）

用語集

CloudWatch Application Signals の主要な用語と概念。

コアコンセプト

アプリケーションパフォーマンスモニタリング (APM)

ソフトウェアアプリケーションのパフォーマンスと可用性を監視・管理する手法。

分散トレーシング

分散システムを流れるリクエストを追跡する手法で、複数のサービスにわたるリクエストのパスとパフォーマンスを把握できます。

サービスマップ

サービスがどのように接続・連携しているかを示す、アプリケーションアーキテクチャの視覚的な表現。

スパン

トレース内の単一の操作で、データベースクエリや API 呼び出しなどの作業単位を表します。

トレース

複数のスパンで構成される、アプリケーション内のリクエストの完全なエンドツーエンドの経路。

インストゥルメンテーション用語

自動インストゥルメンテーション

コードの変更を必要とせず、アプリケーションにオブザーバビリティコードを自動的に追加すること。

手動インストゥルメンテーション

アプリケーションのソースコードにトレースおよびメトリクスコードを明示的に追加すること。

OpenTelemetry

インストゥルメンテーション向けにベンダー中立な API を提供するオープンソースのオブザーバビリティフレームワーク。

サンプリング

オブザーバビリティとコストおよびパフォーマンスのバランスを取るために、トレースするリクエストを選択するプロセス。

コンテキスト伝播

トレースの継続性を維持するために、サービス間でトレースコンテキストを受け渡すメカニズム。

メトリクスとモニタリング

ゴールデンシグナル

アプリケーションの健全性を監視するための4つの主要メトリクス: レイテンシー、スループット、エラーレート、サチュレーション。

レイテンシー

リクエストが処理されるまでにかかる時間で、通常はパーセンタイル (P50、P90、P99) で測定されます。

スループット

リクエストが処理される速度で、通常は1秒あたりのリクエスト数として測定されます。

エラーレート

エラーとなるリクエストの割合。

サチュレーション

システムリソースが使用されている度合い。

サービスレベル目標 (SLO)

サービスプロバイダーとコンシューマーの間で合意されたサービス品質の目標水準。

サービスレベル指標 (SLI)

レイテンシーや可用性など、提供されるサービスレベルの測定可能な側面。

RUM とシンセティクス

Real User Monitoring (RUM)

本番環境におけるWebアプリケーションへの実際のユーザーインタラクションの監視。

Synthetic Monitoring

シミュレートされたユーザーインタラクションを使用したアプリケーションの自動テスト。

Canary

アプリケーションに対して自動テストを実行するSynthetic Monitoringスクリプト。

Core Web Vitals

Web上のユーザーエクスペリエンスを測定するGoogleの指標：LCP、FID、CLS。

Largest Contentful Paint (LCP)

最大のコンテンツ要素が画面にレンダリングされるまでにかかる時間。

First Input Delay (FID)

ユーザーがページに初めてインタラクションを行ってからブラウザが応答するまでの時間。

Cumulative Layout Shift (CLS)

予期しないレイアウトシフトを定量化する、視覚的安定性の指標。

AWS固有の用語

Application Signals

APM、RUM、およびSyntheticsを組み合わせたAWSの統合オブザーバビリティソリューション。

X-Ray

Application Signalsと統合されたAWSの分散トレーシングサービス。

CloudWatch

メトリクス、ログ、およびトレースを収集するAWSの監視・オブザーバビリティサービス。

ADOT (AWS Distro for OpenTelemetry)

AWSサービス向けに最適化されたAWSのOpenTelemetryディストリビューション。

サポートされているランタイムとフレームワーク

Application Signalsがサポートするプログラミング言語とフレームワーク。

自動インストルメンテーションのサポート

互換性およびランタイム／バージョンの対応表については、以下を参照してください。 サポートされているシステム。主な内容：

  • Java： JVM 8/11/17/21/23（フレームワークのサポートはADOT/OpenTelemetry Javaインストルメンテーションに準拠）。
  • Python： Python 3.9以降（フレームワークのサポートはADOT/OpenTelemetry Pythonインストルメンテーションに準拠）。
  • Node.js： Node 14/16/18/20/22（フレームワークのサポートはADOT/OpenTelemetry JSインストルメンテーションに準拠）。
  • .NET： x86-64またはARM64でサポート；Linux x64/ARM64およびWindows Server 2022 x64（ライブラリのサポートはADOT/OpenTelemetry .NETインストルメンテーションに準拠）。
  • PHP/Ruby/Go： OpenTelemetryのゼロコード／自動インストルメンテーションパス経由でサポート（これらの言語向けのADOT SDKは提供されていません）。

注意事項および既知の考慮点

  • Node.js ESM: ESMサポートは限定的です。ドキュメントのESM固有のセットアップに従わない限り、CommonJSを推奨します。
  • Python: コンテナ化されたアプリでは必要になる場合があります PYTHONPATH 設定（既知のOTel自動計装の問題）。
  • .NET: ADOT .NETはAWS SDK for .NET V4に制限があります。完全なサポートにはAWS SDK V3を推奨します。

手動計装

主要なプログラミング言語はすべてOpenTelemetry SDKを通じてサポートされています：

  • Java、Scala、Kotlin
  • Python、Ruby
  • JavaScript/TypeScript、Go
  • .NET（C#、F#、VB.NET）
  • PHP、Rust、C++

設定リファレンス

一般的な設定オプションと環境変数。

環境変数

OpenTelemetry設定：

  • OTEL_SERVICE_NAME - サービスの名前
  • OTEL_TRACES_EXPORTER - トレースエクスポーター（otlp、jaeger、zipkin）
  • OTEL_EXPORTER_OTLP_ENDPOINT - OTLPエンドポイントURL
  • OTEL_TRACES_SAMPLER - サンプリング戦略
  • OTEL_TRACES_SAMPLER_ARG - サンプリングパラメーター

AWS固有の設定：

  • AWS_REGION - AWSリージョン
  • AWS_XRAY_DAEMON_ADDRESS - X-Rayデーモンアドレス
  • OTEL_RESOURCE_ATTRIBUTES - サービス属性（アプリケーションマップのカスタムグループ化にも使用）

サンプリング設定

サンプリング戦略：

  • always_on - すべてのトレースをサンプリングする
  • always_off - トレースをサンプリングしない
  • traceidratio - 比率に基づいてサンプリングする（例：10%の場合は0.1）
  • parentbased_always_on - 親のサンプリング決定を尊重する
  • parentbased_traceidratio - 親ベースの比率サンプリング

コスト最適化

オブザーバビリティコストを管理するための戦略。

サンプリング戦略

  • 本番環境： サンプリングレート5〜10%
  • 開発環境： 50-100% サンプリングレート
  • エラートレース: エラーのあるトレースを常にサンプリングする
  • 高価値オペレーション: クリティカルパスに対して 100% サンプリング

データ保持ポリシー

保持は基盤となるテレメトリストア（例：CloudWatch メトリクス/ログおよび AWS X-Ray トレース）によって制御されます。これらのサービスで保持を設定し、トラブルシューティングおよびコンプライアンス要件に合わせて調整してください。

リソースの最適化

  • バッチエクスポート: ネットワークオーバーヘッドの削減
  • 圧縮: データ圧縮の有効化
  • フィルタリング: 不要なスパンと属性の削除
  • バッファリング: 適切なバッファサイズの使用
