CloudWatch APM Documentation

トラブルシューティング

よくある問題

このセクションでは、CloudWatch Application Signals の実装時に最もよく発生する問題とその解決方法について説明します。

CloudWatch にトレースが表示されない

考えられる原因：

  • サービスに対して Application Signals が有効になっていない
  • IAM 権限が正しくない
  • エージェントが正しく設定されていないか、実行されていない
  • サンプリングレートが低すぎる
  • CloudWatch へのネットワーク接続の問題

解決策：

  1. AWS コンソールで Application Signals が有効になっていることを確認する
  2. IAM 権限に以下が含まれていることを確認する cloudwatch:PutMetricData および xray:PutTraceSegments
  3. エージェントが実行中であり、正しく設定されていることを確認する
  4. テスト用にサンプリングレートを上げる（一時的に 100% に設定する）
  5. VPC セキュリティグループおよび NACL が CloudWatch へのアウトバウンドトラフィックを許可していることを確認する

トレースが欠落しているまたは不完全である

よくある問題：

  • サービス間でコンテキストが伝播されていない
  • 一部のサービスがインストルメント化されていない
  • サンプリングによってスパンが欠落している
  • エージェントの設定の問題

デバッグ手順：

  1. サービスマップを確認してインストルメント化されていないサービスを特定する
  2. トレースコンテキストヘッダーが渡されていることを確認する（例： x-amzn-trace-id)
  3. エージェントログで設定エラーを確認する
  4. サンプリングの問題を除外するために 100% サンプリングでテストする
  5. トレース検索を使用して部分的なトレースを見つける

インストルメンテーションのオーバーヘッドが高い

症状：

  • CPU 使用率の増加
  • メモリ消費量の増加
  • レイテンシーの増加
  • アプリケーションパフォーマンスの低下

最適化戦略：

  • サンプリングレートを下げる（本番環境では 10% から始める）
  • テールベースのサンプリングではなくヘッドベースのサンプリングを使用する
  • 適切なスパン制限を設定する
  • スパンエクスポートをバッチ処理する
  • インストルメンテーションメトリクスを監視する

RUMデータが表示されない

チェックリスト：

  • RUM Webクライアントが正しくインストールおよびロードされている
  • アプリケーションIDと設定が正しい
  • RUMドメインにCORSヘッダーが設定されている
  • コンテンツセキュリティポリシーがRUMスクリプトを許可している
  • ブラウザコンソールにJavaScriptエラーが表示されていない

確認：

// Check if RUM is loaded
console.log(window.cwr);

// Verify configuration
console.log(cwr('getConfig'));

Synthetic Canaryが失敗している

よくある原因：

  • Canary VPCからのネットワーク接続の問題
  • 認証または認可の失敗
  • アプリケーションの変更によるテストスクリプトの破損
  • Canaryランタイムまたは依存関係が古い

デバッグ：

  1. CloudWatch LogsでCanaryログを確認する
  2. Canaryスクリプトをローカルでテストする
  3. VPCルーティングとセキュリティグループを確認する
  4. 必要に応じてCanaryランタイムを更新する
  5. アプリケーションエンドポイントにアクセスできることを確認する

デバッグガイド

複雑な問題を診断するためのステップバイステップガイド。

トレースコンテキスト伝播のデバッグ

ステップ1：インストルメンテーションを確認する

コールチェーン内のすべてのサービスが適切にインストルメント化されていることを確認します：

# Check for trace headers in requests
curl -H "x-amzn-trace-id: Root=1-5759e988-bd862e3fe1be46a994272793;Parent=53995c3f42cd8ad8;Sampled=1" \
     http://your-service/endpoint

ステップ2：エージェント設定を確認する

エージェント設定ファイルと環境変数を確認します。

ステップ3：エンドツーエンドでテストする

テストリクエストを送信し、すべてのサービスを通じてトレースします。

パフォーマンス問題の診断

ボトルネックを特定する：

  1. サービスマップで高レイテンシーのサービスを確認する
  2. トレースタイムラインで遅いスパンを調べる
  3. リソースメトリクス（CPU、メモリ、ディスクI/O）を確認する
  4. ログのエラーパターンを探す
  5. データベースクエリのパフォーマンスを確認する

よくあるボトルネック：

  • データベースクエリの最適化が必要
  • 外部API呼び出しがタイムアウトしている
  • リソースの制約（CPU/メモリ）
  • スレッドプールの枯渇
  • サービス間のネットワークレイテンシー

エージェントのトラブルシューティング

エージェントのステータスを確認する：

# For Java applications
jps -l | grep opentelemetry

# For Python applications
ps aux | grep opentelemetry

# Check agent logs
tail -f /var/log/aws-opentelemetry-agent.log

エージェントの一般的な問題：

  • JVM に対して誤った Java エージェントバージョンを使用している
  • 依存関係またはライブラリが不足している
  • 設定ファイルの構文エラー
  • ログファイルへの書き込み権限の問題

ツールとコマンド

トラブルシューティングに役立つツールとコマンド。

CloudWatch CLI コマンド

# List Application Signals services
aws application-signals list-services

# Get service metrics
aws cloudwatch get-metric-data --metric-data-queries '[
  {
    "Id": "m1",
    "MetricStat": {
      "Metric": {
        "Namespace": "AWS/ApplicationSignals",
        "MetricName": "Latency"
      },
      "Period": 300,
      "Stat": "Average"
    }
  }
]'

# Query traces
aws xray get-trace-summaries --start-time 2024-01-01T00:00:00Z --end-time 2024-01-02T00:00:00Z

ログ分析

CloudWatch Logs Insights を使用してテレメトリデータを分析する：

# Find errors in application logs
fields @timestamp, @message
| filter @message like /ERROR/
| sort @timestamp desc
| limit 100

# Correlate traces with logs
fields @timestamp, traceId, @message
| filter ispresent(traceId)
| sort @timestamp desc
