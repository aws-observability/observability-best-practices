CloudWatch APM Documentation

Features & Capabilities

Application Signals: what it provides

CloudWatch Application Signals is AWS’s application performance monitoring (APM) experience in CloudWatch. After enablement, it automatically collects application telemetry and organizes it by service, operation, and dependency so you can quickly triage operational health and track longer-term performance goals.

  • Standard application metrics: call volume, availability, latency, faults, and errors.
  • Service and dependency topology: an Application Map generated from discovered services and relationships.
  • Deep troubleshooting: drill down from service health into correlated traces and related telemetry.

Application Map (topology + health)

The Application Map visualizes your application’s topology and highlights operational health. It shows connections between clients, canaries, services, and dependencies so operators can understand execution flow and identify where issues propagate.

What you can do

  • See SLO/SLI health: quickly spot services (and upstream/downstream impact) that are not meeting objectives.
  • Drill down: select nodes/edges to open detailed metrics and navigate to service details.
  • Group and filter: use default groupings (Related services, Environment) and create custom groupings using AWS tags or OpenTelemetry resource attributes.
  • Cross-account visibility: view services across multiple AWS accounts (when set up via cross-account observability).

Services, operations, and dependencies

Application Signals structures telemetry into a model that matches how teams debug production systems:

  • Service: your deployable unit (for example, an ECS service, EKS workload, Lambda function).
  • Operation: a request handler within a service (for example, an HTTP route or RPC method).
  • Dependency: downstream calls made by a service (for example, another service or AWS dependency).

This structure makes it easier to answer questions like “which operation regressed?”, “is this an upstream or downstream issue?”, and “which dependencies are contributing to latency or faults?”

Service level objectives (SLOs)

Create and monitor SLOs directly from the telemetry that Application Signals collects. SLOs help you track performance and reliability against targets and surface “at risk” services early.

  • Define SLOs on key SLIs: availability, latency, or other CloudWatch metrics relevant to business goals.
  • Track health: see SLI health status in service lists and the Application Map.
  • Alert: create alarms tied to SLO status/burn so teams can respond before an outage.

Instrumentation (OpenTelemetry-compatible)

Application Signals is OpenTelemetry-compatible and supports multiple ways to instrument applications depending on platform and language.

  • Supported languages: Java, Python, Node.js, and .NET (plus additional runtimes via OpenTelemetry zero-code instrumentation paths).
  • Supported systems: Amazon EKS, Kubernetes, Amazon ECS, Amazon EC2 (and enablement patterns that rely on the CloudWatch agent + AWS Distro for OpenTelemetry).
  • Common setup patterns: AWS Distro for OpenTelemetry (ADOT) SDK/collector or OpenTelemetry SDK + collector, depending on environment.

Works with RUM and Synthetics

Application Signals integrates with CloudWatch RUM and CloudWatch Synthetics. When you enable tracing for those clients/canaries, they can appear in the Application Map and help connect user experience and proactive tests to the backend services they call.

  • RUM clients: connect frontend performance and JS errors to backend service calls.
  • Synthetics canaries: validate critical flows and help detect issues before users do.
