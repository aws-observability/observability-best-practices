CloudWatch APM 문서

Java Spring + OTLP + CloudWatch (실습 연습)

이 가이드는 Java Spring Boot 서비스를 OpenTelemetry로 계측하고, 텔레메트리를 AWS로 내보내며, ECS/Fargate에 배포하고, 트레이스/로그/Application Signals 동작을 확인하는 전체 실습을 문서화합니다.

구현 예제 형태로 작성되었으며 런북 또는 워크숍 핸드아웃으로 사용할 수 있습니다.

1) 이 실습에서 보여주는 내용

  • Java 앱의 엔드투엔드 텔레메트리:
    • 트레이스 (X-Ray / Transaction Search)
    • 로그 (CloudWatch Logs OTLP 엔드포인트 또는 ECS의 awslogs)
    • 메트릭/Application Signals (CloudWatch Agent 사이드카 경로)
  • 다음의 차이점:
    • Micrometer 트레이싱 브릿지 (micrometer-tracing-bridge-otel) - 트레이싱 API 브릿징용
    • Micrometer 레지스트리 (micrometer-registry-cloudwatch2, micrometer-registry-otlp) - 메트릭 전용 싱크
  • 하드코딩된 Java 상수가 아닌 환경 변수(OTEL_*)를 통한 OTLP 엔드포인트 연결.

2) 아키텍처

+----------------------------------------------------------------------------------+
|                               Developer / Runtime Host                           |
|                                                                                  |
|  loadgen.sh ---> Spring Boot app (:8080)                                         |
|                    |                                                             |
|                    +--> @Observed method (DemoService.greet)                     |
|                    |      +--> custom span attrs/events                          |
|                    |      +--> downstream call to httpbin.org                    |
|                    |                                                             |
|                    +--> ADOT Java agent (-javaagent)                             |
|                           |                                                      |
|                           +--> OTLP traces ----> xray.<region>.amazonaws.com     |
|                           +--> OTLP logs ------> logs.<region>.amazonaws.com     |
|                           +--> AppSignals -----> monitoring.<region>.amazonaws.. |
|                                                                                  |
+----------------------------------------------------------------------------------+

4) OTLP 엔드포인트 매핑

시그널 엔드포인트 패턴 일반적인 환경 변수
트레이스 https://xray.<region>.amazonaws.com/v1/traces OTEL_EXPORTER_OTLP_TRACES_ENDPOINT, OTEL_EXPORTER_OTLP_TRACES_PROTOCOL=http/protobuf
메트릭 https://monitoring.<region>.amazonaws.com/v1/metrics OTEL_EXPORTER_OTLP_METRICS_ENDPOINT (또는 사이드카를 통한 App Signals 경로)
로그 https://logs.<region>.amazonaws.com/v1/logs OTEL_EXPORTER_OTLP_LOGS_ENDPOINT, OTEL_EXPORTER_OTLP_LOGS_HEADERS

ECS 사이드카 모드에서 앱 컨테이너는 트레이스/메트릭을 http://localhost:4316/...으로 보내고 CloudWatch Agent가 포워딩을 처리합니다.

5) 이 저장소에서 사용된 계측 모델

5.1 의존성

  • micrometer-tracing-bridge-otel이 pom.xml에 포함되어 있습니다.
  • 트레이싱을 위한 micrometer-registry-cloudwatch2 또는 micrometer-registry-otlp에 대한 의존성이 없습니다.

5.2 자동 + 수동 트레이싱

  • ADOT Java 에이전트의 자동 계측이 프레임워크/클라이언트 스팬을 캡처합니다.
  • DemoService.greet() 내부에 수동 보강이 추가됩니다:
    • 속성: demo.operation, demo.request.name, demo.downstream.status, demo.downstream.elapsed_ms, demo.custom.instrumented
    • 이벤트: demo.greet.start, demo.httpbin.success, demo.httpbin.failure
    • recordException(...) 및 setStatus(ERROR, ...)를 사용한 오류 기록

6) 스크린샷 (트레이스 세부 정보)

워크숍/데모 문서에는 아래 제공된 스크린샷을 사용하세요:

7) AWS 콘솔에서 텔레메트리를 보는 위치

트레이스 (기본)

  1. X-Ray → Traces
  2. 리전: us-east-1
  3. 시간 범위: 최근 15-60분
  4. 서비스: java-spring-otel-poc

사용자 정의 필드의 경우:

  • 트레이스 열기 → 세그먼트 java-spring-otel-poc → Metadata
  • 다음 키를 찾으세요:
    • demo.operation
    • demo.request.name
    • demo.downstream.status
    • demo.downstream.elapsed_ms
    • demo.custom.instrumented

로그

  • ECS 앱 로그: /ecs/java-spring-otel
  • CW 에이전트 사이드카 로그: /ecs/ecs-cwagent-java-spring-otel
  • 로컬 OTLP 로그 그룹 (로컬 실행 시): /aws/otel/java-spring-otel-poc

Application Signals

  • CloudWatch → Application Signals → Services
  • 지속적인 트래픽과 사이드카 경로를 통한 성공적인 메트릭 포워딩 후 java-spring-otel-poc를 찾으세요.

8) 핵심 요점

OpenTelemetry 자동 계측은 스팬을 생성할 수 있지만, 성공적인 AWS 관찰성 결과는 여전히 다음에 달려 있습니다:

  1. 올바른 OTLP 엔드포인트 연결
  2. 올바른 인증/포워딩 경로 (메트릭/App Signals의 경우 사이드카/콜렉터를 통하는 경우가 많음)
  3. 트레이싱 브릿지와 메트릭 레지스트리의 올바른 구분
  4. 지속적인 트래픽과 올바른 콘솔 뷰 (트레이스에는 X-Ray)
