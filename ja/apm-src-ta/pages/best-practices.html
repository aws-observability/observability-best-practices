சிறந்த நடைமுறைகள்

Instrumentation சிறந்த நடைமுறைகள்

உங்கள் observability தரவிலிருந்து மதிப்புள்ள நுண்ணறிவுகளைப் பெற திறம்பட instrumentation அவசியமாகும். குறைந்தபட்ச overhead-ல் அதிகபட்ச மதிப்பை உங்கள் instrumentation வழங்குவதை உறுதிசெய்ய இந்த சிறந்த நடைமுறைகளைப் பின்பற்றவும்.

என்ன Instrument செய்ய வேண்டும்

உங்கள் பயன்பாட்டின் மிக முக்கியமான பகுதிகளை instrument செய்வதில் கவனம் செலுத்தவும்:

உயர் முன்னுரிமை:

  • பொது API endpoints மற்றும் பயனர்-எதிர்நோக்கு செயல்பாடுகள்
  • தரவுத்தள வினவல்கள் மற்றும் வெளிப்புற சேவை அழைப்புகள்
  • வணிக-முக்கிய பணிப்பாய்வுகள் மற்றும் பரிவர்த்தனைகள்
  • பிழை பாதைகள் மற்றும் விதிவிலக்கு கையாளுதல்

நடுத்தர முன்னுரிமை:

  • உள் சேவை-க்கு-சேவை தொடர்பு
  • பின்னணி பணிகள் மற்றும் திட்டமிடப்பட்ட tasks
  • Cache செயல்பாடுகள்

குறைந்த முன்னுரிமை:

  • எளிய utility functions
  • நிலையான உள்ளடக்க வழங்கல்
  • Health check endpoints

பெயரிடும் மரபுகள்

நிலையான பெயரிடுதல் உங்கள் traces-ஐ தேடவும் வடிகட்டவும் எளிதாக்குகிறது:

Span பெயர்கள்:

  • கீழ் எழுத்துக்களை கோடுகளுடன் பயன்படுத்தவும்: http-get-user-profile
  • செயல்பாட்டு வகையை சேர்க்கவும்: db-query-users, redis-get-session
  • விளக்கமாக ஆனால் சுருக்கமாக இருக்கவும்: "process" அல்லது "handle" போன்ற பொதுவான பெயர்களைத் தவிர்க்கவும்

சேவை பெயர்கள்:

  • சூழல்கள் முழுவதும் நிலையான சேவை பெயரிடுதலைப் பயன்படுத்தவும்
  • பல-சூழல் அமைப்புகளுக்கு சேவை பெயரில் சூழலை சேர்க்கவும்
  • உங்கள் நிறுவனத்தின் பெயரிடும் மரபுகளைப் பின்பற்றவும்

Attributes மற்றும் Tags

உங்கள் spans மற்றும் traces-க்கு சூழலை சேர்க்க attributes-ஐ பயன்படுத்தவும்:

நிலையான Attributes:

  • http.method - HTTP method (GET, POST, போன்றவை)
  • http.url - கோரிக்கை URL (PII பற்றி கவனமாக இருக்கவும்)
  • http.status_code - HTTP பதில் நிலை
  • db.system - தரவுத்தள வகை (mysql, postgres, போன்றவை)
  • db.name - தரவுத்தள பெயர்
  • db.operation - தரவுத்தள செயல்பாடு (SELECT, INSERT, போன்றவை)

தனிப்பயன் Attributes:

  • user.id - பயனர் அடையாளங்காட்டி (PII எனில் hash செய்யப்பட்டது)
  • business.operation - வணிக செயல்பாட்டு வகை
  • version - பயன்பாட்டு பதிப்பு
  • environment - deployment சூழல்

PII மற்றும் பாதுகாப்பு கருத்தில் கொள்ள வேண்டியவை

உங்கள் telemetry-ல் உணர்திறன் தரவைப் பாதுகாக்கவும்:

ஒருபோதும் சேர்க்க வேண்டாம்:

  • கடவுச்சொற்கள், API keys, அல்லது அங்கீகார tokens
  • தனிநபர் அடையாளத் தகவல் (PII) - மின்னஞ்சல்கள், தொலைபேசி எண்கள் போன்றவை
  • உணர்திறன் தரவு கொண்ட query parameters-உடன் கூடிய முழு URLs
  • கிரெடிட் கார்ட் எண்கள் அல்லது நிதித் தகவல்

பாதுகாப்பான மாற்று வழிகள்:

  • Hash செய்யப்பட்ட அல்லது tokenized அடையாளங்காட்டிகளைப் பயன்படுத்தவும்
  • URLs-ன் path பகுதியை மட்டும் சேர்க்கவும்
  • குறிப்பிட்ட மதிப்புகளுக்கு பதிலாக பொதுவான நிலை குறிகாட்டிகளைப் பயன்படுத்தவும்
  • Instrumentation நிலையில் data masking-ஐ செயல்படுத்தவும்

Sampling உத்திகள்

Observability கவரேஜை பராமரிக்கும் அதே நேரத்தில் செலவுகளைக் கட்டுப்படுத்த Sampling உதவுகிறது.

Sampling உத்தியைத் தேர்ந்தெடுத்தல்

விகிதம் அடிப்படையிலான Sampling:

  • கோரிக்கைகளின் நிலையான சதவீதத்தை sample செய்யவும் (எ.கா., 10%)
  • செயல்படுத்தவும் புரிந்துகொள்ளவும் எளிமையானது
  • முக்கியமான ஆனால் அரிதான நிகழ்வுகளை தவறவிடலாம்

Tail Sampling:

  • Trace தரவை சேகரித்த பிறகு sampling முடிவுகளை எடுக்கவும்
  • பிழைகள் அல்லது அதிக தாமதம் கொண்ட traces-க்கு முன்னுரிமை அளிக்க முடியும்
  • மிகவும் சிக்கலானது ஆனால் மிகவும் புத்திசாலித்தனமானது

விதி அடிப்படையிலான Sampling:

  • குறிப்பிட்ட நிபந்தனைகளின் அடிப்படையில் sample செய்யவும் (பயனர், endpoint, போன்றவை)
  • முக்கிய பாதைகள் எப்போதும் trace செய்யப்படுவதை உறுதிசெய்யவும்
  • கவனமான விதி வரையறை தேவை

பரிந்துரைக்கப்படும் Sampling விகிதங்கள்

  • Development: முழு தெரிவுநிலைக்கு 100% sampling
  • Staging: சோதனைக்கு 50-100% sampling
  • Production: செலவு-திறன் கவரேஜுக்கு 10-20%
  • அதிக-போக்குவரத்து சேவைகள்: அளவைக் கட்டுப்படுத்த 1-5%
  • முக்கிய பாதைகள்: மற்ற விதிகளைப் பொருட்படுத்தாமல் 100% sampling

செயல்திறன் மேம்படுத்துதல்

Instrumentation-ன் செயல்திறன் தாக்கத்தை குறைக்கவும்.

Overhead-ஐ குறைக்கவும்

  • முடிந்தவரை asynchronous span processing-ஐ பயன்படுத்தவும்
  • Network அழைப்புகளைக் குறைக்க batch span exports-ஐ பயன்படுத்தவும்
  • பொருத்தமான buffer sizes-ஐ கட்டமைக்கவும்
  • Instrumentation செயல்திறன் metrics-ஐ கண்காணிக்கவும்

வள மேலாண்மை

  • Span exports-க்கு நியாயமான timeouts-ஐ அமைக்கவும்
  • Backend தோல்விகளுக்கு circuit breakers-ஐ செயல்படுத்தவும்
  • Collectors-க்கு connection pooling-ஐ பயன்படுத்தவும்
  • பொருத்தமான retry policies-ஐ கட்டமைக்கவும்

எச்சரிக்கை மற்றும் கண்காணிப்பு

பயன்பாட்டு ஆரோக்கியத்தின் மேல் நிற்க திறம்பட alerts-ஐ அமைக்கவும்.

கண்காணிக்க வேண்டிய முக்கிய Metrics

  • தாமதம்: P95/P99 பதில் நேரங்கள்
  • பிழை விகிதம்: தோல்வியடைந்த கோரிக்கைகளின் சதவீதம்
  • செயல்திறன்: வினாடிக்கு கோரிக்கைகள்
  • நிறைவு: வள பயன்பாடு
  • Trace அளவு: சேகரிக்கப்பட்ட traces எண்ணிக்கை

SLO அடிப்படையிலான எச்சரிக்கை

Service Level Objectives-ஐ வரையறுத்து அவை ஆபத்தில் இருக்கும்போது alert செய்யவும்:

  • தாமத SLO: 95% கோரிக்கைகள் < 500ms
  • பிழை SLO: 99.9% வெற்றி விகிதம்
  • கிடைக்கும் தன்மை SLO: 99.95% uptime

செலவு மேலாண்மை

மதிப்பை பராமரிக்கும் அதே நேரத்தில் observability செலவுகளைக் கட்டுப்படுத்தவும்.

செலவு மேம்படுத்துதல் உத்திகள்

  • பொருத்தமான sampling விகிதங்களை செயல்படுத்தவும்
  • தரவு தக்கவைப்பு கொள்கைகளைப் பயன்படுத்தவும்
  • தேவையற்ற spans மற்றும் attributes-ஐ வடிகட்டவும்
  • பழைய தரவை மலிவான சேமிப்பகத்திற்கு archive செய்யவும்
  • தரவு அளவை கண்காணித்து மேம்படுத்தவும்

தரவு தக்கவைப்பு கொள்கைகள்

  • Hot Storage: சமீபத்திய தரவு (7-30 நாட்கள்) செயலில் உள்ள சிக்கல் தீர்வுக்கு
  • Warm Storage: வரலாற்று தரவு (30-90 நாட்கள்) போக்கு பகுப்பாய்வுக்கு
  • Cold Storage: Archive தரவு (90+ நாட்கள்) இணக்கம் மற்றும் அரிய வினவல்களுக்கு
