CloudWatch APM 文档

功能与能力

Application Signals：提供的功能

CloudWatch Application Signals 是 AWS 在 CloudWatch 中的应用性能监控（APM）体验。启用后，它会自动收集应用遥测数据，并按服务、操作和依赖项进行组织，以便您快速分诊运行状况并跟踪长期性能目标。

  • 标准应用指标：调用量、可用性、延迟、故障和错误。
  • 服务和依赖拓扑：根据发现的服务和关系生成的应用程序地图。
  • 深度故障排除：从服务健康状况深入到相关追踪和关联遥测数据。

应用程序地图（拓扑 + 健康状况）

应用程序地图可视化应用程序的拓扑并突出显示运行状况。它显示客户端、金丝雀、服务和依赖项之间的连接，以便运维人员了解执行流程并识别问题传播的位置。

您可以做什么

  • 查看 SLO/SLI 健康状态：快速发现未达到目标的服务（以及上下游影响）。
  • 深入分析：选择节点/边以打开详细指标并导航到服务详情。
  • 分组和过滤：使用默认分组（相关服务、环境）并使用 AWS 标签或 OpenTelemetry 资源属性创建自定义分组。
  • 跨账户可见性：查看多个 AWS 账户中的服务（通过跨账户 Observability 设置时）。

服务、操作和依赖项

Application Signals 将遥测数据构建为与团队调试生产系统方式相匹配的模型：

  • 服务：您的可部署单元（例如 ECS 服务、EKS 工作负载、Lambda 函数）。
  • 操作：服务中的请求处理程序（例如 HTTP 路由或 RPC 方法）。
  • 依赖项：服务发出的下游调用（例如另一个服务或 AWS 依赖项）。

这种结构使您更容易回答诸如"哪个操作发生了回退？"、"这是上游还是下游问题？"以及"哪些依赖项导致了延迟或故障？"等问题。

服务级别目标（SLO）

直接从 Application Signals 收集的遥测数据创建和监控 SLO。SLO 帮助您跟踪性能和可靠性目标，并尽早发现"有风险"的服务。

  • 在关键 SLI 上定义 SLO：可用性、延迟或与业务目标相关的其他 CloudWatch 指标。
  • 跟踪健康状态：在服务列表和应用程序地图中查看 SLI 健康状态。
  • 告警：创建与 SLO 状态/消耗关联的告警，以便团队在故障发生前做出响应。

检测（兼容 OpenTelemetry）

Application Signals 兼容 OpenTelemetry，支持多种方式对应用程序进行检测，具体取决于平台和语言。

  • 支持的语言：Java、Python、Node.js 和 .NET（以及通过 OpenTelemetry 零代码检测路径支持的其他运行时）。
  • 支持的系统：Amazon EKS、Kubernetes、Amazon ECS、Amazon EC2（以及依赖 CloudWatch Agent + AWS Distro for OpenTelemetry 的启用模式）。
  • 常见设置模式：AWS Distro for OpenTelemetry (ADOT) SDK/收集器或 OpenTelemetry SDK + 收集器，具体取决于环境。

与 RUM 和 Synthetics 配合使用

Application Signals 与 CloudWatch RUM 和 CloudWatch Synthetics 集成。当您为这些客户端/金丝雀启用追踪时，它们可以出现在应用程序地图中，帮助将用户体验和主动测试与它们调用的后端服务连接起来。

  • RUM 客户端：将前端性能和 JS 错误与后端服务调用连接起来。
  • Synthetics 金丝雀：验证关键流程并帮助在用户之前检测问题。
