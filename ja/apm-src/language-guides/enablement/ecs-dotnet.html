Overview

This guide provides step-by-step instructions to deploy a .NET Core application on Amazon ECS with AWS Application Signals enabled. No CDK required - pure AWS CLI and Docker.

What you'll accomplish:
  • Deploy a .NET Core app on ECS Fargate
  • Enable AWS Application Signals auto-instrumentation
  • View your service in CloudWatch Application Signals console
  • Monitor service metrics, traces, and operations
Time to complete: 30-45 minutes

Prerequisites

Required Tools

  • AWS CLI v2 (latest version)
  • Docker Desktop
  • .NET 8.0 SDK (for local development)
  • AWS Account with appropriate permissions
  • Text editor

AWS Permissions Required

  • ECS: Full access (create clusters, services, task definitions)
  • ECR: Push images
  • IAM: Create roles and policies
  • CloudWatch: Logs and Application Signals access
  • EC2: VPC, subnet, and security group access
  • SSM: Parameter Store write access

Knowledge Assumed

  • Basic AWS CLI usage
  • Basic Docker commands
  • Basic understanding of ECS concepts
  • Basic .NET Core knowledge

Architecture

Your ECS task will contain 3 containers:

  1. Init Container (init): Copies ADOT .NET auto-instrumentation files
  2. CloudWatch Agent (ecs-cwagent): Collects traces and metrics
  3. Application Container (app): Your .NET Core application
Important Compatibility Note: The AWS Distro for OpenTelemetry (ADOT) SDK for .NET does not support AWS SDK for .NET V4. Use AWS SDK .NET V3 for full Application Signals support. This guide provides a working ECS implementation with basic OpenTelemetry configuration using AWS SDK .NET V3.

Flow:

Init Container (runs once, copies files)

↓

Application starts (auto-instrumented via .NET startup hooks)

↓

Sends traces/metrics to CloudWatch Agent (localhost:4316)

↓

CloudWatch Agent sends to Application Signals

1Enable Application Signals in Your Account

Important: Application Signals must be enabled once per AWS account per region.

Option 1: Enable via AWS Console (Recommended)

  1. Go to AWS Console → CloudWatch → Application Signals
  2. Select your region (e.g., us-east-1)
  3. Click "Get Started" or "Enable Application Signals"
  4. Wait 2-3 minutes for activation

Option 2: Verify if Already Enabled

# Check if service-linked role exists
aws iam get-role \
  --role-name AWSServiceRoleForCloudWatchApplicationSignals \
  --region us-east-1

# If role exists, Application Signals is enabled

Output should show: Role ARN if enabled, error if not.

2.2.1 Project Dependencies

The project uses AWS SDK .NET V3 and OpenTelemetry packages for Application Signals compatibility:

Key Dependencies:

  • AWS SDK .NET V3: AWSSDK.Core, AWSSDK.CloudWatch, AWSSDK.XRay (V3.7.400.1)

Why AWS SDK V3? The AWS Distro for OpenTelemetry (ADOT) SDK for .NET does not support AWS SDK for .NET V4. Using V3 ensures full Application Signals support.

3Build and Push Docker Image

3.1 Set Variables

# Replace with your values
export AWS_REGION="us-east-1"
export AWS_ACCOUNT_ID="YOUR_ACCOUNT_ID"
export AWS_PROFILE="your-profile-name"  # Optional if using default
export APP_NAME="my-appsignals-dotnet-app"

Get your account ID:

export AWS_ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
echo "Account ID: $AWS_ACCOUNT_ID"

3.2 Create ECR Repository

aws ecr create-repository \
  --repository-name ${APP_NAME} \
  --region ${AWS_REGION}

Output: Repository URI (save this)

3.3 Build Docker Image

Important: If you're building on Apple Silicon (M1/M2) Mac, you need to build for the correct architecture. 
cd ~/my-appsignals-dotnet-app

# For Apple Silicon Macs, build for AMD64 architecture
docker build --platform linux/amd64 -t ${APP_NAME} .

# For Intel Macs or Linux, use standard build
# docker build -t ${APP_NAME} .

3.4 Push to ECR

# Login to ECR
aws ecr get-login-password --region ${AWS_REGION} | \
  docker login --username AWS --password-stdin \
  ${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com

# Tag image
docker tag ${APP_NAME}:latest \
  ${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com/${APP_NAME}:latest

# Push image
docker push ${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com/${APP_NAME}:latest

4Create CloudWatch Agent Configuration

4.1 Create Configuration File

cat > /tmp/ecs-cwagent-dotnet.json << 'EOF'
{
  "traces": {
    "traces_collected": {
      "application_signals": {}
    }
  },
  "logs": {
    "metrics_collected": {
      "application_signals": {}
    }
  }
}
EOF

4.2 Upload to SSM Parameter Store

aws ssm put-parameter \
  --name "ecs-cwagent-${APP_NAME}" \
  --type "String" \
  --value "$(cat /tmp/ecs-cwagent-dotnet.json)" \
  --region "${AWS_REGION}" \
  --overwrite

Output: Version number (1 if new, incremented if updating)

5Create IAM Roles

5.1 Create Trust Policy

cat > /tmp/task-trust-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "ecs-tasks.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
EOF

5.2 Create Task Role

aws iam create-role \
  --role-name ${APP_NAME}-ecs-task-role \
  --assume-role-policy-document file:///tmp/task-trust-policy.json

# Attach required policies
aws iam attach-role-policy \
  --role-name ${APP_NAME}-ecs-task-role \
  --policy-arn arn:aws:iam::aws:policy/CloudWatchAgentServerPolicy

aws iam attach-role-policy \
  --role-name ${APP_NAME}-ecs-task-role \
  --policy-arn arn:aws:iam::aws:policy/AWSXRayDaemonWriteAccess

5.3 Create Task Execution Role

aws iam create-role \
  --role-name ${APP_NAME}-ecs-execution-role \
  --assume-role-policy-document file:///tmp/task-trust-policy.json

# Attach policies
aws iam attach-role-policy \
  --role-name ${APP_NAME}-ecs-execution-role \
  --policy-arn arn:aws:iam::aws:policy/service-role/AmazonECSTaskExecutionRolePolicy

aws iam attach-role-policy \
  --role-name ${APP_NAME}-ecs-execution-role \
  --policy-arn arn:aws:iam::aws:policy/AmazonSSMReadOnlyAccess

5.4 Add CloudWatch Logs Permissions

cat > /tmp/logs-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogGroup",
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "arn:aws:logs:*:*:log-group:/ecs/*"
    }
  ]
}
EOF

aws iam put-role-policy \
  --role-name ${APP_NAME}-ecs-execution-role \
  --policy-name CloudWatchLogsPolicy \
  --policy-document file:///tmp/logs-policy.json

export CLUSTER_NAME="${APP_NAME}-cluster"

aws ecs create-cluster \
  --cluster-name ${CLUSTER_NAME} \

7Setup Networking

7.1 Get VPC and Subnets

If you already have a VPC:

# List VPCs
aws ec2 describe-vpcs --region ${AWS_REGION} \
  --query 'Vpcs[*].{VpcId:VpcId,CIDR:CidrBlock,Name:Tags[?Key==`Name`].Value|[0]}'

# Choose your VPC
export VPC_ID="vpc-xxxxx"

# Get private subnets
aws ec2 describe-subnets --region ${AWS_REGION} \
  --filters "Name=vpc-id,Values=${VPC_ID}" \
  --query 'Subnets[?MapPublicIpOnLaunch==`false`].[SubnetId,AvailabilityZone,CidrBlock]'

# Save subnet IDs
export SUBNET_1="subnet-xxxxx"
export SUBNET_2="subnet-yyyyy"

7.2 Create Security Group

export SG_NAME="${APP_NAME}-ecs-sg"

# Create security group
SG_ID=$(aws ec2 create-security-group \
  --group-name ${SG_NAME} \
  --description "Security group for ${APP_NAME} ECS tasks" \
  --vpc-id ${VPC_ID} \
  --region ${AWS_REGION} \
  --query 'GroupId' \
  --output text)

echo "Security Group ID: $SG_ID"
export SG_ID

# Add ingress rule (allow HTTP from anywhere for testing)
aws ec2 authorize-security-group-ingress \
  --group-id ${SG_ID} \
  --protocol tcp \
  --port 8080 \
  --cidr 0.0.0.0/0 \
  --region ${AWS_REGION}
For production: Replace 0.0.0.0/0 with specific IP ranges or load balancer security group.

aws ecs register-task-definition \
  --cli-input-json file:///tmp/app-signals-dotnet-task-def.json \
  --region ${AWS_REGION}

Output: Should show revision 1 with status ACTIVE

9Deploy ECS Service

aws ecs create-service \
  --cluster ${CLUSTER_NAME} \
  --service-name ${APP_NAME}-service \
  --task-definition ${APP_NAME}-task:1 \
  --desired-count 1 \
  --launch-type FARGATE \
  --network-configuration "awsvpcConfiguration={subnets=[${SUBNET_1},${SUBNET_2}],securityGroups=[${SG_ID}],assignPublicIp=ENABLED}" \
  --region ${AWS_REGION}
Note: assignPublicIp=ENABLED is for testing. For production, use a load balancer.

Troubleshooting

Issue: Service Not Appearing in Application Signals

Check:

# 1. Verify metrics are being published
aws cloudwatch list-metrics \
  --namespace ApplicationSignals \
  --region ${AWS_REGION} \
  --dimensions Name=Service,Value=${APP_NAME}

# 2. Check if Application Signals is enabled
aws iam get-role \
  --role-name AWSServiceRoleForCloudWatchApplicationSignals

# 3. Check container logs for errors
aws logs tail /ecs/${APP_NAME} --region ${AWS_REGION} --since 10m

Issue: Init Container Fails

Check:

# Describe task to see init container exit code
aws ecs describe-tasks \
  --cluster ${CLUSTER_NAME} \
  --tasks ${TASK_ARN} \
  --region ${AWS_REGION} \
  --query 'tasks[0].containers[?name==`init`].{name:name,exitCode:exitCode,reason:reason}'

Expected: exitCode: 0

If non-zero: Image might not be available. Verify tag v1.34.0 exists:

docker pull public.ecr.aws/aws-observability/adot-autoinstrumentation-dotnet:v1.34.0

Issue: Application Can't Connect to CloudWatch Agent

Symptoms:

Connection refused to localhost:4316

Check:

  1. Verify CloudWatch agent container is RUNNING
  2. Check container dependency - app should depend on init with condition SUCCESS
  3. Verify environment variables (especially endpoints: http://localhost:4316 and http://localhost:2000)

Issue: No Traces in X-Ray

Common causes:

  1. Missing DOTNET_STARTUP_HOOKS - Critical for .NET auto-instrumentation
  2. Missing DOTNET_ADDITIONAL_DEPS - Required for .NET instrumentation
  3. Missing DOTNET_SHARED_STORE - Required for .NET instrumentation
  4. Missing volume mount - .NET instrumentation files must be accessible

Issue: IAM Permission Errors

Symptoms:

AccessDeniedException: ... logs:CreateLogGroup

Fix: Add CloudWatch Logs permissions to execution role (see Step 5.4)

Issue: "exec format error" in Container

Symptoms:

exec /usr/bin/dotnet: exec format error

Cause: Architecture mismatch - Docker image built on ARM64 (Apple Silicon) but ECS Fargate runs on x86_64.

Fix: Rebuild the Docker image for the correct architecture:

# Rebuild for AMD64 architecture
docker build --platform linux/amd64 -t ${APP_NAME} .

# Tag and push the corrected image
docker tag ${APP_NAME}:latest \
  ${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com/${APP_NAME}:latest

docker push ${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com/${APP_NAME}:latest

# Update the ECS service to use the new image
aws ecs update-service \
  --cluster ${CLUSTER_NAME} \
  --service ${APP_NAME}-service \
  --force-new-deployment \
  --region ${AWS_REGION}

Init Container Details

Image: public.ecr.aws/aws-observability/adot-autoinstrumentation-dotnet:v1.34.0

Why versioned tag, not :latest?

  • The :latest tag doesn't exist for .NET ADOT images
  • Use specific version tags (check https://github.com/aws-observability/aws-otel-dotnet-instrumentation/releases)
  • Current stable: v1.34.0

What it does:

  1. Runs once before app container starts
  2. Copies ADOT .NET instrumentation files from /autoinstrumentation/ to shared volume
  3. Exits with code 0
  4. App container mounts the volume and uses .NET startup hooks to load instrumentation

CloudWatch Agent Configuration

{
  "traces": {

      "application_signals": {}   // Not just "otlp"!
    }
  },

    "metrics_collected": {
      "application_signals": {}

Application Code Requirements

DO:

  • Write normal .NET Core code
  • Use standard libraries (ASP.NET Core, HttpClient, etc.)
  • Keep code clean - no OpenTelemetry imports needed
  • Use standard ASP.NET Core attributes ([ApiController], [HttpGet], etc.)

DON'T:

  • Import OpenTelemetry.* packages
  • Manually configure OpenTelemetry
  • Add manual tracing code (unless needed for custom spans)
  • Install OpenTelemetry dependencies in .csproj
