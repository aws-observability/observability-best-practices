Overview

This guide provides step-by-step instructions to deploy a Java Lambda function with AWS Application Signals enabled. The simplest approach using AWS Lambda console with built-in layer support.

What you'll accomplish:
  • Deploy a Java Lambda function with Application Signals
  • Enable AWS Application Signals auto-instrumentation via Lambda Layer
  • View your service in CloudWatch Application Signals console
  • Monitor service metrics, traces, and operations
Time to complete: 15-20 minutes

Prerequisites

Required Tools

  • AWS CLI v2 (latest version)
  • Java 17+
  • Maven 3.8+
  • AWS Account with appropriate permissions
  • Text editor

AWS Permissions Required

  • Lambda: Create and manage functions
  • IAM: Create roles and policies
  • CloudWatch: Logs and Application Signals access
  • S3: Basic read access (for testing)

Architecture

Your Lambda function will be automatically instrumented using:

  1. AWS Lambda Layer for OpenTelemetry - Provides auto-instrumentation
  2. Application Signals Integration - Built into the layer
  3. Automatic Tracing - HTTP requests, AWS SDK calls, function execution

Step 1: Enable Application Signals in Your Account

Important: Application Signals must be enabled once per AWS account per region.

Option 1: Enable via AWS Console (Recommended)

  1. Go to AWS Console → CloudWatch → Application Signals
  2. Select your region (e.g., us-east-1)
  3. Click "Get Started" or "Enable Application Signals"
  4. Wait 2-3 minutes for activation

Step 2: Prepare Your Java Lambda Function

2.1 Create Application Directory

mkdir -p ~/my-lambda-java-appsignals
cd ~/my-lambda-java-appsignals

2.2 Create Project Structure

mkdir -p src/main/java/com/example/appsignals/controller

2.3 Create pom.xml

Create a Maven POM file with the following dependencies:

  • AWS Lambda Core
  • AWS Lambda Events
  • AWS SDK for S3
  • Spring Boot Web Starter (for local testing)
  • JSON Processing
  • HTTP Client

IMPORTANT: Do NOT add OpenTelemetry dependencies. They're provided by the AWS Lambda Layer.

Step 3: Build Lambda Deployment Package

3.1 Set Variables

export AWS_REGION="us-east-1"
export AWS_ACCOUNT_ID="YOUR_ACCOUNT_ID"
export FUNCTION_NAME="lambda-java"

3.2 Build JAR File

mvn clean package

Step 4: Create IAM Role for Lambda

Create an IAM role with the following policies:

  • AWSLambdaBasicExecutionRole
  • CloudWatchLambdaApplicationSignalsExecutionRolePolicy (CRITICAL!)
  • AmazonS3ReadOnlyAccess (for testing)

4.1 Create IAM Role

aws iam create-role \
  --role-name lambda-java-execution-role \
  --assume-role-policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Principal": {"Service": "lambda.amazonaws.com"},
      "Action": "sts:AssumeRole"
    }]
  }'

4.2 Attach Required Policies

aws iam attach-role-policy \
  --role-name lambda-java-execution-role \
  --policy-arn arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole

aws iam attach-role-policy \
  --role-name lambda-java-execution-role \
  --policy-arn arn:aws:iam::aws:policy/CloudWatchLambdaApplicationSignalsExecutionRolePolicy

aws iam attach-role-policy \
  --role-name lambda-java-execution-role \
  --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess

4.3 Get Role ARN

ROLE_ARN=$(aws iam get-role \
  --role-name lambda-java-execution-role \
  --query 'Role.Arn' \
  --output text)

echo "Role ARN: $ROLE_ARN"

Step 5: Deploy Lambda Function

5.1 Create Lambda Function

aws lambda create-function \
  --function-name lambda-java \
  --runtime java17 \
  --role "$ROLE_ARN" \
  --handler com.example.appsignals.ApplicationSignalsApplication::handleRequest \
  --zip-file fileb://target/lambda-java-appsignals-1.0.0.jar \
  --timeout 30 \
  --memory-size 512

5.2 Add AWS Lambda Layer for OpenTelemetry

For us-east-1:

LAYER_ARN="arn:aws:lambda:us-east-1:615299751070:layer:AWSOpenTelemetryDistroJava:18"

5.3 Configure Environment Variables

aws lambda update-function-configuration \
  --function-name ${FUNCTION_NAME} \
  --environment Variables='{
    "AWS_LAMBDA_EXEC_WRAPPER": "/opt/otel-instrument",
    "OTEL_SERVICE_NAME": "'${FUNCTION_NAME}'",
    "OTEL_RESOURCE_ATTRIBUTES": "service.name='${FUNCTION_NAME}',deployment.environment=lambda:'${AWS_REGION}'"
  }'

Step 6: Test Lambda Function

6.1 Test Basic Functionality

aws lambda invoke \
  --function-name ${FUNCTION_NAME} \
  --payload '{"path": "/", "httpMethod": "GET"}' \
  --region ${AWS_REGION} \
  response.json

6.2 Generate Traffic for Application Signals

for i in {1..50}; do
  aws lambda invoke \
    --function-name ${FUNCTION_NAME} \
    --payload "{\"path\": \"/api/hello/User${i}\", \"httpMethod\": \"GET\"}" \
    --region ${AWS_REGION} \
    /dev/null
done

Step 7: Generate Traffic for Testing

To test Application Signals, you need to generate traffic to your Lambda function. We provide two traffic generator options:

7.1 Python Traffic Generator (Recommended)

Create a Python script to generate realistic traffic patterns:

#!/usr/bin/env python3
import boto3
import json
import time
import random
import sys

def generate_traffic(function_name, region='us-east-1', profile='eksworkstream', num_requests=50):
    # Initialize Lambda client
    session = boto3.Session(profile_name=profile)
    lambda_client = session.client('lambda', region_name=region)
    
    # Test endpoints with different characteristics
    endpoints = [
        {"path": "/", "method": "GET", "name": "Root", "weight": 20},
        {"path": "/health", "method": "GET", "name": "Health", "weight": 15},
        {"path": "/api/hello/User", "method": "GET", "name": "Hello", "weight": 30},
        {"path": "/api/slow", "method": "GET", "name": "Slow", "weight": 10},
        {"path": "/api/error", "method": "GET", "name": "Error", "weight": 10},
        {"path": "/api/external", "method": "GET", "name": "External", "weight": 8},
        {"path": "/api/s3", "method": "GET", "name": "S3", "weight": 5},
        {"path": "/api/database", "method": "GET", "name": "Database", "weight": 2},
    ]
    
    # Create weighted list for random selection
    weighted_endpoints = []
    for endpoint in endpoints:
        weighted_endpoints.extend([endpoint] * endpoint["weight"])
    
    success_count = 0
    error_count = 0
    
    for i in range(1, num_requests + 1):
        endpoint = random.choice(weighted_endpoints)
        
        if endpoint["name"] == "Hello":
            user_name = f"User{random.randint(1, 1000)}"
            path = f"/api/hello/{user_name}"
        else:
            path = endpoint["path"]
        
        payload = {"path": path, "httpMethod": endpoint["method"]}
        
        try:
            response = lambda_client.invoke(
                FunctionName=function_name,
                InvocationType='RequestResponse',
                Payload=json.dumps(payload)
            )
            
            response_payload = json.loads(response['Payload'].read())
            status_code = response_payload.get('statusCode', 200)
            
            if status_code == 200:
                success_count += 1
            else:
                error_count += 1
                
            print(f"Request {i}: {endpoint['name']} | Status: {status_code}")
            time.sleep(random.uniform(0.1, 0.5))
            
        except Exception as e:
            error_count += 1
            print(f"Request {i}: {endpoint['name']} | Error: {str(e)}")
    
    print(f"Summary: {success_count} successful, {error_count} errors")

if __name__ == "__main__":
    function_name = sys.argv[1] if len(sys.argv) > 1 else "lambda-java"
    num_requests = int(sys.argv[2]) if len(sys.argv) > 2 else 50
    generate_traffic(function_name, num_requests=num_requests)

7.2 Shell Script Traffic Generator

Alternative shell script for traffic generation:

#!/bin/bash
FUNCTION_NAME="lambda-java"
NUM_REQUESTS=50

endpoints=(
    "root:/:GET:20"
    "health:/health:GET:15"
    "hello:/api/hello:GET:30"
    "slow:/api/slow:GET:10"
    "error:/api/error:GET:10"
    "external:/api/external:GET:8"
    "s3:/api/s3:GET:5"
    "database:/api/database:GET:2"
)

for i in $(seq 1 $NUM_REQUESTS); do
    endpoint_info=${endpoints[$((RANDOM % ${#endpoints[@]}))]}
    IFS=':' read -r name path method weight <<< "$endpoint_info"
    
    if [ "$name" = "hello" ]; then
        user_name="User$((RANDOM % 1000 + 1))"
        full_path="/api/hello/$user_name"
    else
        full_path="$path"
    fi
    
    payload="{\"path\": \"$full_path\", \"httpMethod\": \"$method\"}"
    
    aws lambda invoke \
        --function-name $FUNCTION_NAME \
        --payload "$payload" \
        /tmp/response.json
    
    echo "Request $i: $name"
    sleep 0.2
done

Step 8: Verify Application Signals

8.1 Check Application Signals Service Discovery

aws application-signals list-services \
  --region ${AWS_REGION} \
  --start-time $(date -u -v-2H +%s) \
  --end-time $(date -u +%s) \
  --query 'ServiceSummaries[*].{Name:KeyAttributes.Name,Environment:KeyAttributes.Environment}'

Expected: Your service should appear with:

  • Name: lambda-java
  • Environment: lambda:us-east-1

Step 9: View in AWS Console

9.1 Console Access

  1. Navigate to: AWS Console → CloudWatch → Application Signals → Services
  2. Look for your service: Service Name: lambda-java
  3. Explore: Service Map, Service Dashboard, Operations, Dependencies

Alternative: Enable via Lambda Console (Simplest Method)

Recommended when experiencing shell issues! This method bypasses all command-line problems and is the most reliable approach.

If you prefer using the AWS Console or encounter shell environment issues:

  1. Go to AWS Lambda Console
  2. Click "Create function"
  3. Choose "Author from scratch"
  4. Function name: lambda-java
  5. Runtime: Java 17
  6. Create function
  7. Upload your JAR file: target/lambda-java-appsignals-1.0.0.jar
  8. Set handler to: com.example.appsignals.ApplicationSignalsApplication::handleRequest
  9. Go to Configuration → Monitoring and operations tools
  10. Click "Edit"
  11. Enable both checkboxes:
    • ✅ "Automatically collect application traces and standard application metrics with Application Signals"
    • ✅ "Automatically collect Lambda service traces for end to end visibility with X-Ray"
  12. Click "Save"

That's it! The console automatically:

  • Adds the AWS Lambda Layer for OpenTelemetry
  • Sets the AWS_LAMBDA_EXEC_WRAPPER environment variable
  • Attaches the required IAM policy

Troubleshooting

Issue: Shell Environment Problems

Problem: Commands fail with errors like parse error near 'cursor_snap_ENV_VARS...' or command not found: dump_zsh_state

Cause: Cursor IDE environment variable injection is corrupting the shell environment

Solutions:

  1. Use a fresh terminal window outside of Cursor IDE
  2. Restart Cursor IDE completely and try again
  3. Use VS Code instead of Cursor IDE
  4. Use the Lambda Console method (recommended for shell issues)

Issue: Role ARN is Empty

Problem: aws: error: argument --role: expected one argument

Cause: The ROLE_ARN variable is empty, indicating IAM role creation failed

Debug Steps:

  1. Check if role exists: 
    aws iam get-role --role-name lambda-java-execution-role
  2. Create role if missing: 
    aws iam create-role \
  --role-name lambda-java-execution-role \
  --assume-role-policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Principal": {"Service": "lambda.amazonaws.com"},
      "Action": "sts:AssumeRole"
    }]
  }'
  3. Get role ARN manually: 
    aws iam get-role \
  --role-name lambda-java-execution-role \
  --query 'Role.Arn' \
  --output text
  4. Use ARN directly in create-function command

Issue: Command Syntax Errors

Common mistakes:

  • Running multiple commands on the same line with periods
  • Using duplicate parameters (e.g., --role twice)
  • Missing quotes around variables

❌ WRONG - Don't do this:

# WRONG: Commands on same line with period
aws iam create-role \ ... }. aws lambda create-function \ ...

# WRONG: Duplicate --role parameter
aws lambda create-function \
  --role ${ROLE_ARN} \
  --role lambda-java-execution-role

# WRONG: Missing quotes around variable
--role $ROLE_ARN

✅ CORRECT - Do this instead:

# Step 1: Create IAM role (separate command)
aws iam create-role \
  --role-name lambda-java-execution-role \
  --assume-role-policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Principal": {"Service": "lambda.amazonaws.com"},
      "Action": "sts:AssumeRole"
    }]
  }'

# Step 2: Attach policies (separate commands)
aws iam attach-role-policy \
  --role-name lambda-java-execution-role \
  --policy-arn arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole

aws iam attach-role-policy \
  --role-name lambda-java-execution-role \
  --policy-arn arn:aws:iam::aws:policy/CloudWatchLambdaApplicationSignalsExecutionRolePolicy

# Step 3: Get role ARN
ROLE_ARN=$(aws iam get-role \
  --role-name lambda-java-execution-role \
  --query 'Role.Arn' \
  --output text)

# Step 4: Create Lambda function (with quotes around variable)
aws lambda create-function \
  --function-name lambda-java \
  --runtime java17 \
  --role "$ROLE_ARN" \
  --handler com.example.appsignals.ApplicationSignalsApplication::handleRequest \
  --zip-file fileb://target/lambda-java-appsignals-1.0.0.jar \
  --timeout 30 \
  --memory-size 512

Issue: Maven Build Failures

Common Maven issues:

  • Shade plugin errors: Use the simplified pom.xml provided
  • Java version mismatch: Ensure Java 17 is installed and configured
  • Dependency conflicts: Remove Spring Boot dependencies if not needed

Issue: Service Not Appearing in Application Signals

Check:

  • Verify Application Signals is enabled in your account
  • Check if the OpenTelemetry layer is attached
  • Verify environment variables are set correctly
  • Wait 5-10 minutes after generating traffic
  • Ensure the IAM policy CloudWatchLambdaApplicationSignalsExecutionRolePolicy is attached

Issue: Lambda Function Errors

Common issues:

  • Missing IAM policy - Ensure CloudWatchLambdaApplicationSignalsExecutionRolePolicy is attached
  • Wrong layer ARN - Verify the layer ARN for your region
  • Missing environment variable - Ensure AWS_LAMBDA_EXEC_WRAPPER=/opt/otel-instrument
  • Handler mismatch - Ensure handler is com.example.appsignals.ApplicationSignalsApplication::handleRequest
  • JAR file not found - Ensure Maven build completed successfully

Issue: Permission Denied Errors

Check your AWS credentials:

  • Verify AWS profile: aws configure list
  • Check caller identity: aws sts get-caller-identity
  • Ensure you have the required IAM permissions
