CloudWatch APM Documentation

Performance Analytics

Standard application metrics

Application Signals publishes standard application metrics to CloudWatch (namespace ApplicationSignals). These metrics are designed for fast triage across services, operations, and dependencies.

  • Latency (ms)
  • Fault (HTTP 5xx and span status errors)
  • Error (HTTP 4xx)
  • Availability is computed in the console as \( (1 - \text{Faults}/\text{Total}) \times 100 \) where Total is derived from SampleCount(Latency). 4xx responses are treated as successful for this availability calculation.

See: Metrics collected by Application Signals.

Dimensions (service vs dependency)

Metrics are emitted with dimensions that let you break down performance by:

  • Service metrics: Environment, Service, Operation
  • Dependency metrics: Environment, Service, Operation plus RemoteService / RemoteOperation (and other remote dimensions depending on platform)

This makes it easier to answer questions like “which operation regressed?” and “is latency driven by a dependency call?”

Runtime metrics (where supported)

Application Signals can also collect runtime metrics for supported runtimes (for example, JVM GC, memory, CPU; Python process metrics; .NET runtime metrics). Runtime metrics are not collected for Node.js.

You can disable runtime metrics collection with OTEL_AWS_APPLICATION_SIGNALS_RUNTIME_ENABLED=false (platform-specific steps are in the docs).
