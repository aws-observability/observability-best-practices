Guides de debogage

Flux de triage rapide (recommande)

  1. Commencez par les SLO (si definis) : identifiez les services/operations/dependances en mauvaise sante.
  2. Utilisez les Services : triez par latence elevee ou taux de fautes pour trouver les principaux responsables.
  3. Ouvrez le detail du service : inspectez les operations et les aretes de dependance ; selectionnez un point de donnees de latence/faute.
  4. Passez aux traces/spans correles : inspectez la chronologie de la trace pour trouver l'appel en aval lent/defaillant.
  5. Confirmez avec les logs/metriques d'execution : correlez les logs d'erreur et JVM/CPU/GC le cas echeant.

Voir : Surveiller la sante operationnelle avec Application Signals.

Utiliser Transaction Search pour les « pics rares »

Pour les pics intermittents, Transaction Search vous aide a analyser les spans par attribut et a explorer les utilisateurs/requetes impactes. Schemas courants :

  • Analyse de groupe par un attribut metier (par exemple, identifiant client) pour trouver qui est impacte.
  • Filtrer par service/operation et duree (p99) pour isoler les valeurs aberrantes.
  • Ouvrir une trace a partir d'un span aberrant et suivre le chemin d'appel de dependance.

Voir : Depannage des problemes applicatifs (Transaction Search).

Verifications courantes « pourquoi ne vois-je pas de donnees ? »

  • Activation du compte : Application Signals doit etre active une fois par compte/region (role lie au service cree).
  • Plage de temps : les services/operations ne s'affichent que s'ils ont une activite dans la fenetre de temps selectionnee.
  • Nommage du service : assurez-vous que service.name / OTEL_RESOURCE_ATTRIBUTES est defini correctement (hors EKS, des noms explicites de service/environnement sont souvent necessaires).
  • Sante du pipeline : CloudWatch Agent/collecteur en fonctionnement et recevant les donnees OTLP ; verifiez les endpoints locaux et les permissions IAM.
