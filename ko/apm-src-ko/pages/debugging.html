빠른 분류 워크플로 (권장)

  1. SLO에서 시작 (정의된 경우): 비정상적인 서비스/오퍼레이션/종속성을 식별합니다.
  2. 서비스 사용: 높은 지연 시간 또는 장애율로 정렬하여 주요 문제 항목을 찾습니다.
  3. 서비스 세부 정보 열기: 오퍼레이션과 종속성 엣지를 검사하고, 지연 시간/장애 데이터 포인트를 선택합니다.
  4. 상관 트레이스/스팬으로 이동: 트레이스 타임라인을 검사하여 느린/실패하는 다운스트림 호출을 찾습니다.
  5. 로그/런타임 메트릭으로 확인: 관련 시 오류 로그 및 JVM/CPU/GC와 상관 분석합니다.

참조: Application Signals로 운영 상태 모니터링.

"드문 급증"에 대한 Transaction Search 사용

간헐적인 급증의 경우, Transaction Search를 통해 속성별로 스팬을 분석하고 영향을 받는 사용자/요청을 드릴다운할 수 있습니다. 일반적인 패턴:

  • 그룹 분석: 비즈니스 속성(예: 고객 ID)별로 그룹화하여 영향을 받는 대상을 찾습니다.
  • 필터링: 서비스/오퍼레이션 및 지속 시간(p99)으로 필터링하여 이상값을 분리합니다.
  • 트레이스 열기: 이상값 스팬에서 트레이스를 열고 종속성 호출 경로를 따라갑니다.

참조: 애플리케이션 문제 해결 (Transaction Search).

일반적인 "데이터가 보이지 않는 이유" 확인 사항

  • 계정 활성화: Application Signals는 계정/리전당 한 번 활성화해야 합니다(서비스 연결 역할 생성).
  • 시간 범위: 서비스/오퍼레이션은 선택한 시간 창 내에 활동이 있는 경우에만 표시됩니다.
  • 서비스 이름 지정: service.name / OTEL_RESOURCE_ATTRIBUTES가 적절하게 설정되어 있는지 확인합니다(비 EKS 환경에서는 명시적인 서비스/환경 이름이 필요한 경우가 많습니다).
  • 파이프라인 상태: CloudWatch Agent/콜렉터가 실행 중이고 OTLP를 수신하는지 확인합니다. 로컬 엔드포인트와 IAM 권한을 확인하세요.

참조: Application Signals 설치 문제 해결.
