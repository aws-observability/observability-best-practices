CloudWatch Application Signals-க்கான வரம்புகள் மற்றும் quotas-ஐ புரிந்துகொள்ளுதல்.

வரம்புகள் மற்றும் quotas

Application Signals CloudWatch மற்றும் AWS X-Ray-ஐ சார்ந்துள்ளது. வரம்புகள் (ingestion, query rates மற்றும் quotas) அந்த சேவைகளால் நிர்வகிக்கப்படுகின்றன மற்றும் காலப்போக்கில் மாறலாம். தற்போதைய values-க்கு அதிகாரப்பூர்வ documentation-ஐ பயன்படுத்துங்கள்.

  • Application Signals: Application Signals docs
  • CloudWatch quotas: CloudWatch service quotas
  • AWS X-Ray quotas: X-Ray API throttling and limits

CloudWatch metric retention (பொதுவான குறிப்பு)

CloudWatch வெவ்வேறு resolutions-ல் metric data-ஐ retain செய்கிறது. முழு retention model-க்கு CloudWatch docs-ஐ பாருங்கள்.

  • 1-நிமிட resolution: 15 நாட்கள்
  • 5-நிமிட resolution: 63 நாட்கள்
  • 1-மணி resolution: 15 மாதங்கள்

Service Quotas

Application Signals:

  • கணக்கிற்கு சேவைகள்: 1,000
  • சேவைக்கு செயல்பாடுகள்: 100
  • தனிப்பயன் metrics: சேவைக்கு 30

RUM:

  • கணக்கிற்கு App monitors: 20
  • வினாடிக்கு Events: 10,000
  • Event size: 64 KB

Synthetics:

  • கணக்கிற்கு Canaries: 100
  • Canary execution நேரம்: அதிகபட்சம் 15 நிமிடங்கள்
  • Canary size: 1 GB (unzipped)

சொற்களஞ்சியம்

CloudWatch Application Signals-ல் முக்கிய சொற்கள் மற்றும் கருத்துக்கள்.

அடிப்படை கருத்துக்கள்

Application Performance Monitoring (APM)

மென்பொருள் பயன்பாடுகளின் செயல்திறன் மற்றும் கிடைக்கும் தன்மையை கண்காணித்து நிர்வகிக்கும் நடைமுறை.

Distributed Tracing

Distributed systems வழியாக கோரிக்கைகள் பயணிக்கும் வழியை கண்காணிக்கும் முறை, பல சேவைகளில் கோரிக்கைகளின் பாதை மற்றும் செயல்திறனைப் புரிந்துகொள்ள உதவுகிறது.

Service Map

சேவைகள் எவ்வாறு இணைகின்றன மற்றும் தொடர்பு கொள்கின்றன என்பதைக் காட்டும் உங்கள் பயன்பாட்டின் architecture-ன் visual representation.

Span

ஒரு trace-க்குள் ஒரு தனி செயல்பாடு, database query அல்லது API call போன்ற ஒரு வேலை அலகை குறிக்கிறது.

Trace

உங்கள் பயன்பாடு வழியாக ஒரு கோரிக்கையின் முழுமையான end-to-end பயணம், பல spans-ஐ உள்ளடக்கியது.

Instrumentation சொற்கள்

Auto-Instrumentation

Code மாற்றங்கள் தேவையில்லாமல் பயன்பாடுகளுக்கு observability code-ஐ தானாக சேர்ப்பது.

Manual Instrumentation

Application source code-க்கு tracing மற்றும் metrics code-ஐ வெளிப்படையாக சேர்ப்பது.

OpenTelemetry

Instrumentation-க்கு vendor-neutral APIs வழங்கும் open-source observability framework.

Sampling

Observability-ஐ செலவு மற்றும் செயல்திறனுடன் சமநிலைப்படுத்த எந்த கோரிக்கைகளை trace செய்ய வேண்டும் என்பதைத் தேர்ந்தெடுக்கும் செயல்முறை.

Context Propagation

Trace continuity-ஐ பராமரிக்க சேவைகளுக்கிடையே trace context-ஐ pass செய்யும் mechanism.

Metrics மற்றும் கண்காணிப்பு

Golden Signals

Application health-ஐ கண்காணிப்பதற்கான நான்கு முக்கிய மெட்ரிக்ஸ்: தாமதம், throughput, பிழை விகிதம் மற்றும் saturation.

தாமதம் (Latency)

ஒரு கோரிக்கை செயலாக்கப்பட எடுக்கும் நேரம், பொதுவாக percentiles-ல் (P50, P90, P99) அளவிடப்படுகிறது.

Throughput

கோரிக்கைகள் செயலாக்கப்படும் விகிதம், பொதுவாக வினாடிக்கு கோரிக்கைகளாக அளவிடப்படுகிறது.

பிழை விகிதம் (Error Rate)

பிழைகளில் முடிவடையும் கோரிக்கைகளின் சதவீதம்.

Saturation

System resources பயன்படுத்தப்படும் அளவு.

Service Level Objective (SLO)

சேவை வழங்குநர்கள் மற்றும் நுகர்வோர் இடையே ஒப்புக்கொள்ளப்பட்ட சேவை தரத்தின் இலக்கு நிலை.

Service Level Indicator (SLI)

தாமதம் அல்லது கிடைக்கும் தன்மை போன்ற வழங்கப்படும் சேவை நிலையின் அளவிடக்கூடிய அம்சம்.

RUM மற்றும் Synthetics

Real User Monitoring (RUM)

Production-ல் web பயன்பாடுகளுடனான உண்மையான பயனர் தொடர்புகளின் கண்காணிப்பு.

Synthetic Monitoring

Simulated user interactions பயன்படுத்தி பயன்பாடுகளின் தானியங்கி testing.

Canary

உங்கள் பயன்பாட்டிற்கு எதிராக தானியங்கி tests இயக்கும் synthetic monitoring script.

Core Web Vitals

Web-ல் பயனர் அனுபவத்தை அளவிடுவதற்கான Google-ன் metrics: LCP, FID மற்றும் CLS.

Largest Contentful Paint (LCP)

Screen-ல் மிகப்பெரிய content element render ஆக எடுக்கும் நேரம்.

First Input Delay (FID)

பயனர் முதலில் ஒரு பக்கத்துடன் interact செய்யும் நேரத்திலிருந்து browser respond செய்யும் நேரம் வரை.

Cumulative Layout Shift (CLS)

Visual stability-ன் அளவீடு, எதிர்பாராத layout shifts-ஐ அளவிடுகிறது.

AWS-குறிப்பிட்ட சொற்கள்

Application Signals

APM, RUM மற்றும் synthetics-ஐ ஒருங்கிணைக்கும் AWS-ன் ஒருங்கிணைந்த observability தீர்வு.

X-Ray

AWS-ன் distributed tracing சேவை, இப்போது Application Signals-உடன் ஒருங்கிணைக்கப்பட்டுள்ளது.

CloudWatch

Metrics, logs மற்றும் traces சேகரிக்கும் AWS-ன் monitoring மற்றும் observability சேவை.

ADOT (AWS Distro for OpenTelemetry)

AWS சேவைகளுக்கு optimized செய்யப்பட்ட AWS-ன் OpenTelemetry distribution.

ஆதரிக்கப்படும் Runtimes மற்றும் Frameworks

Application Signals ஆதரிக்கும் programming languages மற்றும் frameworks.

Auto-Instrumentation ஆதரவு

அதிகாரப்பூர்வ compatibility மற்றும் runtime/version matrix-க்கு, ஆதரிக்கப்படும் அமைப்புகள்-ஐ பாருங்கள். முக்கிய அம்சங்கள்:

  • Java: JVM 8/11/17/21/23 (framework support ADOT/OpenTelemetry Java instrumentation-ஐ பின்பற்றுகிறது).
  • Python: Python 3.9+ (framework support ADOT/OpenTelemetry Python instrumentation-ஐ பின்பற்றுகிறது).
  • Node.js: Node 14/16/18/20/22 (framework support ADOT/OpenTelemetry JS instrumentation-ஐ பின்பற்றுகிறது).
  • .NET: x86-64 அல்லது ARM64-ல் ஆதரிக்கப்படுகிறது; Linux x64/ARM64 மற்றும் Windows Server 2022 x64 (library support ADOT/OpenTelemetry .NET instrumentation-ஐ பின்பற்றுகிறது).
  • PHP/Ruby/Go: OpenTelemetry zero-code/auto-instrumentation paths வழியாக ஆதரிக்கப்படுகிறது (இந்த மொழிகளுக்கு ADOT SDK இல்லை).

குறிப்புகள் மற்றும் அறியப்பட்ட considerations

  • Node.js ESM: ESM support limited; docs-ல் உள்ள ESM-specific setup-ஐ பின்பற்றாவிட்டால் CommonJS-ஐ prefer செய்யுங்கள்.
  • Python: Containerized apps-க்கு PYTHONPATH set செய்ய வேண்டியிருக்கலாம் (அறியப்பட்ட OTel autoinstrumentation சிக்கல்).
  • .NET: ADOT .NET-க்கு AWS SDK for .NET V4-உடன் limitations உள்ளன; முழு support-க்கு AWS SDK V3 பரிந்துரைக்கப்படுகிறது.

Manual Instrumentation

OpenTelemetry SDKs வழியாக அனைத்து முக்கிய programming languages ஆதரிக்கப்படுகின்றன:

  • Java, Scala, Kotlin
  • Python, Ruby
  • JavaScript/TypeScript, Go
  • .NET (C#, F#, VB.NET)
  • PHP, Rust, C++

Configuration Reference

பொதுவான configuration options மற்றும் environment variables.

Environment Variables

OpenTelemetry Configuration:

  • OTEL_SERVICE_NAME - உங்கள் சேவையின் பெயர்
  • OTEL_TRACES_EXPORTER - Trace exporter (otlp, jaeger, zipkin)
  • OTEL_EXPORTER_OTLP_ENDPOINT - OTLP endpoint URL
  • OTEL_TRACES_SAMPLER - Sampling strategy
  • OTEL_TRACES_SAMPLER_ARG - Sampling parameter

AWS-குறிப்பிட்ட Configuration:

  • AWS_REGION - AWS region
  • AWS_XRAY_DAEMON_ADDRESS - X-Ray daemon address
  • OTEL_RESOURCE_ATTRIBUTES - Service attributes (Application Map-ல் custom grouping-க்கும் பயன்படுத்தப்படுகிறது)

Sampling Configuration

Sampling Strategies:

  • always_on - அனைத்து traces-ஐயும் sample செய்யுங்கள்
  • always_off - எந்த traces-ஐயும் sample செய்யாதீர்கள்
  • traceidratio - Ratio அடிப்படையில் sample செய்யுங்கள் (எ.கா., 10%-க்கு 0.1)
  • parentbased_always_on - Parent sampling decision-ஐ மதிக்கவும்
  • parentbased_traceidratio - Parent-based ratio sampling

செலவு Optimization

Observability செலவுகளை கட்டுப்படுத்துவதற்கான strategies.

Sampling Strategies

  • Production: 5-10% sampling rate
  • Development: 50-100% sampling rate
  • Error traces: Errors-உடன் traces-ஐ எப்போதும் sample செய்யுங்கள்
  • High-value operations: முக்கியமான paths-க்கு 100% sampling

Data Retention Policies

Retention அடிப்படை telemetry stores-ஆல் கட்டுப்படுத்தப்படுகிறது (எடுத்துக்காட்டாக, CloudWatch metrics/logs மற்றும் AWS X-Ray traces). அந்த சேவைகளில் retention-ஐ configure செய்து உங்கள் troubleshooting மற்றும் compliance requirements-உடன் align செய்யுங்கள்.

Resource Optimization

  • Batch exports: Network overhead-ஐ குறைக்கவும்
  • Compression: Data compression-ஐ இயக்குங்கள்
  • Filtering: தேவையற்ற spans மற்றும் attributes-ஐ அகற்றுங்கள்
  • Buffering: பொருத்தமான buffer sizes பயன்படுத்துங்கள்
