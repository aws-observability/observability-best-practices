CloudWatch APM 文档

故障排查

常见问题

本节涵盖实施 CloudWatch Application Signals 时最常遇到的问题及其解决方法。

CloudWatch 中未显示链路追踪

可能的原因：

  • 服务未启用 Application Signals
  • IAM 权限不正确
  • 代理未正确配置或未运行
  • 采样率设置过低
  • 到 CloudWatch 的网络连接问题

解决方案：

  1. 在 AWS 控制台中验证 Application Signals 已启用
  2. 检查 IAM 权限是否包含 cloudwatch:PutMetricData 和 xray:PutTraceSegments
  3. 确认代理正在运行且配置正确
  4. 临时将采样率提高到 100% 进行测试
  5. 检查 VPC 安全组和 NACL 是否允许到 CloudWatch 的出站流量

链路追踪缺失或不完整

常见问题：

  • 上下文未在服务间传播
  • 部分服务未插桩
  • 采样丢弃了 span
  • 代理配置问题

调试步骤：

  1. 检查服务拓扑图以识别未插桩的服务
  2. 验证链路追踪上下文头是否被传递（例如 x-amzn-trace-id）
  3. 查看代理日志是否有配置错误
  4. 使用 100% 采样进行测试以排除采样问题
  5. 使用链路追踪搜索查找部分链路追踪

插桩开销过高

症状：

  • CPU 使用率增加
  • 内存消耗增高
  • 延迟增加
  • 应用程序性能下降

优化策略：

  • 降低采样率（生产环境从 10% 开始）
  • 使用基于头部的采样而非尾部采样
  • 配置适当的 span 限制
  • 批量导出 span
  • 监控插桩指标

RUM 数据未显示

检查清单：

  • RUM Web 客户端已正确安装和加载
  • 应用程序 ID 和配置正确
  • 已为 RUM 域配置 CORS 头
  • 内容安全策略允许 RUM 脚本
  • 浏览器控制台无 JavaScript 错误

验证：

// Check if RUM is loaded
console.log(window.cwr);

// Verify configuration
console.log(cwr('getConfig'));

Synthetic 金丝雀失败

常见原因：

  • 金丝雀 VPC 的网络连接问题
  • 认证或授权失败
  • 应用程序变更破坏了测试脚本
  • 金丝雀运行时或依赖项过时

调试：

  1. 查看 CloudWatch Logs 中的金丝雀日志
  2. 在本地测试金丝雀脚本
  3. 检查 VPC 路由和安全组
  4. 如需要则更新金丝雀运行时
  5. 验证应用程序端点可访问

调试指南

用于诊断复杂问题的分步指南。

调试链路追踪上下文传播

步骤 1：验证插桩

检查调用链中的所有服务是否都已正确插桩：

# Check for trace headers in requests
curl -H "x-amzn-trace-id: Root=1-5759e988-bd862e3fe1be46a994272793;Parent=53995c3f42cd8ad8;Sampled=1" \
     http://your-service/endpoint

步骤 2：检查代理配置

验证代理配置文件和环境变量。

步骤 3：端到端测试

发起一个测试请求并追踪它通过所有服务。

性能问题诊断

识别瓶颈：

  1. 检查服务拓扑图中的高延迟服务
  2. 检查链路追踪时间线中的慢速 span
  3. 查看资源指标（CPU、内存、磁盘 I/O）
  4. 在日志中查找错误模式
  5. 检查数据库查询性能

常见瓶颈：

  • 需要优化数据库查询
  • 外部 API 调用超时
  • 资源限制（CPU/内存）
  • 线程池耗尽
  • 服务间网络延迟

代理故障排查

检查代理状态：

# For Java applications
jps -l | grep opentelemetry

# For Python applications
ps aux | grep opentelemetry

# Check agent logs
tail -f /var/log/aws-opentelemetry-agent.log

常见代理问题：

  • Java 代理版本与 JVM 不匹配
  • 缺少依赖项或库
  • 配置文件语法错误
  • 日志文件写入权限问题

工具和命令

用于故障排查的实用工具和命令。

CloudWatch CLI 命令

# List Application Signals services
aws application-signals list-services

# Get service metrics
aws cloudwatch get-metric-data --metric-data-queries '[
  {
    "Id": "m1",
    "MetricStat": {
      "Metric": {
        "Namespace": "AWS/ApplicationSignals",
        "MetricName": "Latency"
      },
      "Period": 300,
      "Stat": "Average"
    }
  }
]'

# Query traces
aws xray get-trace-summaries --start-time 2024-01-01T00:00:00Z --end-time 2024-01-02T00:00:00Z

日志分析

使用 CloudWatch Logs Insights 分析遥测数据：

# Find errors in application logs
fields @timestamp, @message
| filter @message like /ERROR/
| sort @timestamp desc
| limit 100

# Correlate traces with logs
fields @timestamp, traceId, @message
| filter ispresent(traceId)
| sort @timestamp desc
