Debugging Guides

Fast triage workflow (recommended)

  1. Start at SLOs (if defined): identify the unhealthy services/operations/dependencies.
  2. Use Services: sort by high latency or fault rate to find the top offenders.
  3. Open Service detail: inspect operations and dependency edges; pick a latency/fault datapoint.
  4. Jump to correlated traces/spans: inspect the trace timeline to find the slow/failing downstream call.
  5. Confirm with logs/runtime metrics: correlate error logs and JVM/CPU/GC when relevant.

See: Monitor operational health with Application Signals.

Using Transaction Search for “rare spikes”

For intermittent spikes, Transaction Search helps you analyze spans by attribute and drill into impacted users/requests. Common patterns:

  • Group analysis by a business attribute (for example, customer ID) to find who’s impacted.
  • Filter by service/operation and duration (p99) to isolate outliers.
  • Open a trace from an outlier span and follow the dependency call path.

See: Troubleshooting application issues (Transaction Search).

Common “why don’t I see data?” checks

  • Account enablement: Application Signals must be enabled once per account/region (service-linked role created).
  • Time range: services/operations show only if they have activity inside the selected time window.
  • Service naming: ensure service.name / OTEL_RESOURCE_ATTRIBUTES is set appropriately (non-EKS often needs explicit service/environment names).
  • Pipeline health: CloudWatch Agent/collector running and receiving OTLP; verify local endpoints and IAM permissions.

See: Troubleshooting your Application Signals installation.
