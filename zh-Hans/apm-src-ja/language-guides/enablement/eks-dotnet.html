概要

このガイドでは、 ステップバイステップの手順 を提供します。AWS Application Signals の自動計装を使用して、Amazon EKS 上に .NET Core アプリケーションをデプロイします。これは Lambda .NET 関数の EKS 版であり、同じビジネスロジックとエンドポイントを再利用しています。

達成できること：
  • EKS 上に .NET Core アプリをデプロイする
  • AWS Application Signals の自動計装を有効にする
  • CloudWatch Application Signals コンソールでサービスを確認する
  • サービスのメトリクス、トレース、オペレーションを監視する
完了までの時間： 30〜45 分

前提条件

必要なツール

  • AWS CLI v2（最新バージョン）
  • Docker Desktop
  • kubectl（最新バージョン）
  • .NET 8.0 SDK（ローカル開発用）
  • 適切な権限を持つ AWS アカウント
  • テキストエディタ

重要な互換性要件

  • AWS SDK .NET V3：AWS Distro for OpenTelemetry (ADOT) SDK for .NET は AWS SDK for .NET V4 をサポートしていません。Application Signals の完全なサポートには AWS SDK .NET V3 を使用してください。
  • OpenTelemetry パッケージ：Application Signals 統合のために、AWS.DistributedOpenTelemetry と互換性のある OpenTelemetry パッケージを使用してください。

必要な AWS 権限

  • EKS：フルアクセス（クラスター、ノードグループ、アドオンの作成）
  • IAM：ロールおよびポリシーの作成
  • CloudWatch：ログおよび Application Signals へのアクセス
  • ECR：コンテナイメージのプッシュ／プル
  • VPC：VPC リソースの作成と管理

前提とする知識

  • AWS CLI の基本的な使い方
  • Docker の基本的なコマンド
  • kubectl の基本的なコマンド
  • EKSの概念に関する基本的な理解
  • 基本的な.NET Coreの知識

アーキテクチャ

EKS Observabilityアドオン（標準的なアプローチ）: このガイドでは、公式のEKS Observabilityアドオンを使用します。これは、EKSクラスターでApplication Signalsを有効にするための推奨される標準的な方法です。このアドオンは、CloudWatch Agentを手動で設定することなく、テレメトリの収集を自動的に処理します。

EKSクラスターで実行されるもの:

  1. .NET Coreアプリケーション：自動インストルメンテーションアノテーションを含むWeb API
  2. EKS Observabilityアドオン：自動トレースおよびメトリクス収集のための公式AWSアドオン
  3. CloudWatch Agent（アドオンによって管理）：アドオンによって自動的にデプロイおよび設定される

フロー:

EKS Cluster
  → .NET Core App (auto-instrumented via annotation)
  → EKS Observability Add-on
  → CloudWatch Agent (managed)
  → CloudWatch Application Signals
EKS Observabilityアドオンの主なメリット:
  • CloudWatch Agentの手動設定が不要
  • サポートされている言語への自動インストルメンテーション
  • Application Signalsの組み込みサポート
  • デプロイと管理の簡素化

ステップ1：アプリケーションの作成

1.1 プロジェクト構造の作成

mkdir eks-dotnet
cd eks-dotnet

1.2 .NETプロジェクトの作成

作成 Program.cs Lambda関数と同じエンドポイントを持つ:

using Microsoft.AspNetCore.Builder;
using Microsoft.AspNetCore.Hosting;
using Microsoft.Extensions.DependencyInjection;
using Microsoft.Extensions.Hosting;
using Microsoft.Extensions.Logging;

var builder = WebApplication.CreateBuilder(args);

// Add services to the container
builder.Services.AddControllers();
builder.Services.AddEndpointsApiExplorer();
builder.Services.AddSwaggerGen();

var app = builder.Build();

// Configure the HTTP request pipeline
if (app.Environment.IsDevelopment())
{
    app.UseSwagger();
    app.UseSwaggerUI();
}

app.UseRouting();

// Simple endpoints
app.MapGet("/", () => new
{
    message = "AWS Application Signals EKS .NET Demo",
    timestamp = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds(),
    service = "dotnet-eks-service",
    runtime = "dotnet8"
});

app.MapGet("/health", () => new
{
    status = "healthy",
    timestamp = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds(),
    service = "dotnet-eks-service"
});

app.MapGet("/api/hello/{name}", (string name) => new
{
    message = $"Hello, {name}!",
    timestamp = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds(),
    service = "dotnet-eks-service"
});

app.MapGet("/api/slow", async () =>
{
    await Task.Delay(Random.Shared.Next(500, 1500));
    return new
    {
        message = "Slow operation completed",
        timestamp = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds(),
        service = "dotnet-eks-service"
    };
});

app.MapGet("/api/error", () =>
{
    if (Random.Shared.NextDouble() > 0.5)
    {
        throw new Exception("Simulated error for testing Application Signals error tracking");
    }
    return new
    {
        message = "No error occurred",
        timestamp = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds(),
        service = "dotnet-eks-service"
    };
});

app.MapGet("/api/external", async () =>
{
    using var httpClient = new HttpClient();
    try
    {
        var response = await httpClient.GetAsync("https://httpbin.org/delay/1");
        return new
        {
            message = "External call completed",
            external_status_code = (int)response.StatusCode,
            timestamp = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds(),
            service = "dotnet-eks-service"
        };
    }
    catch (Exception ex)
    {
        throw new Exception($"External call failed: {ex.Message}");
    }
});

app.Run();

1.3 Dockerfileの作成

作成 Dockerfile:

FROM mcr.microsoft.com/dotnet/aspnet:8.0 AS base
WORKDIR /app
EXPOSE 8080

FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src
COPY ["Program.cs", "."]
RUN dotnet new webapi --force
RUN dotnet build -c Release -o /app/build

FROM base AS final
WORKDIR /app
COPY --from=build /app/build .
ENTRYPOINT ["dotnet", "Program.dll"]

ステップ3：EKSへのデプロイ

3.1 EKSクラスターの作成

# Create EKS cluster
eksctl create cluster --name appsignals-cluster --region us-east-1 --nodegroup-name workers --node-type t3.medium --nodes 2 --nodes-min 1 --nodes-max 3

# Create namespace
kubectl create namespace appsignals-demo

3.2 EKS Observabilityアドオンのインストール（必須）

重要: EKS ObservabilityアドオンはEKSでApplication Signalsを有効にするための標準的かつ推奨される方法です。このアドオンはテレメトリの収集を自動的に処理し、自動インストルメンテーション機能を提供します。 
# Install the EKS observability add-on (REQUIRED for Application Signals)
aws eks create-addon --cluster-name appsignals-cluster --addon-name amazon-cloudwatch-observability --region us-east-1

# Verify the add-on is installed
aws eks describe-addon --cluster-name appsignals-cluster --addon-name amazon-cloudwatch-observability --region us-east-1
EKS Observabilityアドオンが提供するもの:
  • CloudWatch Agentの自動デプロイ
  • Application Signalsの組み込みサポート
  • Java、Python、Node.js、および.NETへの自動インストルメンテーション
  • 手動設定が不要

ステップ 4: トラフィックの生成とモニタリング

4.1 トラフィックの生成

作成 generate-traffic.sh:

#!/bin/bash

# Port forward the service
kubectl port-forward service/dotnet-eks-service 8080:80 -n appsignals-demo &
PORT_FORWARD_PID=$!

# Wait for port forward to be ready
sleep 10

BASE_URL="http://localhost:8080"

echo "Generating traffic for .NET EKS Application Signals demo..."

# Generate traffic for 5 minutes
for i in {1..300}; do
    echo "Request $i"

    # Root endpoint
    curl -s "$BASE_URL/" > /dev/null

    # Health check
    curl -s "$BASE_URL/health" > /dev/null

    # Hello endpoint
    curl -s "$BASE_URL/api/hello/User$i" > /dev/null

    # Slow endpoint
    curl -s "$BASE_URL/api/slow" > /dev/null

    # Error endpoint
    curl -s "$BASE_URL/api/error" > /dev/null

    # External call
    curl -s "$BASE_URL/api/external" > /dev/null

    sleep 1
done

echo "Traffic generation complete!"

# Clean up port forward
kill $PORT_FORWARD_PID

実行可能にして実行する：

chmod +x generate-traffic.sh
./generate-traffic.sh

4.2 CloudWatch でのモニタリング

  1. CloudWatch コンソールを開く

    • AWS コンソール → CloudWatch に移動する
    • 「Application Signals」に移動する

  2. サービスマップを表示する

    • 「dotnet-eks-service」サービスを探す
    • サービスの依存関係と正常性を確認する

  3. メトリクスを確認する

    • レイテンシー、エラーレート、スループットを表示する
    • Lambda メトリクスと比較する

  4. トレースを分析する

    • サービスをクリックして詳細なトレースを表示する
    • オペレーションレベルのメトリクスを表示する

トラブルシューティング

よくある問題

  1. アプリケーションが起動しない

    • Pod のステータスを確認する： kubectl get pods -n appsignals-demo
    • Pod のログを確認する： kubectl logs -n appsignals-demo -l app=dotnet-eks-service
    • EKS オブザーバビリティアドオンがインストールされていることを確認する

  2. Application Signals にデータが表示されない

    • 自動インストルメンテーションのアノテーションが存在することを確認する
    • 環境変数が正しく設定されていることを確認する
    • EKS オブザーバビリティアドオンが実行中であることを確認する

  3. ポートフォワードの問題

    • サービスが実行中であることを確認する： kubectl get svc -n appsignals-demo
    • Pod の準備が完了していることを確認する： kubectl get pods -n appsignals-demo
