概要

このガイドでは、 ステップバイステップの手順 Amazon EKS 上に Python FastAPI アプリケーションを AWS Application Signals の自動計装とともにデプロイする方法を説明します。これは Lambda Python 関数の EKS 版であり、同じビジネスロジックとエンドポイントを再利用しています。

達成できること：
  • EKS 上に Python FastAPI アプリをデプロイする
  • AWS Application Signals の自動計装を有効にする
  • CloudWatch Application Signals コンソールでサービスを確認する
  • サービスのメトリクス、トレース、オペレーションを監視する
完了までの時間： 30〜45 分

アーキテクチャ

EKS Observability アドオン（標準的なアプローチ）： このガイドでは、公式の EKS Observability アドオンを使用します。これは、EKS クラスターで Application Signals を有効にするための推奨される標準的な方法です。このアドオンは、CloudWatch Agent を手動で設定することなく、テレメトリの収集を自動的に処理します。

EKS クラスターで実行されるコンポーネント：

  1. Python FastAPI アプリケーション：自動インストルメンテーションアノテーションを含む Web API
  2. EKS Observability アドオン：トレースとメトリクス収集を自動化する公式 AWS アドオン
  3. CloudWatch Agent（アドオンによる管理）：アドオンによって自動的にデプロイおよび設定される

フロー：

EKS Cluster
  → Python FastAPI App (auto-instrumented via annotation)
  → EKS Observability Add-on
  → CloudWatch Agent (managed)
  → CloudWatch Application Signals
EKS Observability アドオンの主なメリット：
  • CloudWatch Agent の手動設定が不要
  • サポートされている言語への自動インストルメンテーション
  • Application Signals のビルトインサポート
  • デプロイと管理の簡素化

1.2 Python アプリケーションの作成

作成： app.py Lambda 関数と同じエンドポイントを持つファイル：

"""
AWS Application Signals EKS Python Service
FastAPI application with Application Signals auto-instrumentation
"""
import time
import random
import boto3
import requests
from fastapi import FastAPI, HTTPException
from fastapi.responses import JSONResponse
from typing import Dict, Any
import uvicorn

# Initialize FastAPI app
app = FastAPI(
    title="AWS Application Signals EKS Python Demo",
    description="Python service running on EKS with Application Signals",
    version="1.0.0"
)

# Initialize AWS clients
s3_client = boto3.client('s3')


@app.get("/")
async def root():
    """Root endpoint"""
    return {
        "message": "AWS Application Signals EKS Python Demo",
        "timestamp": time.time(),
        "service": "python-eks-service"
    }


@app.get("/health")
async def health():
    """Health check endpoint"""
    return {
        "status": "healthy",
        "timestamp": time.time(),
        "service": "python-eks-service"
    }


@app.get("/api/hello/{name}")
async def hello(name: str):
    """Personalized greeting with simulated processing"""
    # Simulate some processing time
    time.sleep(random.uniform(0.01, 0.1))
    
    return {
        "message": f"Hello, {name}!",
        "timestamp": time.time(),
        "service": "python-eks-service"
    }


@app.get("/api/slow")
async def slow():
    """Intentionally slow endpoint for latency testing"""
    time.sleep(random.uniform(0.5, 1.5))
    
    return {
        "message": "Slow operation completed",
        "timestamp": time.time(),
        "duration": "1-1.5 seconds",
        "service": "python-eks-service"
    }


@app.get("/api/error")
async def error():
    """Endpoint that randomly raises errors for testing error rates"""
    if random.random() > 0.5:
        raise HTTPException(
            status_code=500, 
            detail="Simulated error for testing Application Signals error tracking"
        )
    
    return {
        "message": "No error occurred",
        "timestamp": time.time(),
        "service": "python-eks-service"
    }


@app.get("/api/external")
async def external():
    """Makes external HTTP call for distributed tracing"""
    try:
        # This will be automatically traced by OpenTelemetry
        response = requests.get("https://httpbin.org/delay/1", timeout=5)
        
        return {
            "message": "External call completed",
            "external_status_code": response.status_code,
            "timestamp": time.time(),
            "service": "python-eks-service"
        }
    except Exception as e:
        raise HTTPException(
            status_code=500,
            detail=f"External call failed: {str(e)}"
        )


@app.get("/api/s3")
async def s3():
    """Makes AWS S3 call for testing AWS SDK instrumentation"""
    try:
        # This will be automatically traced by OpenTelemetry
        response = s3_client.list_buckets()
        bucket_names = [bucket['Name'] for bucket in response['Buckets']]
        
        return {
            "message": "S3 call completed successfully",
            "bucket_count": len(bucket_names),
            "buckets": bucket_names[:5],  # Show first 5 buckets
            "timestamp": time.time(),
            "service": "python-eks-service"
        }
    except Exception as e:
        raise HTTPException(
            status_code=500,
            detail=f"S3 call failed: {str(e)}"
        )


@app.get("/api/metrics")
async def metrics():
    """Custom metrics endpoint"""
    # Simulate some business logic
    processing_time = random.uniform(0.1, 0.5)
    time.sleep(processing_time)
    
    # Simulate different response times based on some logic
    if processing_time > 0.3:
        status = "slow"
    else:
        status = "fast"
    
    return {
        "message": "Metrics endpoint",
        "processing_time": processing_time,
        "status": status,
        "timestamp": time.time(),
        "service": "python-eks-service"
    }


if __name__ == "__main__":
    uvicorn.run(
        "app:app",
        host="0.0.0.0",
        port=8000,
        reload=True,
        log_level="info"
    )

1.4 Dockerfile の作成

作成： Dockerfile:

# Use Python 3.11 slim image
FROM python:3.11-slim

# Set working directory
WORKDIR /app

# Install system dependencies
RUN apt-get update && apt-get install -y \
    gcc \
    && rm -rf /var/lib/apt/lists/*

# Copy requirements first for better caching
COPY requirements.txt .

# Install Python dependencies
RUN pip install --no-cache-dir -r requirements.txt

# Copy application code
COPY app.py .

# Create non-root user
RUN useradd --create-home --shell /bin/bash app && chown -R app:app /app
USER app

# Expose port
EXPOSE 8000

# Health check
HEALTHCHECK --interval=30s --timeout=30s --start-period=5s --retries=3 \
    CMD curl -f http://localhost:8000/health || exit 1

# Run the application
CMD ["uvicorn", "app:app", "--host", "0.0.0.0", "--port", "8000"]

ステップ 3：EKS へのデプロイ

3.1 EKS クラスターの作成

# Create EKS cluster
eksctl create cluster --name appsignals-cluster --region us-east-1 --nodegroup-name workers --node-type t3.medium --nodes 2 --nodes-min 1 --nodes-max 3

# Create namespace
kubectl create namespace appsignals-demo

3.2 EKS Observability アドオンのインストール（必須）

重要： EKS Observability アドオンは、EKS で Application Signals を有効にするための標準かつ推奨される方法です。このアドオンはテレメトリの収集を自動的に処理し、自動インストルメンテーション機能を提供します。 
# Install the EKS observability add-on (REQUIRED for Application Signals)
aws eks create-addon --cluster-name appsignals-cluster --addon-name amazon-cloudwatch-observability --region us-east-1

# Verify the add-on is installed
aws eks describe-addon --cluster-name appsignals-cluster --addon-name amazon-cloudwatch-observability --region us-east-1
EKS Observability アドオンが提供する機能：
  • CloudWatch Agent の自動デプロイ
  • Application Signals のビルトインサポート
  • Java、Python、Node.js、および .NET への自動インストルメンテーション
  • 手動設定が不要

ステップ 4: トラフィックの生成とモニタリング

4.1 トラフィックの生成

作成 generate-traffic.sh:

#!/bin/bash

# Port forward the service
kubectl port-forward service/python-eks-service 8000:80 -n appsignals-demo &
PORT_FORWARD_PID=$!

# Wait for port forward to be ready
sleep 10

BASE_URL="http://localhost:8000"

echo "Generating traffic for Python EKS Application Signals demo..."

# Generate traffic for 5 minutes
for i in {1..300}; do
    echo "Request $i"

    # Root endpoint
    curl -s "$BASE_URL/" > /dev/null

    # Health check
    curl -s "$BASE_URL/health" > /dev/null

    # Hello endpoint
    curl -s "$BASE_URL/api/hello/User$i" > /dev/null

    # Slow endpoint
    curl -s "$BASE_URL/api/slow" > /dev/null

    # Error endpoint
    curl -s "$BASE_URL/api/error" > /dev/null

    # External call
    curl -s "$BASE_URL/api/external" > /dev/null

    # S3 call
    curl -s "$BASE_URL/api/s3" > /dev/null

    # Metrics endpoint
    curl -s "$BASE_URL/api/metrics" > /dev/null

    sleep 1
done

echo "Traffic generation complete!"

# Clean up port forward
kill $PORT_FORWARD_PID

実行可能にして実行する：

chmod +x generate-traffic.sh

4.2 CloudWatch でのモニタリング

  1. CloudWatch コンソールを開く

  2. サービスマップを表示する

  4. トレースを分析する

    • オペレーションレベルのメトリクスを表示する

トラブルシューティング

よくある問題

  1. アプリケーションが起動しない

    • Pod のステータスを確認する： kubectl get pods -n appsignals-demo
    • Pod のログを確認する： kubectl logs -n appsignals-demo -l app=python-eks-service
    • EKS オブザーバビリティアドオンがインストールされていることを確認する

  2. Application Signals にデータが表示されない

    • 自動インストルメンテーションのアノテーションが存在することを確認する
    • 環境変数が正しく設定されていることを確認する
    • EKS オブザーバビリティアドオンが実行中であることを確認する

  3. ポートフォワードの問題

    • サービスが実行中であることを確認する： kubectl get svc -n appsignals-demo
    • Pod の準備ができていることを確認する： kubectl get pods -n appsignals-demo
