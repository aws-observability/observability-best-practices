概要

このガイドでは、 ステップバイステップの手順 を提供します。AWS Application Signals を有効にした Java Lambda 関数をデプロイするための手順です。AWS Lambda コンソールと組み込みレイヤーサポートを使用した最もシンプルなアプローチです。

達成できること：
  • Application Signals を使用した Java Lambda 関数のデプロイ
  • Lambda Layer を介した AWS Application Signals 自動計装の有効化
  • CloudWatch Application Signals コンソールでのサービスの確認
  • サービスのメトリクス、トレース、およびオペレーションの監視
完了までの時間： 15〜20 分

前提条件

必要なツール

  • AWS CLI v2（最新バージョン）
  • Java 17 以上
  • Maven 3.8 以上
  • 適切な権限を持つ AWS アカウント
  • テキストエディタ

必要な AWS 権限

  • Lambda: 関数の作成と管理
  • IAM: ロールとポリシーの作成
  • CloudWatch: ログおよび Application Signals へのアクセス
  • S3: 基本的な読み取りアクセス（テスト用）

アーキテクチャ

Lambda 関数は以下を使用して自動的に計装されます：

  1. AWS Lambda Layer for OpenTelemetry - 自動計装を提供
  2. Application Signals 統合 - レイヤーに組み込み済み
  3. 自動トレーシング - HTTP リクエスト、AWS SDK 呼び出し、関数の実行

ステップ 1: アカウントで Application Signals を有効にする

重要： Application Signals は、AWS アカウントごと、リージョンごとに 1 回有効にする必要があります。

オプション1：AWS コンソールから有効化する（推奨）

  1. AWS コンソール → CloudWatch → Application Signals に移動する
  2. リージョンを選択する（例： us-east-1)
  3. 「Get Started」または「Enable Application Signals」をクリックする
  4. 有効化されるまで2〜3分待つ

ステップ2：Java Lambda 関数を準備する

2.1 アプリケーションディレクトリを作成する

mkdir -p ~/my-lambda-java-appsignals
cd ~/my-lambda-java-appsignals

2.2 プロジェクト構成を作成する

mkdir -p src/main/java/com/example/appsignals/controller

2.3 pom.xml を作成する

以下の依存関係を含む Maven POM ファイルを作成する：

  • AWS Lambda Core
  • AWS Lambda Events
  • AWS SDK for S3
  • Spring Boot Web Starter（ローカルテスト用）
  • JSON 処理
  • HTTP クライアント

重要： OpenTelemetry の依存関係を追加しないでください。これらは AWS Lambda レイヤーによって提供されます。

ステップ3：Lambda デプロイパッケージをビルドする

3.1 変数を設定する

export AWS_REGION="us-east-1"
export AWS_ACCOUNT_ID="YOUR_ACCOUNT_ID"
export FUNCTION_NAME="lambda-java"

3.2 JAR ファイルをビルドする

mvn clean package

ステップ4：Lambda 用の IAM ロールを作成する

以下のポリシーを持つ IAM ロールを作成する：

  • AWSLambdaBasicExecutionRole
  • CloudWatchLambdaApplicationSignalsExecutionRolePolicy （重要！）
  • AmazonS3ReadOnlyAccess （テスト用）

4.1 IAM ロールを作成する

aws iam create-role \
  --role-name lambda-java-execution-role \
  --assume-role-policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Principal": {"Service": "lambda.amazonaws.com"},
      "Action": "sts:AssumeRole"
    }]
  }'

4.2 必要なポリシーをアタッチする

aws iam attach-role-policy \
  --role-name lambda-java-execution-role \
  --policy-arn arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole

aws iam attach-role-policy \
  --role-name lambda-java-execution-role \
  --policy-arn arn:aws:iam::aws:policy/CloudWatchLambdaApplicationSignalsExecutionRolePolicy

aws iam attach-role-policy \
  --role-name lambda-java-execution-role \
  --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess

4.3 ロール ARN を取得する

ROLE_ARN=$(aws iam get-role \
  --role-name lambda-java-execution-role \
  --query 'Role.Arn' \
  --output text)

echo "Role ARN: $ROLE_ARN"

ステップ5：Lambda 関数をデプロイする

5.1 Lambda 関数を作成する

aws lambda create-function \
  --function-name lambda-java \
  --runtime java17 \
  --role "$ROLE_ARN" \
  --handler com.example.appsignals.ApplicationSignalsApplication::handleRequest \
  --zip-file fileb://target/lambda-java-appsignals-1.0.0.jar \
  --timeout 30 \
  --memory-size 512

5.2 OpenTelemetry 用の AWS Lambda レイヤーを追加する

対象： us-east-1:

LAYER_ARN="arn:aws:lambda:us-east-1:615299751070:layer:AWSOpenTelemetryDistroJava:18"

5.3 環境変数を設定する

aws lambda update-function-configuration \
  --function-name ${FUNCTION_NAME} \
  --environment Variables='{
    "AWS_LAMBDA_EXEC_WRAPPER": "/opt/otel-instrument",
    "OTEL_SERVICE_NAME": "'${FUNCTION_NAME}'",
    "OTEL_RESOURCE_ATTRIBUTES": "service.name='${FUNCTION_NAME}',deployment.environment=lambda:'${AWS_REGION}'"
  }'

6.1 基本機能をテストする

ステップ7：テスト用のトラフィックを生成する

Application Signals をテストするには、Lambda 関数へのトラフィックを生成する必要があります。2つのトラフィックジェネレーターオプションを提供しています：

7.1 Python トラフィックジェネレーター（推奨）

現実的なトラフィックパターンを生成する Python スクリプトを作成する：

#!/usr/bin/env python3
import boto3
import json
import time
import random
import sys

def generate_traffic(function_name, region='us-east-1', profile='eksworkstream', num_requests=50):
    # Initialize Lambda client
    session = boto3.Session(profile_name=profile)
    lambda_client = session.client('lambda', region_name=region)
    
    # Test endpoints with different characteristics
    endpoints = [
        {"path": "/", "method": "GET", "name": "Root", "weight": 20},
        {"path": "/health", "method": "GET", "name": "Health", "weight": 15},
        {"path": "/api/hello/User", "method": "GET", "name": "Hello", "weight": 30},
        {"path": "/api/slow", "method": "GET", "name": "Slow", "weight": 10},
        {"path": "/api/error", "method": "GET", "name": "Error", "weight": 10},
        {"path": "/api/external", "method": "GET", "name": "External", "weight": 8},
        {"path": "/api/s3", "method": "GET", "name": "S3", "weight": 5},
        {"path": "/api/database", "method": "GET", "name": "Database", "weight": 2},
    ]
    
    # Create weighted list for random selection
    weighted_endpoints = []
    for endpoint in endpoints:
        weighted_endpoints.extend([endpoint] * endpoint["weight"])
    
    success_count = 0
    error_count = 0
    
    for i in range(1, num_requests + 1):
        endpoint = random.choice(weighted_endpoints)
        
        if endpoint["name"] == "Hello":
            user_name = f"User{random.randint(1, 1000)}"
            path = f"/api/hello/{user_name}"
        else:
            path = endpoint["path"]
        
        payload = {"path": path, "httpMethod": endpoint["method"]}
        
        try:
            response = lambda_client.invoke(
                FunctionName=function_name,
                InvocationType='RequestResponse',
                Payload=json.dumps(payload)
            )
            
            response_payload = json.loads(response['Payload'].read())
            status_code = response_payload.get('statusCode', 200)
            
            if status_code == 200:
                success_count += 1
            else:
                error_count += 1
                
            print(f"Request {i}: {endpoint['name']} | Status: {status_code}")
            time.sleep(random.uniform(0.1, 0.5))
            
        except Exception as e:
            error_count += 1
            print(f"Request {i}: {endpoint['name']} | Error: {str(e)}")
    
    print(f"Summary: {success_count} successful, {error_count} errors")

if __name__ == "__main__":
    function_name = sys.argv[1] if len(sys.argv) > 1 else "lambda-java"
    num_requests = int(sys.argv[2]) if len(sys.argv) > 2 else 50
    generate_traffic(function_name, num_requests=num_requests)

7.2 シェルスクリプトトラフィックジェネレーター

トラフィック生成用の代替シェルスクリプト：

#!/bin/bash
FUNCTION_NAME="lambda-java"
NUM_REQUESTS=50

endpoints=(
    "root:/:GET:20"
    "health:/health:GET:15"
    "hello:/api/hello:GET:30"
    "slow:/api/slow:GET:10"
    "error:/api/error:GET:10"
    "external:/api/external:GET:8"
    "s3:/api/s3:GET:5"
    "database:/api/database:GET:2"
)

for i in $(seq 1 $NUM_REQUESTS); do
    endpoint_info=${endpoints[$((RANDOM % ${#endpoints[@]}))]}
    IFS=':' read -r name path method weight <<< "$endpoint_info"
    
    if [ "$name" = "hello" ]; then
        user_name="User$((RANDOM % 1000 + 1))"
        full_path="/api/hello/$user_name"
    else
        full_path="$path"
    fi
    
    payload="{\"path\": \"$full_path\", \"httpMethod\": \"$method\"}"
    
    aws lambda invoke \
        --function-name $FUNCTION_NAME \
        --payload "$payload" \
        /tmp/response.json
    
    echo "Request $i: $name"
    sleep 0.2
done

  • 重み付きランダム選択： より現実的なトラフィックパターン

  • エラーシミュレーション： 障害テスト用のエラーエンドポイントを含む
  • パフォーマンステスト： レイテンシーテスト用の低速エンドポイントを含む
  • 外部依存関係： 外部 HTTP および S3 呼び出しをテスト
  • 設定可能： リクエスト数とパラメーターを調整可能

ステップ 9：AWS コンソールで表示する

9.1 コンソールアクセス

  1. 移動先： AWS コンソール → CloudWatch → Application Signals → Services
  2. サービスを探す： サービス名： lambda-java
  3. 確認する： Service Map、Service Dashboard、Operations、Dependencies

代替方法：Lambda コンソールから有効化する（最も簡単な方法）

シェルの問題が発生した場合に推奨！ この方法はすべてのコマンドライン問題を回避し、最も信頼性の高いアプローチです。

AWS コンソールを使用したい場合、またはシェル環境の問題が発生した場合：

  1. AWS Lambda コンソールに移動する
  2. 「関数の作成」をクリックする
  3. 「一から作成」を選択する
  4. 関数名： lambda-java
  5. ランタイム: Java 17
  6. 関数を作成する
  7. JAR ファイルをアップロードします。 target/lambda-java-appsignals-1.0.0.jar
  8. ハンドラーを次のように設定します。 com.example.appsignals.ApplicationSignalsApplication::handleRequest
  9. 「設定」→「モニタリングおよびオペレーションツール」に移動します。
  10. 「Edit」をクリックします。
  11. 両方のチェックボックスを有効にします。
    • ✅ "Application Signals を使用してアプリケーショントレースと標準アプリケーションメトリクスを自動的に収集する"
    • ✅ "X-Ray を使用してエンドツーエンドの可視性を実現するために Lambda サービストレースを自動的に収集する"
  12. 「Save」をクリックします。

以上です！ コンソールは自動的に以下を行います。

  • AWS Lambda Layer for OpenTelemetry を追加します。
  • 設定します AWS_LAMBDA_EXEC_WRAPPER 環境変数
  • 必要な IAM ポリシーをアタッチします。

トラブルシューティング

問題: シェル環境の問題

問題： 次のようなエラーでコマンドが失敗します parse error near 'cursor_snap_ENV_VARS...' または command not found: dump_zsh_state

原因： Cursor IDE の環境変数インジェクションがシェル環境を破壊している

ソリューション：

  1. 新しいターミナルウィンドウを使用します。 Cursor IDE の外部
  2. Cursor IDE を再起動します。 完全に実行してから、もう一度お試しください。
  3. VS Code を使用します。 Cursor IDE の代わりに
  4. Lambda コンソールの方法を使用します。 (シェルの問題に推奨)

問題: ロール ARN が空です

問題： aws: error: argument --role: expected one argument

原因： The ROLE_ARN 変数が空であり、IAM ロールの作成が失敗したことを示しています

デバッグ手順：

  1. ロールが存在するか確認します。 
    aws iam get-role --role-name lambda-java-execution-role
  2. ロールが存在しない場合は作成します。 
    aws iam create-role \
  --role-name lambda-java-execution-role \
  --assume-role-policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Principal": {"Service": "lambda.amazonaws.com"},
      "Action": "sts:AssumeRole"
    }]
  }'
  3. ロール ARN を手動で取得します。 
    aws iam get-role \
  --role-name lambda-java-execution-role \
  --query 'Role.Arn' \
  --output text
  4. create-function コマンドで ARN を直接使用する

問題: コマンド構文エラー

よくある間違い：

  • ピリオドを使用して同じ行で複数のコマンドを実行する
  • 重複パラメータの使用（例： --role 2 回)
  • 変数を囲む引用符がありません

❌ WRONG - Don't do this:

# WRONG: Commands on same line with period
aws iam create-role \ ... }. aws lambda create-function \ ...

# WRONG: Duplicate --role parameter
aws lambda create-function \
  --role ${ROLE_ARN} \
  --role lambda-java-execution-role

# WRONG: Missing quotes around variable
--role $ROLE_ARN

✅ 正しい方法 - 代わりにこれを行います。

# Step 1: Create IAM role (separate command)
aws iam create-role \
  --role-name lambda-java-execution-role \
  --assume-role-policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Principal": {"Service": "lambda.amazonaws.com"},
      "Action": "sts:AssumeRole"
    }]
  }'

# Step 2: Attach policies (separate commands)
aws iam attach-role-policy \
  --role-name lambda-java-execution-role \
  --policy-arn arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole

aws iam attach-role-policy \
  --role-name lambda-java-execution-role \
  --policy-arn arn:aws:iam::aws:policy/CloudWatchLambdaApplicationSignalsExecutionRolePolicy

# Step 3: Get role ARN
ROLE_ARN=$(aws iam get-role \
  --role-name lambda-java-execution-role \
  --query 'Role.Arn' \
  --output text)

# Step 4: Create Lambda function (with quotes around variable)
aws lambda create-function \
  --function-name lambda-java \
  --runtime java17 \
  --role "$ROLE_ARN" \
  --handler com.example.appsignals.ApplicationSignalsApplication::handleRequest \
  --zip-file fileb://target/lambda-java-appsignals-1.0.0.jar \
  --timeout 30 \
  --memory-size 512

問題: Maven ビルドの失敗

よくある Maven の問題：

  • Shade プラグインのエラー： 簡略化されたものを使用します。 pom.xml 提供済み
  • Java バージョンの不一致： Java 17 がインストールされ、設定されていることを確認してください。
  • 依存関係の競合： Spring Boot の依存関係が不要な場合は削除します。

問題: Application Signals にサービスが表示されない

確認：

  • Application Signals がアカウントで有効になっていることを確認します
  • OpenTelemetry レイヤーがアタッチされているか確認します。
  • 環境変数が正しく設定されていることを確認します。
  • トラフィックを生成してから 5〜10 分待ちます。
  • IAM ポリシーを確認してください。 CloudWatchLambdaApplicationSignalsExecutionRolePolicy が添付されています

問題: Lambda 関数のエラー

よくある問題：

  • IAM ポリシーが不足しています - 確認してください CloudWatchLambdaApplicationSignalsExecutionRolePolicy が添付されています
  • レイヤー ARN が間違っています - お使いのリージョンのレイヤー ARN を確認してください
  • 環境変数が見つかりません - 確認してください AWS_LAMBDA_EXEC_WRAPPER=/opt/otel-instrument
  • ハンドラーの不一致 - ハンドラーが次のとおりであることを確認してください。 com.example.appsignals.ApplicationSignalsApplication::handleRequest
  • JAR ファイルが見つかりません - Maven ビルドが正常に完了していることを確認してください

問題: 権限拒否エラー

AWS 認証情報を確認します。

  • AWS プロファイルを確認します。 aws configure list
  • 呼び出し元のアイデンティティを確認します。 aws sts get-caller-identity
  • 必要な IAM 権限を持っていることを確認してください。

変数 値 目的
