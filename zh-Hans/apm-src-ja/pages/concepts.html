CloudWatch APM ドキュメント

Concepts & Fundamentals

主要概念

アプリケーションパフォーマンスモニタリング（APM）の基本概念を理解することは、CloudWatch Application Signals を効果的に使用するために不可欠です。このセクションでは、現代のオブザーバビリティの基盤を形成する主要な構成要素について説明します。

分散トレーシング

分散トレーシングとは、分散システムを流れるリクエストを追跡する手法です。マイクロサービスアーキテクチャでは、1 つのユーザーリクエストが複数のサービス、データベース、および外部 API を横断することがあります。分散トレーシングは、この全体的な経路を可視化します。

主要コンポーネント：

  • トレース： システム内におけるリクエストの完全なエンドツーエンドの経路
  • スパン： トレース内の単一の操作（例：データベースクエリや API 呼び出し）
  • トレースコンテキスト： サービス境界をまたいでスパンを接続するメタデータ

トレース、スパン、およびセグメント

これらは分散トレーシングで使用される基本的なデータ構造です：

トレース

トレースは、アプリケーション内におけるリクエストの完全なパスを表します。タイミング情報、メタデータ、およびリクエストの経路を構成するすべてのスパンが含まれます。

スパン

スパンは、トレース内の単一の作業単位を表します。各スパンには以下が含まれます：

  • 名前と操作タイプ
  • 開始および終了タイムスタンプ
  • 追加コンテキストのためのタグと属性
  • 親スパンへの参照

セグメント

AWS X-Ray の用語では、セグメントはトレースデータの基本単位です。各セグメントはスパンに対応し、トレースのその部分で行われた作業に関する情報を含みます。

サービスマップ

サービスマップは、アプリケーションのアーキテクチャと依存関係を視覚的に表現したものです。以下を示します：

  • サービス間の接続関係
  • サービス間のトラフィックフロー
  • 各サービスのパフォーマンスメトリクス
  • エラー率とレイテンシーのパターン

サービスマップはトレースデータから自動的に生成され、システムのトポロジーを一目で把握するのに役立ちます。

インストルメンテーション

インストルメンテーションとは、アプリケーションにオブザーバビリティコードを追加するプロセスです。いくつかのアプローチがあります：

自動インストルメンテーション

自動インストルメンテーションはコードの変更を必要としません。ライブラリとエージェントが一般的なフレームワークおよびライブラリを自動的にインストルメント化します。

手動インストルメンテーション

手動インストルメンテーションは、アプリケーションにトレーシングコードを追加することを伴います。これにより細かい制御が可能になりますが、より多くの開発工数が必要です。

OpenTelemetry

OpenTelemetryは、計装のためのベンダー中立なAPIを提供するオープンソースの可観測性フレームワークです。複数のプログラミング言語をサポートし、さまざまなバックエンドにデータをエクスポートできます。

データモデル

Application Signalsは、テレメトリデータを整理するために構造化されたデータモデルを使用します：

メトリクス

  • レイテンシー： 応答時間の測定値（P50、P90、P99）
  • スループット： リクエストレート（リクエスト毎秒）
  • エラーレート： 失敗したリクエストの割合
  • 飽和度： リソース使用率（CPU、メモリなど）

トレース

スパン、タグ、およびタイミング情報を含む構造化されたトレースデータ。

ログ

完全なコンテキストのためにトレースおよびメトリクスと関連付けられたアプリケーションログ。

サンプリング戦略

サンプリングは、可観測性とコストおよびパフォーマンスのバランスを取るために、どのリクエストをトレースするかを制御します：

ヘッドサンプリング

リクエストのライフサイクルの開始時にトレースするかどうかを決定します。

テールサンプリング

トレースデータを収集した後にサンプリングの判断を行い、リクエストの特性に基づいたよりインテリジェントなサンプリングを可能にします。

レートベースサンプリング

リクエストの一定の割合をトレースします（例：全リクエストの10%）。
