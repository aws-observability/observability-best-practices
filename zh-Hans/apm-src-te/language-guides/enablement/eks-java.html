Overview

This guide provides step-by-step instructions to deploy a Java Spring Boot application on Amazon EKS with AWS Application Signals auto-instrumentation. This is the EKS equivalent of the Lambda Java function, reusing the same business logic and endpoints.

What you'll accomplish:
  • Deploy a Java Spring Boot app on EKS
  • Enable AWS Application Signals auto-instrumentation
  • View your service in CloudWatch Application Signals console
  • Monitor service metrics, traces, and operations
Time to complete: 30-45 minutes

Prerequisites

Required Tools

  • AWS CLI v2 (latest version)
  • Docker Desktop
  • kubectl (latest version)
  • Java 17+ (for local development)
  • Maven 3.6+ (for local development)
  • AWS Account with appropriate permissions
  • Text editor

AWS Permissions Required

  • EKS: Full access (create clusters, node groups, add-ons)
  • IAM: Create roles and policies
  • CloudWatch: Logs and Application Signals access
  • ECR: Push/pull container images
  • VPC: Create and manage VPC resources

Knowledge Assumed

  • Basic AWS CLI usage
  • Basic Docker commands
  • Basic kubectl commands
  • Basic understanding of EKS concepts
  • Basic Java/Spring Boot knowledge

Architecture

EKS Observability Add-on (Standard Approach): This guide uses the official EKS observability add-on, which is the recommended and standard way to enable Application Signals on EKS clusters. The add-on automatically handles telemetry collection without requiring manual CloudWatch Agent configuration.

Your EKS cluster will run:

  1. Java Spring Boot Application: Your web API with auto-instrumentation annotations
  2. EKS Observability Add-on: Official AWS add-on for automatic tracing and metrics collection
  3. CloudWatch Agent (managed by add-on): Automatically deployed and configured by the add-on

Flow:

EKS Cluster
  → Java Spring Boot App (auto-instrumented via annotation)
  → EKS Observability Add-on
  → CloudWatch Agent (managed)
  → CloudWatch Application Signals
Key Benefits of EKS Observability Add-on:
  • No manual CloudWatch Agent configuration required
  • Automatic auto-instrumentation for supported languages
  • Built-in support for Application Signals
  • Simplified deployment and management

1.3 Create the Application Code

Create src/main/java/com/appsignals/JavaEksServiceApplication.java:

package com.appsignals;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;

@SpringBootApplication
public class JavaEksServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(JavaEksServiceApplication.class, args);
    }
}

Create src/main/java/com/appsignals/controller/HelloController.java:

package com.appsignals.controller;

import org.springframework.web.bind.annotation.*;

import java.time.Instant;
import java.util.HashMap;
import java.util.Map;
import java.util.Random;

@RestController
@RequestMapping("/api")
public class HelloController {

    private final Random random = new Random();

    @GetMapping("/")
    public Map<String, Object> root() {
        Map<String, Object> response = new HashMap<>();
        response.put("message", "AWS Application Signals EKS Java Demo");
        response.put("timestamp", Instant.now().toEpochMilli());
        response.put("service", "java-eks-service");
        return response;
    }

    @GetMapping("/health")
    public Map<String, Object> health() {
        Map<String, Object> response = new HashMap<>();
        response.put("status", "healthy");
        response.put("timestamp", Instant.now().toEpochMilli());
        response.put("service", "java-eks-service");
        return response;
    }

    @GetMapping("/hello/{name}")
    public Map<String, Object> hello(@PathVariable String name) {
        try {
            // Simulate some processing time
            Thread.sleep(random.nextInt(100) + 10);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
        
        Map<String, Object> response = new HashMap<>();
        response.put("message", "Hello, " + name + "!");
        response.put("timestamp", Instant.now().toEpochMilli());
        response.put("service", "java-eks-service");
        return response;
    }

    @GetMapping("/slow")
    public Map<String, Object> slow() {
        try {
            // Simulate slow operation
            Thread.sleep(random.nextInt(1000) + 500);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
        
        Map<String, Object> response = new HashMap<>();
        response.put("message", "Slow operation completed");
        response.put("timestamp", Instant.now().toEpochMilli());
        response.put("service", "java-eks-service");
        return response;
    }

    @GetMapping("/error")
    public Map<String, Object> error() {
        if (random.nextBoolean()) {
            throw new RuntimeException("Simulated error for testing Application Signals error tracking");
        }
        
        Map<String, Object> response = new HashMap<>();
        response.put("message", "No error occurred");
        response.put("timestamp", Instant.now().toEpochMilli());
        response.put("service", "java-eks-service");
        return response;
    }

    @GetMapping("/external")
    public Map<String, Object> external() {
        try {
            // Simulate external API call
            Thread.sleep(random.nextInt(2000) + 1000);
            
            Map<String, Object> response = new HashMap<>();
            response.put("message", "External call completed");
            response.put("external_status_code", 200);
            response.put("timestamp", Instant.now().toEpochMilli());
            response.put("service", "java-eks-service");
            return response;
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new RuntimeException("External call failed: " + e.getMessage());
        }
    }
}

1.5 Create Dockerfile

Create Dockerfile:

# Use the official OpenJDK 17 runtime as base image
FROM openjdk:17-jdk-slim AS base
WORKDIR /app
EXPOSE 8080

# Use the OpenJDK 17 SDK for building
FROM openjdk:17-jdk-slim AS build
WORKDIR /app
COPY mvnw .
COPY .mvn .mvn
COPY pom.xml .
RUN ./mvnw dependency:go-offline

COPY src src
RUN ./mvnw clean package -DskipTests

FROM base AS final
WORKDIR /app
COPY --from=build /app/target/*.jar app.jar

# Set environment variables for Application Signals
ENV OTEL_SERVICE_NAME=java-eks-service
ENV OTEL_RESOURCE_ATTRIBUTES=service.name=java-eks-service,service.namespace=appsignals-demo,service.version=1.0.0,deployment.environment=eks

ENTRYPOINT ["java", "-jar", "app.jar"]

Step 3: Deploy to EKS

3.1 Create EKS Cluster

# Create EKS cluster
eksctl create cluster --name appsignals-cluster --region us-east-1 --nodegroup-name workers --node-type t3.medium --nodes 2 --nodes-min 1 --nodes-max 3

# Create namespace
kubectl create namespace appsignals-demo

3.2 Install EKS Observability Add-on (Required)

Important: The EKS observability add-on is the standard and recommended way to enable Application Signals on EKS. This add-on automatically handles telemetry collection and provides auto-instrumentation capabilities. 
# Install the EKS observability add-on (REQUIRED for Application Signals)
aws eks create-addon --cluster-name appsignals-cluster --addon-name amazon-cloudwatch-observability --region us-east-1

# Verify the add-on is installed
aws eks describe-addon --cluster-name appsignals-cluster --addon-name amazon-cloudwatch-observability --region us-east-1
What the EKS Observability Add-on provides:
  • Automatic CloudWatch Agent deployment
  • Built-in support for Application Signals
  • Auto-instrumentation for Java, Python, Node.js, and .NET
  • No manual configuration required

3.4 Deploy Application

Create deployment.yaml:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: java-eks-service
  namespace: appsignals-demo
  labels:
    app: java-eks-service
    service: java
spec:
  replicas: 2
  selector:
    matchLabels:
      app: java-eks-service
  template:
    metadata:
      labels:
        app: java-eks-service
        service: java
      annotations:
        # Enable auto-instrumentation for Java
        instrumentation.opentelemetry.io/inject-java: "true"
        # Prometheus metrics
        prometheus.io/scrape: "true"
        prometheus.io/port: "8080"
        prometheus.io/path: "/actuator/prometheus"
    spec:
      containers:
      - name: java-app
        image: 188524202814.dkr.ecr.us-east-1.amazonaws.com/my-appsignals-java-app:latest
        ports:
        - containerPort: 8080
        env:
        - name: AWS_REGION
          value: "us-east-1"
        - name: OTEL_SERVICE_NAME
          value: "java-eks-service"
        - name: OTEL_RESOURCE_ATTRIBUTES
          value: "service.name=java-eks-service,service.namespace=appsignals-demo,service.version=1.0.0,deployment.environment=eks"
        - name: OTEL_EXPORTER_OTLP_ENDPOINT
          value: "http://cloudwatch-agent.amazon-cloudwatch.svc.cluster.local:4315"
        - name: OTEL_EXPORTER_OTLP_PROTOCOL
          value: "grpc"
        - name: OTEL_EXPORTER_OTLP_TRACES_ENDPOINT
          value: "http://cloudwatch-agent.amazon-cloudwatch.svc.cluster.local:4315"
        - name: OTEL_EXPORTER_OTLP_METRICS_ENDPOINT
          value: "http://cloudwatch-agent.amazon-cloudwatch.svc.cluster.local:4315"
        - name: OTEL_TRACES_EXPORTER
          value: "otlp"
        - name: OTEL_METRICS_EXPORTER
          value: "otlp"
        - name: OTEL_LOGS_EXPORTER
          value: "otlp"
        # Reduce metrics payload size to avoid FRAME_SIZE_ERROR
        - name: OTEL_METRIC_EXPORT_INTERVAL
          value: "30000"
        - name: OTEL_METRIC_EXPORT_TIMEOUT
          value: "10000"
        - name: OTEL_BSP_MAX_EXPORT_BATCH_SIZE
          value: "32"
        - name: OTEL_BSP_EXPORT_TIMEOUT
          value: "10000"
        # Disable metrics entirely to focus on traces
        - name: OTEL_METRICS_EXPORTER
          value: "none"
        - name: OTEL_INSTRUMENTATION_RUNTIME_METRICS_ENABLED
          value: "false"
        - name: OTEL_INSTRUMENTATION_JVM_METRICS_ENABLED
          value: "false"
        resources:
          requests:
            memory: "512Mi"
            cpu: "250m"
          limits:
            memory: "1Gi"
            cpu: "500m"
        livenessProbe:
          httpGet:
            path: /actuator/health
            port: 8080
          initialDelaySeconds: 60
          periodSeconds: 10
        readinessProbe:
          httpGet:
            path: /actuator/health
            port: 8080
          initialDelaySeconds: 30
          periodSeconds: 5
---

metadata:
  name: java-eks-service
  namespace: appsignals-demo
  labels:

kubectl apply -f deployment.yaml

Step 4: Generate Traffic and Monitor

4.1 Generate Traffic

Create generate-traffic.sh:

#!/bin/bash

# Port forward the service
kubectl port-forward service/java-eks-service 8080:80 -n appsignals-demo &
PORT_FORWARD_PID=$!

# Wait for port forward to be ready
sleep 10

BASE_URL="http://localhost:8080"

echo "Generating traffic for Java EKS Application Signals demo..."

# Generate traffic for 5 minutes
for i in {1..300}; do
    echo "Request $i"

    # Root endpoint
    curl -s "$BASE_URL/api/" > /dev/null

    # Health check
    curl -s "$BASE_URL/api/health" > /dev/null

    # Hello endpoint
    curl -s "$BASE_URL/api/hello/User$i" > /dev/null

    # Slow endpoint
    curl -s "$BASE_URL/api/slow" > /dev/null

    # Error endpoint
    curl -s "$BASE_URL/api/error" > /dev/null

    # External call
    curl -s "$BASE_URL/api/external" > /dev/null

    sleep 1
done

echo "Traffic generation complete!"

# Clean up port forward
kill $PORT_FORWARD_PID

Make it executable and run:

chmod +x generate-traffic.sh
./generate-traffic.sh

Troubleshooting

Common Issues

  1. Application won't start

    • Check pod status: kubectl get pods -n appsignals-demo
    • Check pod logs: kubectl logs -n appsignals-demo -l app=java-eks-service
    • Verify EKS observability add-on is installed

  2. No data in Application Signals

    • Verify auto-instrumentation annotation is present
    • Check environment variables are set correctly
    • Ensure EKS observability add-on is running

  3. Port forward issues

    • Check service is running: kubectl get svc -n appsignals-demo
    • Verify pods are ready: kubectl get pods -n appsignals-demo
