概述

本指南提供分步说明，指导您在 Amazon ECS 上部署启用 AWS Application Signals 的 Java Spring Boot 应用。无需 CDK - 仅使用 AWS CLI 和 Docker。

完成目标：
  • 在 ECS Fargate 上部署 Java Spring Boot 应用
  • 启用 AWS Application Signals 自动插桩
  • 在 CloudWatch Application Signals 控制台中查看您的服务
  • 监控服务 metrics、traces 和操作
完成时间： 30-45 分钟

前置条件

必备工具

  • AWS CLI v2（最新版本）
  • Docker Desktop
  • Java 17+（用于本地开发）
  • Maven 3.8+（用于本地开发）
  • 具有适当权限的 AWS 账户
  • 文本编辑器

所需 AWS 权限

  • ECS：完全访问（创建集群、服务、任务定义）
  • ECR：推送镜像
  • IAM：创建角色和策略
  • CloudWatch：Logs 和 Application Signals 访问
  • EC2：VPC、子网和安全组访问
  • SSM：Parameter Store 写入权限

前置知识

  • AWS CLI 基础使用
  • Docker 基础命令
  • ECS 基本概念理解
  • Java/Spring Boot 基础知识

架构

您的 ECS 任务将包含 3 个容器：

  1. Init 容器 (init)：复制 ADOT Java 自动插桩文件
  2. CloudWatch Agent (ecs-cwagent)：收集 traces 和 metrics
  3. 应用容器 (app)：您的 Spring Boot 应用

流程：

Init 容器（运行一次，复制文件）

↓

应用启动（通过 javaagent 自动插桩）

↓

向 CloudWatch Agent 发送 traces/metrics (localhost:4316)

↓

CloudWatch Agent 发送到 Application Signals

1启用 Application Signals in Your Account

重要： 每个 AWS 账户每个区域只需启用一次 Application Signals。

方案 1：通过 AWS 控制台启用（推荐）

  1. 前往 AWS 控制台 → CloudWatch → Application Signals
  2. 选择您的区域（例如 us-east-1)
  3. Click "Get Started" or "启用 Application Signals"
  4. 等待 2-3 分钟激活

方案 2：验证是否已启用

# Check if service-linked role exists
aws iam get-role \
  --role-name AWSServiceRoleForCloudWatchApplicationSignals \
  --region us-east-1

# If role exists, Application Signals is enabled

输出应显示： 如果已启用则显示角色 ARN，否则显示错误。

mkdir -p ~/my-appsignals-java-app

# Copy from the ecs-java folder

  • src/main/resources/application.properties - Spring Boot 配置
  • Dockerfile - 容器配置
  • .dockerignore - Docker 忽略文件

2.3 验证应用结构

3构建并推送 Docker 镜像

3.1 设置变量

# Replace with your values
export AWS_REGION="us-east-1"
export AWS_ACCOUNT_ID="YOUR_ACCOUNT_ID"
export AWS_PROFILE="your-profile-name"  # Optional if using default
export APP_NAME="my-appsignals-java-app"

获取您的账户 ID：

export AWS_ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
echo "Account ID: $AWS_ACCOUNT_ID"

3.2 Create ECR Repository

aws ecr create-repository \
  --repository-name ${APP_NAME} \
  --region ${AWS_REGION}

输出： 存储库 URI（请保存）

3.3 构建 Docker 镜像

cd ~/my-appsignals-java-app
docker build -t ${APP_NAME} .

3.4 推送到 ECR

# Login to ECR
aws ecr get-login-password --region ${AWS_REGION} | \
  docker login --username AWS --password-stdin \
  ${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com

# Tag image
docker tag ${APP_NAME}:latest \
  ${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com/${APP_NAME}:latest

# Push image
docker push ${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com/${APP_NAME}:latest

4创建 CloudWatch Agent 配置

4.1 创建配置文件

cat > /tmp/ecs-cwagent-java.json << 'EOF'
{
  "traces": {
    "traces_collected": {
      "application_signals": {}
    }
  },
  "logs": {
    "metrics_collected": {
      "application_signals": {}
    }
  }
}
EOF

4.2 上传到 SSM Parameter Store

aws ssm put-parameter \
  --name "ecs-cwagent-${APP_NAME}" \
  --type "String" \
  --value "$(cat /tmp/ecs-cwagent-java.json)" \
  --region "${AWS_REGION}" \
  --overwrite

输出： 版本号（新建为 1，更新时递增）

5创建 IAM 角色

5.1 创建信任策略

cat > /tmp/task-trust-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "ecs-tasks.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
EOF

5.2 创建任务角色

aws iam create-role \
  --role-name ${APP_NAME}-ecs-task-role \
  --assume-role-policy-document file:///tmp/task-trust-policy.json

# Attach required policies
aws iam attach-role-policy \
  --role-name ${APP_NAME}-ecs-task-role \
  --policy-arn arn:aws:iam::aws:policy/CloudWatchAgentServerPolicy

aws iam attach-role-policy \
  --role-name ${APP_NAME}-ecs-task-role \
  --policy-arn arn:aws:iam::aws:policy/AWSXRayDaemonWriteAccess

5.3 创建任务执行角色

aws iam create-role \
  --role-name ${APP_NAME}-ecs-execution-role \
  --assume-role-policy-document file:///tmp/task-trust-policy.json

# Attach policies
aws iam attach-role-policy \
  --role-name ${APP_NAME}-ecs-execution-role \
  --policy-arn arn:aws:iam::aws:policy/service-role/AmazonECSTaskExecutionRolePolicy

aws iam attach-role-policy \
  --role-name ${APP_NAME}-ecs-execution-role \
  --policy-arn arn:aws:iam::aws:policy/AmazonSSMReadOnlyAccess

5.4 添加 CloudWatch Logs 权限

cat > /tmp/logs-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogGroup",
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "arn:aws:logs:*:*:log-group:/ecs/*"
    }
  ]
}
EOF

aws iam put-role-policy \
  --role-name ${APP_NAME}-ecs-execution-role \
  --policy-name CloudWatchLogsPolicy \
  --policy-document file:///tmp/logs-policy.json

6创建 ECS 集群

export CLUSTER_NAME="${APP_NAME}-cluster"

aws ecs create-cluster \
  --cluster-name ${CLUSTER_NAME} \
  --region ${AWS_REGION}

输出： 集群 ARN 和状态应为 ACTIVE

7设置网络

7.1 获取 VPC 和子网

如果您已有 VPC：

# List VPCs
aws ec2 describe-vpcs --region ${AWS_REGION} \
  --query 'Vpcs[*].{VpcId:VpcId,CIDR:CidrBlock,Name:Tags[?Key==`Name`].Value|[0]}'

# Choose your VPC
export VPC_ID="vpc-xxxxx"

# Get private subnets
aws ec2 describe-subnets --region ${AWS_REGION} \
  --filters "Name=vpc-id,Values=${VPC_ID}" \
  --query 'Subnets[?MapPublicIpOnLaunch==`false`].[SubnetId,AvailabilityZone,CidrBlock]'

# Save subnet IDs
export SUBNET_1="subnet-xxxxx"
export SUBNET_2="subnet-yyyyy"

7.2 创建安全组

export SG_NAME="${APP_NAME}-ecs-sg"

# Create security group
SG_ID=$(aws ec2 create-security-group \
  --group-name ${SG_NAME} \
  --description "Security group for ${APP_NAME} ECS tasks" \
  --vpc-id ${VPC_ID} \
  --region ${AWS_REGION} \
  --query 'GroupId' \
  --output text)

echo "Security Group ID: $SG_ID"
export SG_ID

# Add ingress rule (allow HTTP from anywhere for testing)
aws ec2 authorize-security-group-ingress \
  --group-id ${SG_ID} \
  --protocol tcp \
  --port 8080 \
  --cidr 0.0.0.0/0 \
  --region ${AWS_REGION}
生产环境： 替换 0.0.0.0/0 为特定 IP 范围或负载均衡器安全组。

9部署 ECS 服务

aws ecs create-service \
  --cluster ${CLUSTER_NAME} \
  --service-name ${APP_NAME}-service \
  --task-definition ${APP_NAME}-task:1 \
  --desired-count 1 \
  --launch-type FARGATE \
  --network-configuration "awsvpcConfiguration={subnets=[${SUBNET_1},${SUBNET_2}],securityGroups=[${SG_ID}],assignPublicIp=ENABLED}" \
  --region ${AWS_REGION}
注意： assignPublicIp=ENABLED 用于测试。生产环境请使用负载均衡器。

10验证部署

10.1 检查服务状态

aws ecs describe-services \
  --cluster ${CLUSTER_NAME} \
  --services ${APP_NAME}-service \
  --region ${AWS_REGION} \
  --query 'services[0].{status:status,runningCount:runningCount,desiredCount:desiredCount}'

预期结果： status: ACTIVE, runningCount: 1

10.2 获取运行中的任务

TASK_ARN=$(aws ecs list-tasks \
  --cluster ${CLUSTER_NAME} \
  --service-name ${APP_NAME}-service \
  --region ${AWS_REGION} \
  --desired-status RUNNING \

aws ecs describe-tasks \
  --cluster ${CLUSTER_NAME} \
  --tasks ${TASK_ARN} \
  --region ${AWS_REGION} \

预期输出：

TASK_IP=$(aws ecs describe-tasks \

11验证 Application Signals

11.1 检查 CloudWatch Logs

# Application logs
aws logs tail /ecs/${APP_NAME} --region ${AWS_REGION} --since 5m --format short

# CloudWatch agent logs
aws logs tail /ecs/ecs-cwagent-${APP_NAME} --region ${AWS_REGION} --since 5m --format short

查找:

  • 应用日志： "Started ApplicationSignalsApplication"
  • Agent 日志： "application_signals", "X-Ray proxy server started"

11.2 查看 Metrics

# Wait 2-3 minutes after first requests, then check
aws cloudwatch list-metrics \
  --namespace ApplicationSignals \
  --region ${AWS_REGION} \
  --dimensions Name=Service,Value=${APP_NAME} \
  --query 'Metrics[*].MetricName' \
  --output text | tr '\t' '\n' | sort -u

预期 metrics：

  • JvmHeapMemoryUsed
  • JvmNonHeapMemoryUsed
  • JvmGcDuration
  • JvmThreadCount
  • Latency
  • Error
  • Fault

11.3 生成流量

# Generate traffic to create traces
for i in {1..100}; do
  curl -s http://${TASK_IP}:8080/api/hello/User${i} > /dev/null
  curl -s http://${TASK_IP}:8080/api/health > /dev/null
  curl -s http://${TASK_IP}:8080/api/database > /dev/null
done

echo "100 requests sent! Wait 5-10 minutes for Application Signals..."

11.4 检查 Application Signals 服务发现

# Wait 5-10 minutes after traffic generation
aws application-signals list-services \
  --region ${AWS_REGION} \
  --start-time $(date -u -v-2H +%s) \
  --end-time $(date -u +%s) \
  --query 'ServiceSummaries[*].{Name:KeyAttributes.Name,Environment:KeyAttributes.Environment}'

预期结果： 您的服务应显示为：

  • Name: ${APP_NAME}
  • 环境： ecs:${AWS_REGION}

11.5 检查服务操作

aws application-signals list-service-operations \
  --region ${AWS_REGION} \
  --start-time $(date -u -v-2H +%s) \
  --end-time $(date -u +%s) \
  --key-attributes Name=${APP_NAME},Type=Service,Environment=ecs:${AWS_REGION} \
  --query 'ServiceOperations[*].Name'

预期操作：

  • GET /api/hello/{name}
  • GET /api/health
  • GET /api/
  • GET /api/database
  • GET /api/external

12在 AWS 控制台中查看

  1. 导航到: 
    AWS Console → CloudWatch → Application Signals → Services
    或直接链接： 
    https://${AWS_REGION}.console.aws.amazon.com/cloudwatch/home?region=${AWS_REGION}#application-signals:services
  2. 查找 your service:
    • 服务名称： 值 of ${APP_NAME}

  3. 浏览：

    • Service Dashboard：延迟、错误、故障
    • Operations：每个端点的 metrics
    • Dependencies：下游服务调用

故障排除

问题：服务未出现在 Application Signals 中

检查：

# 1. Verify metrics are being published
aws cloudwatch list-metrics \
  --namespace ApplicationSignals \
  --region ${AWS_REGION} \
  --dimensions Name=Service,Value=${APP_NAME}

# 2. Check if Application Signals is enabled
aws iam get-role \
  --role-name AWSServiceRoleForCloudWatchApplicationSignals

# 3. Check container logs for errors
aws logs tail /ecs/${APP_NAME} --region ${AWS_REGION} --since 10m

问题：Init 容器失败

检查：

# Describe task to see init container exit code
aws ecs describe-tasks \
  --cluster ${CLUSTER_NAME} \
  --tasks ${TASK_ARN} \
  --region ${AWS_REGION} \
  --query 'tasks[0].containers[?name==`init`].{name:name,exitCode:exitCode,reason:reason}'

预期结果： exitCode: 0

如果非零： 镜像可能不可用。验证标签 v1.34.0 是否存在：

docker pull public.ecr.aws/aws-observability/adot-autoinstrumentation-java:v1.34.0

问题：应用无法连接到 CloudWatch Agent

症状：

Connection refused to localhost:4316

检查：

  1. 验证 CloudWatch Agent 容器正在运行
  2. 检查容器依赖 - app 应该依赖 init 且条件为 SUCCESS
  3. 验证环境变量（特别是端点： http://localhost:4316 and http://localhost:2000)

问题：X-Ray 中无 Traces

常见原因：

  1. Missing JAVA_TOOL_OPTIONS - 对 Java 自动插桩至关重要
  2. Missing OTEL_TRACES_SAMPLER_ARG - 需要 X-Ray 采样器端点
  3. 错误的 javaagent 路径 - 必须指向 /otel-auto-instrumentation-java/opentelemetry-javaagent.jar
  4. 缺少卷挂载 - Java Agent 文件必须可访问

问题：IAM 权限错误

症状：

AccessDeniedException: ... logs:CreateLogGroup

修复： 向执行角色添加 CloudWatch Logs 权限 (see Step 5.4)

关键配置详情

关键环境变量

变量 值 用途
JAVA_TOOL_OPTIONS -javaagent:/otel-auto-instrumentation-java/opentelemetry-javaagent.jar Java 自动插桩 Agent
OTEL_TRACES_SAMPLER xray 使用 X-Ray 采样
OTEL_TRACES_SAMPLER_ARG endpoint=http://localhost:2000 X-Ray 采样器端点
OTEL_AWS_APPLICATION_SIGNALS_ENABLED true 启用 Application Signals
OTEL_EXPORTER_OTLP_TRACES_ENDPOINT http://localhost:4316/v1/traces 向 CW Agent 发送 traces
OTEL_AWS_APPLICATION_SIGNALS_EXPORTER_ENDPOINT http://localhost:4316/v1/metrics 向 CW Agent 发送 metrics
OTEL_RESOURCE_ATTRIBUTES service.name=YOUR_SERVICE,deployment.environment=ecs:REGION 服务标识

Init 容器详情

Image: public.ecr.aws/aws-observability/adot-autoinstrumentation-java:v1.34.0

为什么使用版本标签而不是 :latest?

  • The :latest 标签对 Java ADOT 镜像不存在
  • 使用特定版本标签 (检查 https://github.com/aws-observability/aws-otel-java-instrumentation/releases)
  • 当前稳定版： v1.34.0

功能说明：

  1. 在应用容器启动前运行一次
  2. 从 /autoinstrumentation/ 复制 ADOT Java 插桩文件到共享卷
  3. 以退出码 0 退出
  4. 应用容器挂载该卷并使用 JAVA_TOOL_OPTIONS 来加载 javaagent

CloudWatch Agent 配置

必须包含两个部分：

{
  "traces": {
    "traces_collected": {
      "application_signals": {}   // Not just "otlp"!
    }
  },
  "logs": {
    "metrics_collected": {
      "application_signals": {}
    }
  }
}

应用代码要求

DO:

  • 编写正常的 Spring Boot 代码
  • 使用标准库（Spring Web、WebFlux 等）
  • 保持代码整洁 - 无需 OpenTelemetry 导入
  • 使用标准 Spring Boot 注解（@RestController、@GetMapping 等）

DON'T:

  • Import opentelemetry-* packages
  • 手动配置 OpenTelemetry
  • 添加手动追踪代码（除非需要自定义 span）
  • 在 pom.xml 中安装 OpenTelemetry 依赖

清理资源

删除所有资源：

# Delete ECS service
aws ecs update-service \
  --cluster ${CLUSTER_NAME} \
  --service ${APP_NAME}-service \
  --desired-count 0 \
  --region ${AWS_REGION}

aws ecs delete-service \
  --cluster ${CLUSTER_NAME} \
  --service ${APP_NAME}-service \
  --region ${AWS_REGION}

# Delete ECS cluster
aws ecs delete-cluster \
  --cluster ${CLUSTER_NAME} \
  --region ${AWS_REGION}

# Delete ECR repository
aws ecr delete-repository \
  --repository-name ${APP_NAME} \
  --force \
  --region ${AWS_REGION}

# Delete security group
aws ec2 delete-security-group \
  --group-id ${SG_ID} \
  --region ${AWS_REGION}

# Delete IAM roles
aws iam delete-role \
  --role-name ${APP_NAME}-ecs-task-role

aws iam delete-role \
  --role-name ${APP_NAME}-ecs-execution-role

# Delete SSM parameter
aws ssm delete-parameter \
  --name ecs-cwagent-${APP_NAME} \
  --region ${AWS_REGION}
