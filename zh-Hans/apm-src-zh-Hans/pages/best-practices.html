插桩最佳实践

有效的插桩是从 Observability 数据中获取有价值洞察的关键。遵循这些最佳实践，确保您的插桩在最小化开销的同时提供最大价值。

应该插桩什么

专注于插桩应用程序中最关键的部分：

高优先级：

  • 公共 API 端点和面向用户的操作
  • 数据库查询和外部服务调用
  • 业务关键工作流和事务
  • 错误路径和异常处理

中优先级：

  • 内部服务间通信
  • 后台作业和定时任务
  • 缓存操作

低优先级：

  • 简单工具函数
  • 静态内容服务
  • 健康检查端点

命名规范

一致的命名使搜索和过滤链路追踪更加容易：

Span 名称：

  • 使用小写加连字符：http-get-user-profile
  • 包含操作类型：db-query-users、redis-get-session
  • 描述性但简洁：避免使用"process"或"handle"等通用名称

服务名称：

  • 跨环境使用一致的服务命名
  • 多环境设置中在服务名称中包含环境信息
  • 遵循您组织的命名规范

属性和标签

使用属性为您的 span 和链路追踪添加上下文：

标准属性：

  • http.method - HTTP 方法（GET、POST 等）
  • http.url - 请求 URL（注意 PII）
  • http.status_code - HTTP 响应状态
  • db.system - 数据库类型（mysql、postgres 等）
  • db.name - 数据库名称
  • db.operation - 数据库操作（SELECT、INSERT 等）

自定义属性：

  • user.id - 用户标识符（如果是 PII 则哈希处理）
  • business.operation - 业务操作类型
  • version - 应用程序版本
  • environment - 部署环境

PII 和安全注意事项

保护遥测中的敏感数据：

切勿包含：

  • 密码、API 密钥或认证令牌
  • 个人身份信息 (PII) 如电子邮件、电话号码
  • 包含敏感数据的查询参数的完整 URL
  • 信用卡号或金融信息

安全替代方案：

  • 使用哈希或令牌化标识符
  • 仅包含 URL 的路径部分
  • 使用通用状态指示器代替具体值
  • 在插桩级别实施数据掩码

采样策略

采样有助于在保持 Observability 覆盖范围的同时控制成本。

选择采样策略

基于速率的采样：

  • 采样固定百分比的请求（例如 10%）
  • 实施和理解简单
  • 可能遗漏重要但罕见的事件

尾部采样：

  • 在收集链路追踪数据后做出采样决策
  • 可以优先处理有错误或高延迟的链路追踪
  • 更复杂但更智能

基于规则的采样：

  • 根据特定条件（用户、端点等）进行采样
  • 确保关键路径始终被追踪
  • 需要仔细定义规则

推荐采样率

  • 开发环境：100% 采样以获得完整可见性
  • 预发布环境：50-100% 采样用于测试
  • 生产环境：10-20% 以实现经济高效的覆盖
  • 高流量服务：1-5% 以控制数据量
  • 关键路径：无论其他规则如何均 100% 采样

性能优化

最小化插桩的性能影响。

减少开销

  • 尽可能使用异步 span 处理
  • 批量导出 span 以减少网络调用
  • 配置适当的缓冲区大小
  • 监控插桩性能指标

资源管理

  • 为 span 导出设置合理的超时时间
  • 对后端故障实施断路器
  • 为收集器使用连接池
  • 配置适当的重试策略

告警和监控

设置有效的告警以掌握应用程序健康状况。

需要监控的关键指标

  • 延迟：P95/P99 响应时间
  • 错误率：失败请求的百分比
  • 吞吐量：每秒请求数
  • 饱和度：资源利用率
  • 链路追踪量：收集的链路追踪数量

基于 SLO 的告警

定义服务级别目标并在存在风险时告警：

  • 延迟 SLO：95% 的请求 < 500ms
  • 错误 SLO：99.9% 成功率
  • 可用性 SLO：99.95% 正常运行时间

成本管理

在保持价值的同时控制 Observability 成本。

成本优化策略

  • 实施适当的采样率
  • 使用数据保留策略
  • 过滤不必要的 span 和属性
  • 将旧数据归档到更便宜的存储
  • 监控和优化数据量

数据保留策略

  • 热存储：近期数据（7-30 天）用于活跃故障排查
  • 温存储：历史数据（30-90 天）用于趋势分析
  • 冷存储：归档数据（90 天以上）用于合规和低频查询
