Java Spring + OTLP + CloudWatch (Hands-on Exercise)

This guide documents the full exercise to instrument a Java Spring Boot service with OpenTelemetry, export telemetry to AWS, deploy to ECS/Fargate, and verify traces/logs/Application Signals behavior.

It is written in the style of an implementation example and can be used as a runbook or workshop handout.

1) What this exercise demonstrates

  • End-to-end telemetry from a Java app:
    • Traces (X-Ray / Transaction Search)
    • Logs (CloudWatch Logs OTLP endpoint or awslogs on ECS)
    • Metrics/Application Signals (CloudWatch Agent sidecar path)
  • The difference between:
    • Micrometer tracing bridge (micrometer-tracing-bridge-otel) for tracing API bridging
    • Micrometer registries (micrometer-registry-cloudwatch2, micrometer-registry-otlp) which are metrics-only sinks
  • OTLP endpoint wiring via env vars (OTEL_*), not hardcoded Java constants.

4) OTLP endpoint mapping

Signal Endpoint pattern Typical env vars
Traces https://xray.<region>.amazonaws.com/v1/traces OTEL_EXPORTER_OTLP_TRACES_ENDPOINT, OTEL_EXPORTER_OTLP_TRACES_PROTOCOL=http/protobuf
Metrics https://monitoring.<region>.amazonaws.com/v1/metrics OTEL_EXPORTER_OTLP_METRICS_ENDPOINT (or App Signals path via sidecar)
Logs https://logs.<region>.amazonaws.com/v1/logs OTEL_EXPORTER_OTLP_LOGS_ENDPOINT, OTEL_EXPORTER_OTLP_LOGS_HEADERS

In ECS sidecar mode, app container points traces/metrics to http://localhost:4316/... and the CloudWatch Agent handles forwarding.

5) Instrumentation model used in this repo

5.1 Dependencies

  • micrometer-tracing-bridge-otel is present in pom.xml.
  • No dependency on micrometer-registry-cloudwatch2 or micrometer-registry-otlp for tracing.

5.2 Auto + manual tracing

  • Auto instrumentation from ADOT Java agent captures framework/client spans.
  • Manual enrichment is added inside DemoService.greet():
    • attributes: demo.operation, demo.request.name, demo.downstream.status, demo.downstream.elapsed_ms, demo.custom.instrumented
    • events: demo.greet.start, demo.httpbin.success, demo.httpbin.failure
    • error recording with recordException(...) and setStatus(ERROR, ...)

7) Where to view telemetry in AWS Console

Traces (primary)

  1. X-Ray → Traces
  2. Region: us-east-1
  3. Time range: Last 15–60 min
  4. Service: java-spring-otel-poc

For custom fields:

  • Open a trace → segment java-spring-otel-poc → Metadata
  • Look for keys:
    • demo.operation
    • demo.request.name
    • demo.downstream.status
    • demo.downstream.elapsed_ms
    • demo.custom.instrumented

Logs

  • ECS app logs: /ecs/java-spring-otel
  • CW agent sidecar logs: /ecs/ecs-cwagent-java-spring-otel
  • Local OTLP log group (if running local): /aws/otel/java-spring-otel-poc

Application Signals

  • CloudWatch → Application Signals → Services
  • Look for java-spring-otel-poc after sustained traffic and successful metrics forwarding through sidecar path.

8) Key takeaway

OpenTelemetry auto instrumentation can create spans, but successful AWS observability outcomes still depend on:

  1. Correct OTLP endpoint wiring
  2. Correct auth/forwarding path (often via sidecar/collector for metrics/App Signals)
  3. Correct distinction between tracing bridge vs metrics registries
  4. Sustained traffic and correct console view (X-Ray for traces)
